## 1. What Is It? (Theory)

### "Attention Is All You Need"

In June 2017, Vaswani et al. at Google published a paper asking: what if we keep attention but remove
the recurrence entirely?

The answer: an architecture that:
- Processes entire sequences in parallel (not sequentially like RNNs)
- Captures long-range dependencies directly (not through a compressed hidden state)
- Scales predictably with more compute and data
- Can be trained on massive datasets efficiently

This architecture — the **Transformer** — powers every major LLM you know: GPT, Claude, Gemini, LLaMA,
BERT, T5, Mistral, Qwen. The original paper is arguably the most impactful paper in AI since 1986's backpropagation.

### Why It Replaced Everything for Sequences

| Property | RNN/LSTM | Transformer |
|---|---|---|
| Training parallelism | Sequential (step-by-step) | Fully parallel |
| Long-range dependencies | Bottleneck at hidden state | Direct via attention |
| Scaling with compute | Sub-linear returns | Predictable (scaling laws) |
| Training on long sequences | Slow, gradient issues | Fast, gradient-stable |

## 2. How It Actually Works — Deep Mechanics

### 2.1 The Full Architecture

The original transformer is an encoder-decoder architecture designed for sequence-to-sequence tasks
(translation). Modern LLMs use variants: encoder-only (BERT) or decoder-only (GPT, Claude).

```
                    TRANSFORMER (Encoder-Decoder)
═══════════════════════════════════════════════════════════

     ENCODER (N=6 identical layers)            DECODER (N=6 identical layers)
     ┌────────────────────────────┐            ┌────────────────────────────────┐
     │                            │            │                                │
     │  Input Embedding           │            │  Output Embedding              │
     │  + Positional Encoding     │            │  + Positional Encoding         │
     │          │                 │            │          │                     │
     │  ┌───────────────────┐     │            │  ┌─────────────────────────┐   │
     │  │  Multi-Head        │     │            │  │  Masked Multi-Head       │   │
     │  │  Self-Attention    │     │            │  │  Self-Attention          │   │
     │  └───────────────────┘     │            │  │  (causal mask)           │   │
     │          │ (+ residual)    │            │  └─────────────────────────┘   │
     │  ┌──────────────────┐      │            │          │ (+ residual)        │
     │  │  Layer Norm       │      │            │  ┌─────────────────────────┐   │
     │  └──────────────────┘      │            │  │  Multi-Head              │   │
     │          │                 │            │  │  Cross-Attention          │   │
     │  ┌──────────────────┐      │            │  │  (Q from decoder,        │   │
     │  │  Feed-Forward    │      │  encoder → │  │   K,V from encoder)      │   │
     │  │  Network (FFN)   │      │  outputs   │  └─────────────────────────┘   │
     │  └──────────────────┘      │ ──────────►│          │ (+ residual)        │
     │          │ (+ residual)    │            │  ┌─────────────────────────┐   │
     │  ┌──────────────────┐      │            │  │  Layer Norm              │   │
     │  │  Layer Norm       │      │            │  └─────────────────────────┘   │
     │  └──────────────────┘      │            │          │                     │
     │          │  ×6 layers      │            │  ┌─────────────────────────┐   │
     └────────────────────────────┘            │  │  Feed-Forward Network   │   │
                                               │  └─────────────────────────┘   │
                                               │          │ (+ residual) ×6     │
                                               │  ┌─────────────────────────┐   │
                                               │  │  Linear + Softmax        │   │
                                               │  │  (output probabilities)  │   │
                                               │  └─────────────────────────┘   │
                                               └────────────────────────────────┘
```

### 2.2 Component 1: Token Embeddings + Positional Encoding

**Problem:** Self-attention is permutation-invariant. Swap the word order and attention produces
the same result (just with different indices). But "dog bites man" ≠ "man bites dog".

**Positional encoding** injects position information into the input before attention is applied.

**Sinusoidal positional encoding (original paper):**

```python
PE(pos, 2i)   = sin(pos / 10000^(2i/d_model))
PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model))
```

Each position gets a unique pattern of sines and cosines across the embedding dimensions. The model
can learn to interpret these patterns as position information.

**Why sinusoidal vs learned?**
- Sinusoidal: deterministic, works on sequences longer than training set
- Learned embeddings: one vector per position, trained from data, may not generalize to new lengths

**Modern LLMs use RoPE (Rotary Position Embedding):** instead of adding position to embeddings, it
rotates the query and key vectors by an angle proportional to their position. This naturally encodes
relative positions and generalizes to longer contexts than seen during training.

### 2.3 Component 2: Multi-Head Self-Attention (Encoder)

Each encoder layer applies **full bidirectional self-attention**: every position can attend to every
other position.

```
For a sentence: "The cat sat on the mat"
                  ↑
                 pos 1

When computing the representation of "cat":
  - Attend to "The" (determiner, 0.1)
  - Attend to "cat" (itself, 0.3)
  - Attend to "sat" (verb for cat, 0.4)
  - Attend to "on", "the", "mat" (context, 0.2 combined)
```

The output for "cat" is enriched by its syntactic role, semantic relationships, and positional context.

### 2.4 Component 3: Feed-Forward Network (FFN)

After attention, each position independently goes through a 2-layer MLP (feed-forward network):

```
FFN(x) = max(0, x · W₁ + b₁) · W₂ + b₂

d_model → d_ff → d_model
(512)    (2048)  (512)     ← original paper: d_ff = 4 × d_model
```

The FFN is where most of the model's "knowledge" is stored. Research (Geva et al., 2021 — "Transformer
Feed-Forward Layers Are Key-Value Memories") shows that FFN layers function as key-value memories:
- The first layer weights (W₁) encode "keys" — patterns that match facts
- The second layer weights (W₂) encode "values" — information to add when pattern matches

When GPT "knows" that Paris is the capital of France, that knowledge lives primarily in the FFN weights,
not the attention weights.

### 2.5 Component 4: Layer Normalization

Applied after attention and FFN, before the residual addition:

```
Pre-norm (modern, most LLMs):  x → LayerNorm → Sublayer → x + Sublayer(LayerNorm(x))
Post-norm (original paper):    x → Sublayer → x + Sublayer(x) → LayerNorm
```

**LayerNorm vs BatchNorm:**
- BatchNorm normalizes across the batch dimension (per feature, per batch)
- LayerNorm normalizes across the feature dimension (per token, per example)
- Transformers use LayerNorm because batch sizes and sequence lengths vary; LayerNorm is stable regardless

**Pre-norm vs post-norm:**
- Post-norm (original paper): harder to train deep networks (gradients in early layers are small)
- Pre-norm: more stable training, now standard in GPT, Claude, LLaMA and most modern transformers

### 2.6 Component 5: Residual Connections

Every sublayer (attention + FFN) uses residual connections (same as ResNet, Section 2.1):

```
output = LayerNorm(x + Sublayer(x))
```

This ensures:
- Gradients flow directly through the addition, no squashing
- Deeper transformers (96 layers in GPT-3) can be trained
- The model can learn to use layers selectively (some layers learn nearly identity)

### 2.7 Decoder-Only vs Encoder-Only vs Encoder-Decoder

This is the critical distinction to understand before Phase 3.

**Encoder-only (BERT, RoBERTa):**
- Full bidirectional attention: every token sees every other token
- Output: rich contextual representations of the entire input
- Use for: classification, NER, question answering over known context, embeddings
- Cannot generate — it processes a complete input, doesn't predict next tokens

**Decoder-only (GPT, Claude, LLaMA, Mistral):**
- Causal attention: each token only sees previous tokens
- Output: next-token probabilities
- Use for: text generation, chat, completion, all modern LLMs
- Pre-training task: predict the next token

**Encoder-decoder (T5, BART, original transformer):**
- Encoder: full bidirectional attention over input
- Decoder: causal attention over output + cross-attention to encoder
- Use for: translation, summarization, any "input → output" task
- More complex but better for tasks with a clear source/target distinction

```
Task                    → Architecture
-------------------------------------------------
Sentiment classification → Encoder-only (BERT)
Text generation / chat   → Decoder-only (GPT, Claude)
Translation              → Encoder-decoder (T5, mBART)
Summarization            → Encoder-decoder or decoder-only
Text embeddings          → Encoder-only or special-purpose
Q&A with long context    → Decoder-only (modern LLMs handle this)
```

### 2.8 The Scaling: Parameters and Hyperparameters

For a decoder-only transformer (GPT-style):

```
Hyperparameters:
  d_model     = embedding dimension (768 for GPT-2, 12288 for GPT-3)
  n_layers    = number of transformer blocks (12 for GPT-2, 96 for GPT-3)
  n_heads     = attention heads (12 for GPT-2, 96 for GPT-3)
  d_ff        = FFN hidden dimension (4 × d_model typically)
  vocab_size  = ~50,000 (tokenizer vocabulary)
  context_len = maximum sequence length (2048 for GPT-2, 1M for Claude 3)

Parameter count:
  Embedding:        vocab_size × d_model
  Per layer:        ~12 × d_model²    (roughly)
  Total:            n_layers × 12 × d_model²  (dominant term)

GPT-2   (2019): 12  layers, d=768,   ~117M parameters
GPT-3   (2020): 96  layers, d=12288, ~175B parameters
GPT-4   (2023): ~1.8T parameters (estimated, mixture-of-experts)
Claude 3 (2024): undisclosed, competitive with GPT-4
```

## 3. Real-World Context

### The Models Built on This Architecture

Every major language model is a transformer variant:

| Model | Type | Layers | d_model | Context | Training |
|---|---|---|---|---|---|
| BERT-base | Encoder-only | 12 | 768 | 512 tokens | Masked LM + NSP |
| RoBERTa-large | Encoder-only | 24 | 1024 | 512 tokens | Masked LM (longer training) |
| GPT-2 | Decoder-only | 12–48 | 768–1600 | 1024 tokens | Next token prediction |
| GPT-3 | Decoder-only | 96 | 12288 | 2048 tokens | Next token prediction |
| T5-11B | Encoder-decoder | 24 | 1024 | 512 tokens | Text-to-text |
| LLaMA 3 | Decoder-only | 32–80 | 4096–8192 | 128K tokens | Next token prediction |
| Claude 3.5 Sonnet | Decoder-only | Undisclosed | Undisclosed | 200K tokens | Next token + RLHF |
| Gemini 2.0 Flash | Decoder-only | Undisclosed | Undisclosed | 1M tokens | Multimodal |

All of them share the same core: embedding → N × (attention + FFN + norms + residuals) → output projection.

### Why Understanding the Architecture Matters for AI Engineering

When you hit these problems in production, the architecture tells you the answer:

**"Why does the model lose track of something mentioned early in a long context?"**
→ KV-cache grows linearly, but attention quality degrades at the edges of long contexts. Use retrieval
(RAG) to bring relevant information into the active context window.

**"Why is the model more confident about information in the middle of a passage than at the beginning?"**
→ "Lost in the middle" phenomenon: attention quality degrades for tokens far from either end. Structure
your prompts to put critical information at the start or end, not the middle.

**"Why is inference slow for long outputs?"**
→ Each new token requires recomputing attention over all previous tokens. KV-cache stores previous keys
and values but still grows. Longer generation = more memory and more compute.

**"Why does the model fail at exact lookup but excel at reasoning?"**
→ The FFN is a fuzzy key-value memory, not a database. It stores patterns, not exact facts. Use RAG or
tool use for exact lookup.

## 4. Code Examples

### Complete Decoder-Only Transformer Block

```python
import torch
import torch.nn as nn
import math

class TransformerBlock(nn.Module):
    """
    One decoder-only transformer block (GPT-style):
      - Pre-norm multi-head causal self-attention
      - Pre-norm feed-forward network
      - Residual connections around both
    """
    def __init__(self, d_model: int, n_heads: int, d_ff: int, dropout: float = 0.1):
        super().__init__()
        
        self.norm1 = nn.LayerNorm(d_model)
        self.attn = nn.MultiheadAttention(
            embed_dim=d_model,
            num_heads=n_heads,
            dropout=dropout,
            batch_first=True   # input: [batch, seq, d_model]
        )
        
        self.norm2 = nn.LayerNorm(d_model)
        self.ffn = nn.Sequential(
            nn.Linear(d_model, d_ff),
            nn.GELU(),           # GELU instead of ReLU — standard in GPT-style models
            nn.Dropout(dropout),
            nn.Linear(d_ff, d_model),
            nn.Dropout(dropout),
        )
    
    def forward(self, x: torch.Tensor) -> torch.Tensor:
        seq_len = x.size(1)
        
        # Causal mask: position i can only attend to positions 0..i
        causal_mask = torch.triu(
            torch.ones(seq_len, seq_len, device=x.device), diagonal=1
        ).bool()  # True where BLOCKED
        
        # Pre-norm → attention → residual
        normed = self.norm1(x)
        attn_out, _ = self.attn(
            normed, normed, normed,
            attn_mask=causal_mask,
            is_causal=True
        )
        x = x + attn_out
        
        # Pre-norm → FFN → residual
        x = x + self.ffn(self.norm2(x))
        
        return x


class MiniGPT(nn.Module):
    """
    A minimal GPT-style decoder-only transformer.
    Not production-grade, but shows all the components.
    """
    def __init__(
        self,
        vocab_size: int,
        d_model: int,
        n_layers: int,
        n_heads: int,
        max_seq_len: int,
        dropout: float = 0.1
    ):
        super().__init__()
        d_ff = 4 * d_model  # standard: FFN hidden = 4 × model dim
        
        self.token_embed = nn.Embedding(vocab_size, d_model)
        self.pos_embed = nn.Embedding(max_seq_len, d_model)  # learned positional embeddings
        self.drop = nn.Dropout(dropout)
        
        self.blocks = nn.ModuleList([
            TransformerBlock(d_model, n_heads, d_ff, dropout)
            for _ in range(n_layers)
        ])
        
        self.norm_final = nn.LayerNorm(d_model)
        self.lm_head = nn.Linear(d_model, vocab_size, bias=False)
        
        # Weight tying: embedding and LM head share weights (standard in GPT)
        self.lm_head.weight = self.token_embed.weight
        
        # Initialize weights
        self.apply(self._init_weights)
    
    def _init_weights(self, module):
        if isinstance(module, nn.Linear):
            nn.init.normal_(module.weight, mean=0.0, std=0.02)
            if module.bias is not None:
                nn.init.zeros_(module.bias)
        elif isinstance(module, nn.Embedding):
            nn.init.normal_(module.weight, mean=0.0, std=0.02)
    
    def forward(self, token_ids: torch.Tensor) -> torch.Tensor:
        # token_ids: [batch, seq_len]
        batch, seq_len = token_ids.shape
        
        # Token + positional embeddings
        positions = torch.arange(seq_len, device=token_ids.device)
        x = self.drop(self.token_embed(token_ids) + self.pos_embed(positions))
        
        # N transformer blocks
        for block in self.blocks:
            x = block(x)
        
        x = self.norm_final(x)
        
        # Project to vocabulary: [batch, seq_len, vocab_size]
        logits = self.lm_head(x)
        return logits


# Instantiate a small GPT-style model
model = MiniGPT(
    vocab_size=50257,   # GPT-2 tokenizer vocab size
    d_model=256,
    n_layers=4,
    n_heads=8,
    max_seq_len=1024
)

# Forward pass
input_ids = torch.randint(0, 50257, (2, 64))  # batch=2, seq_len=64
logits = model(input_ids)
print(f"Input:   {input_ids.shape}")   # [2, 64]
print(f"Output:  {logits.shape}")      # [2, 64, 50257]

total_params = sum(p.numel() for p in model.parameters())
print(f"Parameters: {total_params:,}")  # ~10M for this small model
```

### Autoregressive Text Generation

```python
import torch
import torch.nn.functional as F

@torch.no_grad()
def generate(
    model: MiniGPT,
    input_ids: torch.Tensor,   # [1, prompt_len]
    max_new_tokens: int = 50,
    temperature: float = 1.0,  # > 1 = more random, < 1 = more focused
    top_k: int = 50            # sample from top-k most likely tokens
) -> torch.Tensor:
    """Simple autoregressive generation with temperature and top-k sampling."""
    
    model.eval()
    generated = input_ids.clone()
    
    for _ in range(max_new_tokens):
        # Crop to max sequence length if needed
        context = generated[:, -model.pos_embed.num_embeddings:]
        
        # Forward pass: get logits for next token
        logits = model(context)          # [1, seq_len, vocab_size]
        next_token_logits = logits[:, -1, :]  # [1, vocab_size] — only the last position
        
        # Apply temperature
        next_token_logits = next_token_logits / temperature
        
        # Top-k filtering: zero out all but top-k logits
        if top_k > 0:
            top_k_logits, _ = torch.topk(next_token_logits, min(top_k, next_token_logits.size(-1)))
            threshold = top_k_logits[:, -1].unsqueeze(-1)
            next_token_logits = next_token_logits.masked_fill(
                next_token_logits < threshold, float('-inf')
            )
        
        # Sample from distribution
        probs = F.softmax(next_token_logits, dim=-1)
        next_token = torch.multinomial(probs, num_samples=1)  # [1, 1]
        
        generated = torch.cat([generated, next_token], dim=1)
    
    return generated
```

### Positional Encoding Visualization

```python
import torch
import numpy as np
import matplotlib.pyplot as plt

def sinusoidal_positional_encoding(max_len: int, d_model: int) -> torch.Tensor:
    """Original sinusoidal positional encoding from Vaswani et al. 2017."""
    pe = torch.zeros(max_len, d_model)
    position = torch.arange(0, max_len).unsqueeze(1).float()
    
    div_term = torch.exp(
        torch.arange(0, d_model, 2).float() * (-math.log(10000.0) / d_model)
    )
    
    pe[:, 0::2] = torch.sin(position * div_term)  # even indices: sin
    pe[:, 1::2] = torch.cos(position * div_term)  # odd indices: cos
    
    return pe

# Visualize: each position gets a unique pattern
pe = sinusoidal_positional_encoding(50, 64)
# pe[i] is the encoding for position i — unique for every position
# pe[i] and pe[j] have a predictable relationship based on their distance
print(f"PE shape: {pe.shape}")         # [50, 64]
print(f"PE[0] ≠ PE[1]: {not torch.allclose(pe[0], pe[1])}")  # True — different positions
```

## 6. Alternatives & Comparisons

### The Three Transformer Variants Compared

| | Encoder-Only (BERT) | Decoder-Only (GPT/Claude) | Encoder-Decoder (T5) |
|---|---|---|---|
| **Attention direction** | Bidirectional | Causal (left-to-right) | Bidirectional enc, causal dec |
| **Training objective** | Masked LM | Next token prediction | Text-to-text |
| **Generates text?** | No | Yes | Yes |
| **Contextual embeddings** | Excellent | Good (less so) | Via encoder |
| **Best for** | Classification, NER, QA | Chat, generation, completion | Translation, summarization |
| **Examples** | BERT, RoBERTa, DeBERTa | GPT, Claude, LLaMA | T5, BART, mT5 |

### Transformer vs State-Space Models (Mamba, etc.)

A new challenger emerged in 2023–2024: **state-space models** (SSMs) like Mamba.

| Aspect | Transformer | Mamba / SSM |
|---|---|---|
| Inference speed | O(n²) per step (attention) | O(n) per step (no attention) |
| Training | Parallel (fast) | Parallel (fast) |
| Long sequences | Quadratic memory issue | Linear — handles much longer |
| Current quality | State-of-the-art | Competitive but not yet dominant |
| Adoption | Universal | Growing; Mamba-2 promising |

**2026 status:** Transformers still dominate. SSMs are a serious research direction but haven't dethroned
transformers at scale. Most likely future: hybrid architectures (some attention + some SSM layers).

## 7. Trade-offs

### Depth (More Layers)
- More layers = more sequential processing = each layer can build on the previous layer's abstract features
- GPT-3 (96 layers) vs GPT-2 (12 layers): dramatically better reasoning ability
- Cost: O(n_layers) compute at inference

### d_model (Wider Embedding)
- Larger d_model = more capacity per token, richer representations
- Cost: O(d_model²) parameters per layer

### Context Length
- Longer context = can process more information
- Cost: O(n²) attention compute; O(n) KV cache memory
- Quality: attention patterns degrade at edges of very long contexts ("lost in the middle")

### Weight Tying (Embedding = LM Head)
- Ties token embedding weights to output projection weights
- Reduces parameters, often improves performance (same vector space for input and output)
- Used in GPT-2, LLaMA, and most decoder-only models

## 8. When To Use vs. When To Avoid

### Encoder-Only (BERT variants)
✅ Use when: Text classification, NER, question answering (with context), embeddings, semantic similarity
❌ Avoid when: You need to generate text

### Decoder-Only (GPT/Claude variants)
✅ Use when: Text generation, chatbots, code generation, instruction following, any task involving generating content
✅ Use when: Modern single-task NLP (fine-tuned GPT now matches or beats BERT on most tasks)
❌ Avoid when: You need efficient bidirectional embeddings without generation overhead

### Encoder-Decoder (T5/BART)
✅ Use when: Translation, summarization, structured prediction with clear source/target distinction
❌ Avoid when: You only need one direction (encoder-only or decoder-only is simpler)

⚠️ Common mistakes:
- Using BERT for generation — it doesn't support autoregressive generation
- Using GPT with context > its context window — outputs degrade or produce errors
- Not applying the correct attention mask when fine-tuning — padding tokens corrupt training
- Using post-norm when training very deep custom transformers — less stable; use pre-norm
- Not weight-tying the LM head and embedding when training from scratch — wastes parameters

## 9. Senior AI Engineer Perspective

### The Two Places Knowledge Lives in a Transformer

1. **Attention weights:** Learn *where to look* — which tokens influence which tokens. These encode
   syntax, coreference, semantic relationships, and task-specific patterns.

2. **FFN weights:** Store *what to say* — the actual knowledge, facts, patterns. Research (Geva et al.)
   shows FFN layers function as key-value memories. When you prompt "The capital of France is ___",
   the FFN retrieves "Paris" from the pattern encoded during pre-training.

This distinction matters when debugging LLM failures:
- Wrong relationship or coreference → may be an attention issue
- Factually wrong but coherent → likely a knowledge issue in FFN (pre-training data)

### What Every LLM Fine-Tune Is Actually Doing

When you fine-tune a transformer on your task data:
- The attention weights learn your domain's patterns (which tokens matter for your task)
- The FFN weights learn your domain's knowledge and output patterns
- The LM head learns to express outputs in your task's format

LORA (covered in Phase 11) works by adding small adapters to the attention projections. This works
because most task-specific adaptation happens in the attention weights, not the FFN — so you can adapt
the model cheaply by only updating those matrices.

### Reading the Papers

After this section, you can now read and understand most transformer architecture papers:
- When you see "multi-head self-attention" you know exactly what's happening
- When you see "pre-norm" vs "post-norm" you know why the choice matters
- When you see "d_model=4096, n_heads=32, n_layers=32" you can estimate parameter count and compute

The field moves fast. New architecture papers come out weekly. Having the core architecture internalized
means you can parse new ideas as variations on known themes rather than starting from scratch each time.
