## 1. What Is It? (Theory)

### The Student Who Memorized the Answers

Imagine two students preparing for an exam. Student A memorizes every answer from the practice tests.
Student B understands the *concepts* and can solve new problems.

On the practice test: both score 100%.
On the real exam: Student A fails, Student B passes.

This is **overfitting**. Your model is Student A.

- **Overfitting:** Model memorizes training data, performs well on training but poorly on new data
- **Underfitting:** Model is too simple to capture the patterns in the data, performs poorly on both
- **Generalization:** The actual goal — performs well on data it hasn't seen

```
                        Underfitting ←──────────────→ Overfitting
                                         ↑
                                      Just Right

Training loss:     high          decreasing          very low (near zero)
Validation loss:   high          decreasing          high again (diverging)
```

### Why Regularization Exists

A model with enough parameters can memorize any training dataset. Regularization is the set of techniques
that prevent this memorization, encouraging the model to learn general patterns instead.

Think of it as adding a penalty for complexity: "you can fit the training data, but don't get too weird doing it."

## 2. How It Actually Works — Deep Mechanics

### 2.1 Diagnosing the Problem: The Bias-Variance Tradeoff

Every ML error has two sources:

**Bias** = error from wrong assumptions in the model. A linear model trying to fit a curved relationship has
high bias — it's systematically wrong, not just noisy.

**Variance** = error from sensitivity to small fluctuations in training data. A very complex model that
perfectly fits the training noise has high variance — it changes dramatically with different training sets.

```
                High Bias               Low Bias
High Variance   wrong AND noisy         noisy but right on average
Low Variance    consistently wrong      consistently right (goal)
```

- **Underfitting = high bias, low variance.** Fix: more model capacity, more features, less regularization
- **Overfitting = low bias, high variance.** Fix: regularization, more data, simpler model

### 2.2 L1 Regularization (Lasso)

Adds a penalty proportional to the *absolute value* of the weights:

```
Loss_total = Loss_original + λ * Σ|wᵢ|
```

**Effect:** Pushes weights toward zero. Many weights go to *exactly* zero (sparse solution). This is
automatic **feature selection** — the model learns which features don't matter and zeros them out.

**When to use:** High-dimensional data where you suspect many features are irrelevant. Text classification
with many rare words. Gene expression data.

### 2.3 L2 Regularization (Ridge / Weight Decay)

Adds a penalty proportional to the *square* of the weights:

```
Loss_total = Loss_original + λ * Σwᵢ²
```

**Effect:** Pushes all weights toward (but not to) zero. Weights get small but don't disappear. Produces
smoother, more distributed weight values.

**When to use:** Neural networks (this is the `weight_decay` parameter in AdamW). When all features likely
contribute. Linear models where you want regularization but not sparsity.

**The intuition:** Big weights mean the model is very sensitive to small input changes. L2 penalizes that
sensitivity, forcing the model to rely on many small signals rather than a few dominant ones.

### 2.4 Dropout

During training, randomly set a fraction of neuron outputs to zero at each forward pass.

```
Without dropout:
  Input → [n₁=0.8] → [n₂=0.5] → [n₃=0.3] → Output

With dropout (p=0.5):
  Input → [n₁=0.8] → [n₂=0.0] → [n₃=0.3] → Output   (n₂ randomly dropped)
  Input → [n₁=0.0] → [n₂=0.5] → [n₃=0.3] → Output   (n₁ randomly dropped next iteration)
```

**Why it works:** Prevents neurons from co-adapting. Each neuron must learn to be useful on its own because
it can't rely on specific other neurons always being present. The network effectively trains as an ensemble
of many smaller networks.

**Important:** Dropout is only active during training. During inference, all neurons are active, but their
outputs are scaled by (1 - dropout_rate) to maintain the expected value. PyTorch handles this automatically
when you call `model.eval()`.

**Typical values:**
- 0.5 for fully-connected layers (aggressive, works well)
- 0.1–0.2 for transformers (lower dropout is common)
- Not used in batch-normalized layers (they conflict)

### 2.5 Batch Normalization

Normalizes the activations of each layer to have zero mean and unit variance within each mini-batch.

```
For a batch of activations x = [x₁, x₂, ..., xₙ]:
  μ_B = (1/n) * Σxᵢ                    (batch mean)
  σ²_B = (1/n) * Σ(xᵢ - μ_B)²         (batch variance)
  x̂ᵢ = (xᵢ - μ_B) / √(σ²_B + ε)      (normalize)
  yᵢ = γ * x̂ᵢ + β                      (scale + shift, learned)
```

**Why it works:**
1. Prevents internal covariate shift — later layers don't have to adapt to wildly changing distributions
   from earlier layers
2. Allows higher learning rates (training is more stable)
3. Has a mild regularizing effect (noise from batch statistics)

**Important:** BatchNorm has different behavior during training vs inference:
- Training: normalizes using the current mini-batch statistics
- Inference: uses running averages of mean/variance accumulated during training

This is why `model.eval()` matters — BatchNorm switches modes.

### 2.6 Early Stopping

Monitor validation loss during training. Stop when it starts increasing, even if training loss continues to decrease.

```
Epoch  Train Loss  Val Loss
  1     1.20        1.25
  5     0.80        0.82
 10     0.50        0.52
 15     0.30        0.45   ← gap is widening
 20     0.15        0.58   ← overfitting
 25     0.08        0.71   ← STOP, restore weights from epoch 10
```

**Implementation note:** Save model weights at every epoch or when val loss improves. Restore the best checkpoint, not the final checkpoint.

## 3. Evaluation Metrics — The Right Tool for the Right Job

This is where most tutorials give you the definitions but not the *when to use which*. Let me fix that.

### 3.1 The Confusion Matrix

The foundation of all classification metrics. For a binary classifier:

```
                 Predicted Positive    Predicted Negative
Actual Positive:  TP (True Positive)    FN (False Negative)  ← missed real cases
Actual Negative:  FP (False Positive)   TN (True Negative)   ← false alarms
```

### 3.2 Accuracy

```
Accuracy = (TP + TN) / (TP + TN + FP + FN)
```

**The problem:** Useless for imbalanced datasets.

Example: 99% of emails are not spam. A model that classifies every email as "not spam" achieves 99% accuracy
and catches zero spam. Useless.

**When to use:** Only when classes are balanced AND all mistakes have equal cost.

### 3.3 Precision and Recall

```
Precision = TP / (TP + FP)    "Of all predicted positives, how many were actually positive?"
Recall    = TP / (TP + FN)    "Of all actual positives, how many did we catch?"
```

**The tradeoff:** You can't maximize both simultaneously.
- High precision, low recall: very conservative. Only flags things it's sure about. Misses real cases.
- High recall, low precision: very aggressive. Catches everything. Lots of false alarms.

**Real examples:**
- Cancer screening: **maximize recall**. You'd rather flag a healthy person for more tests (FP) than miss
  a cancer case (FN). A missed cancer is catastrophic; a false alarm is just an extra test.
- Spam filter: **maximize precision**. You'd rather miss some spam than delete legitimate email. A false
  positive (important email in spam folder) is worse than a false negative (one spam got through).
- Fraud detection: **maximize recall**. Missing real fraud is expensive. False alarms can be handled by humans.
- Content moderation: depends on context.

### 3.4 F1 Score

```
F1 = 2 * (Precision * Recall) / (Precision + Recall)
```

The harmonic mean of precision and recall. Balances both. Penalizes extreme values (a model that sacrifices
one for the other gets a low F1).

**F-beta score:** If you care more about recall than precision, use F2 (weights recall twice). If precision
matters more, use F0.5.

### 3.5 ROC-AUC

Most classifiers output a score (probability), not a hard label. The threshold for "positive" is adjustable.

**ROC curve:** Plot True Positive Rate (recall) vs False Positive Rate at every possible threshold.
**AUC:** Area under that curve. Summary statistic from 0 to 1.

```
AUC = 1.0  → perfect classifier (never makes mistakes)
AUC = 0.5  → random guessing (useless)
AUC = 0.0  → perfectly wrong (invert predictions)
```

**When to use:** When you care about ranking quality across all thresholds. Good when class imbalance isn't
severe. Doesn't tell you how to set the threshold — for that, use precision-recall curves.

### 3.6 PR-AUC (Precision-Recall AUC)

For imbalanced datasets, PR-AUC is more informative than ROC-AUC.

ROC-AUC can be misleadingly high when the negative class dominates (easy to achieve high TN rate). PR-AUC
focuses only on the positive class — much harder to game with imbalanced data.

**Rule:** When your positive class is rare (fraud detection, rare disease, spam), use PR-AUC over ROC-AUC.

### 3.7 Regression Metrics

| Metric | Formula | When to Use |
|---|---|---|
| **MSE** (Mean Squared Error) | `mean((y - ŷ)²)` | When large errors are especially bad (penalizes outliers heavily) |
| **RMSE** | `sqrt(MSE)` | Same as MSE but in original units (interpretable) |
| **MAE** (Mean Absolute Error) | `mean(|y - ŷ|)` | When all errors are equally bad, more robust to outliers |
| **MAPE** (Mean Absolute % Error) | `mean(|y - ŷ| / y) * 100` | When relative error matters (not when y can be 0) |
| **R²** | `1 - SS_res/SS_tot` | How much variance the model explains (0 = baseline, 1 = perfect) |

## 4. Code Examples

### Detecting Overfitting During Training

```python
import torch
import torch.nn as nn
import torch.optim as optim
import numpy as np

def train_and_monitor(model, X_train, y_train, X_val, y_val, epochs=100):
    optimizer = optim.AdamW(model.parameters(), lr=1e-3, weight_decay=1e-4)
    criterion = nn.BCEWithLogitsLoss()
    
    best_val_loss = float('inf')
    patience = 10
    patience_counter = 0
    best_weights = None
    
    for epoch in range(epochs):
        # Training
        model.train()
        optimizer.zero_grad()
        train_pred = model(X_train).squeeze()
        train_loss = criterion(train_pred, y_train)
        train_loss.backward()
        optimizer.step()
        
        # Validation
        model.eval()
        with torch.no_grad():
            val_pred = model(X_val).squeeze()
            val_loss = criterion(val_pred, y_val).item()
        
        # Early stopping logic
        if val_loss < best_val_loss:
            best_val_loss = val_loss
            best_weights = {k: v.clone() for k, v in model.state_dict().items()}
            patience_counter = 0
        else:
            patience_counter += 1
        
        if (epoch + 1) % 10 == 0:
            print(f"Epoch {epoch+1:3d} | Train: {train_loss.item():.4f} | Val: {val_loss:.4f} | "
                  f"{'✓' if val_loss == best_val_loss else f'patience={patience_counter}'}")
        
        if patience_counter >= patience:
            print(f"Early stopping at epoch {epoch+1}. Restoring best weights.")
            model.load_state_dict(best_weights)
            break
    
    return model
```

### Regularization in Practice

```python
import torch
import torch.nn as nn

class RegularizedMLP(nn.Module):
    def __init__(self, input_dim: int, hidden_dim: int, output_dim: int, dropout_p: float = 0.3):
        super().__init__()
        self.network = nn.Sequential(
            nn.Linear(input_dim, hidden_dim),
            nn.BatchNorm1d(hidden_dim),   # normalize activations
            nn.ReLU(),
            nn.Dropout(p=dropout_p),      # randomly zero out neurons during training
            nn.Linear(hidden_dim, hidden_dim // 2),
            nn.BatchNorm1d(hidden_dim // 2),
            nn.ReLU(),
            nn.Dropout(p=dropout_p),
            nn.Linear(hidden_dim // 2, output_dim)
        )
    
    def forward(self, x: torch.Tensor) -> torch.Tensor:
        return self.network(x)

# L2 regularization via weight_decay in optimizer (AdamW does it correctly)
model = RegularizedMLP(input_dim=20, hidden_dim=128, output_dim=1)
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-3, weight_decay=0.01)

# CRITICAL: switch modes between training and evaluation
model.train()  # dropout ACTIVE, batchnorm uses batch stats
output_train = model(x)

model.eval()   # dropout INACTIVE, batchnorm uses running stats
with torch.no_grad():
    output_eval = model(x)
```

### Complete Evaluation Suite for Classification

```python
from sklearn.metrics import (
    accuracy_score, precision_score, recall_score, f1_score,
    roc_auc_score, average_precision_score, classification_report,
    confusion_matrix, ConfusionMatrixDisplay
)
import numpy as np

def evaluate_classifier(y_true: np.ndarray, y_pred: np.ndarray, y_prob: np.ndarray):
    """
    y_true: ground truth labels (0 or 1)
    y_pred: predicted labels (0 or 1)
    y_prob: predicted probabilities for positive class
    """
    print("=== Classification Evaluation ===\n")
    
    # Core metrics
    print(f"Accuracy:  {accuracy_score(y_true, y_pred):.4f}")
    print(f"Precision: {precision_score(y_true, y_pred):.4f}")
    print(f"Recall:    {recall_score(y_true, y_pred):.4f}")
    print(f"F1 Score:  {f1_score(y_true, y_pred):.4f}")
    print(f"ROC-AUC:   {roc_auc_score(y_true, y_prob):.4f}")
    print(f"PR-AUC:    {average_precision_score(y_true, y_prob):.4f}")
    
    print("\n=== Detailed Report ===")
    print(classification_report(y_true, y_pred, digits=4))
    
    print("=== Confusion Matrix ===")
    cm = confusion_matrix(y_true, y_pred)
    tn, fp, fn, tp = cm.ravel()
    print(f"True Negatives:  {tn}")
    print(f"False Positives: {fp}  ← false alarms")
    print(f"False Negatives: {fn}  ← missed real cases")
    print(f"True Positives:  {tp}")

# Example: fraud detection
np.random.seed(42)
n = 10000
# 2% fraud rate (imbalanced)
y_true = np.random.choice([0, 1], size=n, p=[0.98, 0.02])
# Simulated model scores
y_prob = np.where(y_true == 1, 
                  np.random.uniform(0.6, 0.99, n),  # model gives fraud higher scores
                  np.random.uniform(0.01, 0.4, n))
y_pred = (y_prob > 0.5).astype(int)

evaluate_classifier(y_true, y_pred, y_prob)
# Watch: accuracy will be ~97%+ even if recall is low
# In fraud detection, recall is what matters — not accuracy
```

### Cross-Validation for Robust Evaluation

```python
from sklearn.model_selection import StratifiedKFold, cross_validate
from sklearn.ensemble import GradientBoostingClassifier
from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import Pipeline
import numpy as np

# Build a pipeline that prevents data leakage
# (StandardScaler must be fit only on training folds, not the test fold)
pipeline = Pipeline([
    ('scaler', StandardScaler()),
    ('model', GradientBoostingClassifier(n_estimators=100))
])

# Stratified K-Fold: preserves class balance in each fold
cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)

results = cross_validate(
    pipeline, X, y,
    cv=cv,
    scoring=['accuracy', 'f1', 'roc_auc'],
    return_train_score=True  # to check for overfitting across folds
)

for metric in ['accuracy', 'f1', 'roc_auc']:
    train_scores = results[f'train_{metric}']
    test_scores  = results[f'test_{metric}']
    print(f"{metric:10s}: "
          f"Train {train_scores.mean():.3f} ± {train_scores.std():.3f} | "
          f"Val   {test_scores.mean():.3f} ± {test_scores.std():.3f}")
    
    # Red flag: train >> val means overfitting
```

## 6. Alternatives & Comparisons

### Regularization Technique Comparison

| Technique | Type | Effect | Memory | Best For |
|---|---|---|---|---|
| L1 (Lasso) | Penalty on loss | Sparse weights (feature selection) | None | High-dim sparse tasks |
| L2 / Weight Decay | Penalty on loss | Small distributed weights | None | Neural networks (default) |
| Dropout | Architecture | Ensemble-like, forces redundancy | None | Fully-connected layers |
| BatchNorm | Architecture | Stabilizes activations | Small overhead | Hidden layers (most uses) |
| Early Stopping | Training | Prevents over-training | None | Always do this |
| Data Augmentation | Data | Effectively increases dataset size | None | Images, text, audio |
| Label Smoothing | Loss | Prevents overconfident predictions | None | Classification tasks |

**They stack.** In modern transformers: weight decay + dropout + label smoothing are often all used together.
Start with weight decay + early stopping. Add dropout if you're still overfitting. Add BatchNorm if training is unstable.

### Metric Selection Guide

| Task | Primary Metric | Why |
|---|---|---|
| Balanced classification | F1 or Accuracy | Both meaningful |
| Imbalanced (rare positive) | PR-AUC or Recall | Accuracy is misleading |
| Fraud detection | Recall (then check FP cost) | Missed fraud is expensive |
| Medical diagnosis | Recall | False negatives are dangerous |
| Spam detection | Precision | False positives destroy trust |
| Ranking / recommendation | nDCG, MRR | Cares about order, not just binary |
| Regression | RMSE (interpretable) or R² | Task-dependent |
| Time series | MASE, SMAPE | Better than RMSE for scale-free evaluation |

## 7. Trade-offs

### Regularization Strength (λ)
- Too little → overfitting, high variance
- Too much → underfitting, high bias, model too constrained to fit the data
- **Tune λ with validation set.** Cross-validate over [1e-4, 1e-3, 1e-2, 0.1]

### Dropout Rate
- Too low (< 0.1) → minimal effect
- Too high (> 0.5 for transformers) → too much information destroyed, hard to train
- Standard: 0.1–0.3 for attention layers, 0.3–0.5 for fully-connected layers

### BatchNorm vs LayerNorm
| | BatchNorm | LayerNorm |
|---|---|---|
| Normalizes across | Batch dimension | Feature dimension |
| Requires batched inputs | Yes (breaks with batch_size=1) | No |
| Used in | CNNs, MLPs | Transformers (BERT, GPT) |
| Deployment issue | Running statistics must be tracked | None |

Transformers use LayerNorm, not BatchNorm. The reason: sequence lengths vary, and batch stats are inconsistent.

## 8. When To Use vs. When To Avoid

### Weight Decay (L2)
✅ Use when: Training neural networks with AdamW — almost always
❌ Avoid when: Logistic regression on sparse data (L1 is better there)

### Dropout
✅ Use when: Fully-connected layers in large networks, when overfitting persists after L2
❌ Avoid when: BatchNorm is present in the same layer (they conflict), recurrent layers (use variational dropout)

### BatchNorm
✅ Use when: CNNs, MLPs with large batch sizes
❌ Avoid when: Small batch sizes (< 8), recurrent networks, inference batch_size=1 without running stats

⚠️ Common mistakes:
- Evaluating without `model.eval()` — BatchNorm and Dropout behave differently in training mode
- Not using cross-validation on small datasets — one lucky/unlucky train-val split gives misleading metrics
- Using accuracy on imbalanced datasets — you'll think you have a great model and be wrong
- Not saving the best checkpoint — returning the model from the last epoch, not the best validation epoch
- Data leakage in preprocessing — fitting StandardScaler on the full dataset before splitting

## 9. Senior AI Engineer Perspective

### The Metric Mistake That Kills Products

I've seen teams optimize F1 for months and ship a model that users hated. Why? F1 treats false positives
and false negatives equally. The product didn't. One mistake type cost 100x more than the other in terms of
user experience (a spam filter that deleted a job offer email vs one that let through a spam email).

**Build your business metric first.** Then find the ML metric that approximates it. Don't let the ML
metric become the goal.

### What Validation Loss Actually Tells You

Training loss going down while validation loss flattens or rises is the signal. But it's not always overfitting:
- **Validation loss spikes** but then recovers → learning rate too high, try reducing it
- **Validation loss flatlines early** → model capacity too low, try larger model
- **Validation loss improves but slowly** → could add learning rate schedule
- **Validation loss is consistently 2x train loss** from the start → distribution shift between train and val sets (check your split)

### Cross-Validation Is Not Optional for Small Datasets

A single train/val split on 1,000 examples can give accuracy that varies ±5% depending on which examples
ended up in which split. That's not a model quality signal, that's statistical noise. 5-fold CV gives you
5 estimates and their standard deviation. You can actually trust that.

For LLM fine-tuning, cross-validation is often impractical (too slow). Instead: use multiple random seeds
and average the results.

### The Interview Questions That Reveal If Someone Actually Knows This

- "Your model achieves 99% accuracy on fraud detection. Is that good?"
  (No — if 99% of transactions are legitimate, a model that never predicts fraud also gets 99%. What's the recall? What's the PR-AUC?)
- "Explain the difference between L1 and L2 regularization. When would you choose one over the other?"
  (L1 = sparse, feature selection; L2 = small distributed weights. Choose L1 for high-dimensional sparse data, L2 for neural networks.)
- "Your training loss is 0.1 and your validation loss is 0.9. What are the three most likely causes and how would you diagnose each?"
