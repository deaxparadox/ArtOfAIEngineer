## 1. What Is It? (Theory)

### The Gap Between "Can Predict Text" and "Is a Useful Assistant"

After pre-training, a base model has extraordinary capability. It can write essays, solve math, generate
code, answer questions — if you trigger the right continuation. But it's chaotic: ask it a question
and it might generate 10 more questions. Ask for help and it might roleplay as a character instead.
It has no "I'm an assistant, I should answer helpfully" concept.

**Alignment** is the process of teaching the model:
1. **How to be helpful** — format, tone, following instructions
2. **How to be harmless** — refuse dangerous requests, avoid harmful outputs
3. **How to be honest** — don't claim to know things it doesn't, express uncertainty

The term "alignment" comes from AI safety research: aligning the model's behavior with human values and intentions.

### The Three Stages of Post-Training

In 2026, the standard post-training pipeline is:

```
Base Model (pre-trained)
     │
     ▼ Stage 1: Supervised Fine-Tuning (SFT)
Instruction-Following Model
     │
     ▼ Stage 2: Preference Optimization (DPO or RLHF)
Aligned Assistant Model
     │
     ▼ Stage 3: RLVR (Reasoning models only)
Reasoning-Capable Model (Claude 3.7+, o1, DeepSeek-R1)
```

## 2. How It Actually Works — Deep Mechanics

### 2.1 Supervised Fine-Tuning (SFT)

SFT is the first and simplest alignment step. You collect high-quality examples of the behavior
you want — (instruction, ideal response) pairs — and fine-tune the model to produce those responses.

**Training data format:**

```
[INST] Explain quantum entanglement in simple terms [/INST]
Quantum entanglement is when two particles become connected in such a way that
the state of one instantly affects the other, no matter how far apart they are...

[INST] Write a Python function to reverse a linked list [/INST]
def reverse_linked_list(head):
    prev = None
    current = head
    while current:
        next_node = current.next
        current.next = prev
        prev = current
        current = next_node
    return prev
```

The model learns: "when I see an instruction like this, I should produce a response like this."

**Data quality is everything.** The OpenAI InstructGPT paper used ~13,000 high-quality human
demonstrations to dramatically improve GPT-3's usefulness. More isn't better — better is better.

**What SFT teaches:**
- Response format (assistant tone, markdown, code blocks)
- Instruction following (do what was asked)
- Refusals for clearly harmful requests
- Length calibration (don't over-explain simple things)

**What SFT doesn't fully solve:**
- Nuanced preference (two technically correct responses, one is better)
- Avoiding subtle harms (not obviously harmful but still problematic)
- Calibrated confidence (knowing when to say "I don't know")

### 2.2 RLHF — Reinforcement Learning from Human Feedback

RLHF adds a preference layer on top of SFT. The key insight: humans can recognize good answers
better than they can generate them. Show two responses, ask "which is better?" — that comparison
signal is more informative than writing a gold-standard response.

**The three-step RLHF pipeline:**

```
Step 1: Collect preference data
  For each prompt, generate multiple model responses.
  Human annotators rank/compare them:
    Prompt:    "Explain why the sky is blue"
    Response A: [detailed physics explanation]        ← preferred
    Response B: [vague, hand-wavy non-answer]

Step 2: Train a reward model
  The reward model learns to predict which response humans prefer.
  Input:  (prompt + response)
  Output: scalar reward score

  Training: binary classification — for each (A, B) pair:
    loss = -log(σ(reward(A) - reward(B)))   (Bradley-Terry model)
  
  The reward model approximates "what would a human rate this response?"

Step 3: RL optimization (PPO)
  Use the reward model as the environment reward signal.
  Fine-tune the LLM using Proximal Policy Optimization (PPO):
    - Sample a prompt
    - Generate a response
    - Score it with the reward model
    - Update LLM weights to get higher reward in future
  
  KL constraint: stay close to the SFT model to prevent reward hacking
    Reward_final = reward_model(response) - β * KL(LLM || SFT_model)
```

**Why RLHF works:**
- Human preferences encode subtle quality differences SFT can't capture
- The RL loop lets the model explore and find better responses than any in the training data
- PPO + KL constraint prevents the model from drifting into degenerate behavior

**Why RLHF is hard:**
- Training instability (PPO is notoriously finicky to tune)
- Reward hacking: the model finds ways to maximize reward that don't match human intent
  (e.g., producing longer responses if annotators slightly prefer length)
- Expensive: requires a full reward model + RL training loop
- The reward model can be wrong and the LLM learns to game it

### 2.3 DPO — Direct Preference Optimization

Rafailov et al. (2023) made an observation: you can reformulate RLHF mathematically to remove the
need for a separate reward model and RL training entirely.

**The core insight:** The optimal policy under RLHF can be expressed directly in terms of the
reference policy (SFT model) and human preferences. No separate reward model needed.

**DPO loss:**

```
For each triplet (prompt x, chosen response yᵥ, rejected response yₗ):

L_DPO = -E[log σ(β · log(π(yᵥ|x)/π_ref(yᵥ|x)) - β · log(π(yₗ|x)/π_ref(yₗ|x)))]

Where:
  π       = policy being trained (the LLM)
  π_ref   = reference policy (SFT model, frozen)
  β       = temperature controlling deviation from reference
  σ       = sigmoid function
  yᵥ      = chosen (preferred) response
  yₗ      = rejected response
```

**In plain English:** DPO trains the model to simultaneously:
- Increase the probability of chosen responses relative to the reference model
- Decrease the probability of rejected responses relative to the reference model

**Why DPO replaced RLHF for most labs:**
- No separate reward model to train and maintain
- Standard fine-tuning loop (no RL instability)
- Simpler to implement and debug
- Competitive quality on most benchmarks

**DPO data format:**

```json
{
  "prompt": "What is the capital of France?",
  "chosen": "The capital of France is Paris, which has been the country's capital since 987 AD.",
  "rejected": "France's capital is a city in Europe with the Eiffel Tower."
}
```

### 2.4 RLVR — Reinforcement Learning with Verifiable Rewards

The newest tier (2024–2026), used in OpenAI o1, DeepSeek-R1, Claude 3.7 Sonnet, and similar
"reasoning" models.

**The problem DPO doesn't solve:** DPO relies on human preference judgments. But for complex
reasoning tasks (math, coding, logic), humans can't reliably judge which chain-of-thought is better.
And you can't annotate millions of mathematical proofs.

**RLVR's insight:** For tasks with verifiable answers, use the answer verification itself as the
reward signal. No human judgment needed.

```
For a math problem:
  Prompt:   "Solve: 2x + 5 = 17"
  Response: "<thinking>2x = 12, so x = 6</thinking>\n\nAnswer: x = 6"
  
  Reward:
    If answer == correct_answer: reward = +1
    If answer != correct_answer: reward = -1 (or 0)
  
  No human annotation. The answer is right or wrong.
  
For coding:
  Prompt:   "Write a function that..."
  Response: [code]
  
  Reward:
    Run provided unit tests
    Pass rate → reward signal
```

This is how DeepSeek-R1 and o1-style reasoning models are trained to think in long chains before
answering. The model learns that longer, more careful reasoning leads to more correct answers —
because that's what the verifiable rewards reinforce.

### 2.5 Constitutional AI (Anthropic's Approach)

Anthropic uses a variant called Constitutional AI (CAI):

```
Stage 1: SFT with self-critique
  1. Generate initial responses
  2. Have the model critique its own response based on a "constitution" of principles
  3. Revise the response based on the critique
  4. Fine-tune on the revised responses

Stage 2: RLAIF (RL from AI Feedback)
  Instead of human judges, use the model itself to evaluate responses
  against the constitutional principles.
  This scales preference data collection dramatically.
```

The "constitution" is a set of principles like "Choose the response that is less likely to contain
harmful, unethical, or dishonest content." The model learns to evaluate and improve its own outputs.

## 3. Real-World Context

### What Alignment Actually Produces

The same base model, before and after alignment:

| Prompt | Base Model Response | Aligned Model Response |
|---|---|---|
| "How do I make a bomb?" | [continues with instructions, roleplays, or more questions] | "I can't provide instructions for making explosives. If you're interested in chemistry..." |
| "What is 2+2?" | "What is 2+2? What is 3+3? What is 4+4?" (continues list) | "4" |
| "Summarize this article" | [might generate another article entirely] | [actual summary of the article] |
| "Tell me a joke" | "Tell me a joke. I like jokes. Here is a joke:" [no joke] | [actual joke] |

Alignment is what makes the model an assistant instead of a text predictor.

### The 2026 State: DPO Dominance

Based on research and lab disclosures:
- **Anthropic:** Constitutional AI (RLAIF) + DPO-like preference optimization + RLVR for reasoning
- **OpenAI:** RLHF for earlier models; now largely uses DPO + RLVR (o1, o3)
- **Meta (LLaMA):** DPO + RLVR; open-source alignment recipes in LLaMA alignment papers
- **Google (Gemini):** RLHF + SFT + proprietary alignment methods
- **Mistral, Qwen, DeepSeek:** DPO as primary alignment method; RLVR for their reasoning variants

RLHF with PPO is no longer the dominant method — DPO is simpler, more stable, and competitive in quality.

### The Reward Hacking Problem (A Real Production Issue)

If you deploy a custom RLHF pipeline and notice the model:
- Gets more verbose over time (annotators slightly prefer longer answers)
- Sounds increasingly sycophantic ("Great question! I'd be delighted to help...")
- Adds excessive caveats and disclaimers

These are reward hacking signatures. The model found that these behaviors correlate with higher
human ratings, even if they reduce actual quality. The KL constraint prevents the worst of this,
but it still happens.

**Fix:** Regularly audit your reward model and preference data. Explicitly train against known
hacking behaviors. Use multiple reward signals.

## 4. Code Examples

### SFT with Hugging Face TRL

```python
from transformers import AutoTokenizer, AutoModelForCausalLM
from trl import SFTTrainer, SFTConfig
from datasets import Dataset

# Sample instruction-following dataset
data = [
    {
        "messages": [
            {"role": "user", "content": "What is the boiling point of water?"},
            {"role": "assistant", "content": "Water boils at 100°C (212°F) at standard atmospheric pressure (1 atm / 101.325 kPa). At higher altitudes where pressure is lower, water boils at lower temperatures."}
        ]
    },
    {
        "messages": [
            {"role": "user", "content": "Write a Python function to check if a number is prime."},
            {"role": "assistant", "content": "```python\ndef is_prime(n: int) -> bool:\n    if n < 2:\n        return False\n    for i in range(2, int(n**0.5) + 1):\n        if n % i == 0:\n            return False\n    return True\n```"}
        ]
    },
]

dataset = Dataset.from_list(data)

# Load a small base model for demonstration
model_name = "meta-llama/Llama-3.2-1B"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(model_name)

# SFTTrainer handles the instruction formatting and fine-tuning
trainer = SFTTrainer(
    model=model,
    train_dataset=dataset,
    args=SFTConfig(
        output_dir="./sft-model",
        num_train_epochs=3,
        per_device_train_batch_size=2,
        learning_rate=2e-5,     # key: low LR for fine-tuning
        warmup_ratio=0.1,
        max_seq_length=512,
        report_to="none"
    ),
)

trainer.train()
```

### DPO Training with TRL

```python
from trl import DPOTrainer, DPOConfig
from transformers import AutoTokenizer, AutoModelForCausalLM
from datasets import Dataset

# DPO requires a reference model (the SFT model) and preference pairs
model_name = "./sft-model"  # start from SFT checkpoint

tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(model_name)
ref_model = AutoModelForCausalLM.from_pretrained(model_name)  # frozen reference

# Preference data: (prompt, chosen, rejected) triplets
preference_data = [
    {
        "prompt": "Explain recursion in simple terms.",
        "chosen": "Recursion is when a function calls itself to solve a smaller version of the same problem. Like Russian dolls: each doll contains a smaller version of itself until you reach the smallest one.",
        "rejected": "Recursion means a function calls itself. It uses a base case to stop. The stack is used to store frames."
    },
    {
        "prompt": "How should I handle Python exceptions?",
        "chosen": "Use try/except blocks to catch specific exceptions:\n```python\ntry:\n    result = risky_operation()\nexcept ValueError as e:\n    handle_value_error(e)\nexcept Exception as e:\n    log_and_reraise(e)\n```\nAlways catch specific exceptions rather than bare `except:` — it hides bugs.",
        "rejected": "Use try except. Catch Exception. Don't let errors crash your program."
    },
]

dpo_dataset = Dataset.from_list(preference_data)

# DPO trainer: no reward model needed, no RL instability
trainer = DPOTrainer(
    model=model,
    ref_model=ref_model,   # frozen SFT model as reference
    args=DPOConfig(
        output_dir="./dpo-model",
        num_train_epochs=3,
        per_device_train_batch_size=1,
        learning_rate=5e-7,    # very low LR for DPO
        beta=0.1,              # KL regularization strength
        report_to="none"
    ),
    train_dataset=dpo_dataset,
    tokenizer=tokenizer,
)

trainer.train()
```

### Comparing Reward Model Scores

```python
from transformers import AutoTokenizer, AutoModelForSequenceClassification
import torch

# Load a pre-trained reward model (open source options exist)
# Example: OpenAssistant reward model
rm_name = "OpenAssistant/reward-model-deberta-v3-large-v2"
rm_tokenizer = AutoTokenizer.from_pretrained(rm_name)
reward_model = AutoModelForSequenceClassification.from_pretrained(rm_name)
reward_model.eval()

def score_response(prompt: str, response: str) -> float:
    """Score a prompt+response pair using a reward model."""
    text = prompt + "\n" + response
    inputs = rm_tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    
    with torch.no_grad():
        outputs = reward_model(**inputs)
        score = outputs.logits[0].item()  # scalar reward score
    
    return score

# Compare two responses
prompt = "Explain what machine learning is."

good_response = """
Machine learning is a branch of AI where systems learn from data to improve at tasks
without being explicitly programmed. For example, instead of writing rules to detect
spam, you show a spam filter thousands of spam and non-spam emails and it learns the
patterns itself. The key insight is that good performance comes from learning patterns
in data, not from hard-coded rules.
"""

mediocre_response = "ML is when computers learn stuff from data to do tasks."

print(f"Good response score:     {score_response(prompt, good_response):.3f}")
print(f"Mediocre response score: {score_response(prompt, mediocre_response):.3f}")
# Good response should score higher
```

## 6. Alternatives & Comparisons

### SFT vs RLHF vs DPO vs RLVR

| Method | Data Needed | Engineering Complexity | Quality | Best For |
|---|---|---|---|---|
| **SFT** | (prompt, response) pairs | Low | Good baseline | Format, basic instruction following |
| **RLHF** | Preference pairs + reward model | Very High | SOTA (historically) | Nuanced preference alignment |
| **DPO** | (prompt, chosen, rejected) | Low-Medium | Competitive with RLHF | Default 2026 alignment method |
| **ORPO** | (prompt, chosen, rejected) | Low | Competitive | Combines SFT + DPO in one step |
| **GRPO** | Verifiable rewards | Medium | Strong reasoning | Math, code (reasoning models) |
| **RLVR** | Verifiable answer oracle | Medium | Best for reasoning | o1/R1-style reasoning training |

**The 2026 recommendation:**
- Simple assistant behavior → SFT → DPO (two-stage, lowest complexity)
- Reasoning capability → SFT → DPO → RLVR with verifiable tasks
- Production-scale deployment → consider Constitutional AI / RLAIF to scale preference data cheaply

### DPO vs RLHF in 2026

DPO is now the default for most labs. RLHF (PPO) is still used when:
- You need the full generative flexibility of the RL exploration step
- You have a well-calibrated reward model that's hard to translate into preference pairs
- You're doing online RLHF where new preference data is collected continuously

For most teams building on open models: **use DPO**. It's simpler, more stable, and competitive.

## 7. Trade-offs

### SFT
**Pros:** Simple, stable, fast, well-understood
**Cons:** Limited by quality of demonstrations; can't capture subtle preferences

### RLHF
**Pros:** Can discover behaviors better than any in the SFT data; human preferences are expressive
**Cons:** Training instability, reward hacking, requires reward model, expensive

### DPO
**Pros:** Simpler than RLHF, no reward model, stable training, competitive quality
**Cons:** Slightly less flexible than full RL; can overfit on limited preference data; reduces output diversity

### RLVR
**Pros:** Scales without human annotation; produces strong verifiable reasoning
**Cons:** Only works on tasks with verifiable outcomes (math, code); doesn't help with open-ended tasks

## 8. When To Use vs. When To Avoid

### Applying Alignment When Building Products

✅ If using OpenAI/Anthropic/Gemini APIs: alignment is already done; you're working with aligned models
✅ If fine-tuning an open model for your product: always do SFT → DPO (minimum viable alignment pipeline)
✅ If building a reasoning-heavy product: consider RLVR on your domain's verifiable tasks

❌ Don't skip SFT and go straight to DPO on a base model — DPO needs an SFT model as reference
❌ Don't apply heavy alignment to a model you'll use only for embedding — it wastes alignment effort on a model that won't generate

⚠️ Common mistakes:
- Using DPO with poor-quality rejected responses (if rejected is only slightly worse than chosen, the gradient signal is weak)
- Using a base model instead of SFT model as the reference in DPO
- Forgetting that aligned models can still hallucinate — alignment improves helpfulness/safety, not factual accuracy
- Over-aligning: applying so much safety training that the model refuses legitimate requests

## 9. Senior AI Engineer Perspective

### The Alignment Tax

Every alignment step makes the model safer and more helpful — but often slightly less capable at
raw tasks. Aligned models score slightly lower on certain benchmarks than their base counterparts.
This is sometimes called the **alignment tax**.

In production, this rarely matters. You almost always want an aligned model. But it explains why
researchers sometimes use base models for certain evaluations, and why some capability benchmarks
are measured on base models specifically.

### Sycophancy: The Silent Quality Killer

RLHF/DPO models trained on human preferences learn that humans like agreement. If you push back
on their answer ("Are you sure? I thought it was..."), they'll often capitulate — even when they
were originally correct.

This is **sycophancy**: the model is optimizing for the human's apparent satisfaction, not accuracy.

Practical implication: don't use human "preferred by users" metrics as a proxy for correctness in
high-stakes applications. A confident correct answer and a wishy-washy sycophantic wrong answer can
score the same in human preference ratings.

Anthropic has written about this extensively. Several DPO datasets explicitly include anti-sycophancy
pairs where the chosen response maintains correctness under pushback.

### What Alignment Can't Fix

Alignment improves:
- Helpfulness and format
- Safety (refusing clearly harmful requests)
- Instruction following
- Response style and tone

Alignment cannot fix:
- Factual hallucinations (comes from pre-training knowledge gaps)
- Reasoning failures (comes from architecture limitations)
- Out-of-distribution behavior (if your use case wasn't in alignment data)
- Knowledge cutoff (model still doesn't know events after training)
