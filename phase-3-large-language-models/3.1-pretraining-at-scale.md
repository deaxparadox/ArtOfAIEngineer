## 1. What Is It? (Theory)

### The Core Idea in One Sentence

Pre-training is the phase where a transformer learns to predict the next token on a massive corpus of
text — developing general language understanding, factual knowledge, and reasoning patterns as a
side-effect of getting really good at that one prediction task.

### Why Self-Supervised Learning at Scale Works

In Phase 1.1 we introduced self-supervised learning: the labels come from the data itself.

For language models, the task is simple: **given a sequence of tokens, predict the next one**.

```
Input:   "The capital of France is"
Target:  "Paris"

Input:   "def fibonacci(n):\n    if n <= 1:"
Target:  "\n        return n"

Input:   "The patient presented with acute"
Target:  "myocardial"
```

There are no human labels. The training signal is built into every sequence of text ever written.
The internet alone contains trillions of tokens. Academic papers, books, code, conversations,
Wikipedia — all of it is training data for free.

**The emergent miracle:** A model trained only to predict the next word, at sufficient scale, develops:
- World knowledge (facts about geography, history, science)
- Reasoning ability (logical inference, mathematical steps)
- Code generation (learned from billions of lines of code)
- Multiple languages (learned from multilingual text)
- Instruction following capability (to be unlocked later via alignment)

None of this was explicitly trained. It emerged from the statistical structure of language at scale.

### The Raw Pre-Trained Model Is Not What You Talk To

When pre-training completes, you have a **base model**. It's incredibly capable at predicting text —
but it's not an assistant. Ask it a question and it will continue the text in a plausible way,
which might mean generating another question, a lecture, or a Wikipedia article — not answering yours.

The assistant behavior (ChatGPT, Claude, Gemini) comes from **alignment training** (Section 3.2),
applied after pre-training. Think of pre-training as building the brain; alignment as teaching it
how to behave.

## 2. How It Actually Works — Deep Mechanics

### 2.1 The Data Pipeline

Modern LLMs are trained on datasets of 10–15 trillion tokens for mid-size models; frontier models
(GPT-4, Claude, Gemini) are believed to use 20–50+ trillion tokens.

**Where the data comes from:**

```
Common Crawl          → ~50–60% of typical training data
                         (petabytes of web text, heavily filtered)
Books / Books3        → ~10–20% (long-form coherent text)
Wikipedia / Wikidata  → ~3–5% (factually grounded, structured)
GitHub / Code         → ~10–15% (code understanding + generation)
Scientific papers     → ~3–5% (arXiv, PubMed, semantic scholar)
Other curated sources → varies (news, forums, legal documents, etc.)
Multilingual web text → varies (controls language distribution)
```

**The filtering pipeline (what happens before training):**

```
Raw crawled web data
     │
     ▼ Language identification (keep English, filter gibberish)
     │
     ▼ Quality filtering (remove spam, adult content, low-quality text)
     │
     ▼ Deduplication (exact + fuzzy — removes near-duplicate documents)
     │
     ▼ Heuristic filtering (document length, character ratios, etc.)
     │
     ▼ Classifier-based filtering (ML model predicts quality score)
     │
Clean training corpus (~1–5% of raw web survives)
```

Data quality matters more than data quantity. The Llama 3 technical report noted that quality-filtered
data outperforms larger but noisier datasets. "Garbage in, garbage out" applies at a trillion-token scale.

### 2.2 The Training Objective

**Causal language modeling** (autoregressive next-token prediction):

```python
# For a sequence of tokens [t₁, t₂, t₃, t₄, t₅]:

# Create input/target pairs by shifting by 1:
inputs  = [t₁, t₂, t₃, t₄]
targets = [t₂, t₃, t₄, t₅]

# At each position, predict the next token:
logits = model(inputs)            # shape: [seq_len, vocab_size]
loss   = cross_entropy(logits, targets)   # average across all positions

# The model learns simultaneously:
# - Given t₁, predict t₂
# - Given [t₁, t₂], predict t₃
# - Given [t₁, t₂, t₃], predict t₄
# One forward pass, seq_len training signals
```

This is vastly more data-efficient than supervised learning: every token in every document
generates a training signal. A 1,000-token document gives 999 training examples.

### 2.3 The Infrastructure Scale

Training a frontier LLM requires compute infrastructure most teams will never touch directly.
But understanding it explains capabilities and costs:

**GPT-3 scale (2020, now dated but well-documented):**
- 175B parameters
- 300B tokens of training data
- ~10,000 A100 GPUs running for ~1 month
- Estimated cost: ~$4–12M in compute

**Frontier models (2025–2026, estimated):**
- Hundreds of billions to trillions of parameters
- 20T–50T+ tokens
- Tens of thousands of H100/H200 GPUs
- Estimated cost: $50M–$500M+ per training run

**The distributed training problem:**
A single GPU holds maybe 80GB of memory. A 70B parameter model in float16 needs ~140GB
just for weights — before gradients and optimizer states. Training requires distributing across
hundreds or thousands of GPUs.

```
Three main parallelism strategies:

Data Parallelism:
  Same model on all GPUs
  Different data batches per GPU
  Gradients averaged across GPUs
  → Works until model too big for one GPU

Tensor Parallelism:
  Model layers split across GPUs
  Each GPU holds a shard of each layer
  → For models too large for one GPU

Pipeline Parallelism:
  Different layers on different GPUs
  GPU 1: layers 1–8, GPU 2: layers 9–16, etc.
  → For very deep models, minimizes idle time
```

Modern training frameworks (Megatron-LM, DeepSpeed, FSDP) combine all three strategies.

### 2.4 The Full Modern Training Pipeline (2026)

Pre-training is phase 1 of a multi-phase pipeline. The full pipeline for a frontier model is:

```
Phase 1: Pre-training
   Objective: next-token prediction
   Data: web text, books, code, papers (raw diversity)
   Scale: trillions of tokens
   Result: a capable but not yet useful base model

Phase 2: Continued pre-training / domain adaptation (optional)
   Objective: next-token prediction
   Data: domain-specific corpus (code, medical, legal, etc.)
   Scale: billions of tokens
   Result: base model with stronger domain knowledge

Phase 3: Supervised Fine-Tuning (SFT)
   Objective: next-token prediction on (instruction, response) pairs
   Data: curated instruction-following demonstrations
   Scale: millions of tokens
   Result: model that follows instructions in a helpful format

Phase 4: Preference Optimization (DPO/RLHF)
   Objective: align with human preferences
   Data: (prompt, chosen response, rejected response) triplets
   Scale: hundreds of thousands of comparisons
   Result: model that produces responses humans prefer

Phase 5: RLVR / reasoning training (newer frontier models)
   Objective: improve multi-step reasoning
   Data: verifiable problems (math, code with test cases)
   Scale: millions of problems
   Result: model with improved chain-of-thought reasoning (o1, R1, Claude 3.7+)
```

The 2026 standard pipeline is: pre-train → SFT → DPO → RLVR (for reasoning models).
RLHF with PPO is largely replaced by DPO for most labs due to lower engineering complexity.

### 2.5 What Base Models Know vs Don't Know

Understanding pre-training explains model capabilities and failures:

| Capability | Why It Exists | Why It Fails |
|---|---|---|
| World knowledge | Seen facts repeatedly in text | Knowledge cutoff; rare facts underrepresented |
| Code generation | Trained on millions of GitHub repos | Novel APIs not in training data |
| Reasoning | Seen many examples of reasoning in text | Limited working memory (context window) |
| Multiple languages | Multilingual training data | Low-resource languages underrepresented |
| Instruction following | Partially from pre-train; mostly from SFT | Requires alignment phase |
| Arithmetic | Seen calculations in text | Tokenization makes exact arithmetic hard |
| Current events | Nothing | Knowledge cutoff; must use tools/RAG |

### The Base Model vs Instruct Model Distinction

When you see models like `llama-3-70b` vs `llama-3-70b-instruct`, this is the distinction:

- **Base model:** pre-training output — predicts next text, not designed for chat
- **Instruct model:** base model + SFT + alignment — designed to follow instructions

For production AI engineering, **always use the instruct variant** unless you're doing your own
fine-tuning on the base model (Phase 11).

### What Breaks in Production

1. **Knowledge cutoff surprises:** The model states facts that were true at training time but have
   since changed. Presidents, CEOs, company acquisitions, product versions. Always use RAG for time-sensitive facts.

2. **Training data distribution effects:** The model performs much better on topics heavily represented
   in training data (popular programming languages, English text) than on rare topics (obscure languages,
   niche domains). If you're building for a niche domain, expect to augment with RAG or fine-tuning.

3. **Capability cliffs at context boundary:** Most models work well up to their stated context limit,
   then break sharply. Content near the beginning of an extremely long context gets "lost." Budget
   for ~60–70% of stated context window for reliable results.

## 4. Code Examples

### Running Inference Against a Base Model (Hugging Face)

```python
from transformers import AutoTokenizer, AutoModelForCausalLM
import torch

# Load a small open-source base model (for learning purposes)
model_name = "meta-llama/Llama-3.2-1B"  # smallest LLaMA 3 base model
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype=torch.float16,   # fp16 to save memory
    device_map="auto"            # automatically distributes across available GPUs
)

# Base model: give it text and it CONTINUES it, doesn't answer questions
prompt = "The process of photosynthesis involves"

inputs = tokenizer(prompt, return_tensors="pt").to(model.device)

with torch.no_grad():
    outputs = model.generate(
        **inputs,
        max_new_tokens=100,
        temperature=0.8,
        do_sample=True,
        pad_token_id=tokenizer.eos_token_id
    )

generated = tokenizer.decode(outputs[0], skip_special_tokens=True)
print(generated)
# Output: "The process of photosynthesis involves the conversion of light energy into chemical
# energy stored in glucose. The light-dependent reactions occur in the thylakoid..."
# Note: it continues the text, doesn't say "Here is an explanation of photosynthesis:"
```

### Simulating the Pre-training Data Creation

```python
def create_pretraining_examples(text: str, tokenizer, max_length: int = 512) -> list[dict]:
    """
    Shows how a document becomes many training examples via the sliding window approach.
    Each (input, target) pair is created by shifting by 1.
    """
    # Tokenize the document
    tokens = tokenizer.encode(text)
    
    examples = []
    # Slide window of max_length across the document with stride
    stride = max_length // 2   # 50% overlap to capture cross-boundary patterns
    
    for start in range(0, len(tokens) - 1, stride):
        window = tokens[start : start + max_length + 1]
        if len(window) < 2:
            break
        
        input_ids = window[:-1]   # all tokens except last
        labels    = window[1:]    # all tokens except first (shifted by 1)
        
        examples.append({
            "input_ids": input_ids,
            "labels": labels,      # model must predict each label from all preceding input_ids
            "length": len(input_ids)
        })
    
    return examples

# One paragraph → many training signals
sample = """
Machine learning is a subset of artificial intelligence that enables computers
to learn from data without being explicitly programmed for every task.
"""

# examples = create_pretraining_examples(sample, tokenizer)
# Each example trains the model on one chunk of the document
```

### Measuring Perplexity (the Pre-training Loss)

```python
import torch
import math
from transformers import AutoTokenizer, AutoModelForCausalLM

def compute_perplexity(text: str, model_name: str = "gpt2") -> float:
    """
    Perplexity = exp(average cross-entropy loss) on the text.
    Lower perplexity = model is less "surprised" by the text = better language model.
    
    Typical values:
      Good LLM on news text:  ~5–15
      Humans on their domain: ~3–8
      Random baseline:        ~50K (vocab size)
    """
    tokenizer = AutoTokenizer.from_pretrained(model_name)
    model = AutoModelForCausalLM.from_pretrained(model_name)
    model.eval()
    
    encodings = tokenizer(text, return_tensors="pt")
    input_ids = encodings.input_ids
    
    # Stride through text to handle sequences longer than max context
    max_length = model.config.n_positions   # GPT-2: 1024
    stride = 512
    nlls = []
    
    for begin_loc in range(0, input_ids.size(1), stride):
        end_loc = min(begin_loc + max_length, input_ids.size(1))
        input_ids_chunk = input_ids[:, begin_loc:end_loc]
        
        with torch.no_grad():
            outputs = model(input_ids_chunk, labels=input_ids_chunk)
            nlls.append(outputs.loss)
    
    ppl = torch.exp(torch.stack(nlls).mean())
    return ppl.item()

# Demonstrates: the model assigns lower perplexity (more predictable) to fluent text
fluent   = "The quick brown fox jumps over the lazy dog."
nonsense = "Dog fox the lazy over brown quick jumps the."
# compute_perplexity(fluent)   → ~50 (GPT-2 on unseen text, not fine-tuned)
# compute_perplexity(nonsense) → ~300+ (grammatically wrong = high perplexity)
```

## 6. Alternatives & Comparisons

### Pre-Training Objectives

| Objective | Model | Bidirectional? | Generates? | Notes |
|---|---|---|---|---|
| **Causal LM** (next-token) | GPT, Claude, LLaMA | No | Yes | All modern LLMs |
| **Masked LM** (BERT-style) | BERT, RoBERTa | Yes | No | Best for encoding tasks |
| **Prefix LM** | T5, GLM | Partial | Yes | Encoder-decoder |
| **Diffusion LM** | MDLM, PLAID | Yes | Yes | Emerging; non-autoregressive |

**Why causal LM won:** Autoregressive generation is the natural use case for language. Masked LM requires
modifying inference for generation. The simplicity and universality of "predict the next token"
allowed causal LM to scale to trillion-token corpora with clean, stable training.

### MoE vs Dense Models

**Dense model:** Every token activates all parameters.
**Mixture-of-Experts (MoE):** Each token activates only a subset of "expert" layers.

| Aspect | Dense | MoE |
|---|---|---|
| Total parameters | = Active parameters | Total >> Active |
| Inference compute | Proportional to total params | Proportional to active params only |
| Training efficiency | Standard | Better: same quality at lower active params |
| Memory (inference) | Proportional to total params | Must load all experts; needs more GPU RAM |
| Examples | LLaMA 3 8B/70B | Mixtral 8×7B, LLaMA 4, DeepSeek-V3, GPT-4 (est.) |

DeepSeek-V3 (671B total, 37B active) and LLaMA 4 Scout (17B active out of larger total) demonstrate
that MoE is the efficient frontier in 2026 — more total knowledge, same inference cost.

## 7. Trade-offs

### More Pre-Training Data
**Pros:** Better general knowledge, more language coverage, stronger zero-shot capability
**Cons:** Diminishing returns past optimal compute; more data processing cost; risk of more toxic/biased content

### Larger Model (More Parameters)
**Pros:** More capacity to store knowledge; stronger reasoning at the same data volume
**Cons:** More expensive inference; harder to serve at low latency; alignment is harder

### Longer Context Pre-Training
**Pros:** Model learns long-range dependencies; can be used at long context natively
**Cons:** O(n²) attention makes training extremely expensive; diminishing returns past most-used length

### Compute-Optimal Training (Chinchilla)
The Chinchilla finding (2022): most models at the time were **undertrained** — too large for the amount
of training data. Optimal is roughly 20 tokens per parameter. A 70B model should train on ~1.4T tokens.

But this was compute-optimal for training. Inference-optimal is different: for a model served to millions
of users, you want a *smaller* model trained on *more* data — because inference runs millions of times
but training runs once. LLaMA 3 8B (trained on 15T+ tokens) reflects this insight.

## 8. When To Use vs. When To Avoid

### Using Base Models Directly
✅ Use when: You're doing your own SFT fine-tuning and want the raw base capabilities
✅ Use when: Studying model behavior, running perplexity evaluations
❌ Avoid when: Building user-facing applications — use instruct variants
❌ Avoid when: You need reliable instruction following or safety behavior

⚠️ Common mistakes:
- Using a base model in a chat application (it continues text, doesn't answer)
- Not accounting for knowledge cutoff when building fact-sensitive applications
- Over-trusting model output confidence — the pre-training loss doesn't measure factual accuracy
- Assuming a model trained on more data is always better (data quality > quantity)

## 9. Senior AI Engineer Perspective

### The One Thing That Matters Most About Pre-Training

The training data distribution **is** the model's worldview. Whatever was overrepresented in training data,
the model is confident about. Whatever was underrepresented, the model will confabulate (generate plausible-sounding
but unreliable content).

Practical implication: before deploying any LLM on a task, ask "is this task well-represented in the
training data?" If yes — great. If not — expect hallucinations and use RAG, tools, or fine-tuning to compensate.

### Why "More Parameters = Smarter" Is Wrong

A 70B base model trained on 500B tokens often underperforms a 7B model trained on 10T tokens on benchmarks.
Llama 3 8B outperformed many previous 30B+ models. Parameters are capacity; data is what fills that capacity.

In practice: the Gemma-3 12B model from Google outperforms many 70B models from a year earlier because of
better training data curation, not more parameters. Evaluate actual benchmark performance, not parameter count.

### Interview Questions

- "What is the pre-training objective of GPT-style models?" (next-token prediction via cross-entropy loss)
- "Why can't a base LLM follow instructions straight out of pre-training?" (trained to predict text, not to be an assistant; needs SFT + alignment)
- "What's the difference between a base model and an instruction-tuned model?" (base: continuation; instruct: trained on assistant-style dialogues + aligned)
- "What is perplexity and what does a lower perplexity score mean?" (exp of average loss; lower = model is less surprised by text = better fit)
- "Why do LLMs have a knowledge cutoff, and what are the practical implications?" (pre-training data has a collection date; model doesn't know about events after that date; need RAG or tools for current information)
