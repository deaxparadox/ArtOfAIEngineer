## 1. What Is It? (Theory)

### The Memory Problem

Full fine-tuning of a 7B parameter model requires:
- Model weights: 7B × 2 bytes (fp16) = **14 GB**
- Gradient storage: same as weights = **14 GB**
- Optimizer states (Adam): 2× weights = **28 GB**
- Activations during forward pass: ~**10–20 GB**
- **Total: ~66–76 GB** — requires two A100 GPUs

Full fine-tuning of a 70B model: **~600 GB** — requires 8–16 A100s — approximately
$50,000 worth of hardware for a single training run.

**LoRA** reduces this dramatically by only training a tiny fraction of parameters.
**QLoRA** reduces it further by also quantizing the frozen base model.

### The Key Insight: Low-Rank Adaptation

During fine-tuning, the weight updates to pre-trained models tend to have **low intrinsic rank**.
This means the update matrix ΔW can be approximated by two smaller matrices:

```
Full weight update:
  W_new = W_original + ΔW
  ΔW shape: [d_out × d_in]  (e.g., 4096 × 4096 = 16.7M parameters)

LoRA approximation:
  ΔW ≈ B × A
  B shape: [d_out × r]   (e.g., 4096 × 16 = 65K parameters)
  A shape: [r × d_in]    (e.g., 16 × 4096 = 65K parameters)
  Total: 130K vs 16.7M = 128× fewer parameters for rank r=16

Where r is the rank (typically 4–64):
  Low r (4–16):  fewer parameters, faster training, may underfit
  High r (32–128): more parameters, slower, approaches full fine-tuning quality
```

At r=16, you're training ~0.5–1% of the model's parameters but achieving 95–99% of
full fine-tuning quality.

## 2. How It Actually Works — Deep Mechanics

### 2.1 LoRA Mechanics

```python
import torch
import torch.nn as nn
import math

class LoRALinear(nn.Module):
    """
    Replaces a standard Linear layer with LoRA-augmented version.
    Freezes original weights; trains only A and B matrices.
    """
    
    def __init__(
        self,
        original_layer: nn.Linear,
        r: int = 16,           # rank
        lora_alpha: float = 32, # scaling factor
        lora_dropout: float = 0.05,
    ):
        super().__init__()
        
        d_in  = original_layer.in_features
        d_out = original_layer.out_features
        
        # Freeze original weights
        self.original_weight = original_layer.weight
        self.original_bias   = original_layer.bias
        self.original_weight.requires_grad = False
        
        # Trainable LoRA matrices (tiny)
        self.lora_A = nn.Parameter(torch.empty(r, d_in))
        self.lora_B = nn.Parameter(torch.zeros(d_out, r))   # init B to 0 → ΔW = 0 at start
        
        # Initialize A with Kaiming uniform (standard for linear layers)
        nn.init.kaiming_uniform_(self.lora_A, a=math.sqrt(5))
        
        self.scaling = lora_alpha / r   # scale the adaptation
        self.dropout = nn.Dropout(lora_dropout)
    
    def forward(self, x: torch.Tensor) -> torch.Tensor:
        # Original computation (frozen)
        base_output = nn.functional.linear(x, self.original_weight, self.original_bias)
        
        # LoRA adaptation (trainable)
        lora_output = self.dropout(x) @ self.lora_A.T @ self.lora_B.T
        
        return base_output + lora_output * self.scaling
    
    def merge_weights(self) -> nn.Linear:
        """
        After training, merge LoRA into the base weights.
        Merged model = same as base model but with LoRA adaptation baked in.
        Result: ZERO inference overhead (no separate LoRA forward pass).
        """
        merged_weight = self.original_weight + self.lora_B @ self.lora_A * self.scaling
        merged_layer = nn.Linear(
            self.original_weight.shape[1],
            self.original_weight.shape[0],
            bias=self.original_bias is not None
        )
        merged_layer.weight.data = merged_weight
        if self.original_bias is not None:
            merged_layer.bias.data = self.original_bias
        return merged_layer
```

**Key property:** After training, you can merge LoRA weights into the base model.
The merged model has **identical architecture and identical inference speed** to the base model.
LoRA adds overhead only during training, not inference.

### 2.2 QLoRA: Quantization + LoRA

QLoRA (Dettmers et al., 2023) adds 4-bit quantization to the frozen base model:

```
LoRA:   Train A + B (small) | Base weights stored as fp16 (14GB for 7B model)
QLoRA:  Train A + B (small) | Base weights stored as int4 (3.5GB for 7B model)

4× memory reduction on the frozen base weights.
```

The trick: quantize the frozen model to 4-bit (loses some precision), but train LoRA adapters
in full 16-bit precision. The tiny LoRA updates compensate for the quantization noise.

```python
from transformers import AutoModelForCausalLM, BitsAndBytesConfig
from peft import get_peft_model, LoraConfig, TaskType
import torch

# QLoRA configuration: load model in 4-bit
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,                         # quantize to 4-bit
    bnb_4bit_quant_type="nf4",                 # NormalFloat4: better than int4 for LLMs
    bnb_4bit_compute_dtype=torch.bfloat16,     # compute in bf16 for stability
    bnb_4bit_use_double_quant=True,            # "double quantization": quantize the quantization constants
)

# Load model in 4-bit (QLoRA)
model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Meta-Llama-3-8B",
    quantization_config=bnb_config,
    device_map="auto"
)

# Apply LoRA on top of quantized model
lora_config = LoraConfig(
    r=16,
    lora_alpha=32,
    target_modules=["q_proj", "v_proj", "k_proj", "o_proj",
                    "gate_proj", "up_proj", "down_proj"],   # all linear layers
    lora_dropout=0.05,
    bias="none",
    task_type=TaskType.CAUSAL_LM,
)

model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# Output: trainable params: 83,886,080 || all params: 8,030,261,248 || trainable%: 1.044
# Only ~1% of parameters are trained!

# GPU memory comparison:
# Full fine-tuning 8B model:  ~66 GB  (requires A100 80GB)
# LoRA fine-tuning 8B model:  ~18 GB  (fits on A10G 24GB or RTX 4090 24GB)
# QLoRA fine-tuning 8B model: ~10 GB  (fits on RTX 3090 24GB or even colab T4 16GB)
```

### 2.3 Which Target Modules?

LoRA can be applied to any linear layer in the transformer. Common choices:

```python
# Minimal (attention only — fastest, least capacity)
target_modules_minimal = ["q_proj", "v_proj"]

# Standard (all attention projections — good balance)
target_modules_standard = ["q_proj", "v_proj", "k_proj", "o_proj"]

# Full (attention + FFN — maximum capacity)
target_modules_full = ["q_proj", "v_proj", "k_proj", "o_proj",
                       "gate_proj", "up_proj", "down_proj"]

# For code generation or complex reasoning: use full
# For style/format changes: standard is usually enough
```

### 2.4 Rank Selection

```python
# Rank (r) selection guide:
RANK_GUIDE = {
    "r=4":  "Minimal adaptation; fast; for style/tone changes only",
    "r=8":  "Light adaptation; good for format/output style",
    "r=16": "Standard; good balance for most tasks",
    "r=32": "Higher capacity; for complex domain adaptation",
    "r=64": "Close to full fine-tuning quality; slower training",
    "r=128+": "Approaching full fine-tuning; use if you have large dataset and GPU",
}

# Rule of thumb: start with r=16, lora_alpha=32
# Increase r if training loss doesn't converge or eval shows underfitting
# Decrease r if you see overfitting or want faster training
```

## 3. Full Training Script (LoRA with Axolotl)

For production fine-tuning, Axolotl provides a config-driven approach that handles edge cases:

```yaml
# axolotl_config.yml
base_model: meta-llama/Meta-Llama-3-8B-Instruct
model_type: LlamaForCausalLM
tokenizer_type: AutoTokenizer

load_in_4bit: true   # QLoRA
adapter: lora

lora_r: 16
lora_alpha: 32
lora_dropout: 0.05
lora_target_modules:
  - q_proj
  - v_proj
  - k_proj
  - o_proj

datasets:
  - path: sft_data/train.jsonl
    type: chat_template
val_set_size: 0.05

sequence_len: 2048
sample_packing: false

gradient_accumulation_steps: 4
micro_batch_size: 2
num_epochs: 3
learning_rate: 0.0002
optimizer: adamw_bnb_8bit   # 8-bit Adam from bitsandbytes: saves memory
lr_scheduler: cosine
weight_decay: 0.01

# Output
output_dir: ./output/llama3-8b-sft-v1
save_steps: 100
eval_steps: 100
logging_steps: 10

# W&B tracking
wandb_project: llm-fine-tuning
wandb_run_id: llama3-8b-sft-v1
```

```bash
# Train with Axolotl
pip install axolotl
accelerate launch -m axolotl.cli.train axolotl_config.yml

# After training: merge LoRA into base model for deployment
python -m axolotl.cli.merge_lora axolotl_config.yml
```

## 5. Trade-offs

### LoRA vs Full Fine-Tuning vs QLoRA

| Approach | Quality | Memory | Speed | Inference Overhead |
|---|---|---|---|---|
| Full fine-tuning | Best (100%) | Very high | Slow | None |
| LoRA | 95–99% | High | Moderate | None (after merge) |
| QLoRA | 92–97% | Low | Moderate | None (after dequant merge) |
| QLoRA + int8 inference | 90–95% | Lowest | Fast | None |

**For 95%+ of production use cases: QLoRA is the right choice.** It's what the PEFT paper
authors recommend, and what most teams use in 2026.

## 6. When To Use vs. When To Avoid

### QLoRA (recommended default)
✅ Use when: VRAM is limited (< 40 GB for 7B model)
✅ Use when: Cost is a concern (consumer GPU vs A100)
✅ Use when: Quick experimentation (4-bit loads faster)
❌ Consider full fine-tuning when: Final production quality must be maximized and you have the hardware

⚠️ Common mistakes:
- Wrong target modules for the model architecture — always check the model's config
- Not using `nf4` quantization — int4 is worse quality for LLMs; NF4 is standard
- Forgetting gradient checkpointing — can reduce memory 2× at cost of ~30% training slowdown
- Not merging LoRA before deployment — unmerged LoRA runs slightly slower at inference

## 7. Senior AI Engineer Perspective

### LoRA Changed Who Can Fine-Tune

Before LoRA (2021), fine-tuning a 7B model required A100 GPUs and significant infrastructure.
After LoRA + QLoRA (2022–2023), a single RTX 4090 ($1,500) can fine-tune a 13B model.

This democratized fine-tuning. Teams that couldn't afford model customization now can.
The practical implication for AI engineers: if your task needs fine-tuning, the hardware
barrier is much lower than most people think. A cloud GPU renting at $0.5–$1/hour makes
a 7B QLoRA fine-tuning run affordable for almost any team.

### Rank Is a Quality Dial, Not a Magic Number

Every paper you read will have a "recommended rank" (usually 8 or 16). In practice, the right
rank depends on your task, dataset size, and model. Treat rank as a hyperparameter:
start with r=16, evaluate on your validation set, and tune from there.

If your training loss doesn't converge: increase r.
If your model overfits on small datasets: decrease r.
