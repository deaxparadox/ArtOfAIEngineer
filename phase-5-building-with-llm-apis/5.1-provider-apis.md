## 1. What Is It? (Theory)

### The Three Providers That Matter

In 2026, the production LLM API landscape has three tiers:

**Tier 1 — Frontier closed models (highest quality, highest cost):**
- Anthropic: Claude 4.x family (Sonnet, Opus, Haiku)
- OpenAI: GPT-4o, GPT-5, o3, o4-mini
- Google: Gemini 2.5 Pro, Gemini 2.5 Flash

**Tier 2 — Open-weight hosted models (high quality, lower cost):**
- Together AI, Fireworks, Groq: LLaMA 4, Mistral, Qwen
- AWS Bedrock, Azure AI: hosted open models
- Google Vertex AI: hosted open models

**Tier 3 — Self-hosted (lowest cost, highest operational overhead):**
- vLLM, TGI, Ollama: run your own inference
- Only for high-volume, cost-sensitive, or privacy-required workloads

For most teams building AI products: **start with Tier 1** (fastest to build, best quality),
**move to Tier 2** when cost becomes material, **consider Tier 3** only when you've verified
the economics justify the operational complexity.

## 2. How It Actually Works — Deep Mechanics

### 2.1 The Request/Response Model

Every LLM API follows the same pattern:

```
Client                           Provider Server
  │                                     │
  │ POST /messages                      │
  │ {model, max_tokens, messages, ...}  │
  │ ──────────────────────────────────► │
  │                                     │ (inference runs)
  │ 200 OK                              │
  │ {id, model, content, usage, ...}    │
  │ ◄────────────────────────────────── │
```

**Key fields in every provider's request:**
- `model`: which model to use
- `messages`: the conversation (system + user + assistant turns)
- `max_tokens`: maximum output length (you pay for tokens generated up to this limit)
- `temperature`: randomness (0 = deterministic, 1 = creative)
- `system`: (Anthropic) or first message with role=system (OpenAI)

**Key fields in every response:**
- `content`: the actual output
- `usage`: input_tokens + output_tokens consumed (critical for cost tracking)
- `stop_reason`/`finish_reason`: why generation stopped (end_turn, max_tokens, tool_use)
- `model`: the exact model version used (may differ from what you requested)

### 2.2 Provider-Specific Differences

**Anthropic SDK:**
```python
import anthropic

client = anthropic.Anthropic(api_key="sk-ant-...")  # or ANTHROPIC_API_KEY env var

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system="You are a helpful assistant.",     # separate top-level field
    messages=[
        {"role": "user", "content": "Hello"}
    ]
)

# Access the response
text     = response.content[0].text
in_tok   = response.usage.input_tokens
out_tok  = response.usage.output_tokens
stop     = response.stop_reason   # "end_turn" | "max_tokens" | "tool_use"
```

**OpenAI SDK:**
```python
from openai import OpenAI

client = OpenAI(api_key="sk-...")   # or OPENAI_API_KEY env var

response = client.chat.completions.create(
    model="gpt-4o",
    max_tokens=1024,
    messages=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user",   "content": "Hello"}
    ]
)

# Access the response
text     = response.choices[0].message.content
in_tok   = response.usage.prompt_tokens
out_tok  = response.usage.completion_tokens
stop     = response.choices[0].finish_reason   # "stop" | "length" | "tool_calls"
```

**Google Gemini SDK:**
```python
from google import genai

client = genai.Client(api_key="AIza...")   # or GOOGLE_API_KEY env var

response = client.models.generate_content(
    model="gemini-2.5-flash",
    contents="Hello",
    config=genai.types.GenerateContentConfig(
        system_instruction="You are a helpful assistant.",
        max_output_tokens=1024,
    )
)

text    = response.text
in_tok  = response.usage_metadata.prompt_token_count
out_tok = response.usage_metadata.candidates_token_count
```

### 2.3 Model Naming and Versioning

Provider model names follow different conventions:

| Provider | Pattern | Example | Pinned example |
|---|---|---|---|
| Anthropic | `claude-{family}-{version}` | `claude-sonnet-4-6` | `claude-sonnet-4-6-20251001` |
| OpenAI | `gpt-{series}-{variant}` | `gpt-4o` | `gpt-4o-2024-08-06` |
| Google | `gemini-{version}-{variant}` | `gemini-2.5-flash` | `gemini-2.5-flash-002` |

**Production rule:** Always pin to a specific dated/versioned model ID in production. Unpinned
model names (`claude-sonnet-4-6` without date) may be silently updated by the provider, which
can change output behavior. Pin, then upgrade deliberately after running your eval suite.

## 4. Code Examples

### Provider-Agnostic Wrapper

```python
from __future__ import annotations
import os
from dataclasses import dataclass
from typing import Literal

@dataclass
class LLMResponse:
    text: str
    input_tokens: int
    output_tokens: int
    model: str
    stop_reason: str

Provider = Literal["anthropic", "openai", "gemini"]

def call_llm(
    system: str,
    user: str,
    provider: Provider = "anthropic",
    model: str | None = None,
    max_tokens: int = 1024,
    temperature: float = 0.0,
) -> LLMResponse:
    """
    Provider-agnostic LLM call. Returns a normalized response object.
    Useful for A/B testing providers or building provider fallback logic.
    """
    if provider == "anthropic":
        import anthropic
        client = anthropic.Anthropic()
        m = model or "claude-sonnet-4-6"
        r = client.messages.create(
            model=m, max_tokens=max_tokens,
            system=system, temperature=temperature,
            messages=[{"role": "user", "content": user}]
        )
        return LLMResponse(
            text=r.content[0].text,
            input_tokens=r.usage.input_tokens,
            output_tokens=r.usage.output_tokens,
            model=r.model,
            stop_reason=r.stop_reason,
        )

    elif provider == "openai":
        from openai import OpenAI
        client = OpenAI()
        m = model or "gpt-4o"
        r = client.chat.completions.create(
            model=m, max_tokens=max_tokens, temperature=temperature,
            messages=[
                {"role": "system", "content": system},
                {"role": "user",   "content": user},
            ]
        )
        choice = r.choices[0]
        return LLMResponse(
            text=choice.message.content,
            input_tokens=r.usage.prompt_tokens,
            output_tokens=r.usage.completion_tokens,
            model=r.model,
            stop_reason=choice.finish_reason,
        )

    elif provider == "gemini":
        from google import genai
        client = genai.Client()
        m = model or "gemini-2.5-flash"
        r = client.models.generate_content(
            model=m, contents=user,
            config=genai.types.GenerateContentConfig(
                system_instruction=system,
                max_output_tokens=max_tokens,
                temperature=temperature,
            )
        )
        meta = r.usage_metadata
        return LLMResponse(
            text=r.text,
            input_tokens=meta.prompt_token_count,
            output_tokens=meta.candidates_token_count,
            model=m,
            stop_reason=r.candidates[0].finish_reason.name,
        )

    raise ValueError(f"Unknown provider: {provider}")


# Usage
resp = call_llm(
    system="You are a precise summarizer. Keep it under 3 sentences.",
    user="Explain quantum entanglement.",
    provider="anthropic",
)
print(resp.text)
print(f"Tokens: {resp.input_tokens} in / {resp.output_tokens} out")
print(f"Model: {resp.model}")
```

### Using LiteLLM for Unified Multi-Provider Access

```python
# pip install litellm
import litellm

# LiteLLM provides a unified OpenAI-compatible interface for 100+ providers
# One SDK, all providers

def unified_call(
    system: str,
    user: str,
    model: str = "claude-sonnet-4-6",   # or "gpt-4o", "gemini/gemini-2.5-flash"
) -> str:
    response = litellm.completion(
        model=model,
        messages=[
            {"role": "system", "content": system},
            {"role": "user",   "content": user},
        ]
    )
    return response.choices[0].message.content

# All three providers via the same interface
for model in ["claude-sonnet-4-6", "gpt-4o", "gemini/gemini-2.5-flash"]:
    try:
        result = unified_call("Be brief.", "What is the speed of light?", model=model)
        print(f"{model}: {result[:80]}")
    except Exception as e:
        print(f"{model}: ERROR — {e}")
```

### Checking Model Availability and Context Limits

```python
import anthropic
from dataclasses import dataclass

@dataclass
class ModelInfo:
    id: str
    context_window: int
    max_output: int
    input_cost_per_mtok: float   # per million tokens
    output_cost_per_mtok: float

# Hardcoded reference — update from provider docs periodically
ANTHROPIC_MODELS = {
    "claude-haiku-4-5-20251001":    ModelInfo("claude-haiku-4-5-20251001", 200_000, 8_192, 1.00, 5.00),
    "claude-sonnet-4-6":            ModelInfo("claude-sonnet-4-6",         200_000, 8_192, 3.00, 15.00),
    "claude-opus-4-8":              ModelInfo("claude-opus-4-8",           1_000_000, 32_000, 15.00, 75.00),
}

def estimate_cost(model_id: str, input_tokens: int, output_tokens: int) -> float:
    """Estimate cost in USD for a given usage."""
    info = ANTHROPIC_MODELS.get(model_id)
    if not info:
        return 0.0
    input_cost  = (input_tokens  / 1_000_000) * info.input_cost_per_mtok
    output_cost = (output_tokens / 1_000_000) * info.output_cost_per_mtok
    return input_cost + output_cost

# Example: estimate cost for 1000 typical queries
typical_input  = 800   # tokens
typical_output = 200   # tokens
n_queries      = 1000

for model_id, info in ANTHROPIC_MODELS.items():
    cost = estimate_cost(model_id, typical_input, typical_output) * n_queries
    print(f"{model_id:35s}: ${cost:.2f} / 1000 queries")
```

### The Batch API for Non-Realtime Workloads

```python
import anthropic
import json
import time

client = anthropic.Anthropic()

def batch_process(requests: list[dict], model: str = "claude-haiku-4-5-20251001") -> list[dict]:
    """
    Use the Batches API for 50% cost reduction on non-realtime workloads.
    Ideal for: evaluation runs, data processing, bulk classification.
    Max latency: 24 hours.
    """
    # Create batch
    batch_requests = [
        {
            "custom_id": f"req_{i}",
            "params": {
                "model": model,
                "max_tokens": 256,
                "system": req.get("system", ""),
                "messages": [{"role": "user", "content": req["user"]}]
            }
        }
        for i, req in enumerate(requests)
    ]

    batch = client.beta.messages.batches.create(requests=batch_requests)
    print(f"Batch created: {batch.id}")
    print(f"50% cost savings vs synchronous calls on same model")

    # Poll for completion (production: use webhooks instead)
    while True:
        batch = client.beta.messages.batches.retrieve(batch.id)
        if batch.processing_status == "ended":
            break
        print(f"Status: {batch.processing_status}... waiting")
        time.sleep(30)

    # Collect results
    results = []
    for result in client.beta.messages.batches.results(batch.id):
        if result.result.type == "succeeded":
            results.append({
                "id": result.custom_id,
                "text": result.result.message.content[0].text
            })

    return results


# Example: batch classification of 100 texts
sample_requests = [
    {"system": "Classify sentiment: positive/negative/neutral.", "user": f"Text {i}"}
    for i in range(5)  # small example
]
results = batch_process(sample_requests)
print(f"Processed {len(results)} requests")
```

## 6. Alternatives & Comparisons

### When to Use Which Provider

| Scenario | Recommended | Why |
|---|---|---|
| General assistant, quality-critical | Claude Sonnet 4.6 | Best instruction-following, safest |
| Bulk processing, cost-sensitive | gpt-4o-mini or Claude Haiku | Cheapest frontier models |
| Complex reasoning, math, code | Claude Opus 4.8 or o3 | Best multi-step reasoning |
| Long documents (>200K tokens) | Gemini 2.5 Pro | 2M context window |
| Real-time, low latency | Claude Haiku, Gemini Flash | Fastest response times |
| Privacy / data sovereignty | Self-hosted LLaMA 4 or Mistral | No data leaves your infra |
| Multimodal (image + text) | Gemini 2.5, GPT-4o | Both excellent |
| Non-English text | Gemini 2.5, Qwen 2.5 | Stronger multilingual training |

## 7. Trade-offs

### Closed vs Open-Weight Models

| Factor | Closed (Claude, GPT, Gemini) | Open-weight (LLaMA, Mistral) |
|---|---|---|
| Quality (frontier tasks) | Better | Catching up fast |
| Cost at scale | Per-token pricing | Fixed infra cost |
| Data privacy | Data sent to provider | Stays in your infra |
| SLAs and uptime | Provider-managed | Your responsibility |
| Latest capabilities | Always current | Lags frontier by months |
| Fine-tuning | Limited API options | Full control |

### Single Provider vs Multi-Provider

**Single provider:**
- Simpler codebase, simpler billing
- Risk: provider outage or price increase affects everything

**Multi-provider:**
- Resilience (fallback on outage)
- Cost optimization (route tasks to cheapest capable model)
- Complexity: different APIs, different behavior, different billing

Start single-provider. Add multi-provider routing when you've validated your use cases and need
resilience or cost optimization.

## 8. When To Use vs. When To Avoid

### Batch API
✅ Use when: Evals, bulk classification, data processing, any offline workload
✅ Use when: Cost sensitivity is high and 24-hour turnaround is acceptable
❌ Avoid when: User-facing real-time applications

### API Gateway / Proxy
✅ Use when: Multiple providers, need unified logging and cost tracking
✅ Use when: Enterprise requirements (audit logs, VNET, SLAs)
❌ Avoid when: Simple single-provider projects (adds latency and complexity)

⚠️ Common mistakes:
- Not pinning the model version — silent updates break evals and change behavior
- Hardcoding API keys in code — always use environment variables or a secrets manager
- Not handling `stop_reason == "max_tokens"` — truncated responses are silent failures
- Using the same model for all tasks — routing to cheaper models for simple tasks cuts costs 50–80%

## 9. Senior AI Engineer Perspective

### The Model Routing Decision

The biggest cost and quality lever most teams ignore: **not all tasks need the most expensive model**.

A production system that routes intelligently:
```
Simple classification (intent, sentiment) → haiku / gpt-4o-mini  ($0.15/M input)
Standard generation (chat, summarization) → sonnet / gpt-4o      ($3/M input)
Complex reasoning (analysis, hard coding) → opus / o3            ($15/M input)
```

A 3-tier routing strategy typically cuts costs by 60–80% vs using the premium model for everything,
with no measurable quality difference for the simple tiers.

### The Hidden API Key Rotation Problem

When your API key gets rotated (security incident, key compromise, periodic rotation), every
hardcoded reference to that key breaks simultaneously. Design your key management from day one:
- Environment variables or secrets manager (AWS Secrets Manager, HashiCorp Vault)
- The ability to rotate keys without a code deployment
- Key usage monitoring so you notice anomalies before they become incidents

### Every Response Has a `stop_reason` — Check It

```python
response = client.messages.create(...)
if response.stop_reason == "max_tokens":
    # The output was TRUNCATED — you got an incomplete response
    # Either increase max_tokens or handle the truncation gracefully
    # Returning truncated text to users is a silent quality failure
    pass
```

Most teams never check this. Truncated responses silently degrade quality for users who happen
to trigger long outputs. Build `stop_reason` checking into your response handling from the start.
