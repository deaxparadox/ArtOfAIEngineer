## 1. What Is It? (Theory)

### The Bill That Surprises Everyone

Model API spending doubled from $3.5B to $8.4B between late 2024 and mid-2025. Teams that
launched AI features without cost modeling were regularly 5–10× over budget by the time they
hit meaningful usage.

The patterns that cause this:

```
1. Long system prompt on every request
   1,000-token system prompt × 100,000 requests/month = 100M tokens
   At $3/M (Sonnet) = $300/month from the system prompt alone

2. Output token explosion
   Users ask open-ended questions → model generates 2,000-token responses
   Output tokens cost 4–5× input → a 2,000-token output costs 10× a 400-token output

3. Conversation context accumulation
   Turn 10 of a chat includes all prior 9 turns
   Turn 10 alone may consume 5× the tokens of turn 1

4. No per-user tracking
   One large customer uses 80% of your budget
   You have no visibility until the invoice arrives
```

Understanding token costs is not optional at production scale. It's as fundamental as database
query optimization or cloud storage pricing.

## 2. How It Actually Works — Deep Mechanics

### 2.1 What You're Actually Billed For

```
Every API call bills:
  input_tokens  = (system prompt) + (all prior messages) + (new user message) + (any tool results)
  output_tokens = (model's response text) + (any tool call JSON)

Pricing: input and output billed separately
  Example (Claude Sonnet 4.6): $3.00/M input, $15.00/M output

Cost = (input_tokens / 1,000,000) × input_price
     + (output_tokens / 1,000,000) × output_price
```

**The 4–5× multiplier:** Output tokens cost 4–5× more than input tokens across all major providers.
This means a 1,000-token response costs the same as 4,000–5,000 tokens of input. Design prompts
and max_tokens settings accordingly.

### 2.2 Counting Tokens Before the API Call

```python
import tiktoken
import anthropic

# === OpenAI token counting (tiktoken) ===
enc = tiktoken.encoding_for_model("gpt-4o")

def count_tokens_gpt4o(messages: list[dict]) -> int:
    """Count tokens for a GPT-4o chat completion request."""
    num_tokens = 3  # every reply primed with <|start|>assistant<|message|>
    for message in messages:
        num_tokens += 4  # role + message framing
        for key, value in message.items():
            num_tokens += len(enc.encode(str(value)))
    return num_tokens

# === Anthropic token counting (official API method) ===
client = anthropic.Anthropic()

def count_tokens_anthropic(system: str, messages: list[dict]) -> int:
    """Use Anthropic's official count_tokens endpoint."""
    response = client.messages.count_tokens(
        model="claude-sonnet-4-6",
        system=system,
        messages=messages
    )
    return response.input_tokens

# Estimate before sending
system = "You are a helpful assistant."
messages = [{"role": "user", "content": "Explain quantum computing in detail."}]

estimated_input = count_tokens_anthropic(system, messages)
print(f"Estimated input tokens: {estimated_input}")
print(f"Estimated input cost:   ${estimated_input * 3.00 / 1_000_000:.6f}")
```

### 2.3 Measuring Actual Usage Per Request

```python
import anthropic
from dataclasses import dataclass, field
from datetime import datetime

client = anthropic.Anthropic()

@dataclass
class RequestMetrics:
    timestamp: str
    model: str
    input_tokens: int
    output_tokens: int
    input_cost_usd: float
    output_cost_usd: float
    total_cost_usd: float
    user_id: str | None = None
    feature: str | None = None

# Token prices per million (update these from provider docs)
PRICES = {
    "claude-haiku-4-5-20251001": {"input": 1.00,  "output": 5.00},
    "claude-sonnet-4-6":         {"input": 3.00,  "output": 15.00},
    "claude-opus-4-8":           {"input": 15.00, "output": 75.00},
}

def tracked_call(
    system: str,
    user: str,
    model: str = "claude-sonnet-4-6",
    user_id: str | None = None,
    feature: str | None = None,
) -> tuple[str, RequestMetrics]:
    response = client.messages.create(
        model=model,
        max_tokens=1024,
        system=system,
        messages=[{"role": "user", "content": user}]
    )
    
    in_tok  = response.usage.input_tokens
    out_tok = response.usage.output_tokens
    prices  = PRICES.get(model, {"input": 3.00, "output": 15.00})
    
    in_cost  = in_tok  / 1_000_000 * prices["input"]
    out_cost = out_tok / 1_000_000 * prices["output"]
    
    metrics = RequestMetrics(
        timestamp=datetime.utcnow().isoformat(),
        model=response.model,
        input_tokens=in_tok,
        output_tokens=out_tok,
        input_cost_usd=in_cost,
        output_cost_usd=out_cost,
        total_cost_usd=in_cost + out_cost,
        user_id=user_id,
        feature=feature,
    )
    
    return response.content[0].text, metrics


text, metrics = tracked_call(
    system="Be concise.",
    user="What is recursion?",
    user_id="user_123",
    feature="help_center"
)
print(text[:100])
print(f"Cost: ${metrics.total_cost_usd:.6f} ({metrics.input_tokens} in / {metrics.output_tokens} out)")
```

### 2.4 Monthly Cost Projections

```python
from dataclasses import dataclass

@dataclass
class UsageProjection:
    monthly_requests: int
    avg_input_tokens: int
    avg_output_tokens: int
    model: str = "claude-sonnet-4-6"

def project_monthly_cost(projection: UsageProjection) -> dict:
    prices = PRICES.get(projection.model, {"input": 3.00, "output": 15.00})
    
    total_input  = projection.monthly_requests * projection.avg_input_tokens
    total_output = projection.monthly_requests * projection.avg_output_tokens
    
    input_cost  = total_input  / 1_000_000 * prices["input"]
    output_cost = total_output / 1_000_000 * prices["output"]
    
    # Add 30% buffer for retries, system overhead, context accumulation
    buffered_cost = (input_cost + output_cost) * 1.3
    
    return {
        "model": projection.model,
        "monthly_requests": f"{projection.monthly_requests:,}",
        "total_input_tokens": f"{total_input / 1_000_000:.1f}M",
        "total_output_tokens": f"{total_output / 1_000_000:.1f}M",
        "input_cost": f"${input_cost:.2f}",
        "output_cost": f"${output_cost:.2f}",
        "base_monthly_cost": f"${input_cost + output_cost:.2f}",
        "buffered_monthly_cost": f"${buffered_cost:.2f}",
    }

# Scenario: 10,000 users, 5 requests/day each, average 500 input + 200 output tokens
projection = UsageProjection(
    monthly_requests=10_000 * 5 * 30,  # 1.5M requests/month
    avg_input_tokens=500,
    avg_output_tokens=200,
    model="claude-sonnet-4-6"
)

result = project_monthly_cost(projection)
for k, v in result.items():
    print(f"{k:30s}: {v}")
# At scale: switching from Sonnet to Haiku for simple tasks cuts this ~3-4×
```

### 2.5 Per-User and Per-Feature Cost Attribution

```python
from collections import defaultdict
import json

class CostTracker:
    """
    Track token usage and cost by user_id and feature.
    In production: write to a database, not in-memory.
    """
    
    def __init__(self):
        self.by_user:    dict[str, dict] = defaultdict(lambda: {"input": 0, "output": 0, "cost": 0.0})
        self.by_feature: dict[str, dict] = defaultdict(lambda: {"input": 0, "output": 0, "cost": 0.0})
        self.total = {"input": 0, "output": 0, "cost": 0.0}
    
    def record(self, metrics: RequestMetrics):
        for bucket in [self.by_user[metrics.user_id or "anonymous"],
                       self.by_feature[metrics.feature or "unknown"],
                       self.total]:
            bucket["input"]  += metrics.input_tokens
            bucket["output"] += metrics.output_tokens
            bucket["cost"]   += metrics.total_cost_usd
    
    def report(self):
        print("=== Cost Report ===")
        print(f"\nTotal: ${self.total['cost']:.4f} "
              f"({self.total['input']:,} in / {self.total['output']:,} out)")
        
        print("\nBy Feature:")
        for feature, data in sorted(self.by_feature.items(), key=lambda x: -x[1]["cost"]):
            print(f"  {feature:20s}: ${data['cost']:.4f} ({data['input']:,} in / {data['output']:,} out)")
        
        print("\nTop 5 Users by Cost:")
        sorted_users = sorted(self.by_user.items(), key=lambda x: -x[1]["cost"])[:5]
        for user_id, data in sorted_users:
            print(f"  {user_id:20s}: ${data['cost']:.4f}")

tracker = CostTracker()

# Simulate some requests
import random
features = ["chat", "summarize", "classify", "extract"]
for i in range(20):
    _, metrics = tracked_call(
        system="Be helpful.",
        user=f"Test request {i}",
        user_id=f"user_{random.randint(1, 5)}",
        feature=random.choice(features),
    )
    tracker.record(metrics)

tracker.report()
```

### 2.6 Budget Enforcement

```python
import anthropic
from threading import Lock

client = anthropic.Anthropic()

class BudgetGuard:
    """
    Enforce per-user and global daily cost budgets.
    Thread-safe for concurrent requests.
    """
    
    def __init__(self, daily_user_limit_usd: float = 1.00, daily_global_limit_usd: float = 100.0):
        self.daily_user_limit   = daily_user_limit_usd
        self.daily_global_limit = daily_global_limit_usd
        self.user_spend:  dict[str, float] = defaultdict(float)
        self.global_spend: float = 0.0
        self._lock = Lock()
    
    def can_proceed(self, user_id: str, estimated_cost: float) -> tuple[bool, str]:
        with self._lock:
            if self.global_spend + estimated_cost > self.daily_global_limit:
                return False, "Global daily budget exceeded"
            if self.user_spend[user_id] + estimated_cost > self.daily_user_limit:
                return False, f"User daily budget exceeded (${self.user_spend[user_id]:.4f} spent)"
            return True, ""
    
    def record_spend(self, user_id: str, cost: float):
        with self._lock:
            self.user_spend[user_id] += cost
            self.global_spend        += cost
    
    def reset_daily(self):
        with self._lock:
            self.user_spend.clear()
            self.global_spend = 0.0

guard = BudgetGuard(daily_user_limit_usd=0.50, daily_global_limit_usd=50.0)

def budget_checked_call(user_id: str, prompt: str) -> str | None:
    # Pre-check with rough estimate (input tokens × unit cost)
    rough_estimate = len(prompt.split()) * 1.3 / 1_000_000 * 3.0 + 0.002  # rough output estimate
    
    allowed, reason = guard.can_proceed(user_id, rough_estimate)
    if not allowed:
        return f"Request blocked: {reason}"
    
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=512,
        messages=[{"role": "user", "content": prompt}]
    )
    
    # Record actual spend
    in_tok  = response.usage.input_tokens
    out_tok = response.usage.output_tokens
    actual_cost = (in_tok * 3.0 + out_tok * 15.0) / 1_000_000
    guard.record_spend(user_id, actual_cost)
    
    return response.content[0].text
```

## 4. Cost Optimization Strategies

### The Six Levers

| Lever | Potential Savings | Effort |
|---|---|---|
| 1. Prompt caching (repeated system prompts) | 50–90% on cached tokens | Low |
| 2. Model routing (use cheaper model for simple tasks) | 60–80% on simple task costs | Medium |
| 3. Response length limits | 30–50% on output costs | Low |
| 4. Batch API (offline workloads) | 50% flat | Low |
| 5. Context compression (summarize old turns) | 30–60% for long conversations | Medium |
| 6. Caching identical queries | 90%+ on repeated questions | High |

```python
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=150,   # aggressive limit reduces output cost

# Lever 5: Context compression for long conversations
def compress_conversation(messages: list[dict], threshold: int = 4000) -> list[dict]:
    """Summarize older messages when conversation grows too long."""
    total_tokens = sum(len(m["content"].split()) * 1.3 for m in messages)
    
    if total_tokens < threshold:
        return messages
    
    # Summarize all but last 4 messages
    to_compress = messages[:-4]
    recent      = messages[-4:]
    
    summary_resp = client.messages.create(
        model="claude-haiku-4-5-20251001",  # cheap model for summarization
        max_tokens=300,
        messages=[{
            "role": "user",
            "content": f"Summarize this conversation in 2–3 sentences:\n"
                       + "\n".join(f"{m['role']}: {m['content'][:200]}" for m in to_compress)
        }]
    )
    
    compressed = [{"role": "user", "content": f"[Conversation summary]: {summary_resp.content[0].text}"}]
    return compressed + recent
```

## 6. Trade-offs

### Aggressive `max_tokens` Limits

**Lower `max_tokens`:**
- Caps output cost
- Risk: responses get truncated if model wants to be longer
- Check `stop_reason == "max_tokens"` to detect truncation

**Higher `max_tokens`:**
- Never truncates (up to the limit)
- You only pay for what's generated (unused max_tokens aren't billed)
- Risk: model generates unnecessarily long responses

**Rule:** Set `max_tokens` based on your task's reasonable maximum, not the model's hard limit.
A classification task rarely needs more than 20 tokens. A code generation task might need 2,000.

### Cheaper Model vs Premium Model

Running everything on the premium model is the easiest choice that produces the worst unit economics.
Running everything on the cheapest model degrades quality unexpectedly.

**Practical approach:** Start with the premium model. Build evals. Then systematically test cheaper
models against your eval suite until you find the cheapest model that passes. Route that task to
the cheaper model going forward.

## 7. When To Use vs. When To Avoid

### Per-User Cost Tracking
✅ Always: Any product charging users, any SaaS, any freemium model
✅ Always: When a small number of users could consume disproportionate costs (power users)
❌ Can skip: Pure internal tooling with no cost-per-user concern

### Batch API
✅ Use when: Eval runs, data processing pipelines, non-realtime generation
✅ Use when: Cost is the primary concern and 24-hour delay is acceptable
❌ Don't use when: User is waiting for the response

⚠️ Common mistakes:
- Setting `max_tokens` to the model's maximum for every request — you pay for what's generated, but you're giving the model permission to ramble
- Not tracking output token costs separately — they're 4–5× more expensive and often the dominant cost
- Forgetting that conversation context accumulates — turn 20 of a chat is processing all prior turns too
- Not adding a buffer (1.5–2×) to cost projections — retries, overhead, and longer-than-expected outputs eat into the estimate

## 8. Senior AI Engineer Perspective

### The 10× Cost Shock

Almost every team that deploys an LLM feature without cost modeling experiences a 10× shock:
they estimated based on average request size, but their heaviest users consume 10–50× average.
A few power users can dominate your monthly bill.

Defense: per-user cost attribution + budget limits + anomaly alerts when a user or feature
spends more than 2–3× their normal rate.

### Output Tokens Are the Hidden Monster

Teams focus on input costs because that's where prompt engineering effort goes (reducing prompt length,
improving few-shot efficiency). But for most use cases, **output tokens cost 4–5× more per token
and often represent 50–80% of total cost**.

The output cost levers:
1. `max_tokens` limit — don't let the model ramble
2. System prompt instructions — "answer concisely in 2–3 sentences"
3. Model routing — for classification/extraction, output tokens are few; cost is mainly input
4. Caching responses to identical or near-identical queries

### The 1000-Request Mental Model

Before finalizing any LLM feature:

```
Take your average prompt:
  System prompt: ~800 tokens
  User message:  ~200 tokens
  Response:      ~400 tokens

Per request on Claude Sonnet:
  Input:  1,000 tokens × $3.00/M  = $0.003
  Output: 400 tokens × $15.00/M   = $0.006
  Total:                           = $0.009 per request

At 100,000 requests/month:        = $900/month

Can you afford $900/month for this feature?
If usage grows 10× in 3 months?   = $9,000/month
Is that sustainable?
```

Do this math before shipping. Not after.
