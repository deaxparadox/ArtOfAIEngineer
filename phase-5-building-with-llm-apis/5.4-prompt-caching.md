## 1. What Is It? (Theory)

### The Insight

Every time you call the LLM API with a system prompt, the provider:
1. Tokenizes the system prompt
2. Runs it through the model to compute key-value (KV) attention pairs
3. Bills you for those input tokens

If your system prompt is the same across 10,000 requests today, you're paying for step 1–3
ten thousand times. Prompt caching lets you pay for it once and reuse the KV-cache across requests.

**Savings:**
- Anthropic: cached tokens billed at $0.30/M (vs $3.00/M) — **90% off**
- OpenAI: automatic caching, 50% off for cached prefixes ≥ 1,024 tokens
- Google: Context Caching API, ~75% off for cached content

For any application with a long, repeated system prompt, caching is one of the highest-ROI
optimizations available. Five minutes to implement, 50–90% cost reduction.

### What Gets Cached

Caching works on **prefixes** — the beginning of the prompt that remains constant across requests.

```
Request 1: [System Prompt 2000 tokens] + [User message A 50 tokens]
Request 2: [System Prompt 2000 tokens] + [User message B 50 tokens]
Request 3: [System Prompt 2000 tokens] + [User message C 100 tokens]

Without caching: 3 × 2000 = 6,000 system prompt tokens billed
With caching:    2000 tokens (first write) + 0 + 0 = 2,000 system prompt tokens billed
                 2000 cache write tokens (one-time overhead)
                 200 new user tokens billed normally

Net savings: ~66% on this example
```

The longer the repeated prefix and the more requests, the higher the savings.

## 2. How It Actually Works — Deep Mechanics

### 2.1 Anthropic: Explicit Cache Control

Anthropic requires you to explicitly mark content for caching using `cache_control`:

```python
import anthropic

client = anthropic.Anthropic()

# Mark the system prompt for caching
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": "You are a senior Python engineer...\n[2000 tokens of instructions]",
            "cache_control": {"type": "ephemeral"}  # cache this block
        }
    ],
    messages=[{"role": "user", "content": "How do I write a decorator?"}]
)

# Check cache performance in usage
usage = response.usage
print(f"Input tokens:         {usage.input_tokens}")
print(f"Cache write tokens:   {getattr(usage, 'cache_creation_input_tokens', 0)}")
print(f"Cache read tokens:    {getattr(usage, 'cache_read_input_tokens', 0)}")
```

**On first call:** `cache_creation_input_tokens` will be non-zero — you're paying a small
write overhead (25% more than normal for 5-minute cache TTL).

**On subsequent calls within 5 minutes:** `cache_read_input_tokens` will show the cached tokens
billed at 90% off.

**TTL options:**
- `"ephemeral"` → 5 minutes (default, cheapest to write)
- You can also use the longer-lived 1-hour cache by specifying TTL in the beta headers

### 2.2 Cache Placement Rules

The cache breakpoint must be at a consistent position across requests. Anthropic caches everything
**up to and including** the cache_control marker.

```python
# CORRECT: cache_control at end of stable prefix
system=[
    {"type": "text", "text": "[stable 2000 token system prompt]", "cache_control": {"type": "ephemeral"}},
    # Any dynamic content goes AFTER the cache marker or in the user turn
]

# WRONG: cache_control before dynamic content
system=[
    {"type": "text", "text": "[stable prefix]", "cache_control": {"type": "ephemeral"}},
    {"type": "text", "text": f"[dynamic content: {variable}]"},   # This breaks caching
    # The prefix ending at the cache_control point is valid, but the
    # dynamic content after it changes every request
]
```

**Rule:** Everything before (and including) the `cache_control` marker must be identical
across requests. Dynamic content (user documents, retrieved context) goes after or in the user turn.

### 2.3 Multiple Cache Breakpoints

You can have up to 4 cache breakpoints in one request. This enables caching at multiple levels:

```python
import anthropic

client = anthropic.Anthropic()

REFERENCE_DOC = "..." * 5000   # 5000-token reference document

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": "You are a technical writer. [Instructions]",
            "cache_control": {"type": "ephemeral"}   # Cache 1: system instructions
        },
        {
            "type": "text",
            "text": REFERENCE_DOC,
            "cache_control": {"type": "ephemeral"}   # Cache 2: reference document
        }
    ],
    messages=[{"role": "user", "content": "Summarize section 3 of the reference doc."}]
)
```

First call: pays full price + cache write overhead for both blocks.
Subsequent calls: both blocks served from cache at 10% of normal cost.

### 2.4 OpenAI: Automatic Caching

OpenAI handles caching automatically — no code changes required:

```python
from openai import OpenAI

client = OpenAI()

# No special configuration needed
# OpenAI automatically caches prompt prefixes ≥ 1,024 tokens
# 50% discount applied automatically when cache hits

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "system",  "content": "[1500 token system prompt — will be cached automatically]"},
        {"role": "user",    "content": "User question here"}
    ]
)

# Check if caching happened
if hasattr(response.usage, "prompt_tokens_details"):
    cached = response.usage.prompt_tokens_details.cached_tokens
    print(f"Cached tokens: {cached}")
```

**OpenAI caching requirements:**
- Prompt prefix must be ≥ 1,024 tokens to qualify for caching
- Cache persists for 5–60 minutes (varies by load)
- Cache key = exact token sequence — even one token difference = cache miss

**Structural implication:** Keep all stable content at the beginning of your messages array.
Variable content at the end. Any insertion into the middle of a cached prefix invalidates the cache.

### 2.5 Google: Context Caching API

Google's approach is more explicit — you create a named cache object and reference it:

```python
from google import genai
import datetime

client = genai.Client()

# Create a cache for large stable content
cache = client.caches.create(
    model="gemini-2.5-flash",
    config=genai.types.CreateCachedContentConfig(
        display_name="product-documentation-cache",
        system_instruction="You are a product documentation assistant...",
        contents=[
            genai.types.Content(
                role="user",
                parts=[genai.types.Part(text="[10,000 tokens of product docs]")]
            )
        ],
        ttl=datetime.timedelta(hours=1)
    )
)

print(f"Cache created: {cache.name}")

# Reference the cache in subsequent requests
response = client.models.generate_content(
    model="gemini-2.5-flash",
    contents="What does section 3 say about authentication?",
    config=genai.types.GenerateContentConfig(
        cached_content=cache.name
    )
)
```

## 3. Real-World Context

### Where Caching Pays Off

| Use Case | System Prompt Size | Monthly Requests | Monthly Savings (Sonnet) |
|---|---|---|---|
| Customer support bot | 1,000 tokens | 100,000 | ~$270 |
| Code review tool (with style guide) | 5,000 tokens | 50,000 | ~$675 |
| Document Q&A over product docs | 15,000 tokens | 30,000 | ~$1,215 |
| Medical reference assistant | 20,000 tokens | 20,000 | ~$1,080 |
| Legal document analysis | 30,000 tokens | 10,000 | ~$810 |

The pattern: **larger system prompts + higher request volume = larger absolute savings**.
For simple apps with short prompts, the savings are real but small. For enterprise tools with
detailed instructions and high volume, caching is often the #1 cost optimization available.

### Real Production Deployment

**Anthropic's stated case:** A team with a 2,000-token system prompt handling 10,000 daily
requests. Without caching: `2,000 × 10,000 × $3/M = $60/day = $1,800/month`.
With caching: cache write on first use, reads at 10% = ~$180/month. **$1,620/month saved**.

## 4. Code Examples

### Measuring Cache Performance

```python
import anthropic
import time

client = anthropic.Anthropic()

LONG_SYSTEM = """You are an expert Python code reviewer specializing in:
1. Security vulnerabilities (SQL injection, XSS, path traversal, etc.)
2. Performance anti-patterns (N+1 queries, unnecessary loops, memory leaks)
3. Code style and Python best practices (PEP 8, type hints, docstrings)
4. Error handling completeness
5. Testing coverage gaps

For each review, structure your response as:
- CRITICAL: Issues that must be fixed before deployment
- WARNING: Issues that should be fixed soon
- SUGGESTION: Minor improvements for code quality
""" + ("Additional context about reviewing practices.\n" * 100)   # ~2000 tokens total

def call_with_cache_metrics(code: str, call_number: int) -> dict:
    start = time.time()
    
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=512,
        system=[{
            "type": "text",
            "text": LONG_SYSTEM,
            "cache_control": {"type": "ephemeral"}
        }],
        messages=[{"role": "user", "content": f"Review this code:\n```python\n{code}\n```"}]
    )
    
    elapsed = time.time() - start
    usage   = response.usage
    
    cache_write = getattr(usage, "cache_creation_input_tokens", 0)
    cache_read  = getattr(usage, "cache_read_input_tokens", 0)
    normal_in   = usage.input_tokens - cache_write - cache_read
    
    # Cost calculation
    cost = (normal_in * 3.00 + cache_write * 3.75 + cache_read * 0.30 + usage.output_tokens * 15.00) / 1_000_000
    
    print(f"\nCall {call_number}:")
    print(f"  Elapsed:      {elapsed:.2f}s")
    print(f"  Normal input: {normal_in:,} tokens (${normal_in * 3.00 / 1_000_000:.4f})")
    print(f"  Cache write:  {cache_write:,} tokens (${cache_write * 3.75 / 1_000_000:.4f})")
    print(f"  Cache read:   {cache_read:,} tokens (${cache_read * 0.30 / 1_000_000:.4f})")
    print(f"  Output:       {usage.output_tokens:,} tokens (${usage.output_tokens * 15.00 / 1_000_000:.4f})")
    print(f"  Total cost:   ${cost:.4f}")
    
    return {"elapsed": elapsed, "cost": cost, "cache_hit": cache_read > 0}

sample_code = """
def get_user(user_id):
    query = f"SELECT * FROM users WHERE id = {user_id}"
    return db.execute(query)
"""

# First call: cache MISS (cache is populated)
r1 = call_with_cache_metrics(sample_code, 1)

# Wait just a moment, then second call: cache HIT
time.sleep(1)
r2 = call_with_cache_metrics(sample_code, 2)

print(f"\nCost reduction: {(1 - r2['cost']/r1['cost']) * 100:.0f}%")
```

### Document-Augmented Q&A with Caching

```python
import anthropic

client = anthropic.Anthropic()

class CachedDocumentQA:
    """
    Q&A over a document with the document cached.
    Subsequent questions over the same document pay ~10% of initial token cost.
    """
    
    def __init__(self, document_text: str, domain: str = "general"):
        self.document  = document_text
        self.domain    = domain
        self._system   = [
            {
                "type": "text",
                "text": f"You are an expert {domain} analyst. Answer questions about the provided document. "
                        f"Always cite specific sections when answering.\n\n"
                        f"DOCUMENT:\n{document_text}",
                "cache_control": {"type": "ephemeral"}   # cache document + instructions
            }
        ]
    
    def ask(self, question: str) -> str:
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=512,
            system=self._system,
            messages=[{"role": "user", "content": question}]
        )
        
        usage  = response.usage
        cached = getattr(usage, "cache_read_input_tokens", 0)
        
        if cached > 0:
            print(f"  [Cache HIT: {cached:,} tokens served from cache]")
        
        return response.content[0].text
    
    def multi_turn_ask(self, questions: list[str]) -> list[str]:
        """Ask multiple questions about the same document efficiently."""
        return [self.ask(q) for q in questions]

# Usage: load a long document once, ask many questions cheaply
long_doc = "Chapter 1: Introduction to Machine Learning...\n" * 500  # ~2500 tokens
qa = CachedDocumentQA(long_doc, domain="technical")

questions = [
    "What are the main topics covered in chapter 1?",
    "What examples are given in the introduction?",
    "Summarize the key points from the first section.",
]

for q in questions:
    print(f"\nQ: {q}")
    print(f"A: {qa.ask(q)[:150]}...")
# Questions 2 and 3 benefit from cache hit on the document
```

## 6. Alternatives & Comparisons

### Cache Types Comparison

| Approach | Mechanism | Savings | Complexity |
|---|---|---|---|
| **Prompt caching** (provider-level) | Provider caches KV pairs | 50–90% on stable prefix | Low |
| **Response caching** (application-level) | Cache identical (prompt → response) pairs in Redis/DB | 100% for exact repeats | Medium |
| **Semantic caching** | Embed queries, find similar cached responses | 90%+ for near-duplicate queries | High |

**Combination:** Use both prompt caching (cheap stable prefix) + response caching (deduplicate
identical queries). Many enterprise tools get 70–80% cache hit rates on FAQ-style queries.

## 7. Trade-offs

### Cache TTL (5 min vs 1 hour for Anthropic)

| TTL | Write cost | Hit cost | Best for |
|---|---|---|---|
| 5 min (ephemeral) | 1.25× normal input | 0.10× | High-frequency apps; cache stays warm |
| 1 hour | 2× normal input | 0.10× | Sporadic usage; long-document analysis |

For bursty traffic patterns (e.g., business hours usage), 1-hour caching is worth the higher
write cost. For 24/7 steady traffic, 5-minute ephemeral caching provides the same hit rate
at lower overhead.

### What Breaks Caching

Anything that changes the exact token sequence before the cache marker breaks the cache:
- Adding a timestamp to the system prompt
- Interpolating user-specific data into the system prompt
- Changing word order or formatting
- Model version updates may invalidate provider-side caches

**Design rule:** Treat the cached prefix as a configuration file — static, versioned, not dynamic.

## 8. When To Use vs. When To Avoid

### Prompt Caching
✅ Always use when: System prompt > 1,024 tokens and request volume > 100/day
✅ High value when: Large reference documents (product docs, legal text, code context) shared across queries
✅ Use for: Eval runs (same system prompt, many test inputs)
❌ Skip when: System prompt is short (<500 tokens) — absolute savings are minimal
❌ Skip when: Every request has a unique "system prompt" — caching requires stable prefix

⚠️ Common mistakes:
- Inserting dynamic content (user ID, timestamp) before the cache marker — kills all cache hits
- Not checking cache_read_input_tokens in usage — you can't verify caching is working without this
- Assuming caching "just works" without verification — the provider may not cache if TTL expired or tokens are below threshold
- Forgetting that OpenAI's minimum is 1,024 tokens — a 900-token system prompt gets zero caching

## 9. Senior AI Engineer Perspective

### The "Stable Prefix" Architectural Decision

Prompt caching forces a good architectural discipline: separate stable configuration from
dynamic request context.

```
Good architecture:
  Stable:  system instructions + knowledge base + examples    → CACHED
  Dynamic: user message + retrieved context + conversation    → NOT cached

Bad architecture:
  System prompt includes: f"User {user_id} from {company}, current time {timestamp}..."
  → Every request is unique, cache miss every time
```

The discipline to keep your stable prefix truly stable pays off in two ways: cheaper API calls
and more predictable model behavior (the model sees the same instructions every time, not slightly
different versions with updated timestamps).

### Caching Is the First Optimization to Reach For

When teams ask "how do we cut our LLM costs?", the answer is almost always:

1. Enable prompt caching → 50–90% savings on system prompt tokens (30 minutes of work)
2. Route simple tasks to cheaper models → 60–80% savings on those tasks (1–2 days of work)
3. Batch non-realtime workloads → 50% flat savings (1 day of work)

Caching alone often justifies the entire AI platform budget optimization effort. Implement it
before anything else.
