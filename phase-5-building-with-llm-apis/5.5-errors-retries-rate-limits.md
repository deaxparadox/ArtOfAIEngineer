## 1. What Is It? (Theory)

### LLM APIs Are Unreliable by Design

LLM providers operate at the frontier of compute infrastructure. Unlike a simple database query
or a REST API returning static data, LLM inference:
- Consumes variable GPU compute per request
- Runs on clusters that can be overloaded
- Is subject to rate limits enforced at multiple granularities
- Can time out for very long generations

**The numbers that matter:**
- Rate limit errors (429s): the most common error in any high-volume LLM application
- Provider overload (529): happens during peak hours or viral traffic events
- Timeouts: especially for long-context requests or streaming
- Model unavailability (503): rare but happens during deployments

**What "retry without jitter" does:** If 1,000 clients all hit a 429 at the same time and all
retry with the same exponential backoff schedule, they all retry simultaneously — creating
synchronized "thundering herd" waves that immediately re-trigger the rate limit. Jitter breaks
this synchronization.

## 2. How It Actually Works — Deep Mechanics

### 2.1 Error Taxonomy

```
HTTP 400 Bad Request       → Invalid request (wrong model name, missing required field)
HTTP 401 Unauthorized      → Invalid or expired API key
HTTP 403 Forbidden         → Request violates content policy
HTTP 404 Not Found         → Model doesn't exist or endpoint wrong
HTTP 422 Unprocessable     → Request format is valid but semantically wrong (Anthropic-specific)
HTTP 429 Too Many Requests → Rate limit exceeded (tokens/min or requests/min)
HTTP 500 Internal Server   → Provider-side error — transient, retry
HTTP 503 Service Unavailable → Provider overloaded or in maintenance — retry
HTTP 529 Overloaded        → Anthropic-specific: system overloaded — retry
Timeout                    → Request took too long — may or may not have generated
```

**Retryable errors:** 429, 500, 503, 529, Timeout
**Non-retryable errors:** 400, 401, 403, 404, 422

Retrying a 401 is pointless — your key is wrong. Retrying a 403 is pointless — your content
violates policy. Retrying a 429 with backoff will eventually succeed.

### 2.2 Exponential Backoff with Full Jitter

```
Simple exponential:  wait = 2^attempt × base_delay
Exponential + jitter: wait = random(0, 2^attempt × base_delay)

Example sequence (base_delay=1s, max=32s):
  Attempt 1 failed → wait random(0, 2s)    → retry
  Attempt 2 failed → wait random(0, 4s)    → retry
  Attempt 3 failed → wait random(0, 8s)    → retry
  Attempt 4 failed → wait random(0, 16s)   → retry
  Attempt 5 failed → wait random(0, 32s)   → retry
  Attempt 6 failed → raise exception
```

**Why full jitter (vs half-jitter or capped jitter)?** Full jitter (`random(0, cap)`) distributes
retries uniformly, preventing all clients from retrying simultaneously. This is the AWS recommended
approach for distributed systems.

### 2.3 The tenacity Library (Production Retry Pattern)

```python
import anthropic
from tenacity import (
    retry,
    stop_after_attempt,
    wait_random_exponential,
    retry_if_exception_type,
    before_sleep_log,
)
import logging

logger = logging.getLogger(__name__)
client = anthropic.Anthropic()

# Retryable exceptions
RETRYABLE = (
    anthropic.RateLimitError,
    anthropic.InternalServerError,
    anthropic.APIConnectionError,
    anthropic.APITimeoutError,
)

@retry(
    retry=retry_if_exception_type(RETRYABLE),
    wait=wait_random_exponential(multiplier=1, min=1, max=60),
    stop=stop_after_attempt(6),
    before_sleep=before_sleep_log(logger, logging.WARNING),
)
def resilient_call(system: str, user: str, model: str = "claude-sonnet-4-6") -> str:
    """LLM call with exponential backoff retry on transient failures."""
    response = client.messages.create(
        model=model,
        max_tokens=1024,
        system=system,
        messages=[{"role": "user", "content": user}]
    )
    return response.content[0].text

# Usage — handles transient errors transparently
try:
    result = resilient_call("Be helpful.", "What is Python?")
    print(result)
except anthropic.RateLimitError:
    print("Rate limit persists after 6 attempts — escalate or queue")
except anthropic.AuthenticationError:
    print("API key invalid — non-retryable")
except anthropic.BadRequestError as e:
    print(f"Request invalid: {e}")
```

### 2.4 Rate Limit Handling

LLM providers enforce rate limits on two dimensions:

```
RPM (Requests Per Minute):     e.g., 60 RPM for a starter tier
TPM (Tokens Per Minute):       e.g., 100,000 TPM for a starter tier

You can hit either limit:
  - Many small requests: RPM limit hits first
  - Few large requests:  TPM limit hits first
  - Mixed: depends on size distribution
```

**Reading rate limit headers:**
```python
import anthropic

client = anthropic.Anthropic()

try:
    response = client.messages.create(...)
except anthropic.RateLimitError as e:
    # The response headers contain useful information
    print(f"Status: {e.status_code}")
    # Some providers include Retry-After header
    retry_after = e.response.headers.get("retry-after", "unknown")
    print(f"Retry after: {retry_after}s")
    # Anthropic-specific headers
    limit      = e.response.headers.get("anthropic-ratelimit-requests-limit")
    remaining  = e.response.headers.get("anthropic-ratelimit-requests-remaining")
    reset_time = e.response.headers.get("anthropic-ratelimit-requests-reset")
    print(f"Limit: {limit}, Remaining: {remaining}, Resets at: {reset_time}")
```

**Proactive rate limit management:**

```python
import time
from threading import Lock
from collections import deque

class RateLimiter:
    """
    Token bucket rate limiter.
    Enforces RPM and TPM limits client-side before they hit the API.
    """
    
    def __init__(self, rpm: int = 60, tpm: int = 100_000):
        self.rpm     = rpm
        self.tpm     = tpm
        self.request_times: deque = deque()
        self.token_usage:   deque = deque()   # (timestamp, token_count)
        self._lock = Lock()
    
    def acquire(self, estimated_tokens: int = 1000) -> None:
        """Block until we're within rate limits. Call before each API request."""
        with self._lock:
            now = time.time()
            window = 60.0
            
            # Clean old entries
            while self.request_times and now - self.request_times[0] > window:
                self.request_times.popleft()
            while self.token_usage and now - self.token_usage[0][0] > window:
                self.token_usage.popleft()
            
            # Check RPM
            if len(self.request_times) >= self.rpm:
                sleep_time = window - (now - self.request_times[0]) + 0.1
                time.sleep(max(0, sleep_time))
            
            # Check TPM
            current_tpm = sum(t for _, t in self.token_usage)
            if current_tpm + estimated_tokens > self.tpm:
                sleep_time = window - (now - self.token_usage[0][0]) + 0.1
                time.sleep(max(0, sleep_time))
            
            # Record this request
            self.request_times.append(time.time())
            self.token_usage.append((time.time(), estimated_tokens))

limiter = RateLimiter(rpm=50, tpm=90_000)

def rate_limited_call(prompt: str) -> str:
    limiter.acquire(estimated_tokens=len(prompt.split()) * 2)   # rough estimate
    return resilient_call("Be concise.", prompt)
```

### 2.5 Circuit Breaker Pattern

A circuit breaker prevents hammering a failing service. When error rate exceeds a threshold,
"open" the circuit: fail fast without calling the provider. After a cooldown, "half-open" and
test if the service recovered.

```python
import time
from enum import Enum

class CircuitState(Enum):
    CLOSED   = "closed"    # Normal operation
    OPEN     = "open"      # Failing fast — not calling the provider
    HALF_OPEN = "half_open" # Testing if recovery happened

class CircuitBreaker:
    def __init__(
        self,
        failure_threshold: int = 5,    # open after N failures
        recovery_timeout: float = 60.0, # seconds before trying again
        success_threshold: int = 2,     # close after N successes in half-open
    ):
        self.failure_threshold = failure_threshold
        self.recovery_timeout  = recovery_timeout
        self.success_threshold = success_threshold
        self.state             = CircuitState.CLOSED
        self.failure_count     = 0
        self.success_count     = 0
        self.last_failure_time = 0.0
    
    def call(self, fn, *args, **kwargs):
        if self.state == CircuitState.OPEN:
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = CircuitState.HALF_OPEN
                self.success_count = 0
            else:
                raise RuntimeError("Circuit breaker OPEN — service unavailable")
        
        try:
            result = fn(*args, **kwargs)
            self._on_success()
            return result
        except Exception as e:
            self._on_failure()
            raise
    
    def _on_success(self):
        if self.state == CircuitState.HALF_OPEN:
            self.success_count += 1
            if self.success_count >= self.success_threshold:
                self.state = CircuitState.CLOSED
                self.failure_count = 0
        elif self.state == CircuitState.CLOSED:
            self.failure_count = 0
    
    def _on_failure(self):
        self.failure_count += 1
        self.last_failure_time = time.time()
        if self.failure_count >= self.failure_threshold:
            self.state = CircuitState.OPEN
            print(f"Circuit OPENED after {self.failure_count} failures")
```

### 2.6 Multi-Provider Fallback

```python
import anthropic
from openai import OpenAI

anthropic_client = anthropic.Anthropic()
openai_client    = OpenAI()

def call_with_fallback(system: str, user: str) -> tuple[str, str]:
    """
    Try primary provider (Anthropic), fall back to secondary (OpenAI) on failure.
    Returns (response_text, provider_used).
    """
    
    # Try primary
    try:
        resp = anthropic_client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            system=system,
            messages=[{"role": "user", "content": user}]
        )
        return resp.content[0].text, "anthropic"
    
    except (anthropic.RateLimitError, anthropic.InternalServerError,
            anthropic.APIConnectionError) as e:
        print(f"Primary (Anthropic) failed: {type(e).__name__} — trying fallback")
    
    # Try fallback
    try:
        resp = openai_client.chat.completions.create(
            model="gpt-4o",
            max_tokens=1024,
            messages=[
                {"role": "system", "content": system},
                {"role": "user",   "content": user}
            ]
        )
        return resp.choices[0].message.content, "openai"
    
    except Exception as e:
        raise RuntimeError(f"Both providers failed. Last error: {e}") from e

text, provider = call_with_fallback("Be concise.", "What is machine learning?")
print(f"Response from {provider}: {text[:100]}")
```

### 2.7 Async Retry with asyncio

```python
import asyncio
import random
import anthropic

async_client = anthropic.AsyncAnthropic()

async def async_call_with_retry(
    system: str,
    user: str,
    max_retries: int = 5,
    base_delay: float = 1.0,
    max_delay: float = 60.0,
) -> str:
    """Async LLM call with exponential backoff and jitter."""
    
    for attempt in range(max_retries):
        try:
            response = await async_client.messages.create(
                model="claude-sonnet-4-6",
                max_tokens=1024,
                system=system,
                messages=[{"role": "user", "content": user}]
            )
            return response.content[0].text
        
        except (anthropic.RateLimitError, anthropic.InternalServerError,
                anthropic.APIConnectionError, anthropic.APITimeoutError) as e:
            
            if attempt == max_retries - 1:
                raise  # exhausted retries
            
            # Full jitter: random delay in [0, cap]
            cap   = min(max_delay, base_delay * (2 ** attempt))
            delay = random.uniform(0, cap)
            print(f"Attempt {attempt+1} failed ({type(e).__name__}). Retrying in {delay:.1f}s")
            await asyncio.sleep(delay)
        
        except (anthropic.AuthenticationError, anthropic.BadRequestError):
            raise  # non-retryable

# Usage in async context
async def main():
    result = await async_call_with_retry("Be helpful.", "What is Python?")
    print(result[:100])

asyncio.run(main())
```

### Production Incident: The Thundering Herd

A real pattern: a product launches, hits viral traction, suddenly 10,000 concurrent users hit
the API simultaneously. All requests get 429s. All clients retry with the same exponential
backoff (no jitter). They all retry at T+2s, hit 429 again. All retry at T+4s. The synchronized
retries continue for minutes, amplifying the overload.

**Fix:** Jitter (spreads retries across time), queue-based rate limiting (limits concurrent in-flight
requests), and circuit breakers (stops trying when the service is clearly overwhelmed).

## 4. Code Examples

### Complete Production Client

```python
import anthropic
import asyncio
import random
import time
import logging
from dataclasses import dataclass
from typing import Optional

logger = logging.getLogger(__name__)

@dataclass
class CallConfig:
    model:        str   = "claude-sonnet-4-6"
    max_tokens:   int   = 1024
    max_retries:  int   = 5
    base_delay:   float = 1.0
    max_delay:    float = 60.0
    timeout:      float = 30.0

class ProductionLLMClient:
    """
    Production-grade LLM client with:
    - Exponential backoff + jitter
    - Circuit breaker
    - Request/response logging
    - Cost tracking
    """
    
    RETRYABLE_STATUS_CODES = {429, 500, 503, 529}
    
    def __init__(self, api_key: Optional[str] = None):
        self.client  = anthropic.Anthropic(api_key=api_key, timeout=60.0)
        self.breaker = CircuitBreaker(failure_threshold=10, recovery_timeout=120)
    
    def call(self, system: str, user: str, cfg: CallConfig = CallConfig()) -> str:
        return self.breaker.call(self._call_with_retry, system, user, cfg)
    
    def _call_with_retry(self, system: str, user: str, cfg: CallConfig) -> str:
        for attempt in range(cfg.max_retries):
            try:
                start    = time.monotonic()
                response = self.client.messages.create(
                    model=cfg.model, max_tokens=cfg.max_tokens,
                    system=system,
                    messages=[{"role": "user", "content": user}]
                )
                elapsed = time.monotonic() - start
                
                logger.info(
                    "LLM call success",
                    extra={
                        "model": cfg.model, "attempt": attempt + 1,
                        "elapsed_s": round(elapsed, 2),
                        "input_tokens": response.usage.input_tokens,
                        "output_tokens": response.usage.output_tokens,
                        "stop_reason": response.stop_reason,
                    }
                )
                
                if response.stop_reason == "max_tokens":
                    logger.warning("Response truncated at max_tokens")
                
                return response.content[0].text
            
            except anthropic.APIStatusError as e:
                if e.status_code not in self.RETRYABLE_STATUS_CODES:
                    logger.error(f"Non-retryable error {e.status_code}: {e}")
                    raise
                
                if attempt == cfg.max_retries - 1:
                    logger.error(f"Exhausted {cfg.max_retries} retries. Last error: {e}")
                    raise
                
                delay = min(cfg.max_delay, random.uniform(0, cfg.base_delay * (2 ** attempt)))
                logger.warning(f"Retryable error {e.status_code}. Attempt {attempt+1}. Sleeping {delay:.1f}s")
                time.sleep(delay)

client = ProductionLLMClient()
result = client.call(
    system="You are a concise assistant.",
    user="What is the capital of Germany?"
)
print(result)
```

## 6. Alternatives & Comparisons

### Retry Strategies

| Strategy | Algorithm | Best For |
|---|---|---|
| Fixed delay | Always wait N seconds | Simple; bad for thundering herd |
| Exponential | 2^n × base | Better; still thundering herd risk |
| Exponential + jitter | random(0, 2^n × base) | Production standard |
| Linear backoff | n × base | Gentler; less common |
| Immediate retry (once) | No wait | For idempotent ops with fast recovery |

**Always use exponential + jitter in production.** Fixed and exponential-without-jitter cause
thundering herds.

### Multi-Provider vs Single-Provider

**Single provider with resilience:**
- Simpler; one SDK, one billing
- Fallback on overload is impossible
- Good for most early-stage products

**Multi-provider with failover:**
- Resilient to provider outages
- Complex; behavior may differ between providers
- Necessary for high-availability products (> 99.9% uptime requirements)

## 7. Trade-offs

### Retry Count vs Latency

More retries = higher chance of eventual success, but higher user-perceived latency.
For user-facing features, consider fail-fast after 2–3 retries and show an error rather than
making the user wait 60+ seconds for the 5th retry.

For background/async workloads: 5–7 retries with longer backoff is fine.

### Circuit Breaker Sensitivity

**Too sensitive (low threshold):** Opens circuit on normal transient errors, degrading
user experience unnecessarily.

**Too permissive (high threshold):** Keeps hammering a failing service, wasting resources
and delaying failure detection.

**Tune based on:** Error rate baseline for your workload. If your normal error rate is 0.5%,
set the circuit-open threshold at 5–10%. If normal is 2%, set it at 15–20%.

## 8. When To Use vs. When To Avoid

### Retry Logic
✅ Always implement: Any production system calling LLM APIs
✅ Always use jitter: For any retry pattern with multiple concurrent clients
❌ Don't retry: 4xx non-429 errors (your request is wrong, not the provider)
❌ Don't retry: Content policy violations (retrying won't change the refusal)

### Circuit Breaker
✅ Use when: High-volume production traffic where sustained failures would cascade
✅ Use when: The LLM is in a critical path (the feature breaks if the LLM fails)
❌ Can skip for: Low-volume internal tools where simple retry suffices

⚠️ Common mistakes:
- Retrying without jitter — thundering herd re-triggers the rate limit immediately
- Retrying non-retryable errors — wastes time, never succeeds
- Not logging retry attempts — you can't diagnose rate limit patterns without this data
- Setting `max_retries` too high for user-facing features — 60-second user wait is a bad UX
- Ignoring `Retry-After` headers — the provider tells you how long to wait; use it

## 9. Senior AI Engineer Perspective

### The "Budget for Retries" Mental Model

Every retry consumes tokens (if the request succeeds on the Nth try) and time. Build retry
cost into your token budget:

```
Expected cost per "logical request":
  = (p_success_first_try × cost_first_try)
  + (1 - p_success) × (p_success_second_try × cost_second_try)
  + ...

If p(success first try) = 0.97 and p(success second try given failure) = 0.90:
  Expected calls ≈ 1.03
  Budget factor: 1.03× your nominal cost estimate

At 429 spikes (p_success = 0.7 per try):
  Expected calls ≈ 1.4
  Budget factor: 1.4×
```

This is why cost projections should include a 1.3–1.5× buffer.

### Observability Is Half the Resilience Story

Retry logic prevents failures from reaching users. But you also need to **know when retries
are happening** — they're a signal that something is wrong:

- Sudden spike in 429s → you're approaching your rate limit, consider increasing your plan
- Persistent 500s → provider is having issues, consider switching to fallback
- Rising retry rate on one endpoint → that endpoint is getting more traffic, rebalance

Instrument every retry attempt with logging. Aggregate into a dashboard. Set alerts on retry rate.
Without this, you're flying blind and will discover problems from user complaints, not metrics.
