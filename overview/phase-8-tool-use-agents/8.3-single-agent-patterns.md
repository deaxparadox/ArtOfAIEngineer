## 1. What Is It? (Theory)

### Why Patterns Matter

A "loop that calls tools" is not an agent. An agent is a system that:
1. **Reasons** about what to do next
2. **Acts** by calling tools
3. **Observes** the results
4. **Decides** whether to continue or finish

The pattern you choose determines how the agent reasons, how it handles failures, and how
predictable its behavior is. There are two dominant production patterns:

**ReAct** (Reason + Act): Think, act, observe, repeat. The model reasons at every step.
**Plan-and-Execute**: Plan first, then execute the plan. Separates strategy from tactics.

Each has strengths and weaknesses. Knowing when to use which is a key AI engineering skill.

## 2. ReAct — Reason + Act

### 2.1 The Pattern

ReAct (Yao et al., 2023) interleaves reasoning traces with tool calls. At each step:

```
Thought:  What do I need to do next?
Action:   Call a tool
Observation: The tool result
Thought:  What did I learn? What next?
Action:   Call another tool (or finish)
...
Final Answer: The synthesized response
```

The "Thought" step is internal reasoning that happens inside the model's extended output
before the tool call. Some implementations make this explicit (prompt the model to produce
a Thought before each tool call); modern frontier models do this implicitly.

### 2.2 Implementation

```python
import anthropic
import json

client = anthropic.Anthropic()

def react_agent(
    task: str,
    tools: list[dict],
    tool_executors: dict,
    max_steps: int = 10,
    verbose: bool = True,
) -> dict:
    """
    ReAct agent: Reason → Act → Observe at each step.
    The model drives the loop; it calls tools and reasons about results.
    """
    system = """You are a capable assistant that solves tasks step by step.

For each step:
1. Think about what information you need or what action to take
2. Use the available tools to gather information or take actions
3. Reason about what you learned and what to do next
4. Continue until you have enough information to give a complete, accurate answer

Be precise: only call tools when you actually need new information.
Don't call the same tool twice with the same arguments."""
    
    messages = [{"role": "user", "content": task}]
    step_trace = []
    
    for step in range(max_steps):
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=2048,
            system=system,
            tools=tools,
            messages=messages
        )
        
        # Extract any text reasoning the model produced before tool calls
        thinking = " ".join(
            b.text for b in response.content
            if hasattr(b, "text") and b.text
        )
        
        step_record = {
            "step":         step + 1,
            "stop_reason":  response.stop_reason,
            "thinking":     thinking[:200] if thinking else "",
            "tool_calls":   [],
        }
        
        if verbose and thinking:
            print(f"\n[Step {step+1}] Thought: {thinking[:150]}...")
        
        # Terminal: model finished reasoning and is responding
        if response.stop_reason == "end_turn":
            step_trace.append(step_record)
            return {
                "answer":      thinking,
                "steps":       len(step_trace),
                "trace":       step_trace,
                "total_steps": step + 1,
            }
        
        # Continue: model wants to call tools
        if response.stop_reason == "tool_use":
            messages.append({"role": "assistant", "content": response.content})
            
            tool_results = []
            for block in response.content:
                if block.type != "tool_use":
                    continue
                
                if verbose:
                    print(f"  → Tool: {block.name}({json.dumps(block.input)[:80]})")
                
                executor = tool_executors.get(block.name)
                try:
                    result = executor(**block.input) if executor else {"error": f"No executor for {block.name}"}
                except Exception as e:
                    result = {"error": str(e), "tool": block.name}
                
                if verbose:
                    print(f"  ← Result: {str(result)[:100]}")
                
                step_record["tool_calls"].append({
                    "tool":   block.name,
                    "input":  block.input,
                    "result": str(result)[:200]
                })
                
                tool_results.append({
                    "type":        "tool_result",
                    "tool_use_id": block.id,
                    "content":     json.dumps(result)
                })
            
            step_trace.append(step_record)
            messages.append({"role": "user", "content": tool_results})
    
    return {
        "answer":      "Max steps reached without finishing.",
        "steps":       max_steps,
        "trace":       step_trace,
        "total_steps": max_steps,
    }
```

### 2.3 ReAct with Reflexion (Production Extension)

Reflexion adds self-correction: when the agent fails, it reflects on what went wrong and
retries with a revised approach. This is the "ReAct + Reflexion" production stack.

```python
def reflexion_react_agent(
    task: str,
    tools: list[dict],
    tool_executors: dict,
    validate_fn,   # callable: (answer: str) -> tuple[bool, str]  # (passed, feedback)
    max_attempts: int = 3,
) -> dict:
    """
    ReAct with Reflexion: retry with self-critique on failure.
    """
    reflection_context = ""
    
    for attempt in range(max_attempts):
        # Build task with any reflection from prior attempts
        task_with_reflection = task
        if reflection_context:
            task_with_reflection += f"\n\nIMPORTANT: Previous attempt failed. Reflection:\n{reflection_context}"
        
        result = react_agent(task_with_reflection, tools, tool_executors)
        
        # Check if the answer is good
        passed, feedback = validate_fn(result["answer"])
        
        if passed:
            return {**result, "attempts": attempt + 1, "final": True}
        
        # Generate reflection on failure
        reflection_response = client.messages.create(
            model="claude-haiku-4-5-20251001",
            max_tokens=256,
            messages=[{
                "role": "user",
                "content": f"""Task: {task}
My attempt: {result['answer']}
Feedback: {feedback}

In 2-3 sentences, what went wrong and how should I approach this differently?"""
            }]
        )
        
        reflection_context = reflection_response.content[0].text
        print(f"\n[Attempt {attempt+1} failed] Reflection: {reflection_context[:100]}...")
    
    return {**result, "attempts": max_attempts, "final": False}
```

## 3. Plan-and-Execute

### 3.1 The Pattern

Plan-and-Execute separates reasoning into two phases:

```
PLANNING PHASE (strategic — uses a capable planner model):
  "Here is the task. Generate a step-by-step plan."
  → Plan: [Step 1, Step 2, Step 3, ...]

EXECUTION PHASE (tactical — can use a cheaper executor):
  For each step in the plan:
    → Execute the step (may involve tool calls)
    → If execution fails, trigger replanning
  → Synthesize final answer from step results
```

**When Plan-and-Execute outperforms ReAct:**
- Complex, multi-step tasks where all steps are known upfront
- Tasks where you can use a cheap/fast model for execution
- When you need the plan to be reviewable/auditable before execution
- When individual steps can be executed in parallel

```python
def plan_and_execute_agent(
    task: str,
    tools: list[dict],
    tool_executors: dict,
    planner_model: str = "claude-sonnet-4-6",    # strategic
    executor_model: str = "claude-haiku-4-5-20251001",  # cheaper for individual steps
) -> dict:
    """
    Plan-and-Execute agent: plan first, then execute each step.
    Uses cheaper executor model for individual steps.
    """
    
    # PHASE 1: Plan
    plan_response = client.messages.create(
        model=planner_model,
        max_tokens=1024,
        system="Create a clear, step-by-step plan to complete the task. "
               "Each step should be a single, concrete action. "
               "Output as a numbered list. Be specific about what information to gather or verify.",
        messages=[{"role": "user", "content": f"Task: {task}"}]
    )
    
    plan_text = plan_response.content[0].text
    steps = [
        line.strip().lstrip("0123456789.)- ")
        for line in plan_text.split("\n")
        if line.strip() and line.strip()[0].isdigit()
    ]
    
    print(f"Plan ({len(steps)} steps):")
    for i, step in enumerate(steps, 1):
        print(f"  {i}. {step}")
    
    # PHASE 2: Execute each step
    step_results = []
    
    for i, step in enumerate(steps, 1):
        print(f"\nExecuting step {i}: {step[:80]}...")
        
        # Build context from prior step results
        prior_context = ""
        if step_results:
            prior_context = "Results from previous steps:\n" + "\n".join(
                f"Step {j+1}: {r['result'][:200]}" for j, r in enumerate(step_results)
            )
        
        # Execute this step (may involve tool calls)
        step_response = client.messages.create(
            model=executor_model,
            max_tokens=1024,
            system=f"Execute this specific step as part of a larger task.\n\n{prior_context}",
            tools=tools,
            messages=[{
                "role": "user",
                "content": f"Task step: {step}\nOriginal task context: {task[:200]}"
            }]
        )
        
        # Handle tool calls within this step
        step_messages = [{"role": "user", "content": f"Task step: {step}"}]
        
        while step_response.stop_reason == "tool_use":
            step_messages.append({"role": "assistant", "content": step_response.content})
            tool_results = []
            
            for block in step_response.content:
                if block.type == "tool_use":
                    executor = tool_executors.get(block.name)
                    try:
                        result = executor(**block.input) if executor else {"error": "Unknown tool"}
                    except Exception as e:
                        result = {"error": str(e)}
                    
                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": json.dumps(result)
                    })
            
            step_messages.append({"role": "user", "content": tool_results})
            step_response = client.messages.create(
                model=executor_model, max_tokens=1024, tools=tools, messages=step_messages
            )
        
        step_result = next(
            (b.text for b in step_response.content if hasattr(b, "text")), ""
        )
        step_results.append({"step": step, "result": step_result})
        print(f"  ✓ {step_result[:100]}")
    
    # PHASE 3: Synthesize
    synthesis = client.messages.create(
        model=planner_model,
        max_tokens=2048,
        system="Synthesize these step results into a complete, coherent answer to the original task.",
        messages=[{
            "role": "user",
            "content": f"Task: {task}\n\nStep results:\n" + "\n".join(
                f"Step {i+1} ({r['step']}): {r['result']}"
                for i, r in enumerate(step_results)
            )
        }]
    )
    
    return {
        "answer":       synthesis.content[0].text,
        "plan":         steps,
        "step_results": step_results,
    }
```

## 4. Comparing the Patterns

| Aspect | ReAct | Plan-and-Execute |
|---|---|---|
| **Planning** | Implicit — emerges from reasoning | Explicit — plan is generated first |
| **Adaptability** | High — responds to each observation | Lower — plan is fixed unless replanning triggered |
| **Predictability** | Lower — hard to know what steps it will take | Higher — you can review the plan before execution |
| **Cost** | One model for everything | Can use cheap model for execution |
| **Parallelism** | Sequential steps | Steps can be parallelized |
| **Debugging** | Trace each observation | Inspect the plan; trace step failures |
| **Best for** | Complex reasoning where steps depend on prior results | Tasks with known structure; parallelizable steps |

**The 2026 consensus:** ReAct is the right default for most single-agent tasks.
Upgrade to Plan-and-Execute when:
- Tasks have predictable structure (you could enumerate the steps in advance)
- You want cost savings by using smaller executor models
- Individual steps can run in parallel (Plan-and-Execute + asyncio)
- The plan needs to be auditable/explainable

## 5. Real-World Context

### When to Use Which Pattern

| Task Type | Pattern | Why |
|---|---|---|
| Multi-hop Q&A | ReAct | Steps depend on prior answers |
| Structured data extraction | Plan-and-Execute | Steps known upfront, parallelizable |
| Research synthesis | ReAct | Adaptive: depends on what's found |
| Report generation | Plan-and-Execute | Clear structure: gather → process → write |
| Debugging a system | ReAct | Unknown steps until you see the error |
| Batch data processing | Plan-and-Execute | Same steps for each item |

### The Performance Numbers (2026)

From Digital Applied's 2026 agent architecture survey:
- Plan-and-Execute achieved **92% task completion** with **3.6× speedup** over ReAct
  on structured, multi-step tasks (because of cheap executor model + parallelism)
- ReAct outperformed Plan-and-Execute on **adaptive tasks** where early results changed
  the path forward

## 6. Code Examples

### ReAct for Research

```python
# A practical ReAct agent for research tasks

research_tools = [
    {
        "name": "search_web",
        "description": "Search the web for current information on any topic. Returns a list of relevant snippets.",
        "input_schema": {
            "type": "object",
            "properties": {
                "query": {"type": "string", "description": "Search query. Be specific."},
                "max_results": {"type": "integer", "default": 5}
            },
            "required": ["query"]
        }
    },
    {
        "name": "get_page_content",
        "description": "Fetch and extract text content from a specific URL.",
        "input_schema": {
            "type": "object",
            "properties": {
                "url": {"type": "string", "description": "The URL to fetch"}
            },
            "required": ["url"]
        }
    },
    {
        "name": "calculate",
        "description": "Evaluate a mathematical expression. Use for all arithmetic, percentages, conversions.",
        "input_schema": {
            "type": "object",
            "properties": {
                "expression": {
                    "type": "string",
                    "description": "Math expression, e.g. '(150 * 0.08) + 150' or '12.5 / 3.14159'"
                }
            },
            "required": ["expression"]
        }
    }
]

def run_research_agent(question: str) -> str:
    result = react_agent(
        task=question,
        tools=research_tools,
        tool_executors={
            "search_web":       lambda query, max_results=5: {"results": [f"Search result {i} for {query}" for i in range(max_results)]},  # stub
            "get_page_content": lambda url: {"content": f"Content from {url}"},   # stub
            "calculate":        lambda expression: {"result": eval(expression, {"__builtins__": {}})},
        },
        max_steps=8,
        verbose=True,
    )
    return result["answer"]

answer = run_research_agent(
    "What is 15% of the average of 1250, 1380, and 1175?"
)
print(f"\nFinal Answer: {answer}")
```

## 7. When To Use vs. When To Avoid

### ReAct
✅ Use as the default for any single-agent task
✅ Use when each step depends on the previous step's result
✅ Use when the task structure is unknown upfront
❌ Upgrade to Plan-and-Execute when tasks are predictably structured

### Plan-and-Execute
✅ Use when you want cost savings from cheap executor models
✅ Use when individual task steps can run in parallel
✅ Use when the plan needs human review before execution
❌ Avoid when early steps frequently invalidate later steps (high replanning cost)

⚠️ Common mistakes:
- Not setting max_steps — agents loop forever
- Using expensive models for every execution step — use Haiku/gpt-4o-mini for simple steps
- Not handling tool execution errors — always catch and return {"error": "..."} to the model
- Not logging the full trace — you cannot debug agent failures without the step-by-step history

## 8. Senior AI Engineer Perspective

### Build ReAct First, Then Measure

Every team building agents should start with ReAct. It's simpler to implement and debug.
Before moving to Plan-and-Execute:
1. Measure: what's the failure rate on your specific tasks?
2. Categorize: is the failure from wrong reasoning or missing capabilities?
3. Profile: what fraction of tasks have fixed structure that could be pre-planned?
4. Only if #3 is high (>60%): consider Plan-and-Execute

The 3.6× speedup from Plan-and-Execute is real — but only on tasks it's suited for.
On adaptive tasks, it's often slower (because replanning adds overhead).

### The Trace Is Your Debugger

For any production agent, persist the full execution trace:
- Which tools were called, with what arguments
- What each tool returned
- What the model decided after each observation
- How many steps it took

When an agent fails in production, the trace is often the only way to understand why.
Without it, you're debugging a black box.
