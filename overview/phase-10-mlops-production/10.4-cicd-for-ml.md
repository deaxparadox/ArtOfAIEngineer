## 1. What Is It? (Theory)

### The ML CI/CD Problem

Traditional software CI/CD is well-understood: commit → build → test → deploy.
Each step has clear pass/fail criteria.

ML CI/CD has the same structure but harder gate criteria:
- **Build:** Does the training pipeline run without errors?
- **Test:** Does the new model pass quality thresholds on the eval dataset?
- **Deploy:** Does the model serve correctly in staging?
- **Release:** Is quality better than the current production version?

The challenge: ML tests are statistical, not deterministic. "Does this model pass?" has
a probabilistic answer, not a binary one. The pipeline must account for this.

### The Four ML CI/CD Gates

```
GATE 1: Data Validation
  Before training starts, validate that data is:
  - Complete (expected row count, column count)
  - Not corrupted (checksums, format validation)
  - Not significantly drifted from previous version
  Fail fast here — expensive training shouldn't start on bad data.

GATE 2: Model Validation
  After training, before staging:
  - Eval score above minimum threshold
  - No regression from baseline (score not significantly worse)
  - Safety checks pass (no harmful outputs on adversarial test set)
  - Format/output validation (correct output schema)

GATE 3: Staging Integration Test
  After staging deployment:
  - Load test (can it handle expected traffic?)
  - Integration test (works with all downstream services?)
  - Shadow traffic comparison (similar quality to production?)

GATE 4: Canary Release
  After production deployment begins:
  - Monitor error rate, latency, quality for 15–30 minutes
  - Auto-rollback if metrics degrade beyond threshold
  - Gradually increase traffic if stable
```

## 2. How It Actually Works — Deep Mechanics

### 2.1 GitHub Actions Pipeline for LLM Features

```yaml
# .github/workflows/llm-cicd.yml
name: LLM Feature CI/CD

on:
  push:
    branches: [main]
    paths:
      - 'prompts/**'
      - 'src/ai/**'
      - 'fine_tuning/**'
  pull_request:
    paths:
      - 'prompts/**'
      - 'src/ai/**'

env:
  ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
  LANGFUSE_SECRET_KEY: ${{ secrets.LANGFUSE_SECRET_KEY }}

jobs:
  # ── GATE 1: Fast checks (run on every PR) ──────────────────────────
  fast-checks:
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: {python-version: '3.11'}
      - run: pip install -r requirements-dev.txt
      
      # Lint prompts
      - name: Validate prompt files
        run: python scripts/validate_prompts.py
      
      # Schema validation
      - name: Validate output schemas
        run: python -m pytest tests/test_schemas.py -x -q
      
      # Canary cases (fastest eval subset — 10 cases)
      - name: Canary eval (blocking)
        run: |
          python -m pytest tests/eval/test_canary.py -v \
            --model=claude-sonnet-4-6 \
            --timeout=60
        # If canaries fail, don't even run the full suite

  # ── GATE 2: Full eval suite (run on merge to main) ─────────────────
  full-eval:
    runs-on: ubuntu-latest
    needs: fast-checks
    if: github.ref == 'refs/heads/main'
    timeout-minutes: 30
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: {python-version: '3.11'}
      - run: pip install -r requirements-dev.txt

      - name: Run full regression suite
        id: eval
        run: |
          python scripts/run_eval.py \
            --model=claude-sonnet-4-6 \
            --dataset=evals/golden_dataset.json \
            --baseline=.eval_baselines/latest.json \
            --output=eval_results/summary.json \
            --threshold=0.75 \
            --max-regression=0.1

      - name: Upload eval results
        uses: actions/upload-artifact@v4
        with:
          name: eval-results
          path: eval_results/

      - name: Check eval gate
        run: |
          python scripts/check_eval_gate.py eval_results/summary.json
          # Fails the job if quality below threshold or regression detected

      - name: Update baseline if improved
        if: success()
        run: |
          python scripts/update_baseline.py \
            --results=eval_results/summary.json \
            --baseline=.eval_baselines/latest.json
          git config user.email "ci@company.com"
          git config user.name "CI Bot"
          git add .eval_baselines/
          git diff --staged --quiet || git commit -m "ci: update eval baseline [skip ci]"
          git push

  # ── GATE 3: Staging deployment ─────────────────────────────────────
  deploy-staging:
    runs-on: ubuntu-latest
    needs: full-eval
    environment: staging
    steps:
      - uses: actions/checkout@v4
      
      - name: Deploy to staging
        run: |
          # Deploy the new prompt/model config to staging environment
          python scripts/deploy.py --env=staging --version=${{ github.sha }}
      
      - name: Run staging integration tests
        run: |
          python -m pytest tests/integration/ -v \
            --base-url=https://staging.api.company.com \
            --timeout=120
      
      - name: Shadow traffic test (compare staging vs production)
        run: |
          python scripts/shadow_compare.py \
            --staging-url=https://staging.api.company.com \
            --prod-url=https://api.company.com \
            --n-requests=100

  # ── GATE 4: Production canary ───────────────────────────────────────
  deploy-production:
    runs-on: ubuntu-latest
    needs: deploy-staging
    environment: production   # requires manual approval
    steps:
      - uses: actions/checkout@v4
      
      - name: Canary deployment (5% traffic)
        run: |
          python scripts/deploy.py \
            --env=production \
            --version=${{ github.sha }} \
            --traffic-pct=5
      
      - name: Monitor canary (15 minutes)
        run: |
          python scripts/monitor_canary.py \
            --duration=900 \
            --error-threshold=0.02 \
            --quality-threshold=3.5 \
            --rollback-on-failure
      
      - name: Full production rollout
        if: success()
        run: |
          python scripts/deploy.py \
            --env=production \
            --version=${{ github.sha }} \
            --traffic-pct=100
```

### 2.2 Deployment Gate Script

```python
# scripts/check_eval_gate.py
import json
import sys

def check_eval_gate(results_path: str) -> bool:
    """
    Hard gates that block deployment.
    Returns True if deployment should proceed.
    """
    with open(results_path) as f:
        results = json.load(f)
    
    failures = []
    
    # Gate 1: Minimum pass rate
    if results["det_pass_rate"] < 0.75:
        failures.append(
            f"Pass rate {results['det_pass_rate']:.0%} below minimum 75%"
        )
    
    # Gate 2: No canary failures
    canary_failures = [r for r in results.get("canary_results", []) if not r["passed"]]
    if canary_failures:
        failures.append(
            f"CANARY FAILURE: {len(canary_failures)} canary cases failed — BLOCKING DEPLOYMENT"
        )
    
    # Gate 3: No statistical regression from baseline
    regression = results.get("regression")
    if regression and regression.get("regression_detected"):
        failures.append(f"REGRESSION: {regression['message']}")
    
    # Gate 4: Mean quality score above floor
    if results.get("mean_judge_score", 5) < 3.0:
        failures.append(
            f"Quality score {results['mean_judge_score']:.2f} below minimum 3.0"
        )
    
    if failures:
        print("❌ DEPLOYMENT BLOCKED:")
        for f in failures:
            print(f"  - {f}")
        return False
    
    print("✅ All eval gates passed. Deployment approved.")
    print(f"  Pass rate:      {results['det_pass_rate']:.0%}")
    print(f"  Quality score:  {results.get('mean_judge_score', 'N/A')}")
    return True

if __name__ == "__main__":
    results_path = sys.argv[1]
    success = check_eval_gate(results_path)
    sys.exit(0 if success else 1)
```

### 2.3 Canary Monitoring Script

```python
# scripts/monitor_canary.py
import time
import requests
import sys
import statistics
from datetime import datetime

def monitor_canary(
    duration_seconds: int = 900,  # 15 minutes
    error_threshold: float = 0.02,
    quality_threshold: float = 3.5,
    check_interval: int = 30,
    auto_rollback: bool = True,
) -> bool:
    """
    Monitor a canary deployment and trigger rollback if metrics degrade.
    Returns True if canary is healthy after full duration.
    """
    print(f"Monitoring canary for {duration_seconds}s...")
    start_time = time.time()
    error_counts  = []
    quality_scores = []
    
    while time.time() - start_time < duration_seconds:
        # Check real-time metrics from observability platform
        metrics = get_current_metrics()
        
        error_rate    = metrics.get("error_rate_1min", 0)
        quality_score = metrics.get("rolling_quality_score_1min", 5.0)
        
        error_counts.append(error_rate)
        quality_scores.append(quality_score)
        
        elapsed = int(time.time() - start_time)
        print(f"  [{elapsed:4d}s] error_rate={error_rate:.2%} quality={quality_score:.2f}")
        
        # Check thresholds
        if error_rate > error_threshold:
            print(f"⚠️  Error rate {error_rate:.2%} exceeds threshold {error_threshold:.2%}")
            if auto_rollback:
                rollback()
                return False
        
        if quality_score < quality_threshold:
            print(f"⚠️  Quality score {quality_score:.2f} below threshold {quality_threshold:.2f}")
            if auto_rollback:
                rollback()
                return False
        
        time.sleep(check_interval)
    
    # Final assessment
    avg_error_rate    = statistics.mean(error_counts)
    avg_quality_score = statistics.mean(quality_scores)
    
    print(f"\nCanary monitoring complete:")
    print(f"  Avg error rate:    {avg_error_rate:.2%}")
    print(f"  Avg quality score: {avg_quality_score:.2f}")
    
    return avg_error_rate <= error_threshold and avg_quality_score >= quality_threshold
```

## 3. Real-World Context

### The Netflix Scale (Aspirational Benchmark)

Netflix pushes 300 model updates daily across their recommendation infrastructure, with
each deployment automatically validated, tested, and monitored without human intervention.

For most teams, the realistic target is:
- Every PR triggers fast eval checks (10 cases, <5 min)
- Every merge to main triggers full eval (100 cases, <15 min)
- Human approval required for production deployments
- Automated canary monitoring with auto-rollback

Even at this simpler level, CI/CD significantly reduces the time from "prompt changed"
to "confident deployment" from days to hours.

### The Deployment Environment Hierarchy

```
Development:
  - Local testing; no eval required
  - Fast iteration; quick feedback

Staging (pre-production):
  - Full eval suite must pass
  - Integration tests must pass
  - Mirrors production config exactly
  - Shadow traffic comparison optional

Production:
  - Canary deployment (5–10%)
  - Monitored for 15–30 minutes before full rollout
  - Automatic rollback capability required
  - Human approval gate (in most orgs)
```

## 5. Trade-offs

### Fast Eval vs Full Eval

| | Fast eval (canaries only) | Full eval suite |
|---|---|---|
| Duration | 2–5 minutes | 10–30 minutes |
| Blocking? | Yes (every PR) | Yes (merge to main) |
| Coverage | Critical paths only | Full distribution |
| False positive rate | Low | Low |
| Miss rate | Higher | Lower |

Run both: fast on every PR, full on every merge. The two-tier approach gives fast feedback
without blocking CI while maintaining thorough regression detection for production.

## 6. Senior AI Engineer Perspective

### The Eval Gate Threshold Is a Business Decision

Setting the threshold at 0.75 pass rate vs 0.80 is not purely a technical decision.
It's a tradeoff between:
- **Higher threshold:** Fewer quality regressions reach production; more deployments blocked
- **Lower threshold:** Faster iteration; more quality issues reach users

Involve product and business stakeholders in setting the threshold. "What's the acceptable
quality degradation vs deployment velocity tradeoff?" is a business question, not an
engineering one. Engineers can quantify the options; stakeholders choose.

### Test the Rollback Procedure in Staging Monthly

A rollback that hasn't been tested is a rollback that won't work when you need it.
At least monthly, deploy a known-bad version to staging and verify:
1. The canary monitoring detects the degradation
2. The auto-rollback fires correctly
3. The stable version resumes serving

Do this as a fire drill. The first time you test rollback should not be during a production incident.
