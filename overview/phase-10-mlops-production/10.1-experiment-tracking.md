## 1. What Is It? (Theory)

### The Experiment Graveyard Problem

Every AI team runs experiments. Most experiments leave behind only muscle memory and Slack messages:
- "We tried X and it didn't work" — but what exactly was X? With what parameters?
- "Version 3 was best" — which version is that? What was in it?
- "The one with the higher learning rate" — compared to which baseline?

Without experiment tracking, your team rediscovers the same failures repeatedly. Progress is
measured in weeks, not days. And when you need to reproduce a result from 3 months ago —
the one that's now in production — you can't.

**Experiment tracking** records every run's parameters, metrics, code version, and artifacts.
It turns your experiments from one-time events into a searchable, reproducible history.

### What to Track

```
PARAMETERS (inputs to the experiment):
  - Model name and version (e.g., claude-sonnet-4-6)
  - Hyperparameters (learning rate, batch size, num_epochs)
  - Prompt version / system prompt hash
  - Embedding model and chunk size (for RAG)
  - Dataset version

METRICS (outputs of the experiment):
  - Eval scores (faithfulness, context recall, answer relevancy)
  - Pass rates by category
  - Latency (p50, p95)
  - Cost per request
  - Loss curves (for training runs)

ARTIFACTS (files produced):
  - Model weights or adapter
  - Fine-tuned model checkpoint
  - Generated predictions/outputs
  - Confusion matrices, plots

METADATA:
  - Git commit hash
  - Environment (Python version, library versions)
  - Timestamp and duration
  - Who ran it and why
```

## 2. MLflow 3.0 (2025) — The Open-Source Choice

MLflow released version 3.0 in June 2025, pivoting to become a unified AI engineering platform
that covers both classical ML and GenAI/LLM workflows. It's now the most comprehensive
open-source ML lifecycle platform.

```python
# pip install mlflow anthropic

import mlflow
import mlflow.anthropic  # GenAI integration (MLflow 3.0+)
import anthropic
import json
from datetime import datetime

# Configure MLflow tracking server (or use local default)
mlflow.set_tracking_uri("http://localhost:5000")   # or "sqlite:///mlflow.db" for local
mlflow.set_experiment("rag-pipeline-optimization")

client = anthropic.Anthropic()

def run_rag_experiment(
    chunk_size: int,
    top_k: int,
    embedding_model: str,
    eval_cases: list[dict],
    experiment_name: str = ""
) -> dict:
    """Run a RAG experiment and track everything with MLflow."""
    
    with mlflow.start_run(run_name=experiment_name or f"rag_cs{chunk_size}_k{top_k}"):
        # Log parameters
        mlflow.log_params({
            "chunk_size":      chunk_size,
            "chunk_overlap":   chunk_size // 8,
            "top_k":           top_k,
            "embedding_model": embedding_model,
            "llm_model":       "claude-sonnet-4-6",
            "n_eval_cases":    len(eval_cases),
        })
        
        # Also log environment metadata
        mlflow.set_tags({
            "git_commit": get_git_commit(),
            "run_by":     "ai_team",
            "purpose":    "chunk_size_ablation",
        })
        
        # Run evaluation
        scores = []
        costs  = []
        
        for case in eval_cases:
            # Build RAG pipeline with current params
            chunks   = retrieve(case["question"], top_k=top_k)
            response = client.messages.create(
                model="claude-sonnet-4-6", max_tokens=512,
                messages=[{"role": "user", "content": f"Context:\n{build_context(chunks)}\n\nQ: {case['question']}"}]
            )
            
            answer   = response.content[0].text
            score    = judge_answer(case["question"], answer, case["reference"])
            cost     = (response.usage.input_tokens * 3 + response.usage.output_tokens * 15) / 1_000_000
            
            scores.append(score)
            costs.append(cost)
        
        # Log metrics
        mlflow.log_metrics({
            "mean_quality_score": sum(scores) / len(scores),
            "min_quality_score":  min(scores),
            "pass_rate":          sum(1 for s in scores if s >= 3.5) / len(scores),
            "mean_cost_usd":      sum(costs) / len(costs),
            "total_cost_usd":     sum(costs),
        })
        
        # Log artifacts: save detailed results as JSON
        results = {
            "params": {"chunk_size": chunk_size, "top_k": top_k},
            "scores": scores,
            "costs":  costs,
            "cases":  [{"question": c["question"], "score": s} for c, s in zip(eval_cases, scores)]
        }
        with open("results.json", "w") as f:
            json.dump(results, f, indent=2)
        mlflow.log_artifact("results.json")
        
        mean_score = sum(scores) / len(scores)
        print(f"  chunk_size={chunk_size}, top_k={top_k}: score={mean_score:.3f}")
        return {"mean_score": mean_score, "pass_rate": sum(1 for s in scores if s >= 3.5) / len(scores)}

# Helper stubs
def get_git_commit() -> str:
    import subprocess
    try:
        return subprocess.check_output(["git", "rev-parse", "--short", "HEAD"]).decode().strip()
    except Exception:
        return "unknown"

def retrieve(question: str, top_k: int) -> list[dict]:
    return [{"text": f"Sample chunk {i} for {question[:30]}"} for i in range(top_k)]

def build_context(chunks: list[dict]) -> str:
    return "\n".join(c["text"] for c in chunks)

def judge_answer(question: str, answer: str, reference: str) -> float:
    return 4.0  # stub; in practice: call LLM judge

# Run ablation study
print("Running chunk size ablation...")
for chunk_size in [256, 512, 1024]:
    for top_k in [3, 5, 10]:
        run_rag_experiment(
            chunk_size=chunk_size,
            top_k=top_k,
            embedding_model="voyage-3-large",
            eval_cases=[{"question": f"Test question {i}", "reference": f"Answer {i}"} for i in range(5)],
            experiment_name=f"cs{chunk_size}_k{top_k}"
        )

print("\nView results: mlflow ui --port 5000")
```

## 3. Weights & Biases (W&B) — The Visualization Choice

W&B is preferred for teams that want best-in-class experiment visualization and collaboration.
Acquired by CoreWeave in March 2025, it's positioned as a key AI training infrastructure platform.

```python
# pip install wandb

import wandb
import numpy as np

# Initialize a run
run = wandb.init(
    project="llm-optimization",
    name="rag-chunk-ablation",
    config={
        "chunk_size":      512,
        "top_k":           5,
        "embedding_model": "voyage-3-large",
        "llm_model":       "claude-sonnet-4-6",
    }
)

# Log metrics during training / evaluation
scores_over_time = [3.2, 3.5, 3.7, 3.8, 3.9, 4.0, 4.1]
for step, score in enumerate(scores_over_time):
    wandb.log({
        "quality_score": score,
        "cost_per_query": 0.005 - step * 0.0003,   # improving cost efficiency
        "step": step
    })

# Log a summary table
table = wandb.Table(
    columns=["question", "answer", "score"],
    data=[
        ["What is the return policy?", "30 days for unopened items.", 4.5],
        ["Can I return a sale item?",  "Sale items are final.",      4.8],
    ]
)
wandb.log({"eval_results": table})

# Log artifacts (model outputs, plots, etc.)
wandb.log_artifact("model_outputs.json", name="eval-outputs", type="dataset")

run.finish()

# W&B Sweeps: hyperparameter optimization
sweep_config = {
    "method": "bayes",    # bayesian optimization
    "metric": {"name": "quality_score", "goal": "maximize"},
    "parameters": {
        "chunk_size": {"values": [256, 512, 1024]},
        "top_k":      {"values": [3, 5, 10]},
        "temperature":{"min": 0.0, "max": 0.5},
    }
}

def sweep_train():
    run = wandb.init()
    config = wandb.config
    
    result = run_rag_experiment(
        chunk_size=config.chunk_size,
        top_k=config.top_k,
        embedding_model="voyage-3-large",
        eval_cases=[...]
    )
    
    wandb.log({"quality_score": result["mean_score"]})
    run.finish()

sweep_id = wandb.sweep(sweep_config, project="rag-optimization")
wandb.agent(sweep_id, function=sweep_train, count=20)
```

## 4. LLM-Specific Tracking: Prompt Versioning

For LLM applications, tracking the prompt is as important as tracking model parameters:

```python
import hashlib
import json
from dataclasses import dataclass
from datetime import datetime

@dataclass
class PromptVersion:
    version_id: str
    content:    str
    created_at: str
    author:     str
    change_note: str
    eval_score:  float = 0.0
    in_production: bool = False

class PromptRegistry:
    """
    Lightweight prompt version tracking.
    For production: use Langfuse prompts, PromptLayer, or Braintrust.
    """
    
    def __init__(self, path: str = "prompt_registry.json"):
        self.path = path
        self.versions: dict[str, PromptVersion] = self._load()
    
    def _load(self) -> dict:
        try:
            with open(self.path) as f:
                data = json.load(f)
            return {k: PromptVersion(**v) for k, v in data.items()}
        except FileNotFoundError:
            return {}
    
    def register(self, content: str, author: str, change_note: str) -> str:
        """Register a new prompt version. Returns version_id."""
        version_id = hashlib.md5(content.encode()).hexdigest()[:8]
        
        self.versions[version_id] = PromptVersion(
            version_id  = version_id,
            content     = content,
            created_at  = datetime.utcnow().isoformat(),
            author      = author,
            change_note = change_note
        )
        
        with open(self.path, "w") as f:
            json.dump({k: vars(v) for k, v in self.versions.items()}, f, indent=2)
        
        return version_id
    
    def promote(self, version_id: str, eval_score: float):
        """Mark a version as production-ready after eval passes."""
        if version_id not in self.versions:
            raise ValueError(f"Unknown version: {version_id}")
        
        # Demote current production version
        for v in self.versions.values():
            v.in_production = False
        
        self.versions[version_id].eval_score = eval_score
        self.versions[version_id].in_production = True
        
        with open(self.path, "w") as f:
            json.dump({k: vars(v) for k, v in self.versions.items()}, f, indent=2)
        
        print(f"Promoted {version_id} to production (score: {eval_score:.2f})")
    
    def current(self) -> PromptVersion | None:
        """Return the current production prompt."""
        return next((v for v in self.versions.values() if v.in_production), None)

# Usage
registry = PromptRegistry()
v1 = registry.register(
    content="You are a helpful assistant. Answer concisely.",
    author="team",
    change_note="Initial prompt"
)
v2 = registry.register(
    content="You are a helpful assistant. Answer concisely. Always cite sources.",
    author="team",
    change_note="Added citation requirement based on eval feedback"
)

# After running eval on v2:
registry.promote(v2, eval_score=4.2)
```

## 7. When To Use vs. When To Avoid

### Experiment Tracking
✅ Always: Any iterative development where you're comparing options
✅ Always: Fine-tuning runs — the most expensive experiments you'll run
✅ Always: When multiple team members are running experiments
❌ Can skip: One-off analysis or throwaway prototypes

⚠️ Common mistakes:
- Not logging the git commit hash — you can't reproduce without knowing the code state
- Logging too many metrics (noise) — pick the 3–5 metrics that actually matter
- Not logging the dataset version — parameters can be identical but data can differ
- Running experiments without a baseline — you can't measure improvement without one

## 8. Senior AI Engineer Perspective

### The "Best Run" Is Worthless Without Reproducibility

Finding the best hyperparameter combination is only valuable if you can reproduce it. The combination
of: git commit + dataset version + parameter set + environment = reproducible result.
Tracking any of these without the others is insufficient.

The minimum reproducibility contract: if I give you a run ID, you can reproduce the result
within ±2% on the key metric. If you can't guarantee this, your tracking is incomplete.

### Track the Negative Results Too

The most valuable experiments are the ones that didn't work and explain why. A note saying
"tried chunk_size=2048: quality degraded significantly on short queries, embeddings too diffuse"
saves the next engineer 2 days of work. Log it.
