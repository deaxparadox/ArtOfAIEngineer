## 1. What Is It? (Theory)

### Why AI Features Need A/B Testing

You've made a change: new prompt, new model, new chunking strategy, new retrieval config.
Your evals say it's better. But:
- Evals are offline; users are live
- Evals test a sample; users generate the full distribution
- A higher eval score doesn't always translate to better user outcomes

**A/B testing** validates the improvement in production, with real users, measured on business
metrics that actually matter — not just eval scores.

### The A/B Testing → AI Complexity

Standard A/B testing is well-understood for deterministic systems (button colors, page layouts).
For AI systems, three complications arise:

1. **Non-determinism:** The same user on the same AI variant gets different responses each
   time. The "treatment" is a distribution, not a fixed value.

2. **Novelty effect:** Users might prefer a new AI system simply because it's different,
   not because it's better. This fades after a week.

3. **Quality vs preference divergence:** Users may prefer the AI that agrees with them
   (sycophantic) over the one that gives the correct but unwelcome answer.

These complications don't make A/B testing impossible — they make it require more careful design.

## 2. How It Actually Works — Deep Mechanics

### 2.1 Randomization and Assignment

```python
import hashlib
from dataclasses import dataclass

@dataclass
class ExperimentConfig:
    experiment_id:   str
    control_name:    str   # "A" or descriptive name
    treatment_name:  str   # "B" or descriptive name
    treatment_pct:   float # fraction of users to assign to treatment (e.g., 0.5)
    
    # Metrics to measure
    primary_metric:    str   # the ONE metric this experiment is designed to move
    guardrail_metrics: list[str]  # must not degrade

def assign_variant(user_id: str, experiment: ExperimentConfig) -> str:
    """
    Deterministic, consistent user assignment.
    Same user always gets the same variant within an experiment.
    Uses hashing for even distribution without a database.
    """
    hash_input = f"{experiment.experiment_id}:{user_id}"
    hash_value = int(hashlib.md5(hash_input.encode()).hexdigest(), 16)
    
    # Convert hash to 0–1 range
    bucket = (hash_value % 10000) / 10000.0
    
    if bucket < experiment.treatment_pct:
        return experiment.treatment_name
    return experiment.control_name


# Example: test whether a new system prompt improves task completion rate
experiment = ExperimentConfig(
    experiment_id    = "prompt_v3_test_20260611",
    control_name     = "control",
    treatment_name   = "treatment",
    treatment_pct    = 0.5,     # 50/50 split
    primary_metric   = "task_completion_rate",
    guardrail_metrics= ["helpfulness_score", "error_rate", "latency_p95"]
)

CONFIGS = {
    "control": {
        "model": "claude-sonnet-4-6",
        "system": "You are a helpful customer support assistant.",
    },
    "treatment": {
        "model": "claude-sonnet-4-6",
        "system": "You are a helpful customer support assistant. Always cite sources with [Source N].",
    },
}

def handle_request(user_id: str, question: str) -> dict:
    variant = assign_variant(user_id, experiment)
    config  = CONFIGS[variant]
    
    # ... call the model ...
    
    return {
        "answer":    "...",
        "variant":   variant,
        "exp_id":    experiment.experiment_id,
    }
```

### 2.2 Collecting Metrics

For each request, collect the signals needed to evaluate the experiment:

```python
from dataclasses import dataclass
from datetime import datetime
from typing import Optional

@dataclass
class ExperimentObservation:
    """One observation in the A/B test."""
    
    # Identity
    experiment_id: str
    user_id:       str
    session_id:    str
    variant:       str        # "control" | "treatment"
    timestamp:     str
    
    # Quality metrics (from LLM judge, sampled)
    quality_score:        Optional[float] = None    # 1–5 from judge
    faithfulness_score:   Optional[float] = None    # 0–1
    
    # User behavior signals (from frontend)
    thumbs_up:            Optional[bool]  = None    # explicit feedback
    thumbs_down:          Optional[bool]  = None
    follow_up_question:   Optional[bool]  = None    # did user ask follow-up? (signal of incomplete answer)
    session_ended_after:  Optional[bool]  = None    # did user end the session? (proxy for completion)
    copied_response:      Optional[bool]  = None    # did user copy the answer? (positive signal)
    
    # Business metrics
    task_completed:       Optional[bool]  = None    # did the AI help complete the task?
    conversion_happened:  Optional[bool]  = None    # downstream business outcome
    
    # Guardrails
    latency_ms:           float = 0.0
    cost_usd:             float = 0.0
    is_error:             bool = False

# In production: log every observation to a database or analytics warehouse
def log_observation(obs: ExperimentObservation, db):
    db.insert("experiment_observations", vars(obs))
```

### 2.3 Statistical Analysis

```python
import numpy as np
from scipy import stats
from dataclasses import dataclass

@dataclass
class ExperimentResults:
    control_n:        int
    treatment_n:      int
    control_mean:     float
    treatment_mean:   float
    delta:            float      # treatment - control
    delta_pct:        float      # relative change
    p_value:          float
    ci_lower:         float      # 95% confidence interval lower bound
    ci_upper:         float      # 95% confidence interval upper bound
    is_significant:   bool
    recommendation:   str

def analyze_experiment(
    control_scores:   list[float],
    treatment_scores: list[float],
    alpha: float = 0.05,   # significance level (5% false positive rate)
    min_effect_size: float = 0.05,  # minimum meaningful change (5%)
) -> ExperimentResults:
    """
    Analyze A/B test results with statistical rigor.
    
    Uses Welch's t-test (doesn't assume equal variance).
    """
    control   = np.array(control_scores)
    treatment = np.array(treatment_scores)
    
    control_mean   = np.mean(control)
    treatment_mean = np.mean(treatment)
    delta          = treatment_mean - control_mean
    delta_pct      = delta / control_mean if control_mean != 0 else 0
    
    # Welch's t-test
    t_stat, p_value = stats.ttest_ind(treatment, control, equal_var=False)
    
    # 95% confidence interval for the difference
    se = np.sqrt(np.var(treatment, ddof=1)/len(treatment) +
                 np.var(control, ddof=1)/len(control))
    ci_lower = delta - 1.96 * se
    ci_upper = delta + 1.96 * se
    
    is_significant = p_value < alpha and abs(delta_pct) >= min_effect_size
    
    if is_significant and delta > 0:
        recommendation = f"SHIP: Treatment improved by {delta_pct:+.1%} (p={p_value:.3f})"
    elif is_significant and delta < 0:
        recommendation = f"REJECT: Treatment degraded by {delta_pct:+.1%} (p={p_value:.3f})"
    elif p_value < alpha:
        recommendation = f"INCONCLUSIVE: Statistically significant but effect too small ({delta_pct:+.1%})"
    else:
        recommendation = f"INSUFFICIENT_DATA: p={p_value:.3f} (need more data)"
    
    return ExperimentResults(
        control_n=len(control), treatment_n=len(treatment),
        control_mean=round(float(control_mean), 3),
        treatment_mean=round(float(treatment_mean), 3),
        delta=round(float(delta), 3),
        delta_pct=round(float(delta_pct), 4),
        p_value=round(float(p_value), 4),
        ci_lower=round(float(ci_lower), 3),
        ci_upper=round(float(ci_upper), 3),
        is_significant=is_significant,
        recommendation=recommendation,
    )

# Sample analysis
import random
random.seed(42)

# Simulate 200 observations per variant
control_scores   = [random.gauss(3.8, 0.6) for _ in range(200)]
treatment_scores = [random.gauss(4.1, 0.6) for _ in range(200)]   # slightly better

results = analyze_experiment(control_scores, treatment_scores)
print(f"Control mean:   {results.control_mean:.3f}")
print(f"Treatment mean: {results.treatment_mean:.3f}")
print(f"Delta:          {results.delta_pct:+.1%}")
print(f"P-value:        {results.p_value:.4f}")
print(f"95% CI:         [{results.ci_lower:.3f}, {results.ci_upper:.3f}]")
print(f"Recommendation: {results.recommendation}")
```

### 2.4 Power Analysis: How Many Users Do You Need?

Before running the experiment, calculate the required sample size:

```python
from scipy import stats
import math

def calculate_sample_size(
    baseline_mean:   float,   # current control metric value
    min_effect_pct:  float,   # minimum detectable effect (e.g., 0.05 for 5%)
    std_dev:         float,   # standard deviation of the metric
    alpha:           float = 0.05,   # false positive rate
    power:           float = 0.80,   # 1 - false negative rate
) -> dict:
    """
    Calculate how many users you need per variant.
    Run this BEFORE starting the experiment.
    """
    min_delta = baseline_mean * min_effect_pct
    
    # Standard formula for two-sample t-test
    z_alpha = stats.norm.ppf(1 - alpha/2)   # two-tailed
    z_beta  = stats.norm.ppf(power)
    
    n = 2 * ((z_alpha + z_beta) * std_dev / min_delta) ** 2
    n_per_variant = math.ceil(n)
    
    return {
        "n_per_variant": n_per_variant,
        "total_n":       n_per_variant * 2,
        "baseline_mean": baseline_mean,
        "min_effect_pct": f"{min_effect_pct:.0%}",
        "power": f"{power:.0%}",
        "note": f"Need {n_per_variant:,} observations per variant to detect a {min_effect_pct:.0%} change with {power:.0%} power"
    }

# Example: task completion rate experiment
result = calculate_sample_size(
    baseline_mean  = 0.65,    # current 65% task completion
    min_effect_pct = 0.05,    # want to detect a 5% improvement (to 68.25%)
    std_dev        = 0.47,    # sqrt(p*(1-p)) for binary metric
    alpha          = 0.05,
    power          = 0.80,
)

print(f"\nRequired sample size:")
for k, v in result.items():
    print(f"  {k}: {v}")

# At 1,000 users/day, this takes 5 days per variant = 10 days total
```

### 2.5 Guardrail Metrics — What Must Not Degrade

A/B tests can optimize one metric at the expense of others. Guardrail metrics prevent this:

```python
def check_guardrails(
    control_data:   dict[str, list[float]],
    treatment_data: dict[str, list[float]],
    guardrail_specs: dict[str, dict],  # {metric_name: {"max_degradation": 0.02}}
) -> dict:
    """
    Check that guardrail metrics haven't degraded significantly.
    Returns False if any guardrail is violated.
    """
    violations = []
    
    for metric, spec in guardrail_specs.items():
        if metric not in control_data or metric not in treatment_data:
            continue
        
        ctrl_mean = np.mean(control_data[metric])
        trt_mean  = np.mean(treatment_data[metric])
        delta_pct = (trt_mean - ctrl_mean) / ctrl_mean
        
        max_degradation = spec.get("max_degradation", 0.02)   # default: 2% max degradation
        
        if delta_pct < -max_degradation:
            violations.append({
                "metric":       metric,
                "control_mean": round(ctrl_mean, 3),
                "treatment_mean": round(trt_mean, 3),
                "delta_pct":    f"{delta_pct:.1%}",
                "max_allowed":  f"-{max_degradation:.0%}",
                "violated":     True
            })
    
    return {
        "guardrails_passed": len(violations) == 0,
        "violations":        violations,
    }

# Usage
guardrail_check = check_guardrails(
    control_data={
        "helpfulness_score": control_scores,
        "error_rate":        [0.01] * 200,
        "latency_ms":        [250.0] * 200,
    },
    treatment_data={
        "helpfulness_score": treatment_scores,
        "error_rate":        [0.012] * 200,   # slightly higher — within tolerance?
        "latency_ms":        [350.0] * 200,   # 40% slower — violation!
    },
    guardrail_specs={
        "helpfulness_score": {"max_degradation": 0.05},
        "error_rate":        {"max_degradation": 0.50},   # 50% relative increase OK
        "latency_ms":        {"max_degradation": 0.20},   # max 20% slower
    }
)

if not guardrail_check["guardrails_passed"]:
    print("❌ Guardrail violations — do not ship:")
    for v in guardrail_check["violations"]:
        print(f"  {v['metric']}: {v['delta_pct']} (max {v['max_allowed']})")
```

## 3. Real-World Context

### The Minimum Experiment Duration

**Run for at least 7 days.** Why:
- Day-of-week effects: users behave differently on Monday vs Friday
- Novelty effect fades after day 2–3: initial enthusiasm for the new system skews early results
- Low-traffic periods: if you get 50% of traffic on weekdays and 20% on weekends, a 3-day test oversample weekdays

**Stop early rules:**
Never stop an experiment early because "the trend looks good." This is the most common
source of false positives in AI A/B testing. Stopping early when significance is reached
inflates your false positive rate to ~30–40%, not the stated 5%.

Use pre-registered stopping rules: commit to the sample size before starting, and only
stop when that sample size is reached.

### What Metrics Actually Work for AI

| Metric | Validity | Notes |
|---|---|---|
| Explicit feedback (thumbs up/down) | Medium | Users rarely give feedback; biased |
| Task completion (binary) | High | Did the AI actually help? Best leading indicator |
| Session end after AI response | Medium | Proxy for "got what I needed" |
| Follow-up question rate | Medium | High rate = answer was incomplete |
| Business conversion | High | Ultimate outcome; high noise |
| LLM judge quality | Medium | Fast; biased; good for trends |

**The metric hierarchy:** Business outcome > Task completion > Behavioral signals > LLM judge score

## 5. Trade-offs

### Eval-Based Improvement vs A/B Test

| | Offline eval only | A/B test |
|---|---|---|
| Speed | Fast (hours) | Slow (days to weeks) |
| Cost | Low | Medium (extra monitoring) |
| Validity | Lower (eval ≠ real user) | Higher (real user outcomes) |
| Safety | Can test anything | Limited to safe changes |

**Rule:** Use offline evals to filter out bad changes early. Use A/B tests to confirm
improvements before full production rollout. Don't A/B test every small change — reserve
it for significant prompt/model changes.
