## 1. What Is It? (Theory)

### Why Streaming Exists

Without streaming, the interaction looks like:

```
User sends request
    │
    ▼ (2–30 seconds of waiting with nothing)
    │
Entire response arrives at once
    │
UI displays everything simultaneously
```

This is terrible UX. Users trained by ChatGPT and Claude expect to see tokens appearing as
they're generated. A 10-second wait followed by a wall of text feels broken. Streaming fixes this:

```
User sends request
    │
    ▼ (200ms — first token arrives)
Token 1 displays
    ▼ (20ms — next token)
Token 2 displays
    ▼ (20ms per token, continuous)
...
Response complete — total elapsed time same, but UX is far better
```

**The performance truth:** Streaming doesn't make total generation faster. It makes the perceived
latency dramatically better because users see progress immediately. The first token arrives in
~200ms; without streaming, you'd wait for all tokens (potentially 10–30s) before seeing anything.

### Server-Sent Events (SSE): The Protocol

All major LLM providers stream via **Server-Sent Events** — a simple HTTP standard where the
server sends a continuous stream of `data:` messages:

```
HTTP/1.1 200 OK
Content-Type: text/event-stream
Transfer-Encoding: chunked

data: {"type": "content_block_delta", "delta": {"text": "The"}}

data: {"type": "content_block_delta", "delta": {"text": " quick"}}

data: {"type": "content_block_delta", "delta": {"text": " brown"}}

data: [DONE]
```

The provider SDKs handle parsing this for you. You don't need to implement SSE parsing manually.

## 2. How It Actually Works — Deep Mechanics

### 2.1 Basic Streaming with Anthropic

```python
import anthropic

client = anthropic.Anthropic()

# Method 1: Using the context manager (recommended)
with client.messages.stream(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Write a haiku about autumn."}]
) as stream:
    # text_stream yields one text chunk at a time
    for text_chunk in stream.text_stream:
        print(text_chunk, end="", flush=True)
    
    # After the stream, you can access the complete response
    final_message = stream.get_final_message()
    print(f"\n\nTokens: {final_message.usage.input_tokens} in / {final_message.usage.output_tokens} out")
```

### 2.2 Basic Streaming with OpenAI

```python
from openai import OpenAI

client = OpenAI()

stream = client.chat.completions.create(
    model="gpt-4o",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Write a haiku about autumn."}],
    stream=True
)

full_text = ""
for chunk in stream:
    delta = chunk.choices[0].delta
    if delta.content:
        print(delta.content, end="", flush=True)
        full_text += delta.content

print()  # newline after stream
```

### 2.3 Async Streaming (Production Pattern)

Synchronous streaming blocks the thread for the entire generation. For web servers handling
concurrent requests, you need async:

```python
import anthropic
import asyncio

# Async client
client = anthropic.AsyncAnthropic()

async def stream_response(prompt: str) -> str:
    """Async streaming — doesn't block other coroutines while waiting for tokens."""
    full_text = ""
    
    async with client.messages.stream(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        messages=[{"role": "user", "content": prompt}]
    ) as stream:
        async for text_chunk in stream.text_stream:
            full_text += text_chunk
            # In a web app: yield text_chunk to the client
    
    return full_text

# Run multiple concurrent streams without blocking
async def process_many(prompts: list[str]) -> list[str]:
    tasks = [stream_response(p) for p in prompts]
    return await asyncio.gather(*tasks)   # all run concurrently

# Results come back as each stream completes, not one at a time
responses = asyncio.run(process_many(["Prompt 1", "Prompt 2", "Prompt 3"]))
```

### 2.4 Streaming in a FastAPI Server

The production pattern: SSE from your backend to your frontend.

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import anthropic
import json

app = FastAPI()
client = anthropic.AsyncAnthropic()

@app.post("/chat/stream")
async def chat_stream(request: dict):
    """Stream LLM response to the browser via SSE."""
    
    async def generate():
        async with client.messages.stream(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            system=request.get("system", ""),
            messages=request["messages"]
        ) as stream:
            async for text_chunk in stream.text_stream:
                # Format as SSE event
                yield f"data: {json.dumps({'text': text_chunk})}\n\n"
            
            # Send final usage stats
            final = stream.get_final_message()
            yield f"data: {json.dumps({'done': True, 'usage': {'input': final.usage.input_tokens, 'output': final.usage.output_tokens}})}\n\n"
    
    return StreamingResponse(
        generate(),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "Connection": "keep-alive",
            "X-Accel-Buffering": "no"   # disable nginx buffering
        }
    )
```

**Frontend consuming the SSE:**

```javascript
const response = await fetch('/chat/stream', {
    method: 'POST',
    headers: {'Content-Type': 'application/json'},
    body: JSON.stringify({messages: [{role: 'user', content: userInput}]})
});

const reader = response.body.getReader();
const decoder = new TextDecoder();

while (true) {
    const {value, done} = await reader.read();
    if (done) break;
    
    const lines = decoder.decode(value).split('\n');
    for (const line of lines) {
        if (line.startsWith('data: ')) {
            const data = JSON.parse(line.slice(6));
            if (data.text) {
                // Append text to UI
                document.getElementById('response').textContent += data.text;
            }
            if (data.done) {
                console.log('Usage:', data.usage);
            }
        }
    }
}
```

### 2.5 Handling Streaming Errors Gracefully

Streams can fail mid-way. Handle this properly:

```python
import anthropic
from anthropic import APIError, APITimeoutError, APIConnectionError

client = anthropic.Anthropic()

def stream_with_error_handling(prompt: str) -> str:
    """Stream with graceful degradation on errors."""
    full_text = ""
    
    try:
        with client.messages.stream(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            messages=[{"role": "user", "content": prompt}]
        ) as stream:
            for text_chunk in stream.text_stream:
                full_text += text_chunk
    
    except APITimeoutError:
        if full_text:
            # Return partial response rather than nothing
            return full_text + " [response timed out]"
        raise  # Re-raise if we got nothing
    
    except APIConnectionError:
        if full_text:
            return full_text + " [connection lost]"
        raise
    
    except APIError as e:
        if e.status_code == 529:  # Anthropic overloaded
            raise  # Retry upstream
        raise
    
    return full_text
```

### 2.6 Event Types in Streaming

Anthropic's streaming is richer than just text chunks. Events include:

```python
import anthropic

client = anthropic.Anthropic()

with client.messages.stream(
    model="claude-sonnet-4-6",
    max_tokens=512,
    messages=[{"role": "user", "content": "Count from 1 to 5."}]
) as stream:
    for event in stream:
        # Different event types in the stream
        if event.type == "message_start":
            print(f"Message started, model: {event.message.model}")
        
        elif event.type == "content_block_start":
            print(f"Content block started")
        
        elif event.type == "content_block_delta":
            if event.delta.type == "text_delta":
                print(event.delta.text, end="", flush=True)
        
        elif event.type == "content_block_stop":
            print("\n[block ended]")
        
        elif event.type == "message_delta":
            # Final token counts and stop reason
            if hasattr(event, "usage"):
                print(f"\nOutput tokens: {event.usage.output_tokens}")
        
        elif event.type == "message_stop":
            print("[stream ended]")
```

## 3. Real-World Context

### Production Streaming Architecture

```
Browser/Client
    │  POST /chat  (JSON body)
    ▼
Load Balancer (nginx)
    │  Proxied to app server
    │  nginx: proxy_buffering off;  ← critical for SSE
    ▼
App Server (FastAPI/Flask/Express)
    │  Async SSE generator
    ▼
LLM Provider API
    │  Streams tokens back
    ▼  (tokens flow back up the chain)
Browser renders tokens as they arrive
```

**Critical infrastructure note:** Most reverse proxies (nginx, CloudFlare, AWS ALB) buffer
responses by default. SSE requires disabling this buffering or the browser receives the entire
response in one chunk — defeating the purpose. Configure:
- nginx: `proxy_buffering off;`
- CloudFlare: disable caching for streaming endpoints
- AWS ALB: streaming typically works without changes, but verify

### Real User Impact

Teams that implement streaming report:
- Users' perceived satisfaction increases significantly (they feel the app is "working")
- Bounce rates on generation pages drop
- Users are more likely to read long outputs (they see them build up rather than appearing all at once)

For any user-facing generation, streaming is not optional in 2026.

### Time-to-First-Token (TTFT) Benchmarks (2026)

| Model | Median TTFT | Notes |
|---|---|---|
| claude-haiku-4-5 | ~150ms | Fastest Anthropic |
| claude-sonnet-4-6 | ~300ms | Good for most uses |
| gpt-4o-mini | ~200ms | Fastest OpenAI |
| gpt-4o | ~400ms | Standard |
| gemini-2.5-flash | ~150ms | Fastest Gemini |
| gemini-2.5-pro | ~500ms | Premium model |

TTFT varies by: prompt length (longer prompt = slower TTFT), time of day (peak hours = higher),
your geographic distance from the provider's inference endpoint.

## 4. Code Examples

### Async Batch Streaming (Concurrent Requests)

```python
import anthropic
import asyncio
from dataclasses import dataclass

client = anthropic.AsyncAnthropic()

@dataclass
class StreamResult:
    index: int
    text: str
    input_tokens: int
    output_tokens: int

async def stream_one(index: int, prompt: str, system: str = "") -> StreamResult:
    """Stream a single request asynchronously."""
    full_text = ""
    final = None
    
    async with client.messages.stream(
        model="claude-haiku-4-5-20251001",
        max_tokens=512,
        system=system,
        messages=[{"role": "user", "content": prompt}]
    ) as stream:
        async for chunk in stream.text_stream:
            full_text += chunk
        final = stream.get_final_message()
    
    return StreamResult(
        index=index,
        text=full_text,
        input_tokens=final.usage.input_tokens,
        output_tokens=final.usage.output_tokens,
    )

async def parallel_streams(prompts: list[str], system: str = "") -> list[StreamResult]:
    """
    Stream N requests concurrently.
    Total time ≈ slowest single request (not sum of all).
    """
    tasks = [stream_one(i, p, system) for i, p in enumerate(prompts)]
    results = await asyncio.gather(*tasks, return_exceptions=True)
    
    # Filter out failures, log errors
    valid = []
    for i, r in enumerate(results):
        if isinstance(r, Exception):
            print(f"Request {i} failed: {r}")
        else:
            valid.append(r)
    
    return sorted(valid, key=lambda x: x.index)

# Process 5 requests concurrently — takes ~max_time, not ~5×time
prompts = [
    "Summarize quantum computing in one sentence.",
    "What is the capital of Brazil?",
    "Write a Python function to reverse a string.",
    "Name three famous Renaissance painters.",
    "Explain blockchain in plain English.",
]

results = asyncio.run(parallel_streams(prompts, system="Be very concise."))
for r in results:
    total_cost_usd = (r.input_tokens * 1.00 + r.output_tokens * 5.00) / 1_000_000
    print(f"[{r.index}] {r.text[:80]} (${total_cost_usd:.6f})")
```

### Streaming to Multiple Destinations

```python
import anthropic
import asyncio

client = anthropic.AsyncAnthropic()

async def stream_and_broadcast(prompt: str, destinations: list):
    """
    Stream from LLM and broadcast each chunk to multiple destinations.
    Use case: stream to websocket + save to database simultaneously.
    """
    
    async with client.messages.stream(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        messages=[{"role": "user", "content": prompt}]
    ) as stream:
        async for chunk in stream.text_stream:
            # Broadcast to all destinations concurrently
            await asyncio.gather(*[d(chunk) for d in destinations])
        
        final = stream.get_final_message()
        return final

async def save_to_db(chunk: str):
    """Simulated DB save."""
    pass  # await db.append(chunk)

async def send_to_websocket(chunk: str):
    """Simulated websocket send."""
    print(chunk, end="", flush=True)  # simulating websocket

asyncio.run(stream_and_broadcast(
    "Write a 3-sentence story about a robot.",
    destinations=[save_to_db, send_to_websocket]
))
```

### Streaming Token Budget Guard

```python
import anthropic

client = anthropic.Anthropic()

def stream_with_budget(
    prompt: str,
    max_output_tokens: int = 500,
    budget_tokens: int = 400    # stop generating before hitting the hard limit
) -> tuple[str, bool]:
    """
    Stream with a soft budget: stop early when approaching the token limit.
    Prevents truncated mid-sentence responses.
    Returns: (text, was_complete)
    """
    full_text = ""
    output_tokens = 0
    stopped_early = False
    
    with client.messages.stream(
        model="claude-sonnet-4-6",
        max_tokens=max_output_tokens,
        messages=[{"role": "user", "content": prompt}]
    ) as stream:
        for event in stream:
            if event.type == "content_block_delta":
                if hasattr(event.delta, "text"):
                    full_text += event.delta.text
                    output_tokens += 1  # approximate; actual count from usage
            
            elif event.type == "message_delta":
                if hasattr(event, "usage"):
                    output_tokens = event.usage.output_tokens
                    if output_tokens >= budget_tokens and full_text.endswith((".", "!", "?")):
                        # Stop at a sentence boundary before hard cutoff
                        stopped_early = True
                        break
    
    return full_text, not stopped_early

text, complete = stream_with_budget("Explain the history of the internet in detail.")
print(text)
print(f"\nResponse complete: {complete}")
```

## 6. Alternatives & Comparisons

### SSE vs WebSockets for LLM Streaming

| Aspect | SSE | WebSockets |
|---|---|---|
| Direction | Server → Client (one-way) | Bidirectional |
| Reconnection | Automatic | Manual |
| Infrastructure | Standard HTTP (CDN-friendly) | Needs WS-aware proxy |
| Complexity | Low | Higher |
| Best for LLM | Yes — generation is one-way | Overkill unless you need two-way push |

SSE is the right choice for LLM response streaming. WebSockets add complexity without benefit
for this specific use case.

### Streaming vs Non-Streaming

| Scenario | Use Streaming | Don't Stream |
|---|---|---|
| User-facing chat/generation | Always | — |
| API-to-API (your backend calls LLM) | Optional (simpler without) | Default for programmatic use |
| Batch processing | No | Yes — non-streaming is simpler |
| When output is short (<50 tokens) | Skip (TTFT dominates) | Yes |

## 7. Trade-offs

### Async vs Sync

**Synchronous streaming:**
- Simpler code; easier to debug
- Blocks the thread for the full generation duration
- Fine for CLI tools, scripts, batch processing

**Async streaming:**
- Required for web servers handling concurrent requests
- Doesn't block other requests while waiting for tokens
- More complex; requires understanding of async/await

**Rule:** Use sync for scripts and offline tools. Use async for anything serving multiple users.

### Streaming Granularity

Providers stream at different granularities — some emit character-by-character, others word-by-word
chunks. For UI purposes, character-level streaming feels more natural. For programmatic parsing,
larger chunks are more efficient.

---

## 8. When To Use vs. When To Avoid

### Streaming
✅ Always use for: User-facing text generation — chat, writing assistants, code generation
✅ Use for: Long generations (>200 tokens) where latency matters
❌ Skip for: Structured output extraction where you need the full JSON before processing
❌ Skip for: Batch offline workloads — non-streaming is simpler and equivalent cost

⚠️ Common mistakes:
- Not disabling proxy buffering — SSE arrives in one chunk instead of streaming
- Not handling stream disconnects — browsers disconnect on tab switch; server must handle this
- Not tracking token usage from the final stream event — usage isn't available until the stream completes
- Using sync streaming in a FastAPI route — blocks the event loop under concurrent load
- Not handling `APITimeoutError` during streams — streams can be interrupted mid-generation

## 9. Senior AI Engineer Perspective

### The 80% of Streaming is Infrastructure

The API call is 20% of the work. The rest is:
- nginx/proxy configuration to prevent buffering
- Frontend SSE parsing and error handling
- Reconnection logic (what happens if the stream drops?)
- Progress indicators and loading states
- Token counting from stream events for billing

Most teams get the API call right and ship broken streaming because they forget the infrastructure.
Test streaming end-to-end through your entire stack before calling it done.

### Streaming and Cost Budgets

Streaming makes it easy to forget that every token costs money while the stream is running.
A user who leaves a tab open while a 10,000-token stream runs is costing you money even if
they never read it.

Production patterns:
- Set `max_tokens` conservatively — don't generate unboundedly
- Detect tab/session closure and cancel the stream server-side
- Soft budget: stop generating at a sentence boundary before hitting `max_tokens`

### The Real-Time Expected UX

In 2026, users who've been using ChatGPT and Claude expect typing-speed token delivery.
If your app shows nothing for 5 seconds then dumps text, you've already lost them.

Two design principles:
1. Show the first token as fast as possible (optimize for TTFT, not total generation time)
2. Show progress even before the first LLM token (skeleton loaders, "thinking..." indicators)
