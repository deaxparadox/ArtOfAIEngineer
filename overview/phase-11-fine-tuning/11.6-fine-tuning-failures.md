## 1. Failure 1: Training on Insufficient Data

**Symptom:** Training loss goes down. Val loss goes down. Eval scores barely improve or even
slightly degrade. The fine-tuned model produces responses in the training style but with
inconsistent quality.

**Root cause:** With fewer than ~300 examples (for simple tasks) or ~1,000+ (for complex tasks),
the model memorizes the small training set rather than learning generalizable patterns.

```python
# Diagnostic: check if val loss suspiciously low relative to task diversity
if val_loss < 0.3 and n_training_examples < 200:
    print("WARNING: May be overfitting to tiny dataset")
    print("  Train loss:", train_loss)
    print("  Val loss:", val_loss)
    print("  Gap (overfitting signal):", train_loss - val_loss)
```

**Fixes:**
1. Collect more examples — there's no shortcut for data scarcity
2. Use data augmentation: rephrase questions, vary answer styles
3. Use a smaller model that needs less data (7B → 1B)
4. Consider few-shot prompting instead — works better than fine-tuning with tiny data

## 2. Failure 2: Noisy or Inconsistent Labels

**Symptom:** Training converges but the model's behavior is unpredictable. Sometimes it
follows the pattern from examples, sometimes it reverts to base model behavior, sometimes
it does something unexpected.

**Root cause:** Your training examples contain contradictions. Example A shows one behavior,
example B shows a different behavior for a similar input. The model can't resolve the contradiction,
so it interpolates between them or behaves erratically.

```python
def detect_label_inconsistencies(examples: list[dict]) -> list[dict]:
    """
    Find examples with very similar inputs but different outputs.
    These are potential label inconsistencies.
    """
    from sentence_transformers import SentenceTransformer
    import numpy as np
    
    model = SentenceTransformer("all-MiniLM-L6-v2")
    
    inputs = [ex["messages"][-2]["content"] for ex in examples]   # user messages
    outputs = [ex["messages"][-1]["content"] for ex in examples]  # assistant responses
    
    input_embeddings = model.encode(inputs, normalize_embeddings=True)
    
    inconsistencies = []
    
    for i in range(len(examples)):
        for j in range(i+1, len(examples)):
            similarity = float(np.dot(input_embeddings[i], input_embeddings[j]))
            
            if similarity > 0.92:   # very similar inputs
                # Check if outputs are very different
                out_embeddings = model.encode([outputs[i], outputs[j]], normalize_embeddings=True)
                output_sim = float(np.dot(out_embeddings[0], out_embeddings[1]))
                
                if output_sim < 0.5:   # very different outputs
                    inconsistencies.append({
                        "index_1": i,
                        "index_2": j,
                        "input_sim": round(similarity, 3),
                        "output_sim": round(output_sim, 3),
                        "input_1": inputs[i][:80],
                        "input_2": inputs[j][:80],
                        "output_1": outputs[i][:80],
                        "output_2": outputs[j][:80],
                    })
    
    return inconsistencies[:10]   # return top 10 potential issues
```

**Fixes:**
1. Run the consistency checker and manually review flagged pairs
2. Establish a labeling rubric with examples of correct behavior
3. Have multiple annotators label the same examples; measure inter-annotator agreement
4. If annotators disagree, the example is ambiguous — remove or rewrite it

## 3. Failure 3: Catastrophic Forgetting

**Symptom:** The fine-tuned model performs well on the target task but has noticeably degraded
performance on general language capabilities — reasoning, instruction following, general knowledge.

**Root cause:** Fine-tuning on a narrow, homogeneous dataset shifts the model's weight distribution
away from the capabilities learned during pre-training. The narrow training signal overwrites
or suppresses more general patterns.

Empirically documented: fine-tuning causes ~10% MMLU degradation on reasoning tasks when training
on narrow-domain data.

```python
# Prevention strategies:

# Strategy 1: Replay buffer — include general examples in fine-tuning data
# Mix domain-specific examples with general instruction-following examples
mixed_dataset = (
    domain_specific_examples +                       # 80% target task
    general_instruction_examples[:len(domain_specific_examples) // 4]  # 20% general
)

# Strategy 2: Low learning rate reduces forgetting
# Smaller steps = less dramatic weight updates = better preservation of prior knowledge
training_args.learning_rate = 1e-5   # even lower than default 2e-4 for LoRA

# Strategy 3: Fewer epochs
# More epochs = more forgetting. Stop when val loss plateaus.
training_args.num_train_epochs = 1   # try 1 epoch first

# Strategy 4: LoRA (not full fine-tuning)
# LoRA only updates ~1% of parameters → much less forgetting than full fine-tuning
# This is one of the strongest arguments for LoRA in production
```

## 4. Failure 4: Wrong Chat Template

**Symptom:** Training completes, but the fine-tuned model produces garbled output, ignores
instructions, or randomly includes template tokens in its responses.

**Root cause:** Used the wrong chat format for the model's tokenizer. This is surprisingly common.
Each model family has a specific expected format. LLaMA-3 format ≠ Mistral format ≠ ChatML.

```python
from transformers import AutoTokenizer

# ALWAYS use the tokenizer's apply_chat_template
tokenizer = AutoTokenizer.from_pretrained("meta-llama/Meta-Llama-3-8B-Instruct")

# WRONG: manually format
wrong_format = f"[INST] {user_message} [/INST] {assistant_response}"
# This is LLaMA-2 format, NOT LLaMA-3 format

# RIGHT: use the tokenizer
correct_format = tokenizer.apply_chat_template(
    conversation=[
        {"role": "system",    "content": system_prompt},
        {"role": "user",      "content": user_message},
        {"role": "assistant", "content": assistant_response},
    ],
    tokenize=False,
    add_generation_prompt=False  # False for training data
)

# For OpenAI fine-tuning: their API handles formatting automatically
# For Hugging Face/TRL: use SFTTrainer with chat_template format
# (SFTTrainer calls apply_chat_template internally)
```

**Fixes:**
1. Always use `tokenizer.apply_chat_template()` — never format manually
2. Verify the formatted data looks right by printing 3–5 examples before training
3. Check the model card or tokenizer config for the expected format

## 5. Failure 5: Overfitting (Too Many Epochs)

**Symptom:** Training loss reaches near-zero. Val loss bottomed out at epoch 2 and has been
increasing since. The fine-tuned model gives great answers to questions that look exactly like
training examples, but struggles on slightly different phrasings.

**Root cause:** Training for too many epochs causes the model to memorize the training set
instead of learning generalizable patterns.

```python
# Monitor val loss during training — stop when it starts increasing
class EarlyStoppingCallback:
    def __init__(self, patience: int = 2, min_delta: float = 0.001):
        self.patience   = patience
        self.min_delta  = min_delta
        self.best_loss  = float('inf')
        self.wait       = 0
    
    def should_stop(self, val_loss: float) -> bool:
        if val_loss < self.best_loss - self.min_delta:
            self.best_loss = val_loss
            self.wait      = 0
            return False
        
        self.wait += 1
        if self.wait >= self.patience:
            print(f"Early stopping: val loss not improving after {self.patience} epochs")
            return True
        
        return False

# TRL's TrainingArguments supports early stopping:
from transformers import EarlyStoppingCallback as HFEarlyStop

training_args = SFTConfig(
    load_best_model_at_end = True,
    metric_for_best_model  = "eval_loss",
    greater_is_better      = False,
    eval_steps             = 100,
    save_steps             = 100,
)
trainer.add_callback(HFEarlyStop(early_stopping_patience=2))
```

## 6. Failure 6: Wrong Loss Masking (Prompt Included in Loss)

**Symptom:** The fine-tuned model tends to repeat or echo the prompt in its responses.
Or: training converges quickly to low loss but the model doesn't learn the target behavior.

**Root cause:** If loss masking is not applied correctly, the model is trained to predict both
the prompt AND the response. This teaches it to "complete the prompt" rather than "respond to it."

```python
# Verify that your training framework masks the prompt

# TRL's SFTTrainer handles this correctly with chat_template format:
# It automatically masks the user/system turns and only trains on assistant turns

# Manual verification: check which token IDs have loss computed
from transformers import AutoTokenizer
import torch

tokenizer = AutoTokenizer.from_pretrained("meta-llama/Meta-Llama-3-8B-Instruct")

conversation = [
    {"role": "user",      "content": "What is the capital of France?"},
    {"role": "assistant", "content": "The capital of France is Paris."},
]

# Tokenize with apply_chat_template
formatted = tokenizer.apply_chat_template(
    conversation, tokenize=True, return_dict=True
)

# The labels should be -100 (masked) for non-assistant tokens
# -100 is PyTorch's convention for "ignore this token in loss"
input_ids = formatted["input_ids"]
labels = formatted.get("labels", input_ids)   # SFTTrainer sets labels = input_ids with masking

# Count: how many tokens are unmasked (contribute to loss)?
n_unmasked = sum(1 for l in labels if l != -100)
n_total    = len(labels)
print(f"Tokens: total={n_total}, unmasked (assistant)={n_unmasked}, masked={n_total - n_unmasked}")
# Good: n_unmasked should be roughly proportional to the assistant response length
# Bad: if n_unmasked == n_total, loss masking is not applied
```

## 7. The Fine-Tuning Checklist

Run through this before every training job:

```python
PRETRAIN_CHECKLIST = [
    # DATA QUALITY
    "[ ] Read 50+ random examples from training set — do they all look correct?",
    "[ ] Run detect_label_inconsistencies() — resolve flagged conflicts",
    "[ ] Verify no duplicate examples (content-hash dedup)",
    "[ ] Check response length distribution — outliers indicate errors",
    "[ ] Verify system prompt matches production exactly",
    
    # FORMAT
    "[ ] Used tokenizer.apply_chat_template() — not manual formatting",
    "[ ] Printed 3 formatted examples — they look like the model's expected format",
    "[ ] Loss masking is active — assistant turns train, prompt turns don't",
    
    # TRAINING SETUP
    "[ ] Val split exists (at least 5-10% of data)",
    "[ ] Early stopping configured",
    "[ ] W&B / MLflow tracking configured",
    "[ ] Start with n_epochs=1 for first run; increase only if needed",
    
    # EVALUATION
    "[ ] Eval dataset ready (task-specific + general capability cases)",
    "[ ] Baseline scores computed for the base model",
    "[ ] Deployment criteria defined: what improvement is 'good enough'?",
]

for item in PRETRAIN_CHECKLIST:
    print(item)
```
