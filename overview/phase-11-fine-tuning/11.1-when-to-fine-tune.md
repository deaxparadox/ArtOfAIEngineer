## 1. What Is It? (Theory)

### The Most Common Mistake in Fine-Tuning

Teams reach for fine-tuning too early. They have a prompt that works 70% of the time, they're
frustrated, and they decide fine-tuning will fix it. They spend two weeks collecting data,
three days training, and discover the fine-tuned model works 72% of the time — barely better
than a well-designed prompt they haven't tried yet.

**The decision rule:** Fine-tune only after you've fully explored prompting and RAG.
The upgrade cost of prompting is near-zero. The cost of fine-tuning (data, compute, eval)
is significant. Exhaust cheap options first.

### What Fine-Tuning Actually Changes

Fine-tuning modifies the model's *weights* — the billions of numbers that determine how it
processes text. This is fundamentally different from prompting:

```
PROMPTING:           Provides context → model uses existing capabilities
FINE-TUNING:         Changes capabilities → model learns new behaviors from examples
RAG:                 Provides information → model answers from retrieved content

These are not competing alternatives — they're complementary.
The right system often uses all three:
  Fine-tune for behavior/style
  Prompt for task specification
  RAG for knowledge retrieval
```

## 2. The Decision Framework

### 2.1 The Decision Tree

```
Is the base model capable of the task at all?
  NO → fine-tuning won't help either; need a different approach
  YES → continue...

Have you tried a well-engineered prompt with few-shot examples?
  NO → try this first; it's free and takes an afternoon
  YES → continue...

Does the task require external/private/current knowledge?
  YES → add RAG before fine-tuning; this often solves it
  NO → continue...

What is the specific failure mode?
  A) Wrong output FORMAT (JSON structure, style, tone)
     → Fine-tuning is a good fit
  B) Wrong KNOWLEDGE (model doesn't know domain facts)  
     → RAG is better; fine-tuning doesn't reliably add knowledge
  C) Wrong BEHAVIOR (model doesn't follow instructions reliably)
     → Fine-tuning is a good fit; DPO is especially good here
  D) Wrong REASONING (model makes logical errors)
     → Fine-tuning rarely helps; use CoT prompting or a more capable model

Do you have 500+ high-quality labeled examples?
  NO → not enough data; collect more before fine-tuning
  YES → fine-tuning is likely worth trying

Does inference cost matter at scale?
  YES → fine-tuning a small model can match a large model at lower cost
  NO → prompting a larger model may be fine
```

### 2.2 When Fine-Tuning Definitely Helps

**Format and style consistency:**
```python
# Problem: even with clear instructions, the model sometimes outputs
# {"result": "..."} and sometimes "Here is the result: ..."
# Fine-tuning on hundreds of examples of the correct format
# → Model consistently outputs {"result": "..."} every time
```

**Domain vocabulary and terminology:**
```python
# Problem: model paraphrases medical terms instead of using exact codes
# Fine-tuned on clinical notes → model uses correct ICD-10 codes naturally
```

**Efficiency at scale:**
```python
# Problem: achieving target quality requires a 5,000-token few-shot prompt
# Fine-tuning teaches the task → can use a 200-token prompt
# 5,000 tokens × 1M requests × $3/M = $15,000 saved per month
```

**Consistent persona/tone:**
```python
# Problem: the model keeps slipping out of character
# Fine-tuning on brand-voice examples → consistent tone without prompting tricks
```

**Instruction following in specific domains:**
```python
# Problem: model understands instructions but not domain-specific conventions
# Fine-tuning on legal, medical, or scientific text → follows domain norms
```

### 2.3 When Fine-Tuning Doesn't Help

**Adding knowledge:**
Fine-tuning cannot reliably add factual knowledge to a model. Fine-tuning on medical papers
doesn't make a model "know medicine" — it makes it produce text that *sounds like* medical
writing. The factual accuracy often doesn't improve as expected.

```
Trying to add knowledge → Use RAG instead
Fine-tuning for knowledge = expensive, unreliable, wrong approach
```

**Fixing reasoning errors:**
If a model makes systematic logical errors, fine-tuning on examples of correct reasoning
often doesn't fix the underlying reasoning capability. The model memorizes the training
examples instead of learning the reasoning pattern.

```
Fixing reasoning → Use chain-of-thought prompting, a more capable model,
                   or RLVR (reinforcement learning with verifiable rewards)
```

**When you have very few examples:**
Fine-tuning with fewer than ~100–300 examples typically leads to overfitting or negligible
improvement. The model memorizes the tiny training set rather than generalizing.

**When the issue is unclear:**
If you can't specify precisely what "good" looks like in enough examples to train on,
fine-tuning will optimize toward something undefined.

### 2.4 The Three-Way Decision: Fine-Tune vs RAG vs Both

| Goal | Best Approach |
|---|---|
| Answer questions about private documents | RAG |
| Answer questions about current events | RAG with live search |
| Consistent output format/schema | Fine-tuning |
| Domain vocabulary and style | Fine-tuning |
| Domain-specific Q&A with facts | RAG + fine-tuning |
| Reduce prompt length at scale | Fine-tuning |
| Specific tone and persona | Fine-tuning |
| Factual accuracy on domain knowledge | RAG (not fine-tuning alone) |

## 3. Real-World Context

### ComplyAdvantage Case Study

ComplyAdvantage (financial crime AI) faced a decision: their model was producing inconsistent
compliance classifications. They tried prompt engineering extensively but format consistency
remained an issue at scale.

The decision framework said: "wrong output format → fine-tuning is a good fit."

After fine-tuning on ~2,000 labeled compliance decisions, they achieved 98%+ format consistency
(up from ~82% with prompting), and reduced their system prompt from 4,000 tokens to 800 tokens,
saving ~$15K/month at their API call volume.

**The lesson:** They didn't fine-tune to add compliance knowledge (they still use RAG for that).
They fine-tuned specifically for the format/behavior problem.

### The Typical Fine-Tuning ROI Timeline

```
Week 1: Data collection and preparation (most expensive step)
Week 2: First fine-tuning run and evaluation
Week 3: Second iteration (data quality improvements, eval feedback)
Month 2: Production deployment and monitoring
Month 3+: ROI positive if:
  - Inference cost savings exceed fine-tuning cost, OR
  - Quality improvement is measurable and business-impactful
```

For most teams at moderate scale: fine-tuning pays off in 2–4 months if the quality gain
is real and the API call volume is significant.

## 4. Code Examples

### Decision Checklist

```python
from dataclasses import dataclass

@dataclass
class FineTuneDecision:
    tried_prompt_engineering:    bool
    tried_few_shot:              bool
    tried_rag:                   bool
    failure_mode:                str   # "format" | "knowledge" | "behavior" | "reasoning"
    n_available_examples:        int
    monthly_api_calls:           int
    quality_gap:                 float   # 0-1: how far current system is from target
    
    def recommend(self) -> str:
        if not self.tried_prompt_engineering or not self.tried_few_shot:
            return "PROMPT FIRST: Haven't exhausted prompt engineering + few-shot"
        
        if self.failure_mode == "knowledge" and not self.tried_rag:
            return "TRY RAG: Knowledge problems need retrieval, not fine-tuning"
        
        if self.failure_mode == "reasoning":
            return "FINE-TUNING WON'T HELP: Try CoT prompting or a stronger base model"
        
        if self.n_available_examples < 300:
            return f"COLLECT MORE DATA: {self.n_available_examples} examples is insufficient (need 300+)"
        
        if self.failure_mode in ("format", "behavior") and self.n_available_examples >= 300:
            # Estimate ROI
            tokens_saved_per_call = 3000   # rough: shorter prompt after fine-tuning
            monthly_savings = self.monthly_api_calls * tokens_saved_per_call * 3.0 / 1_000_000
            return (f"FINE-TUNE: Good candidate. "
                    f"Estimated monthly API savings: ${monthly_savings:.0f}/month. "
                    f"Data collection + training cost: ~$500-5000 (one-time)")
        
        return "CONTINUE PROMPTING: Edge case — analyze the specific failure mode further"

# Example evaluations
cases = [
    FineTuneDecision(True, True, True, "format", 2000, 100_000, 0.15),
    FineTuneDecision(True, False, False, "knowledge", 50, 10_000, 0.30),
    FineTuneDecision(True, True, True, "reasoning", 500, 50_000, 0.25),
    FineTuneDecision(False, False, False, "behavior", 1000, 80_000, 0.20),
]

for case in cases:
    print(f"\nFailure: {case.failure_mode} | Examples: {case.n_available_examples}")
    print(f"  → {case.recommend()}")
```

### Fine-Tuning Options by Approach

| Approach | When to Use | Tools |

## 6. Trade-offs

### Fine-Tuning vs Larger Model

| | Fine-tune small model | Use larger model |
|---|---|---|
| Inference cost | Lower | Higher |
| Quality ceiling | Limited by base model | Better reasoning capability |
| Time to production | Weeks | Hours (just change the model) |
| Maintenance | Must re-fine-tune on model updates | Just upgrade model |
| Data requirement | Yes | No |

**Rule:** If a 3× more expensive larger model solves the problem acceptably, the cost of the
larger model may be less than the engineering cost of fine-tuning a smaller one. Calculate before committing.

## 7. Senior AI Engineer Perspective

### The "Fine-Tuning Will Fix It" Trap

The most expensive mistake in LLM engineering: using fine-tuning to fix problems that prompting
or RAG could solve, while ignoring the real issue.

Real example: a team spent 3 weeks fine-tuning a model to produce better legal summaries.
After deployment, the quality was similar to their prompt-engineered baseline. Post-mortem
revealed: the actual problem was poor document chunking in their RAG pipeline. The model
was generating from poorly retrieved context. No fine-tuning could fix that.

**Before fine-tuning, run your full eval suite on the current system and identify the specific
failure mode with precision. "The model is bad" is not a fine-tuning plan.**

### When Fine-Tuning Is the Right Answer

After working through this decision framework honestly, fine-tuning IS the right answer when:
- You have a specific, measurable format/behavior problem
- You have enough high-quality examples
- You've verified that prompting can't reliably solve it
- The volume or cost justifies the investment

When these conditions are met, fine-tuning produces dramatic, durable improvements that no
prompt engineering can match. The key is honest assessment of the conditions.
