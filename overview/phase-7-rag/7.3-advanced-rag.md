## 1. What Is It? (Theory)

### When to Graduate from Naive RAG

After building and measuring naive RAG, you'll encounter specific failure patterns. Advanced RAG
addresses each pattern with a targeted technique:

```
Failure: Query phrasing doesn't match document phrasing
Fix:     Query expansion — generate multiple phrasings

Failure: Short/ambiguous queries retrieve wrong chunks
Fix:     HyDE — generate a hypothetical answer, embed that instead

Failure: Correct chunks are retrieved but ranked too low
Fix:     Reranking — precision-focused second-stage scoring

Failure: Keyword terms (codes, names) not in top-K results
Fix:     Hybrid search (covered in Phase 6.6)

Failure: Answer requires multiple chunks with different info
Fix:     Multi-query retrieval — retrieve from N different sub-queries
```

Advanced RAG is a menu of techniques, not a monolithic upgrade. Apply the technique that targets
your specific measured failure mode.

## 2. How It Actually Works — Deep Mechanics

### 2.1 Reranking

**The problem with bi-encoder retrieval:** Vector similarity finds semantically related text,
but "related" is not the same as "directly answers this question." A chunk about "dogs" is
related to a query about "pet training" but may be less relevant than a specific chunk about
"how to train puppies."

**Cross-encoder rerankers** read the full (query, document) pair together — unlike bi-encoders
which encode them separately. This allows the model to assess precise relevance.

```
Pipeline:
  Query → Vector retrieval → Top-50 candidates → Cross-encoder → Top-5 final

Cost:
  Vector retrieval: 1 embedding + 1 ANN search = fast
  Cross-encoder:    50 × (query + chunk) forward passes = slow but only on 50 items
```

```python
import anthropic
import cohere
from sentence_transformers import SentenceTransformer, CrossEncoder

embed_model  = SentenceTransformer("BAAI/bge-large-en-v1.5")
cross_encoder = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")   # free, fast
# OR: use Cohere Rerank API (managed, production-grade)

co = cohere.Client()  # for Cohere Rerank

def retrieve_and_rerank(
    query: str,
    all_chunks: list[dict],   # {"text": ..., "source": ..., ...}
    first_stage_k: int = 20,  # retrieve this many from vector DB
    final_k: int = 5,         # return this many after reranking
    use_api: bool = False,     # True = Cohere API, False = local cross-encoder
) -> list[dict]:
    """
    Two-stage retrieval: bi-encoder for recall, cross-encoder for precision.
    """
    # Stage 1: Fast vector retrieval (recall-focused)
    q_vec = embed_model.encode([query], normalize_embeddings=True)[0]
    texts = [c["text"] for c in all_chunks]
    import numpy as np
    embeddings = embed_model.encode(texts, normalize_embeddings=True)
    scores_1 = embeddings @ q_vec
    top_indices = np.argsort(-scores_1)[:first_stage_k]
    candidates = [all_chunks[i] for i in top_indices]
    
    # Stage 2: Slow reranking (precision-focused)
    if use_api:
        # Cohere Rerank API — production-grade, ~$0.001/query at top-50
        response = co.rerank(
            model="rerank-v3.5",
            query=query,
            documents=[c["text"] for c in candidates],
            top_n=final_k
        )
        return [candidates[r.index] for r in response.results]
    
    else:
        # Local cross-encoder — free, adds ~100–200ms
        pairs = [(query, c["text"]) for c in candidates]
        rerank_scores = cross_encoder.predict(pairs)
        ranked = sorted(zip(candidates, rerank_scores), key=lambda x: -x[1])
        return [chunk for chunk, _ in ranked[:final_k]]
```

### 2.2 Query Expansion (Multi-Query)

**The problem:** The user's query uses different words than the relevant documents. "How do I
fix a login issue?" may not match a document about "authenticating users" or "troubleshooting
OAuth errors."

**Query expansion** generates multiple alternative phrasings of the original query and runs
retrieval for each, then merges results with deduplication.

```python
import anthropic
import numpy as np

client = anthropic.Anthropic()

def generate_query_variants(query: str, n_variants: int = 4) -> list[str]:
    """
    Use LLM to generate alternative phrasings of the query.
    These variants probe different regions of the embedding space.
    """
    response = client.messages.create(
        model="claude-haiku-4-5-20251001",   # cheap model for this auxiliary task
        max_tokens=256,
        messages=[{
            "role": "user",
            "content": f"""Generate {n_variants} different phrasings of this search query.
Each variant should express the same information need but use different words or structure.
Output only the variants, one per line. No explanations.

Original query: {query}"""
        }]
    )
    variants = [q.strip() for q in response.content[0].text.strip().split("\n") if q.strip()]
    return [query] + variants[:n_variants]   # include original + variants


def multi_query_retrieve(
    query: str,
    embed_fn,     # callable: (list[str]) -> np.ndarray
    all_chunks: list[dict],
    n_variants: int = 3,
    k_per_query: int = 10,
    final_k: int = 5,
) -> list[dict]:
    """
    Retrieve using multiple query variants, merge with deduplication.
    """
    variants = generate_query_variants(query, n_variants)
    print(f"Query variants: {variants}")
    
    # Retrieve for each variant
    texts = [c["text"] for c in all_chunks]
    all_embeddings = embed_fn(texts)
    
    seen_indices = set()
    merged_results: list[tuple[dict, float]] = []
    
    for variant in variants:
        q_vec = embed_fn([variant])[0]
        scores = all_embeddings @ q_vec
        top_k = np.argsort(-scores)[:k_per_query]
        
        for idx in top_k:
            if idx not in seen_indices:
                seen_indices.add(idx)
                merged_results.append((all_chunks[idx], float(scores[idx])))
    
    # Sort merged results by score, take top final_k
    merged_results.sort(key=lambda x: -x[1])
    return [chunk for chunk, _ in merged_results[:final_k]]
```

### 2.3 HyDE — Hypothetical Document Embeddings

**The problem:** Queries and documents live in different parts of the embedding space.
A short question ("What is photosynthesis?") embeds very differently from a long explanatory
passage ("Photosynthesis is the biological process by which plants...").

**HyDE's insight:** Generate a hypothetical document that *would* answer the query, then embed
that document instead. The hypothetical document lives in the same part of the embedding space
as real answer documents.

```
Standard:  embed("What is photosynthesis?") → query vector → search
HyDE:      "What is photosynthesis?" 
           → LLM generates: "Photosynthesis is the process by which plants..."
           → embed(hypothetical_answer) → search
           → The embedding of a hypothetical answer is closer to real answers than the question
```

```python
def hyde_retrieve(
    query: str,
    embed_fn,
    all_chunks: list[dict],
    k: int = 5,
) -> list[dict]:
    """
    HyDE: Generate a hypothetical answer, embed it, search for similar chunks.
    """
    # Step 1: Generate a hypothetical answer
    response = client.messages.create(
        model="claude-haiku-4-5-20251001",  # cheap; we don't need accuracy here, just plausible text
        max_tokens=256,
        system="Write a concise, plausible answer to the question. "
               "If unsure, write what an answer might look like. "
               "This is for retrieval purposes only, not final output.",
        messages=[{"role": "user", "content": f"Question: {query}"}]
    )
    hypothetical_answer = response.content[0].text
    print(f"Hypothetical answer: {hypothetical_answer[:100]}...")
    
    # Step 2: Embed the hypothetical answer (not the original query)
    texts = [c["text"] for c in all_chunks]
    all_embeddings = embed_fn(texts)
    
    h_vec = embed_fn([hypothetical_answer])[0]
    scores = all_embeddings @ h_vec
    top_indices = np.argsort(-scores)[:k]
    
    return [all_chunks[i] for i in top_indices]

# When to use HyDE:
# ✓ Short, ambiguous, or underspecified queries
# ✓ When query and document phrasing are very different
# ✗ When the query is already specific and well-phrased
# ✗ Time-critical applications (adds 1 extra LLM call per query)
# Note: hallucinations in the hypothetical answer CAN reduce retrieval quality
```

### 2.4 Contextual Compression

**The problem:** Even when the right chunk is retrieved, it may contain irrelevant surrounding
text that dilutes the signal and wastes LLM context tokens.

**Solution:** After retrieval, use an LLM to compress each chunk to only the parts relevant
to the query.

```python
def compress_context(query: str, chunk_text: str) -> str:
    """
    Extract only the query-relevant portion from a retrieved chunk.
    Reduces context token usage and improves signal/noise ratio.
    """
    response = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=256,
        system="Extract only the sentences from the provided text that are relevant to the question. "
               "If nothing is relevant, output: 'NOT RELEVANT'",
        messages=[{"role": "user", "content": f"Question: {query}\n\nText:\n{chunk_text}"}]
    )
    compressed = response.content[0].text.strip()
    
    if compressed == "NOT RELEVANT":
        return None   # Signal to skip this chunk
    return compressed

def retrieve_and_compress(
    query: str, retrieved_chunks: list[dict]
) -> list[dict]:
    """Filter out irrelevant chunks, compress relevant ones."""
    compressed = []
    for chunk in retrieved_chunks:
        compressed_text = compress_context(query, chunk["text"])
        if compressed_text:
            compressed.append({**chunk, "text": compressed_text, "original_text": chunk["text"]})
    return compressed

# Cost: 1 LLM call per retrieved chunk (use cheap/fast model)
# Benefit: 30-60% token reduction; improved answer faithfulness
```

### 2.5 Combining Techniques: The Advanced RAG Stack

```python
def advanced_rag_query(
    query: str,
    all_chunks: list[dict],
    embed_fn,
    use_hyde: bool = False,
    use_multi_query: bool = True,
    use_reranker: bool = True,
    use_compression: bool = False,
    first_stage_k: int = 20,
    final_k: int = 5,
) -> dict:
    """
    Configurable advanced RAG pipeline.
    Enable techniques based on your measured failure modes.
    """
    # Step 1: Pre-retrieval (expand or transform the query)
    if use_hyde:
        effective_query = generate_hyde_text(query)
    else:
        effective_query = query
    
    # Step 2: Retrieval
    if use_multi_query:
        chunks = multi_query_retrieve(query, embed_fn, all_chunks, k_per_query=first_stage_k, final_k=first_stage_k)
    else:
        q_vec = embed_fn([effective_query])[0]
        scores = embed_fn([c["text"] for c in all_chunks]) @ q_vec
        top_idx = np.argsort(-scores)[:first_stage_k]
        chunks = [all_chunks[i] for i in top_idx]
    
    # Step 3: Post-retrieval
    if use_reranker:
        pairs = [(query, c["text"]) for c in chunks]
        rerank_scores = cross_encoder.predict(pairs)
        chunks = [c for c, _ in sorted(zip(chunks, rerank_scores), key=lambda x: -x[1])[:final_k]]
    else:
        chunks = chunks[:final_k]
    
    if use_compression:
        chunks = retrieve_and_compress(query, chunks)
    
    # Step 4: Generate
    context = build_context(chunks)
    answer  = generate_answer(query, context)
    
    return {
        "query":    query,
        "answer":   answer,
        "chunks":   chunks,
        "pipeline": {
            "hyde":        use_hyde,
            "multi_query": use_multi_query,
            "reranker":    use_reranker,
            "compression": use_compression,
        }
    }
```

## 3. Real-World Context

### Measured Impact of Each Technique

Based on production benchmarks and 2025 research:

| Technique | Recall@5 Delta | Latency Added | Cost Added |
|---|---|---|---|
| Reranking (top-20 → 5) | +8–15% | +50–200ms | ~$0.001/query (API) |
| Multi-query (3 variants) | +10–18% | +100–300ms | ~$0.002/query |
| HyDE | +5–12% (variable) | +100–200ms | ~$0.001/query |
| Hybrid search (Phase 6.6) | +10–20% | +2–5ms | Low |
| Contextual compression | -0–5% recall, +faithfulness | +50–150ms | ~$0.003/query |

**The practical ranking of techniques by ROI:**
1. Hybrid search (Section 6.6) — biggest impact, lowest overhead
2. Reranking — consistent improvement, reasonable overhead
3. Multi-query — good for short/ambiguous queries, higher cost
4. HyDE — situationally useful; test first
5. Contextual compression — use only when context length is the bottleneck

## 4. Full Production Advanced RAG Example

```python
from qdrant_client import QdrantClient
from qdrant_client.models import SearchRequest
import cohere
import anthropic

# Production-grade advanced RAG: hybrid search + reranking
qdrant_client = QdrantClient(url="http://localhost:6333")
cohere_client = cohere.Client()
claude_client = anthropic.Anthropic()

def production_rag_query(query: str, collection: str = "knowledge_base") -> dict:
    """
    Production pipeline:
    1. Hybrid retrieval (dense + BM25 via Qdrant fusion)
    2. Cohere reranking
    3. Claude generation with strict grounding
    """
    # 1. Hybrid retrieval via Qdrant's built-in fusion
    from qdrant_client.models import Prefetch, FusionQuery, Fusion
    from sentence_transformers import SentenceTransformer
    
    embed = SentenceTransformer("BAAI/bge-large-en-v1.5")
    q_vec = embed.encode([query], normalize_embeddings=True)[0].tolist()
    
    # Dense + sparse fusion with RRF
    results = qdrant_client.query_points(
        collection_name=collection,
        prefetch=[
            Prefetch(query=q_vec, using="dense", limit=20),
            # Add sparse vector for hybrid search if collection has sparse vectors
        ],
        query=FusionQuery(fusion=Fusion.RRF),
        limit=20,
        with_payload=True,
    )
    
    candidates = [r.payload for r in results.points]
    
    # 2. Cohere reranking (precision stage)
    rerank_response = cohere_client.rerank(
        model="rerank-v3.5",
        query=query,
        documents=[c["text"] for c in candidates],
        top_n=5
    )
    
    final_chunks = [candidates[r.index] for r in rerank_response.results]
    
    # 3. Grounded generation
    context = "\n\n---\n\n".join(
        f"[Source {i+1}]\n{c['text']}"
        for i, c in enumerate(final_chunks)
    )
    
    response = claude_client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system="""Answer questions based only on provided sources.
Cite with [Source N]. Say "not in provided documents" if the answer isn't there.""",
        messages=[{"role": "user", "content": f"Sources:\n{context}\n\nQuestion: {query}"}]
    )
    
    return {
        "answer":     response.content[0].text,
        "sources":    [c.get("source", "") for c in final_chunks],
        "candidates": len(candidates),
        "reranked_to": 5,
        "usage": {
            "input_tokens":  response.usage.input_tokens,
            "output_tokens": response.usage.output_tokens,
        }
    }
```

## 6. Trade-offs

### Recall vs Precision

Advanced RAG techniques often improve one at the cost of the other:

| Technique | Recall | Precision |
|---|---|---|
| Reranking | Neutral or slight gain | High improvement |
| Multi-query | High improvement | Slight decrease (more noise) |
| HyDE | Moderate improvement | Variable |
| Compression | Slight decrease | Improvement |
| Hybrid search | High improvement | Slight improvement |

**In practice:** Multi-query increases recall (finds more relevant things). Reranking increases
precision (filters to the best). **Use them together:** multi-query first, then rerank.

### When More Is Not Better

Adding all advanced techniques simultaneously:
- Increases latency significantly (3–5 LLM calls before generation)
- Increases cost (query variants + reranking)
- Adds complexity that's harder to debug

**Principle:** Add one technique at a time. Measure after each addition. Keep only what improves
your specific measured failure mode.

## 7. When To Use vs. When To Avoid

### Reranking
✅ Always add to production RAG — low overhead, consistent improvement
✅ Use Cohere Rerank API for fast production deployment
❌ Skip only if latency budget is extremely tight (< 200ms total)

### Multi-Query
✅ Use when: Queries are short, ambiguous, or domain-specific terminology varies
✅ Use when: Recall@5 is low on your eval set
❌ Skip when: Queries are already long and specific; adds cost without benefit

### HyDE
✅ Use when: Q&A over technical documentation with very different query/document styles
✅ Use when: Multi-query alone isn't enough
❌ Skip when: Query domain is already well-matched to document domain
❌ Skip when: Hypothetical answers would consistently hallucinate (verify empirically)

⚠️ Common mistakes:
- Adding advanced techniques before measuring naive RAG quality — you don't know what you're fixing
- Using HyDE when hallucinations in the hypothetical answer corrupt retrieval
- Reranking with a tiny first-stage pool (< 10) — reranking can't fix missed recall
- Not A/B testing advanced vs naive on the same eval set — you need evidence of improvement

## 8. Senior AI Engineer Perspective

### The Rule: Measure Before You Optimize

Every technique in this section costs tokens, latency, or complexity. Before adding any of them,
establish your baseline:
1. Build naive RAG
2. Create an eval set (50+ query-answer pairs that are representative of real user queries)
3. Measure Recall@5 and answer faithfulness
4. Identify which failure mode is most common
5. Apply the technique targeting that failure mode
6. Measure again — did it actually improve?

Teams that skip this process end up with complex pipelines that aren't actually better than naive RAG.

### The HyDE Trap

HyDE sounds clever and the paper results are promising. In practice, it's temperamental:
- If the LLM generates a plausible but wrong hypothetical answer, retrieval is misled
- It adds one full LLM call per query (cost + latency)
- Results vary heavily by domain and embedding model

Test HyDE on your specific data before committing to it. For general-purpose Q&A over
well-structured documentation, simple multi-query often outperforms HyDE at lower cost.
