## 1. What Is It? (Theory)

### Why You Need a Serving Framework

Running inference with `model.generate()` in a Jupyter notebook doesn't scale to production.
A production serving framework handles:

- **Request queuing:** Multiple concurrent users; requests must wait without blocking others
- **Batching:** Group multiple requests together to maximize GPU utilization
- **Memory management:** Efficiently allocate and reuse GPU memory across requests
- **API surface:** Expose model as HTTP/gRPC endpoint with standard interface
- **Monitoring:** Track latency, throughput, errors
- **Model management:** Load/unload models, manage multiple model versions

Without a serving framework, a naive implementation leaves 80–90% of GPU capacity idle.

### The 2026 Landscape

**vLLM** won the LLM serving race for pure-LLM workloads. It's the default choice.
**TGI** (Text Generation Inference by Hugging Face) moved to maintenance mode in December 2025;
new projects should use vLLM or SGLang.
**Triton** remains the standard for multi-model GPU platforms.
**TorchServe** is still used for PyTorch models generally but rarely the first choice for LLMs.

## 2. How It Actually Works — Deep Mechanics

### 2.1 vLLM — The LLM Serving Standard

vLLM introduced two key innovations: **PagedAttention** (memory-efficient KV-cache) and
**continuous batching** (dynamic request scheduling). Together they achieve 20–23× higher
throughput than naive serving.

**Quick start:**

```bash
# Install
pip install vllm

# Serve a model (OpenAI-compatible API)
python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Llama-3-8B-Instruct \
    --dtype bfloat16 \
    --tensor-parallel-size 1 \   # GPUs to use
    --gpu-memory-utilization 0.85 \
    --max-model-len 8192 \
    --port 8000

# That's it — you now have an OpenAI-compatible API on port 8000
```

**Use it with the OpenAI SDK:**

```python
from openai import OpenAI

# Point the OpenAI client at your vLLM server
client = OpenAI(
    api_key="not-needed",
    base_url="http://localhost:8000/v1"
)

response = client.chat.completions.create(
    model="meta-llama/Llama-3-8B-Instruct",
    messages=[{"role": "user", "content": "What is the capital of Japan?"}],
    max_tokens=100
)
print(response.choices[0].message.content)
```

**Production Docker deployment:**

```dockerfile
# Dockerfile.vllm
```

**Key vLLM configuration options:**

```python
from vllm import LLM, SamplingParams

# Programmatic API (vs CLI)
llm = LLM(
    model="meta-llama/Llama-3-8B-Instruct",
    dtype="bfloat16",
    gpu_memory_utilization=0.85,   # leave 15% for system
    max_model_len=8192,            # max sequence length
    tensor_parallel_size=1,        # number of GPUs
    enforce_eager=False,           # use CUDA graphs (faster) vs eager mode
    
    # Quantization options (see Section 12.3)
    quantization=None,             # or "awq", "gptq", "fp8"
    
    # KV-cache quantization (reduce memory)
    kv_cache_dtype="auto",         # or "fp8"
)

sampling_params = SamplingParams(
    temperature=0.7,
    top_p=0.95,
    max_tokens=512,
    stop=["<|eot_id|>"],   # model-specific stop tokens
)

# Batch inference (offline — not server mode)
prompts = [
    "What is machine learning?",
    "Explain quantum computing.",
    "Write a Python function to reverse a list.",
]
outputs = llm.generate(prompts, sampling_params)

for output in outputs:
    print(f"Prompt: {output.prompt[:50]}...")
    print(f"Response: {output.outputs[0].text[:100]}")
    print(f"Tokens generated: {len(output.outputs[0].token_ids)}\n")
```

### 2.2 SGLang — The Alternative for Complex Workflows

SGLang (Stanford, 2024) optimizes for complex multi-call workflows like agents and RAG:

```bash
pip install sglang

# Serve with SGLang
python -m sglang.launch_server \
    --model-path meta-llama/Llama-3-8B-Instruct \
    --port 30000
```

SGLang's key differentiator: **RadixAttention** — a KV-cache sharing mechanism that dramatically
reduces memory usage when many requests share common prefixes (long system prompts, RAG context).

### 2.3 NVIDIA Triton Inference Server

Triton is the production standard when you need to serve multiple models (not just LLMs)
on the same GPU infrastructure:

```python
# Triton serves models defined in a model repository
# Directory structure:
# model_repository/
#   llama3_8b/
#     config.pbtxt
#     1/
#       model.pt  (or model.onnx, TensorRT engine, etc.)

# config.pbtxt example
TRITON_CONFIG = """
name: "llama3_8b"
backend: "python"
max_batch_size: 8

input [
  {
    name: "input_ids"
    data_type: TYPE_INT64
    dims: [-1, -1]   # [batch_size, sequence_length]
  }
]

output [
  {
    name: "output_ids"
    data_type: TYPE_INT64
    dims: [-1, -1]
  }
]

dynamic_batching {
  preferred_batch_size: [4, 8]
  max_queue_delay_microseconds: 100
}
"""

# Client
import tritonclient.http as httpclient
import numpy as np

client = httpclient.InferenceServerClient(url="localhost:8000")

# Create input
input_data = np.array([[1, 2, 3, 4, 5]], dtype=np.int64)
inputs = [httpclient.InferInput("input_ids", input_data.shape, "INT64")]
inputs[0].set_data_from_numpy(input_data)

# Infer
result = client.infer(model_name="llama3_8b", inputs=inputs)
output = result.as_numpy("output_ids")
```

### 2.4 TorchServe — PyTorch's Native Serving

```bash
# Install
pip install torchserve torch-model-archiver

# Archive the model
torch-model-archiver \
    --model-name llm_model \
    --version 1.0 \
    --model-file model.py \
    --handler handler.py \
    --extra-files "config.json,tokenizer.json" \
    --output-dir model_store

# Serve
torchserve \
    --start \
    --ncs \
    --model-store model_store \
    --models llm_model=llm_model.mar
```

## 3. Framework Comparison

### Head-to-Head: What to Choose

| Framework | Best For | Key Feature | Limitation |
|---|---|---|---|
| **vLLM** | Pure LLM serving; default choice | PagedAttention + continuous batching | LLMs only |
| **SGLang** | Agent/RAG workflows; prefix sharing | RadixAttention for shared prefixes | Newer, smaller community |
| **Triton** | Multi-model GPU platform | Heterogeneous model serving | Complex setup for LLMs alone |
| **TorchServe** | General PyTorch models | PyTorch-native | Not LLM-optimized |
| **TGI** | Legacy systems | Was the previous standard | Maintenance mode since Dec 2025 |
| **llama.cpp** | CPU / local inference | Runs on CPU with quantization | Lower throughput than GPU options |
| **Ollama** | Developer local use | Simplest setup | Not production-grade |

**The 2026 recommendation:**
- Single LLM in production → **vLLM**
- Complex agent workflows with shared prefixes → **SGLang**
- Multiple models (LLM + vision + embedding) on same GPUs → **Triton**
- Local development → **Ollama** (wraps llama.cpp)

## 4. Production Deployment Pattern

```python
# Complete production setup with health checks and monitoring

from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
import httpx
import os

app = FastAPI()

VLLM_BASE_URL = os.getenv("VLLM_URL", "http://localhost:8000")

class ChatRequest(BaseModel):
    messages: list[dict]
    model:     str = "meta-llama/Llama-3-8B-Instruct"
    max_tokens: int = 512

@app.get("/health")
async def health():
    """Check if vLLM is healthy."""
    try:
        async with httpx.AsyncClient() as client:
            resp = await client.get(f"{VLLM_BASE_URL}/health", timeout=5)
            resp.raise_for_status()
        return {"status": "healthy", "vllm": "ok"}
    except Exception as e:
        raise HTTPException(503, f"vLLM unhealthy: {e}")

@app.post("/chat")
async def chat(request: ChatRequest):
    """Proxy requests to vLLM with error handling."""
    async with httpx.AsyncClient(timeout=60) as client:
        try:
            resp = await client.post(
                f"{VLLM_BASE_URL}/v1/chat/completions",
                json={
                    "model":      request.model,
                    "messages":   request.messages,
                    "max_tokens": request.max_tokens,
                }
            )
            resp.raise_for_status()
            return resp.json()
        except httpx.TimeoutException:
            raise HTTPException(504, "Inference timeout")
        except httpx.HTTPStatusError as e:
            raise HTTPException(e.response.status_code, str(e))
```

## 5. Senior AI Engineer Perspective

### vLLM Is the Starting Point, Not the Only Option

vLLM is the right default. But understand its limitations:
- It's optimized for throughput; for latency-sensitive use cases, SGLang or custom optimization may win
- Multi-GPU setup requires careful configuration of tensor_parallel_size
- Very long context (>200K tokens) puts pressure on PagedAttention; investigate if you see memory issues

Start with vLLM and its defaults. Profile before optimizing.

### The OpenAI-Compatible API Is a Gift

Every major serving framework (vLLM, SGLang, TGI, Ollama) exposes an OpenAI-compatible API.
This means: your application code that calls `openai.chat.completions.create()` works unchanged
when you point `base_url` at your self-hosted server. This is the most important API design
decision in the LLM serving ecosystem.
