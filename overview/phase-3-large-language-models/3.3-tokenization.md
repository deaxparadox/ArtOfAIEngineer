## 1. What Is It? (Theory)

### The Translation Layer No One Talks About

Neural networks work with numbers. Text is characters. Tokenization is the translation layer between them.

But it's not as simple as "assign each character a number." That works for fixed alphabets, but
breaks on the diversity of human language: 150,000+ Unicode characters, technical terms, code
syntax, emojis, rare words, multiple scripts.

**The tokenizer's job:**
1. Split text into chunks called **tokens**
2. Map each chunk to an integer ID
3. (At output time) Map integer IDs back to text

**Why this layer is underestimated:**
Tokenization decisions have deep and non-obvious consequences for model behavior. Bugs that look
like "the model can't count" or "it's bad at math" are often tokenization problems in disguise.

### The Token Is Not the Word

This is the most important thing to internalize about tokens:

```
"tokenization" → ["token", "ization"]           — 2 tokens, not 1 word
"ChatGPT"      → ["Chat", "G", "PT"]            — 3 tokens, not 1 word
"2024"         → ["2024"]                       — 1 token (frequent number)
"1742839"      → ["174", "283", "9"]            — 3 tokens (rare number)
"hello"        → ["hello"]                      — 1 token
"Ġhello"       → ["hello"] with leading space   — same token, space embedded
"résumé"       → ["r", "é", "sum", "é"]         — 4 tokens (accented chars)
```

When you ask "how many tokens is my prompt?", it is **never** the word count. It's typically
1.3–1.5× the word count for typical English text — but can be much higher for code, non-English
text, or numbers.

## 2. How It Actually Works — Deep Mechanics

### 2.1 Byte Pair Encoding (BPE) — The Algorithm Behind GPT, Claude, LLaMA

BPE was originally a data compression algorithm (Gage 1994). Sennrich et al. (2016) adapted it
for neural machine translation. It's now the dominant tokenization algorithm for LLMs.

**The BPE algorithm:**

```
Training phase (learns the vocabulary from the corpus):

1. Start with all individual bytes as the initial vocabulary
   (256 possible values — handles ANY Unicode text)

2. Count all adjacent pairs in the corpus
   Example corpus: "low low lower lowest"
   Pairs: (l,o)=4, (o,w)=4, (w,' ')=2, (w,e)=2, (e,r)=1, (e,s)=1...

3. Merge the most frequent pair into a new token
   Most frequent: (l,o) → merge into "lo"
   New corpus:  "lo w lo w lo wer lo west"
   New vocab:   {l, o, w, e, r, s, t, lo, ...}

4. Repeat until vocabulary reaches target size (32K–256K)
   Next merge: (lo,w) → "low"
   Then: (low,' ') → "low " (with trailing space)
   And so on...

Encoding phase (tokenizing new text):
  Apply the learned merges in the SAME ORDER as training.
  "lower" → l,o,w,e,r → lo,w,e,r → low,e,r → lower (if in vocab) or low,er
```

**Why bytes and not characters?**

Byte-level BPE starts with 256 bytes, not Unicode characters. Every possible text is encodeable
without an "unknown token" — the worst case is falling back to individual bytes. This is how GPT-4's
`cl100k_base` tokenizer handles any Unicode: if a character isn't in the learned vocabulary, it
falls back to its UTF-8 bytes.

### 2.2 The Vocabulary

Modern tokenizer vocabulary sizes:

| Model | Tokenizer | Vocab Size |
|---|---|---|
| GPT-2 | `gpt2` | 50,257 |
| GPT-3.5 / GPT-4 | `cl100k_base` | 100,277 |
| GPT-4o | `o200k_base` | 200,019 |
| LLaMA 2 | `llama` (SentencePiece BPE) | 32,000 |
| LLaMA 3 | `llama3` (tiktoken BPE) | 128,256 |
| Gemma | `gemma` (SentencePiece BPE) | 256,000 |
| Claude | Anthropic internal | ~100K (estimated) |

Larger vocab = fewer tokens per text = faster inference + less context used, but larger embedding
table and harder to learn rare tokens.

### 2.3 Why Tokenization Causes Surprising Model Failures

**Failure 1: Letter counting is broken**

```
User: "How many 'r's are in 'strawberry'?"
Model: "There are 2 r's in strawberry."
Correct answer: 3

Why: "strawberry" → ["straw", "berry"] — 2 tokens
The model operates on tokens, not characters.
It can't "see" individual letters without extra reasoning.
```

**Failure 2: Arithmetic on unfamiliar numbers**

```
"123 + 456 = ?"      → model does fine (these numbers are common in training)
"1847293 + 3829471 = ?"  → model makes errors

Why: 1847293 → ["184", "72", "93"] — 3 tokens
Addition requires carrying across token boundaries, which attention doesn't naturally do.
```

**Failure 3: Non-English text is more expensive**

```
"Hello world" (English)  → ~2 tokens
"Bonjour monde" (French) → ~2 tokens
"你好世界" (Chinese)       → ~4–8 tokens (BPE trained on less Chinese text)
"مرحبا بالعالم" (Arabic) → ~8–12 tokens

For the same semantic content, non-English text uses 2–4× more tokens.
This means more context usage and higher cost.
```

**Failure 4: Code tokenization quirks**

```python
# Python indentation uses spaces
"    " (4 spaces) → might be ["    "] (1 token) or ["  ", "  "] (2 tokens)

# Depending on tokenizer, this can cause subtle inconsistencies in code generation
# because the model may not "see" indentation the same way as whitespace count
```

**Failure 5: Token boundary effects on generation**

Some words tokenize to the same prefix as different words. The model must predict the next token —
not the next character. This creates cases where stopping at a token boundary produces unexpected output.

### 2.4 Special Tokens

Every tokenizer has special tokens that signal structure:

| Token | Common ID | Meaning |
|---|---|---|
| `<|endoftext|>` | 50256 (GPT-2) | End of document during training |
| `<s>` | 1 (LLaMA) | Beginning of sequence |
| `</s>` | 2 (LLaMA) | End of sequence |
| `[INST]`, `[/INST]` | LLaMA 2 | Instruction markers |
| `<|im_start|>`, `<|im_end|>` | ChatML format | Message boundaries |
| `<|begin_of_text|>` | 128000 (LLaMA 3) | Start of text |
| `<|eot_id|>` | LLaMA 3 | End of turn |
| `<pad>` | varies | Padding for batch alignment |

These matter when you're building prompt templates or fine-tuning. Using the wrong special tokens
for a model causes garbled output because the model interprets them differently.

### 2.5 Chat Templates: How Tokenizers Format Conversations

Modern instruct models expect a specific format for multi-turn conversation. The tokenizer's
`apply_chat_template` method handles this:

```
LLaMA 3 format:
<|begin_of_text|>
<|start_header_id|>system<|end_header_id|>
You are a helpful assistant.<|eot_id|>
<|start_header_id|>user<|end_header_id|>
What is Python?<|eot_id|>
<|start_header_id|>assistant<|end_header_id|>
Python is a high-level programming language...

ChatML format (GPT-4 style):
<|im_start|>system
You are a helpful assistant.<|im_end|>
<|im_start|>user
What is Python?<|im_end|>
<|im_start|>assistant
Python is a high-level programming language...
```

Using the wrong chat template for a model is one of the most common fine-tuning bugs. The model was
trained to expect a specific format; if you feed it a different one, outputs are degraded.

## 3. Real-World Context

### The Token Count Affects Everything

| What It Affects | How |
|---|---|
| **Cost** | API pricing is per token (input + output) |
| **Latency** | More tokens = slower generation |
| **Context usage** | Long prompts eat into available context window |
| **Quality on long contexts** | Models degrade near the context limit |
| **Fine-tuning data prep** | Sequences too long must be truncated |
| **RAG chunk sizing** | Chunks sized in tokens, not words |

### Real Token Counting Examples

```
A typical tweet (280 chars)      → ~50–80 tokens
A GPT-4 API call with a          → ~500 tokens input
  medium system prompt + query
One page of dense text           → ~500–700 tokens
GPT-4's context limit (128K)     → ~96,000 words (~330 pages of text)
Claude Sonnet's context (200K)   → ~150,000 words (~500 pages)
```

### Production Impact of Tokenization

At Stripe-scale or ChatGPT-scale:
- Tokenization runs on every request, before the model even sees the input
- Tokenization is CPU-bound — fast (microseconds), but parallelism matters at scale
- Wrong tokenization = wrong input = wrong output; no error, just silent degradation

## 4. Code Examples

### Exploring Tokenization with tiktoken (OpenAI) and HF Tokenizers

```python
import tiktoken
from transformers import AutoTokenizer

# ===== OpenAI's tiktoken (GPT-4, Claude-compatible) =====
enc = tiktoken.get_encoding("cl100k_base")   # GPT-3.5 / GPT-4 tokenizer

text = "The tokenization of 'strawberry' is surprising to many users."
tokens = enc.encode(text)
print(f"Text:      {text}")
print(f"Token IDs: {tokens}")
print(f"Count:     {len(tokens)} tokens")

# Decode individual tokens to see the splits
print("\nToken breakdown:")
for token_id in tokens:
    print(f"  {token_id:6d} → '{enc.decode([token_id])}'")


# ===== Hugging Face tokenizer (LLaMA 3) =====
tokenizer = AutoTokenizer.from_pretrained("meta-llama/Meta-Llama-3-8B")

examples = [
    "hello",
    "Hello",               # capital H is different!
    "strawberry",
    "2024",
    "1847293",
    "你好世界",              # Chinese
    "def fibonacci(n):",   # code
    "résumé",              # accented characters
]

print("\n=== LLaMA 3 Tokenizer ===")
for text in examples:
    tokens = tokenizer.encode(text, add_special_tokens=False)
    pieces = [tokenizer.decode([t]) for t in tokens]
    print(f"'{text}' → {pieces} ({len(tokens)} tokens)")
```

### Counting Tokens Before an API Call (Cost Estimation)

```python
import tiktoken
from typing import list

def count_tokens_openai(messages: list[dict], model: str = "gpt-4o") -> int:
    """
    Count tokens for a chat completion request (matches OpenAI billing).
    Based on OpenAI's official token counting guide.
    """
    encoding = tiktoken.encoding_for_model(model)
    
    # Overhead per message (role + message structure)
    tokens_per_message = 3
    tokens_per_name = 1
    
    num_tokens = 0
    for message in messages:
        num_tokens += tokens_per_message
        for key, value in message.items():
            num_tokens += len(encoding.encode(str(value)))
            if key == "name":
                num_tokens += tokens_per_name
    
    num_tokens += 3  # every reply is primed with <|start|>assistant<|message|>
    return num_tokens

# Example: estimate cost before sending
messages = [
    {"role": "system", "content": "You are a helpful assistant specializing in Python."},
    {"role": "user", "content": "Explain list comprehensions with 3 examples."}
]

input_tokens = count_tokens_openai(messages, model="gpt-4o")
print(f"Input tokens:  {input_tokens}")
print(f"Estimated input cost (gpt-4o @ $2.50/1M): ${input_tokens * 2.50 / 1_000_000:.6f}")


def count_tokens_anthropic(text: str) -> int:
    """
    For Anthropic models — use the official token counting API.
    """
    import anthropic
    client = anthropic.Anthropic()
    
    response = client.messages.count_tokens(
        model="claude-sonnet-4-6",
        messages=[{"role": "user", "content": text}]
    )
    return response.input_tokens
```

### Chat Template Application

```python
from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("meta-llama/Meta-Llama-3-8B-Instruct")

messages = [
    {"role": "system", "content": "You are a helpful coding assistant."},
    {"role": "user",   "content": "How do I read a JSON file in Python?"},
]

# apply_chat_template formats the conversation correctly for this model
# This is what you MUST do when building prompts for instruct models
formatted = tokenizer.apply_chat_template(
    messages,
    tokenize=False,        # return string, not token IDs
    add_generation_prompt=True  # add the assistant prefix to trigger generation
)

print("=== Formatted prompt ===")
print(formatted)
print(f"\nToken count: {len(tokenizer.encode(formatted))}")

# The output shows the LLaMA 3 specific format:
# <|begin_of_text|><|start_header_id|>system<|end_header_id|>
# You are a helpful coding assistant.<|eot_id|>
# <|start_header_id|>user<|end_header_id|>
# How do I read a JSON file in Python?<|eot_id|>
# <|start_header_id|>assistant<|end_header_id|>
```

### Demonstrating the Strawberry Problem

```python
import tiktoken

enc = tiktoken.get_encoding("cl100k_base")

# Demonstrate why LLMs struggle with letter counting
word = "strawberry"
token_ids = enc.encode(word)
tokens = [enc.decode([t]) for t in token_ids]
print(f"Word: '{word}'")
print(f"Tokens: {tokens}")   # ['straw', 'berry']
print(f"Token count: {len(tokens)}")

# The model sees ['straw', 'berry'], not individual characters
# When asked "how many r's?", it must reason about characters WITHIN tokens
# This is harder than it looks — the model doesn't natively "see" subtoken characters

# Test some surprising tokenizations
surprises = [
    ("SolidGoldMagikarp", "Famous adversarial token that caused GPT-3 to hallucinate"),
    ("ChatGPT", "Not one token"),
    (" cat", "Leading space changes the token"),
    ("cat",   "Different from ' cat'"),
]

for text, note in surprises:
    tokens = [enc.decode([t]) for t in enc.encode(text)]
    print(f"\n'{text}' ({note}):")
    print(f"  → {tokens} ({len(tokens)} tokens)")
```

## 6. Alternatives & Comparisons

### BPE vs WordPiece vs Unigram

| Algorithm | Used By | Key Difference |
|---|---|---|
| **BPE** | GPT-2/3/4, LLaMA, Claude | Bottom-up: merges most frequent pairs |
| **WordPiece** | BERT, DistilBERT | Similar to BPE but uses likelihood not frequency |
| **Unigram** | T5, ALBERT, some multilingual | Top-down: starts large, removes least useful tokens |
| **Byte-level BPE** | GPT-2+, LLaMA 3 | BPE on bytes: handles all Unicode, no unknown tokens |
| **Character-level** | Some older models | Every char = 1 token; very long sequences |

**In practice:** For LLMs, byte-level BPE (or its SentencePiece equivalent) is standard. The
technical differences between BPE and WordPiece rarely matter for downstream tasks.

### Vocabulary Size Trade-offs

| Vocab Size | Tokens per Text | Embedding Table | Rare Token Quality |
|---|---|---|---|
| Small (32K) | More | Smaller | Fewer, better-learned tokens |
| Medium (100K) | Average | Medium | Good balance |
| Large (256K) | Fewer | Larger | More direct word-level tokens |

LLaMA 3 moving from 32K (LLaMA 2) to 128K vocabulary was a major quality improvement — more
multilingual text represented as single tokens instead of byte fallbacks.

## 7. Trade-offs

### Larger Vocabulary
**Pros:** Fewer tokens per text = faster inference, better multilingual coverage, whole words as tokens
**Cons:** Larger embedding matrix, rare tokens get less training signal, larger model size

### Smaller Vocabulary
**Pros:** Every token well-trained, smaller model, consistent coverage
**Cons:** More tokens per text = slower inference, worse multilingual handling

### Character-Level vs Subword
Character-level was popular before BPE. Subword (BPE) is better because:
- Characters have no semantic content on their own
- Subword tokens can carry meaning (e.g., "un-" prefix)
- Much shorter sequences = faster training and inference
- Better generalization to unseen words (shared subwords)

## 8. When To Use vs. When To Avoid

### Counting Tokens Before API Calls
✅ Always count before sending large prompts — API limits are in tokens, not words
✅ Count when estimating costs for production workloads
✅ Use when sizing RAG chunks (chunk in tokens, not characters)

### Understanding Tokenization When Debugging Model Behavior
✅ When model fails at seemingly simple tasks (counting letters, arithmetic on large numbers)
✅ When non-English content performs worse than expected
✅ When fine-tuning data appears corrupted (wrong special tokens)

⚠️ Common mistakes:
- Estimating costs in words instead of tokens (words × 1.3–1.5 for English, higher for non-English)
- Not using `apply_chat_template` when building prompts for instruct models
- Mixing tokenizers (counting tokens with tiktoken, then sending to a LLaMA model which uses a different tokenizer)
- Assuming "1 token ≈ 1 word" — it's 1 token ≈ 0.75 words for English, less for other languages
- Truncating at a character limit instead of a token limit in preprocessing

## 9. Senior AI Engineer Perspective

### The Arithmetic Failure Isn't Stupidity

When GPT-4 makes an arithmetic mistake on a large number, junior engineers often say "the model
is bad at math." The reality is more nuanced: the model is operating on tokens that don't map
cleanly to digits. `1847293` is `["184", "72", "93"]` — the model must carry across token
boundaries in a way that plain attention doesn't naturally support.

This is why tools (code interpreters, calculators) exist in AI systems. For exact computation,
use a tool. The model's job is to reason about which computation to do, not to execute it reliably.

### The Embedding Table Is 20–30% of Small Model Parameters

For a LLaMA 3 8B model with a 128K vocabulary and 4096-dimensional embeddings:
- Embedding table: 128,000 × 4,096 = ~524M parameters
- That's ~6.5% of an 8B model — significant

For a 1B model, the embedding table can be 50%+ of total parameters. This is why vocabulary
choice and weight tying (sharing embedding and LM head weights) matter so much at small model scales.

### Tokenization Is the Hidden Latency Cost

At a model serving company at scale:
- The transformer forward pass: 50–200ms
- Tokenization: 0.1–5ms
- Detokenization: 0.1–1ms

Sounds trivial. But at 10,000 requests/second with long prompts, tokenization CPU usage adds up.
`tiktoken` uses a Rust implementation specifically because Python tokenization was a bottleneck at
OpenAI's scale. If you're building a high-throughput serving system, profile tokenization separately.
