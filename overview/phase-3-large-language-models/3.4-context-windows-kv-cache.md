## 1. What Is It? (Theory)

### Context Window: What the Model Can "See"

The **context window** (also called context length or context limit) is the total number of tokens
a model can process in a single forward pass — the sum of prompt tokens plus generated tokens.

Think of it as the model's working memory. Everything inside the context window is "in view."
Everything outside doesn't exist to the model. It has no memory of previous conversations, no
awareness of what you asked last week, no knowledge of the 50,001st token of a 50,000-token document.

```
Context Window (e.g., 128K tokens):
┌────────────────────────────────────────────────────────────────────────┐
│ System Prompt | Conversation History | New User Message | Response...  │
│ ←─────────────────────── 128,000 tokens max ─────────────────────────→ │
└────────────────────────────────────────────────────────────────────────┘
                                                                          ↑
                                                                   Token 128,001:
                                                                   doesn't exist
```

### KV-Cache: Why Generation Gets Faster (Not Slower) as Context Grows

When generating token-by-token, a naive implementation would re-run the full attention computation
for all previous tokens at every step. Token 1000 would require processing tokens 1–999 from scratch.

**KV-cache** eliminates this redundancy by storing the key and value matrices from every previous
token, reusing them at each generation step.

```
Without KV-cache (naive):
  Generating token 1000:
    → Recompute attention over tokens 1–999 (999 forward passes)
    → Time grows quadratically with generated length

With KV-cache:
  Generating token 1000:
    → Q (query) for token 1000 = 1 new computation
    → K, V from tokens 1–999 = RETRIEVED FROM CACHE (already computed)
    → Attention = Q × cached K, V
    → Time is nearly constant per step
```

The KV-cache is what makes autoregressive generation practical at any meaningful length.

## 2. How It Actually Works — Deep Mechanics

### 2.1 The Two Phases of LLM Inference

**Prefill phase:** Process the entire prompt in one parallel pass. Compute and cache all K, V
matrices for every prompt token. This is fast because all prompt tokens are processed simultaneously.

```
Prompt: "Write a haiku about autumn."  (7 tokens)

Prefill:
  Token 1 ("Write"):   compute Q₁, K₁, V₁  → cache K₁, V₁
  Token 2 ("a"):       compute Q₂, K₂, V₂  → cache K₂, V₂
  ...all done in ONE parallel forward pass...
  Token 7 ("."):       compute Q₇, K₇, V₇  → cache K₇, V₇

KV cache now contains: [(K₁,V₁), (K₂,V₂), ..., (K₇,V₇)]
```

**Decode phase:** Generate tokens one at a time. Each step adds one token to the cache.

```
Decode step 1 (generate "Leaves"):
  → compute Q₈ for position 8
  → retrieve cached K₁...K₇, V₁...V₇
  → compute attention: Q₈ · [K₁...K₇]ᵀ → weights → weighted sum of [V₁...V₇]
  → pass through FFN → predict next token → "Leaves"
  → add K₈, V₈ to cache

Decode step 2 (generate "fall"):
  → compute Q₉ for position 9
  → retrieve cached K₁...K₈, V₁...V₈
  → ... and so on
```

### 2.2 KV-Cache Memory: The Real Constraint

KV-cache size grows linearly with context length. For each token, each layer, each attention head:

```
KV-cache size (bytes) = 
  seq_len × n_layers × 2 × n_heads × head_dim × bytes_per_element

For LLaMA 3-8B (FP16):
  n_layers = 32
  n_heads = 32
  head_dim = 128
  bytes = 2 (float16)

  Per token: 32 × 2 × 32 × 128 × 2 = 524,288 bytes = 0.5MB per token

  At 10,000 tokens: 0.5MB × 10,000 = 5GB of KV-cache
  At 128K tokens:   0.5MB × 128,000 = 64GB — exceeds an A100!
```

At 128K tokens, a 7B model's KV-cache (~64GB) exceeds a single A100 80GB GPU capacity.
This is the fundamental constraint behind context window limits in production.

**KV-cache memory ≫ model weights memory for long contexts.**

### 2.3 Context Window Sizes (2026)

| Model | Context | KV-cache at Full Context | Notes |
|---|---|---|---|
| GPT-3.5 Turbo | 16K | ~8GB (7B-scale equivalent) | Legacy |
| GPT-4o | 128K | Large | Flash Attention helps |

| Claude Opus 4.8 | 1M | Requires streaming/compression | Extreme long context |
| Gemini 2.5 Pro | 2M | Requires specialized infra | Largest context commercially |
| LLaMA 3-8B | 128K | ~64GB at full context | Requires multi-GPU |
| LLaMA 3-70B | 128K | Much larger | Requires many A100s |
| LLaMA 4 Scout | 10M | Research frontier | Specialized architecture |

### 2.4 The "Lost in the Middle" Problem

Having a large context window doesn't mean all of it is used equally well.

Research (Liu et al., 2023, "Lost in the Middle") showed that models perform best when relevant
information is at the **beginning** or **end** of the context, and worst when it's in the **middle**:

```
Performance by position of relevant info in long context:

100% ─────────────────────────────────────────────────────
      ↑                                                  ↑
      │  High perf at start                  High perf at end
 60%  │
      │              Low perf in middle
      │             ▼───────────────▼
 40%  └─────────────────────────────────────────────────
      Start         Middle                           End
                    of context
```

**Practical implication:** In RAG systems, put the most relevant chunks at the beginning or end
of the context. Don't bury the key information in a wall of text in the middle.

### 2.5 KV-Cache Optimization Techniques (2025–2026)

**PagedAttention (vLLM):**
KV-cache allocated in non-contiguous "pages" (like OS virtual memory) instead of one contiguous
block. Prevents fragmentation; allows serving many requests with different lengths simultaneously.
vLLM uses PagedAttention by default — it's why vLLM is ~10× more memory-efficient than naive serving.

**Quantized KV-cache:**
Store KV-cache in INT8 or even FP4 instead of FP16. Cuts memory by 2–4×, slight quality drop.
NVFP4 quantization (2025) cuts KV-cache to 25% of original size with < 1% accuracy loss.

**KV-cache eviction:**
For very long contexts, evict the KV pairs with lowest attention scores (they're rarely attended to).
Keeps the most important context, discards the least. ForesightKV (2025) predicts which tokens
will be needed later and preserves them.

**Prefix caching (prompt caching):**
If many requests share a common prefix (e.g., a long system prompt), cache the KV pairs for
that prefix once and reuse across requests. Anthropic, OpenAI, and Google all support this.

```
Request 1: [System Prompt (2000 tokens)] [User question A (20 tokens)]
Request 2: [System Prompt (2000 tokens)] [User question B (20 tokens)]
Request 3: [System Prompt (2000 tokens)] [User question C (30 tokens)]

Without prefix caching: Compute 2000+20 + 2000+20 + 2000+30 = 6070 tokens
With prefix caching:    Compute 2000 once + 20 + 20 + 30 = 2070 tokens
Cost savings: ~66%
```

## 3. Real-World Context

### Why Context Windows Grew So Fast

```
2020: GPT-3 — 4,096 tokens
2022: GPT-3.5 Turbo — 4,096 → 16,384 tokens
2023: GPT-4 — 8,192 → 32,768 → 128,000 tokens
2024: Claude 3 — 200,000 tokens
2024: Gemini 1.5 Pro — 1,000,000 tokens
2025: Claude 3.5/4 — 200,000 tokens
2025: Gemini 2.5 Pro — 2,000,000 tokens
2026: LLaMA 4 Scout — 10,000,000 tokens (research)
```

Three engineering breakthroughs made this possible:
1. **Flash Attention** (2022) — reduced attention memory from O(n²) to O(n)
2. **RoPE / ALiBi** positional encodings — generalize to longer sequences than trained
3. **KV-cache quantization and eviction** — fit longer contexts in available GPU memory

### Production Serving Considerations

**Time to first token (TTFT):** Dominated by prefill. Longer prompt → slower first token.
For interactive apps, keep prompts short or pre-fill asynchronously.

**Tokens per second (TPS):** Dominated by decode. Roughly constant per step with KV-cache.
But more concurrent users sharing a GPU = each user's TPS drops.

**Memory capacity:** Each additional concurrent request adds one KV-cache. At 128K context,
serving 4 simultaneous requests requires 4× the KV-cache memory.

### The Practical Context Budget

Don't use 100% of the context window in production. Rules of thumb:
- Use 60–70% max for reliable results (quality degrades near the limit)
- Reserve 10% for the model's output
- Leave headroom for longer-than-expected responses

## 4. Code Examples

### Counting Tokens and Estimating Context Usage

```python
import anthropic
import tiktoken

def analyze_context_usage(
    system_prompt: str,
    messages: list[dict],
    max_context: int = 200_000  # Claude Sonnet 4.6
) -> dict:
    """Estimate context usage before sending."""
    
    enc = tiktoken.get_encoding("cl100k_base")  # approximate for Claude
    
    system_tokens = len(enc.encode(system_prompt))
    message_tokens = sum(
        len(enc.encode(m["content"])) for m in messages
    )
    total = system_tokens + message_tokens
    
    return {
        "system_tokens":  system_tokens,
        "message_tokens": message_tokens,
        "total_tokens":   total,
        "context_used_pct": f"{100 * total / max_context:.1f}%",
        "tokens_remaining": max_context - total,
        "warning": total > max_context * 0.7  # alert at 70% usage
    }

# Example
system = "You are a helpful assistant that analyzes code for bugs."
messages = [
    {"role": "user", "content": "Review this 500-line Python module..."}
]

stats = analyze_context_usage(system, messages)
for k, v in stats.items():
    print(f"{k:25s}: {v}")
```

### Using Anthropic Prompt Caching (Reduces Cost at Long Contexts)

```python
import anthropic

client = anthropic.Anthropic()

# Long system prompt that repeats across many requests
# Use prompt caching to avoid re-processing it every time
LONG_SYSTEM = """
You are an expert code reviewer specializing in Python security vulnerabilities.
You look for: SQL injection, XSS, path traversal, insecure deserialization,
hardcoded credentials, weak cryptography, SSRF, and command injection.

[... 2000 more tokens of detailed guidelines ...]
"""

def review_code_with_caching(code: str) -> str:
    """
    The system prompt is cached after first use.
    Only new user messages are billed at full rate.
    """
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system=[
            {
                "type": "text",
                "text": LONG_SYSTEM,
                "cache_control": {"type": "ephemeral"}  # cache this block
            }
        ],
        messages=[
            {"role": "user", "content": f"Review this code:\n\n```python\n{code}\n```"}
        ]
    )
    
    # First call: full input tokens billed, cache written
    # Subsequent calls: cached tokens billed at ~10% of normal rate
    usage = response.usage
    print(f"Input tokens: {usage.input_tokens}")
    print(f"Cache write tokens: {getattr(usage, 'cache_creation_input_tokens', 0)}")
    print(f"Cache read tokens: {getattr(usage, 'cache_read_input_tokens', 0)}")
    
    return response.content[0].text

# Simulated multiple requests with same system prompt
code_snippet_1 = "password = 'admin123'"
code_snippet_2 = "query = f'SELECT * FROM users WHERE id={user_input}'"

result1 = review_code_with_caching(code_snippet_1)  # cache MISS — full cost
result2 = review_code_with_caching(code_snippet_2)  # cache HIT — ~90% cheaper
```

### Implementing Simple Rolling Context for Long Conversations

```python
import anthropic
from collections import deque

class RollingContextChat:
    """
    Maintains a conversation within a context budget by dropping oldest messages.
    Ensures system prompt and recent messages always fit.
    """
    
    def __init__(
        self,
        system_prompt: str,
        model: str = "claude-sonnet-4-6",
        max_context_tokens: int = 150_000,  # leave headroom in 200K window
        max_output_tokens: int = 4096
    ):
        self.client = anthropic.Anthropic()
        self.system_prompt = system_prompt
        self.model = model
        self.max_context = max_context_tokens
        self.max_output = max_output_tokens
        self.messages: deque = deque()
        
        # Reserve tokens for system prompt and output
        system_tokens = self._estimate_tokens(system_prompt)
        self.available_for_messages = max_context_tokens - system_tokens - max_output_tokens
    
    def _estimate_tokens(self, text: str) -> int:
        # Simple estimate: 1 token ≈ 4 characters
        return len(text) // 4
    
    def _total_message_tokens(self) -> int:
        return sum(self._estimate_tokens(m["content"]) for m in self.messages)
    
    def _trim_to_fit(self):
        """Drop oldest messages until we fit in context."""
        while self._total_message_tokens() > self.available_for_messages:
            if len(self.messages) <= 2:  # keep at least last exchange
                break
            self.messages.popleft()  # drop oldest user message
            if self.messages:
                self.messages.popleft()  # drop its corresponding assistant reply
    
    def chat(self, user_message: str) -> str:
        self.messages.append({"role": "user", "content": user_message})
        self._trim_to_fit()
        
        response = self.client.messages.create(
            model=self.model,
            max_tokens=self.max_output,
            system=self.system_prompt,
            messages=list(self.messages)
        )
        
        assistant_reply = response.content[0].text
        self.messages.append({"role": "assistant", "content": assistant_reply})
        
        print(f"[Context: ~{self._total_message_tokens():,} tokens used]")
        return assistant_reply


# Usage
chat = RollingContextChat(
    system_prompt="You are a Python expert. Be concise.",
    max_context_tokens=100_000
)

reply1 = chat.chat("What's the difference between list and tuple?")
print(reply1[:200])

reply2 = chat.chat("When should I use a generator instead?")
print(reply2[:200])
```

## 6. Alternatives & Comparisons

### Long Context vs RAG

When you have more information than fits in context, you have two options:

| Approach | Context Window | RAG |
|---|---|---|
| **How it works** | Put everything in the prompt | Retrieve only relevant chunks |
| **Best for** | Documents that must be fully analyzed | Large knowledge bases |
| **Cost** | High (long context = many tokens) | Lower (only relevant chunks billed) |
| **Quality** | Best if info fits well | Depends on retrieval quality |
| **Speed** | Slow for very long contexts | Fast (short context each call) |
| **Lost in middle** | Yes — quality degrades | Less; relevant info at front |

**Rule of thumb (2026):**
- < 50 pages and you need comprehensive analysis → long context
- Large knowledge base, targeted Q&A → RAG
- Both possible → hybrid (RAG + long context for top chunks)

### Sliding Window vs Full Attention

| Method | Memory | Quality | Use Case |
|---|---|---|---|
| Full attention | O(n²) | Best | Standard transformers |
| Flash Attention | O(n) memory, same compute | Same as full | All modern LLMs |
| Sliding window | O(n·w) | Miss long-range | Longformer; local attention |
| Sparse attention | O(n√n) | Good | BigBird; global tokens |
| Linear attention | O(n) | Approximate | Research; lower quality |

## 7. Trade-offs

### Larger Context Window
**Pros:** Process longer documents; longer conversations; better multi-document reasoning
**Cons:** Higher cost (more tokens billed), slower time-to-first-token, more GPU memory, "lost in middle" degrades quality

### KV-Cache Eviction
**Pros:** Fit longer contexts in fixed GPU memory
**Cons:** Evicted tokens can't be "remembered"; wrong eviction destroys relevant context

### Prompt Caching
**Pros:** Huge cost savings on repeated system prompts; faster TTFT for cached portions
**Cons:** Cache miss on first use; cache expires (5 min for Anthropic); only applies to identical prefix

## 8. When To Use vs. When To Avoid

### Full Context Window
✅ When: Analyzing a complete document, long code review, book summarization
✅ When: You need the model to reason across distant parts of the same document
❌ When: Simple Q&A where RAG suffices — don't pay for 200K tokens if 2K is enough
❌ When: Latency matters — prefill of long prompts adds seconds to TTFT

### Prompt Caching
✅ Always enable when: System prompt > 500 tokens and many requests per session
✅ When: Running evals over many inputs with the same system prompt
❌ When: Every request has a completely unique prefix (no cache hits)

⚠️ Common mistakes:
- Not enabling prompt caching with long system prompts (leaving 90% cost savings on the table)
- Using 100% of context window (quality degrades near the limit)
- Putting critical information in the middle of a long context (it gets "lost")
- Building with a 128K model then scaling to 1M context without profiling memory/cost impact
- Forgetting that KV-cache is per-request — high concurrency with long contexts exhausts GPU memory fast

## 9. Senior AI Engineer Perspective

### The Context Window Is Not Free Working Memory

Junior engineers treat the context window like RAM: "just put everything in there." This ignores:

1. **Cost:** 200K tokens at $3/M = $0.60 per request. At 1M requests/month = $600K/month.
2. **Quality:** The "lost in the middle" effect is real. Dense middle contexts are harder to attend to.
3. **Latency:** Prefill of 100K tokens takes 1–3 seconds. Users notice.

The engineering discipline is: **put only what the model needs for the current task in context.**
RAG, tool use, and conversation compression are context management strategies, not workarounds.

### KV-Cache Is the Bottleneck You Don't Know About

When your model serving starts slowing down and memory usage climbs, it's almost never the weights
(they're static). It's almost always the KV-cache growing faster than you anticipated.

At any serving scale: monitor KV-cache memory usage separately from model weight memory. They have
very different scaling behaviors — weights are fixed, KV-cache grows with load × context length.

### Why 10M Context Is Hard to Use Well

LLaMA 4 Scout supports 10M token context. In theory, you could put an entire codebase in context.
In practice:
- Time to first token at 10M tokens: minutes, not seconds
- Memory requirement: hundreds of GB of KV-cache
- Quality: "lost in the middle" at 10M tokens is severe

The frontier of long-context AI is not "put everything in context." It's building systems that
efficiently retrieve, compress, and present exactly the right information from large corpora —
while using the smallest context necessary for the task.
