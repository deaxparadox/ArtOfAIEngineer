## 1. What Is It? (Theory)

### The Limits of Fixed Pipelines

Naive and advanced RAG follow a fixed sequence:
```
Query → Retrieve(K chunks) → Generate(answer from chunks)
```

This pipeline has hard-coded decisions:
- How many chunks to retrieve? Fixed (e.g., K=5)
- Which knowledge base to search? Fixed
- Does the query need retrieval at all? Always retrieves
- Should it ask follow-up questions? Never

For simple Q&A, this is fine. For complex, multi-hop questions, it breaks:

```
"Compare our Q3 2024 revenue to Q3 2023, and explain the main drivers of change."

Fixed pipeline:
  → Retrieves 5 chunks (probably from one time period or mixes both)
  → LLM tries to answer with incomplete information
  → Gets it wrong or gives a partial answer

Agentic RAG:
  → Recognizes this is a multi-part question
  → Retrieves Q3 2024 revenue data
  → Retrieves Q3 2023 revenue data
  → Identifies gap: needs to know "drivers of change"
  → Retrieves relevant analysis documents
  → Synthesizes answer from all three retrievals
```

**Agentic RAG:** The model decides dynamically what to retrieve, when, and from where.
Retrieval is a tool the agent calls, not a fixed step in a pipeline.

### The Conceptual Shift

```
Fixed pipeline:           Query → RETRIEVAL → LLM → Answer
                                  (always, fixed K, one query)

Agentic RAG:              Query → LLM → "I need to retrieve X" → RETRIEVAL
                                    ↑                                   │
                                    └────────────────────────────────────┘
                                    (loop until answer or max iterations)
```

The LLM drives the process. It reads the query, decides if retrieval is needed, issues
targeted sub-queries, reads results, decides if more retrieval is needed, and synthesizes
the final answer.

## 2. How It Actually Works — Deep Mechanics

### 2.1 Retrieval as a Tool

In agentic RAG, retrieval is implemented as a tool the LLM can call:

```python
import anthropic
import numpy as np
from sentence_transformers import SentenceTransformer

client = anthropic.Anthropic()
embed  = SentenceTransformer("BAAI/bge-large-en-v1.5")

# Our knowledge base (in production: a Qdrant/pgvector collection)
KNOWLEDGE_BASE = [
    {"id": "1", "text": "Q3 2024 revenue was $2.1B, up 12% year-over-year.", "source": "q3_2024_report.pdf"},
    {"id": "2", "text": "Q3 2023 revenue was $1.875B.", "source": "q3_2023_report.pdf"},
    {"id": "3", "text": "Revenue growth was driven by product line expansion and international markets.", "source": "q3_2024_analysis.pdf"},
    {"id": "4", "text": "The company launched 3 new products in Q2 2024.", "source": "product_launch.pdf"},
    {"id": "5", "text": "Employee headcount grew 8% to 12,400 in 2024.", "source": "hr_report.pdf"},
]

kb_embeddings = embed.encode([d["text"] for d in KNOWLEDGE_BASE], normalize_embeddings=True)

def search_knowledge_base(query: str, k: int = 3) -> list[dict]:
    """Retrieval tool: search the knowledge base and return top-K results."""
    q_vec = embed.encode([query], normalize_embeddings=True)[0]
    scores = kb_embeddings @ q_vec
    top_k = np.argsort(-scores)[:k]
    return [
        {"content": KNOWLEDGE_BASE[i]["text"], "source": KNOWLEDGE_BASE[i]["source"], "score": float(scores[i])}
        for i in top_k
    ]

# Define retrieval as a tool the LLM can call
RETRIEVAL_TOOL = {
    "name": "search_knowledge_base",
    "description": "Search the company knowledge base for relevant information. "
                   "Use targeted, specific queries rather than the full user question. "
                   "Call this multiple times with different queries to gather complete information.",
    "input_schema": {
        "type": "object",
        "properties": {
            "query": {
                "type": "string",
                "description": "Specific search query. Use focused terms, not the full question."
            },
            "k": {
                "type": "integer",
                "description": "Number of results to retrieve (1-5)",
                "default": 3
            }
        },
        "required": ["query"]
    }
}
```

### 2.2 The Agentic Loop

```python
def agentic_rag(user_question: str, max_iterations: int = 5) -> dict:
    """
    Agentic RAG: the model dynamically decides what to retrieve and when.
    """
    messages = [{"role": "user", "content": user_question}]
    all_retrieved_contexts: list[dict] = []
    iteration = 0
    
    system = """You are a research assistant with access to a knowledge base.

To answer complex questions:
1. Break the question into sub-parts
2. Use search_knowledge_base for EACH sub-part with a targeted query
3. Read the results and identify what you still need to know
4. Search again if needed for missing information
5. Only generate a final answer when you have sufficient information

Cite sources using [source_name] for every factual claim."""
    
    while iteration < max_iterations:
        iteration += 1
        
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            system=system,
            tools=[RETRIEVAL_TOOL],
            messages=messages
        )
        
        # If model is done, return the answer
        if response.stop_reason == "end_turn":
            final_answer = next(
                (b.text for b in response.content if hasattr(b, "text")), ""
            )
            return {
                "answer":     final_answer,
                "iterations": iteration,
                "contexts":   all_retrieved_contexts,
            }
        
        # If model called a tool, execute retrieval
        if response.stop_reason == "tool_use":
            messages.append({"role": "assistant", "content": response.content})
            
            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    search_query = block.input["query"]
                    k = block.input.get("k", 3)
                    
                    print(f"  [Iteration {iteration}] Searching: '{search_query}'")
                    results = search_knowledge_base(search_query, k=k)
                    all_retrieved_contexts.extend(results)
                    
                    # Format results for the model
                    result_text = "\n".join(
                        f"- [{r['source']}] {r['content']}" for r in results
                    )
                    
                    tool_results.append({
                        "type":        "tool_result",
                        "tool_use_id": block.id,
                        "content":     result_text if results else "No relevant results found."
                    })
            
            messages.append({"role": "user", "content": tool_results})
    
    return {"answer": "Max iterations reached without complete answer.", "iterations": iteration}

# Test with multi-hop question
result = agentic_rag(
    "Compare Q3 2024 to Q3 2023 revenue and explain the main drivers of growth."
)
print(f"\nAnswer (after {result['iterations']} retrieval iterations):")
print(result["answer"])
print(f"\nRetrieved {len(result['contexts'])} context pieces total")
```

### 2.3 Multi-Source Routing

Agentic RAG shines when you have multiple knowledge sources and the model routes queries to
the right one:

```python
# Multiple retrieval tools for different knowledge sources
TOOLS = [
    {
        "name": "search_product_docs",
        "description": "Search product documentation, user guides, API references, and feature specs.",
        "input_schema": {"type": "object", "properties": {"query": {"type": "string"}}, "required": ["query"]}
    },
    {
        "name": "search_support_history",
        "description": "Search past customer support tickets and resolutions for similar issues.",
        "input_schema": {"type": "object", "properties": {"query": {"type": "string"}}, "required": ["query"]}
    },
    {
        "name": "search_policy_docs",
        "description": "Search company policies, SLAs, and legal agreements.",
        "input_schema": {"type": "object", "properties": {"query": {"type": "string"}}, "required": ["query"]}
    },
    {
        "name": "search_web",
        "description": "Search the public internet for current information. Use only for public data, not internal docs.",
        "input_schema": {"type": "object", "properties": {"query": {"type": "string"}}, "required": ["query"]}
    }
]

# The model automatically routes to the right knowledge source:
# "How do I configure the OAuth integration?" → search_product_docs
# "What's our SLA for enterprise customers?" → search_policy_docs
# "Other users reported similar login issues?" → search_support_history
# "Latest OAuth 2.0 best practices?" → search_web
```

### 2.4 Self-Questioning RAG

A pattern where the agent generates sub-questions from the original question and retrieves
for each:

```python
def self_ask_rag(question: str) -> dict:
    """
    Self-Ask RAG: explicitly generate sub-questions, retrieve per sub-question,
    then synthesize the final answer.
    """
    
    # Step 1: Decompose the question
    decomp = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=256,
        messages=[{"role": "user", "content": f"""Decompose this into 2-4 sub-questions needed to answer it.
Output only the sub-questions, one per line.

Question: {question}"""}]
    )
    sub_questions = [q.strip() for q in decomp.content[0].text.strip().split("\n") if q.strip()]
    print(f"Sub-questions: {sub_questions}")
    
    # Step 2: Retrieve for each sub-question
    all_context_parts = []
    for sq in sub_questions:
        results = search_knowledge_base(sq, k=3)
        context_for_sq = "\n".join(f"- {r['content']}" for r in results)
        all_context_parts.append(f"Sub-question: {sq}\nRelevant information:\n{context_for_sq}")
    
    combined_context = "\n\n".join(all_context_parts)
    
    # Step 3: Generate final answer from all gathered context
    answer = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system="Answer the question using the provided research. Cite sources.",
        messages=[{"role": "user", "content": f"Research gathered:\n{combined_context}\n\nOriginal question: {question}"}]
    )
    
    return {
        "answer":         answer.content[0].text,
        "sub_questions":  sub_questions,
        "context_parts":  all_context_parts,
    }
```

## 3. Real-World Context

### When Agentic RAG Is Worth the Complexity

**Use agentic RAG when your queries:**
- Require information from multiple documents or time periods
- Need to follow chains of references ("find all contracts that reference policy X")
- Are ambiguous and require clarification before retrieval
- Mix factual lookup with reasoning steps
- Need adaptive depth (sometimes 1 retrieval is enough, sometimes 5 are needed)

**Stick with fixed RAG when:**
- Most questions are single-hop (answer is in one chunk)
- Latency is critical (agentic adds multiple LLM call rounds)
- Cost is a constraint (each iteration costs tokens)
- Reliability matters more than capability (agents have higher failure rate)

### Real Production Examples

| Use Case | Why Agentic |
|---|---|
| Legal contract review | "What provisions in Exhibit A contradict section 3?" — multi-hop across document |
| Financial analysis | "Compare YoY performance for each product line" — multiple targeted retrievals |
| Technical debugging | "Trace this error through the system" — follow references across docs |
| Research synthesis | "Summarize what we know about competitor X" — multiple source consolidation |
| Compliance checking | "Does this document comply with all relevant regulations?" — multi-rule lookup |

### The Reliability Cost

Agentic systems have a higher failure rate than fixed pipelines:
- The model may retrieve irrelevant things
- The model may loop (retrieve the same thing repeatedly)
- Complex queries may exceed `max_iterations`
- Each additional tool call is an opportunity for an error

For high-stakes applications: prefer deterministic fixed pipelines with thorough validation.
For research/analysis tools: agentic RAG's capability justifies the reliability trade-off.

## 4. Code Examples

### Agentic RAG with Iteration Tracking

```python
import anthropic
from dataclasses import dataclass, field

client = anthropic.Anthropic()

@dataclass
class AgentTrace:
    question:    str
    iterations:  list[dict] = field(default_factory=list)
    final_answer: str = ""
    total_tokens: int = 0

def traced_agentic_rag(question: str) -> AgentTrace:
    """Agentic RAG with full trace for debugging."""
    trace = AgentTrace(question=question)
    messages = [{"role": "user", "content": question}]
    
    for i in range(6):   # max 6 iterations
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            system="Answer using search_knowledge_base. Search multiple times for complete info.",
            tools=[RETRIEVAL_TOOL],
            messages=messages
        )
        
        trace.total_tokens += response.usage.input_tokens + response.usage.output_tokens
        
        iteration_record = {
            "iteration": i + 1,
            "stop_reason": response.stop_reason,
            "tool_calls": [],
            "text": ""
        }
        
        if response.stop_reason == "end_turn":
            trace.final_answer = next(
                (b.text for b in response.content if hasattr(b, "text")), ""
            )
            iteration_record["text"] = trace.final_answer
            trace.iterations.append(iteration_record)
            break
        
        if response.stop_reason == "tool_use":
            messages.append({"role": "assistant", "content": response.content})
            tool_results = []
            
            for block in response.content:
                if block.type == "tool_use":
                    results = search_knowledge_base(block.input["query"])
                    result_text = "\n".join(f"- {r['content']}" for r in results)
                    
                    iteration_record["tool_calls"].append({
                        "query": block.input["query"],
                        "results_count": len(results),
                        "top_result": results[0]["content"][:100] if results else ""
                    })
                    
                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": result_text
                    })
            
            messages.append({"role": "user", "content": tool_results})
            trace.iterations.append(iteration_record)
    
    return trace

# Debug an agentic RAG query
trace = traced_agentic_rag(
    "What drove revenue growth from Q3 2023 to Q3 2024?"
)

print(f"Answer: {trace.final_answer}")
print(f"Iterations: {len(trace.iterations)}")
print(f"Total tokens: {trace.total_tokens}")
for it in trace.iterations:
    for tc in it.get("tool_calls", []):
        print(f"  Iter {it['iteration']}: searched '{tc['query']}'")
```

## 6. Trade-offs

### Agentic vs Fixed Pipeline

| Aspect | Fixed Pipeline | Agentic RAG |
|---|---|---|
| Latency | Predictable (1–2s) | Variable (2–10s) |
| Cost per query | Predictable (~$0.005) | Variable (up to ~$0.05) |
| Reliability | High | Lower (each step can fail) |
| Capability (simple Q) | Good | Good (but wasteful) |
| Capability (complex Q) | Limited | Excellent |
| Debuggability | Easy | Harder |

**Decision:** Default to fixed pipeline. Upgrade to agentic only when user feedback or evals
show that multi-hop or routing is needed.

## 7. When To Use vs. When To Avoid

### Agentic RAG
✅ Use when: Questions require synthesizing information from multiple documents
✅ Use when: You have multiple knowledge sources and need smart routing
✅ Use when: Questions vary widely in complexity (simple → complex)
❌ Avoid when: High-availability, low-latency requirements
❌ Avoid when: Most queries are single-hop (overkill)
❌ Avoid when: Cost per query is tightly constrained

⚠️ Common mistakes:
- Not setting a `max_iterations` limit — agents can loop indefinitely
- Giving the agent too many tools — analysis paralysis, wrong tool selection
- Not logging which tools are called — impossible to debug failures
- Using expensive models for auxiliary calls (use Haiku for sub-query generation, Sonnet for final answer)

## 8. Senior AI Engineer Perspective

### The Maximum Iterations Trap

Every production agentic system needs a hard iteration limit. Without it:
- A confused agent re-retrieves the same thing in a loop
- A complex question exhausts context with repeated retrievals
- Costs explode unexpectedly

Set `max_iterations = 5–7` for production. If the agent can't answer in 7 iterations, fail
gracefully rather than letting it run indefinitely.

### Agentic RAG Is Not Smarter — It's More Adaptive

A common misconception: agentic RAG produces better answers because the model is "smarter."
It's not. The same model is used. Agentic RAG produces better answers for complex questions
because it can gather more targeted information before generating.

For questions where one well-placed retrieval gives all needed information, agentic RAG adds
latency and cost with no quality benefit. The intelligence is in using it selectively.
