## 1. What Is It? (Theory)

### The Field Guide to RAG Failures

RAG systems fail in predictable, repeatable ways. Once you've seen enough production RAG systems,
you recognize the patterns immediately. This section is a diagnostic guide: describe the symptom,
identify the root cause, apply the targeted fix.

**The core finding:** 73% of RAG failures are retrieval failures, not generation failures.
Fix retrieval first. Only then optimize generation.

The failure modes are organized into two categories:
- **Retrieval failures** (the right content wasn't found)
- **Generation failures** (the right content was found but the answer is wrong)

## 2. Retrieval Failure Modes

### Failure 1: The Semantic Gap (Most Common)

**Symptom:** User asks "How do I fix a login issue?" but the relevant document says
"Troubleshooting authentication errors." The vector search returns unrelated content.

**Root cause:** The query and document don't share vocabulary. The embedding model finds
"login" and "authentication" related but the specific phrasing gaps are large enough to
miss the relevant document.

**Diagnosis:** Low context recall. Relevant documents exist in the index but don't appear
in top-K results.

**Fixes:**
```python
# Fix 1: Query expansion — generate multiple phrasings
variants = generate_query_variants("How do I fix a login issue?")
# → ["login troubleshooting", "authentication error resolution", 
#    "sign-in problem fix", "access issues debugging"]
# Retrieve for all, merge with RRF

# Fix 2: Hybrid search — BM25 catches "authentication" as keyword
# even if semantic similarity is lower

# Fix 3: Better embedding model — Voyage 3-large understands more domain synonyms
# Switch from all-MiniLM to BAAI/bge-large or voyage-3-large

# Fix 4: Keyword boosting — explicitly add synonyms to the query
enhanced_query = f"{query} {' '.join(domain_synonyms[query_topic])}"
```

### Failure 2: The Chunk Boundary Problem

**Symptom:** User asks "What is the refund timeline?" but the answer is split across two chunks:
- Chunk 1: "...our return policy covers 30 days. Refunds are..."
- Chunk 2: "...processed within 5-7 business days to the original payment method."

Only Chunk 1 is retrieved. The answer is incomplete.

**Root cause:** The chunk boundary splits a semantically connected piece of information.

**Fixes:**
```python
# Fix 1: Increase chunk overlap
splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,
    chunk_overlap=128,  # Increase from 64 → 128 tokens overlap
)

# Fix 2: Parent-child retrieval — retrieve the parent (larger) chunk
# that contains both child chunks

# Fix 3: Retrieve adjacent chunks when the top-1 chunk has a high score
def retrieve_with_neighbors(
    query: str,
    top_chunk: dict,
    all_chunks: list[dict],
    neighbors: int = 1
) -> list[dict]:
    idx = next(i for i, c in enumerate(all_chunks) if c["chunk_id"] == top_chunk["chunk_id"])
    start = max(0, idx - neighbors)
    end   = min(len(all_chunks), idx + neighbors + 1)
    return all_chunks[start:end]
```

### Failure 3: The Exact Identifier Miss

**Symptom:** User searches "error code 404 in payment gateway" but the document says
"HTTP 404 status code during checkout." The semantic search returns documents about
"payment errors" in general, not the specific 404 case.

**Root cause:** Exact technical identifiers (error codes, product IDs, version numbers)
aren't captured well by dense embeddings.

**Fixes:**
```python
# Fix 1: Hybrid search — BM25 catches exact terms
# This is the single biggest fix for exact identifier misses

# Fix 2: Metadata filtering — index error codes as structured metadata
chunk_payload = {
    "text": "HTTP 404 during checkout...",
    "error_codes": ["404"],        # extracted at indexing time
    "components": ["payment"],
}

# Then filter by error code at query time:
results = qdrant.search(
    collection_name="docs",
    query_vector=q_vec,
    query_filter=Filter(
        should=[FieldCondition(key="error_codes", match=MatchValue(value="404"))]
    ),
    limit=5
)

# Fix 3: Query preprocessing — expand "error code 404" to include
# "HTTP 404", "404 Not Found", "404 status" etc.
```

### Failure 4: The Stale Knowledge Problem

**Symptom:** The RAG system answers with outdated information. "Our product doesn't support
feature X" when feature X was added 3 months ago.

**Root cause:** The knowledge base hasn't been updated since the feature was released.

**Fixes:**
```python
# Fix 1: Regular re-indexing pipeline
import schedule

def reindex_changed_documents():
    """Find documents modified since last index, re-chunk, re-embed, upsert."""
    changed_docs = document_store.find_updated_since(last_index_time)
    for doc in changed_docs:
        chunks = chunk_document(doc.text, doc.id)
        embeddings = embed_model.encode([c.text for c in chunks])
        qdrant.upsert(collection_name=COLLECTION, points=[...])

# Schedule daily re-indexing
schedule.every().day.at("02:00").do(reindex_changed_documents)

# Fix 2: Index metadata with timestamps, filter by recency
results = qdrant.search(
    collection_name="docs",
    query_vector=q_vec,
    query_filter=Filter(
        must=[FieldCondition(key="updated_at", range=Range(gte=thirty_days_ago))]
    ),
)

# Fix 3: Add "last updated" dates to documents in the index
# and include them in the LLM context:
context_item = f"[Source: {source} | Updated: {updated_date}]\n{text}"
```

### Failure 5: The Multi-Document Answer Problem

**Symptom:** "What's the total cost of product A + product B + installation?" requires
information from three separate documents. Top-5 retrieval returns 3 chunks about product A
and 2 irrelevant chunks.

**Root cause:** Single-query retrieval optimizes for relevance to the query, which may favor
one topic. The query covers multiple topics that need separate retrieval.

**Fixes:**
```python
# Fix 1: Agentic RAG — let the model retrieve each component separately
# (Section 7.4)

# Fix 2: Self-Ask decomposition
def self_ask_retrieve(complex_query: str) -> list[dict]:
    # Decompose: ["price of product A?", "price of product B?", "installation cost?"]
    sub_queries = decompose_query(complex_query)
    
    all_chunks = []
    for sq in sub_queries:
        chunks = retrieve(sq, k=3)
        all_chunks.extend(chunks)
    
    return deduplicate(all_chunks)

# Fix 3: Increase top-K to cast a wider net
# (but compensate with reranking to maintain precision)
results = retrieve(query, k=15)  # cast wide
results = rerank(query, results, final_k=5)  # then narrow precisely
```

## 3. Generation Failure Modes

### Failure 6: The Faithful Ignorance (Context Not Used)

**Symptom:** Correct chunks are retrieved but the LLM generates an answer from its training
memory instead of the provided context. The chunks say one thing; the answer says another.

**Root cause:**
1. System prompt grounding instructions are weak or ambiguous
2. Training memory is very strong for the topic — overrides provided context
3. Context is too long — relevant chunks are "lost in the middle"

**Fixes:**
```python
# Fix 1: Stronger grounding instruction
STRICT_SYSTEM = """You must ONLY use the provided [Source N] documents.
Do not use any information from your training.
If the answer is not in the sources, say exactly: "Not in provided sources."
Every factual claim must have a [Source N] citation. No exceptions."""

# Fix 2: Assistant prefill — start the response with a citation
messages.append({"role": "assistant", "content": "Based on the provided sources: [Source 1]"})

# Fix 3: Ask the model to verify after generating
verification_prompt = f"""Review your answer and check each claim:
Is it supported by the provided sources? If not, remove it.

Original answer: {answer}
Sources: {context}

Revised, fully-grounded answer:"""

# Fix 4: Reduce context noise — use contextual compression to
# remove irrelevant parts of each chunk before sending to LLM
```

### Failure 7: The Hallucination Despite Context

**Symptom:** The correct chunk is retrieved and in context, but the LLM adds information
not in the chunk. "The refund takes 5-7 business days and can be expedited for $10."
(The "$10 expedite" is invented — only the 5-7 days is in the context.)

**Root cause:** The LLM's completion tendency fills in plausible details that aren't actually
in the context. Common with sycophantic or "helpfully elaborating" behavior.

**Fixes:**
```python
# Fix 1: Claim extraction + verification
def verify_answer(answer: str, context: str) -> dict:
    """Check every claim in the answer against the context."""
    claims = extract_claims(answer)       # split answer into atomic claims
    for claim in claims:
        if not is_supported_by(claim, context):
            # Flag or remove the unsupported claim
            pass

# Fix 2: Temperature = 0 for factual Q&A
# Higher temperature → more creative → more hallucination
response = client.messages.create(
    model="claude-sonnet-4-6",
    temperature=0,   # deterministic; less confabulation
    ...
)

# Fix 3: Citation format forces the model to think about support
SYSTEM = """Format: 
For each sentence in your answer: [sentence] [Source N].
If you cannot cite a source, omit the information."""

# Fix 4: Self-check post-generation
# Ask the model to flag any claims it's not sure are in the sources
```

### Failure 8: The Contradictory Sources Problem

**Symptom:** Retrieved chunks contradict each other. Version 1.0 docs say "timeout is 30 seconds";
version 2.0 docs say "timeout is 60 seconds." The answer averages them or picks one arbitrarily.

**Root cause:** Multiple document versions or contradictory sources in the index.

**Fixes:**
```python
# Fix 1: Version metadata + recency-weighted retrieval
# Prefer more recent documents for configuration/fact queries
results = qdrant.search(
    query_vector=q_vec,
    query_filter=Filter(
        must=[FieldCondition(key="doc_version", range=Range(gte=2.0))]
    ),
)

# Fix 2: Explicit contradiction handling in the system prompt
SYSTEM = """If sources contradict each other, explicitly note it:
'Source 1 (v1.0, 2023) states X, while Source 3 (v2.0, 2024) states Y.
The more recent version states Y.'"""

# Fix 3: Source deduplication — don't index multiple versions of the same doc
# Only keep the most recent version in the index

# Fix 4: Date-aware indexing
def should_supersede(old_chunk: dict, new_chunk: dict) -> bool:
    """If new doc is an update of the old, remove old chunk."""
    return (new_chunk["doc_id"] == old_chunk["doc_id"] and
            new_chunk["version"] > old_chunk["version"])
```

### Failure 9: The Long Context Quality Degradation

**Symptom:** RAG works well with 3 chunks. When increased to 10 chunks for more coverage,
answer quality drops. The model uses chunks 1 and 10 but ignores chunks 5-8.

**Root cause:** "Lost in the middle" phenomenon — attention quality degrades for content
in the middle of a long context.

**Fixes:**
```python
# Fix 1: Put the most relevant chunks at the START of context (not middle)
def build_context_ranked_first(chunks: list[dict], query: str) -> str:
    # Sort by relevance score (highest first = start of context)
    sorted_chunks = sorted(chunks, key=lambda x: x.get("score", 0), reverse=True)
    return "\n\n---\n\n".join(
        f"[Source {i+1} — Relevance: {c.get('score', 0):.2f}]\n{c['text']}"
        for i, c in enumerate(sorted_chunks)
    )

# Fix 2: Reduce K — fewer, better chunks > more, noisier chunks
# Start with K=3 before increasing

# Fix 3: Contextual compression — distill each chunk to only the
# query-relevant sentences before building context

# Fix 4: Use a model with better long-context handling
# Claude with extended context, Gemini 2.5 Pro (best long-context)
```

### Failure 10: The Multi-Turn Context Drift

**Symptom:** In a multi-turn conversation, the RAG system stops using the knowledge base
and starts answering from memory. "Earlier you said X, but now you're saying Y."

**Root cause:** As the conversation grows, the conversation history dominates the context.
The retrieved chunks get "pushed out" or the model prioritizes conversational continuity
over document grounding.

**Fixes:**
```python
# Fix 1: Re-retrieve on every turn (not just the first)
def multi_turn_rag(conversation_history: list[dict], new_user_message: str) -> str:
    # Don't use only the initial question for retrieval
    # Use the latest message (+ conversation summary if needed)
    retrieval_query = synthesize_retrieval_query(conversation_history, new_user_message)
    chunks = retrieve(retrieval_query, k=5)
    
    # Always include fresh retrieved context, not just conversation history
    context = build_context(chunks)
    
    # System prompt always references the current sources
    system = f"""You are answering based on these current sources:
{context}

Use conversation history for context but answer FROM the sources, not from memory."""
    
    return generate(system, conversation_history + [{"role": "user", "content": new_user_message}])

# Fix 2: Conversation compression — summarize old turns to save space for context
# Fix 3: Conversation reset threshold — after N turns, start fresh context
```

## 4. The Diagnostic Decision Tree

```
User reports: "The RAG answer is wrong"
                │
                ▼
Was the correct information in the retrieved chunks?
  NO  → Retrieval failure
        ├─ Low vocabulary overlap → Failure 1 (Semantic Gap) → Hybrid search, query expansion
        ├─ Answer spans chunks → Failure 2 (Chunk Boundary) → More overlap, parent-child
        ├─ Exact identifier missing → Failure 3 (Identifier Miss) → Hybrid search, metadata
        ├─ Information is new → Failure 4 (Stale Knowledge) → Re-indexing pipeline
        └─ Question is multi-part → Failure 5 (Multi-Document) → Agentic RAG, self-ask
  
  YES → Generation failure
        ├─ LLM ignores context → Failure 6 (Context Not Used) → Stronger grounding prompt
        ├─ LLM adds extra info → Failure 7 (Hallucination) → Claim verification, temp=0
        ├─ Sources conflict → Failure 8 (Contradictions) → Version metadata, dedup
        ├─ Context too long → Failure 9 (Lost in Middle) → Reduce K, rank-first context
        └─ Multi-turn drift → Failure 10 (Context Drift) → Re-retrieve each turn
```

## 5. Code Examples

### Automated Failure Mode Detection

```python
import anthropic
import json

client = anthropic.Anthropic()

def diagnose_rag_failure(
    question: str,
    retrieved_chunks: list[dict],
    generated_answer: str,
    ground_truth: str,
) -> dict:
    """
    Use an LLM to classify which failure mode is present.
    Useful for automated triage of eval failures.
    """
    context = "\n".join(c["text"] for c in retrieved_chunks)
    
    response = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=256,
        messages=[{"role": "user", "content": f"""Analyze this RAG failure:

Question: {question}
Ground truth answer: {ground_truth}
Retrieved context: {context[:500]}...
Generated answer: {generated_answer}

Classify the primary failure mode:
1. Retrieval miss: relevant info not in retrieved chunks
2. Chunk boundary: answer spans two chunks
3. Identifier miss: exact term/code not retrieved
4. Stale data: info is outdated
5. Context ignored: correct chunks retrieved but not used
6. Hallucination: info added beyond what's in context
7. Contradiction: conflicting sources present
8. Context lost: relevant chunk in context but overwhelmed by noise

Output JSON: {{"failure_mode": N, "explanation": "...", "suggested_fix": "..."}}"""}]
    )
    
    try:
        return json.loads(response.content[0].text)
    except json.JSONDecodeError:
        return {"failure_mode": 0, "explanation": response.content[0].text}

# Triage eval failures automatically
for result in eval_failures:
    diagnosis = diagnose_rag_failure(
        result["question"],
        result["retrieved_chunks"],
        result["answer"],
        result["ground_truth"]
    )
    print(f"Failure mode {diagnosis['failure_mode']}: {diagnosis['explanation']}")
```

## 6. The Failure Mode → Fix Mapping

| Failure Mode | Primary Fix | Secondary Fix |
|---|---|---|
| Semantic gap | Hybrid search + query expansion | Better embedding model |
| Chunk boundary | Parent-child retrieval + overlap | Sentence-aware chunking |
| Identifier miss | Hybrid search | Metadata extraction + filtering |
| Stale knowledge | Automated re-indexing | Timestamp filtering |
| Multi-document | Agentic RAG | Self-ask decomposition |
| Context ignored | Stronger grounding prompt | Assertion prefill |
| Hallucination | Claim verification | temperature=0 |
| Contradictions | Version metadata | Source deduplication |
| Lost in middle | Rank-first context | Smaller K + reranking |
| Multi-turn drift | Re-retrieve each turn | Context compression |

---

## 7. The Priority Ordering

When multiple failure modes are present, fix in this order:

```
1. Retrieval recall < 0.75
   Fix retrieval first. Nothing else matters until retrieval works.

2. Context precision < 0.70
   Reduce noise. Too many irrelevant chunks contaminate generation.

3. Faithfulness < 0.85
   Fix generation. Model is not using the context correctly.

4. Answer relevancy < 0.80
   Fix both. Usually indicates both retrieval and generation issues.
```
