## 1. What Is It? (Theory)

### Why Most Teams Skip This (And Why That's Catastrophic)

Evaluation is the most commonly skipped step in RAG development. Teams:
1. Build a RAG pipeline
2. Test it manually on 5–10 examples
3. "Looks good!" → ship to production
4. Receive user complaints 3 weeks later

The problem: 5–10 manual examples don't cover the distribution of real queries. A pipeline
can look excellent on cherry-picked examples while failing on 30% of real user queries.

**Without evals, you're flying blind.** When you change the embedding model, chunk size, or
retrieval strategy, you have no way to know if it got better or worse. Every optimization is
guess-and-check.

With evals: you have a measurement system. Every change produces a number. You know when
you're improving.

### The Four Things You Need to Measure

```
RAG pipeline:
  Query → [RETRIEVAL] → [GENERATION] → Answer

Retrieval metrics (is the right content being found?):
  1. Context Recall:    Is the information needed to answer the query in the retrieved chunks?
  2. Context Precision: Are the retrieved chunks actually relevant to the query?

Generation metrics (is the LLM using the context correctly?):
  3. Answer Faithfulness:  Does the answer make only claims supported by the retrieved chunks?
  4. Answer Relevance:     Does the answer actually address the user's question?
```

This 2×2 matrix covers every major RAG failure mode. More on this in Section 7.6.

## 2. How It Actually Works — Deep Mechanics

### 2.1 RAGAS: The Standard RAG Evaluation Framework

RAGAS (RAG Assessment) is the most widely-used framework for automated RAG evaluation.
It uses an LLM-as-judge approach to score each metric.

```python
# pip install ragas datasets
from ragas import evaluate
from ragas.metrics import (
    context_recall,
    context_precision,
    faithfulness,
    answer_relevancy,
)
from datasets import Dataset
import anthropic
from langchain_anthropic import ChatAnthropic
from langchain_voyageai import VoyageAIEmbeddings

# RAGAS uses LangChain wrappers internally
llm_judge    = ChatAnthropic(model="claude-sonnet-4-6")
embed_model  = VoyageAIEmbeddings(model="voyage-3-large")

# Build an evaluation dataset
# Format: question, answer, contexts (retrieved chunks), ground_truth (reference answer)
eval_data = {
    "question": [
        "What is our return policy for sale items?",
        "How long does standard shipping take?",
    ],
    "answer": [
        # What your RAG system actually answered
        "Sale items are final sale and cannot be returned. [Source 1]",
        "Standard shipping takes 5-7 business days. Express shipping takes 2 days.",
    ],
    "contexts": [
        # What your RAG system retrieved
        [
            "Sale items are final sale and cannot be returned or exchanged.",
            "Our return policy allows returns within 30 days of purchase.",
        ],
        [
            "Standard shipping takes 5-7 business days.",
            "Express shipping (2-day) is available for an extra $15.",
        ],
    ],
    "ground_truth": [
        # The reference answer (what a human expert would say)
        "Sale items are final sale and cannot be returned.",
        "Standard shipping takes 5-7 business days.",
    ],
}

dataset = Dataset.from_dict(eval_data)

# Run RAGAS evaluation
results = evaluate(
    dataset=dataset,
    metrics=[
        context_recall,       # requires ground_truth
        context_precision,    # requires ground_truth
        faithfulness,         # compares answer to contexts
        answer_relevancy,     # compares answer to question
    ],
    llm=llm_judge,
    embeddings=embed_model,
)

print(results)
# Output:
# context_recall:    0.95   (how much of the needed info was retrieved)
# context_precision: 0.78   (how relevant were the retrieved chunks)
# faithfulness:      0.92   (how grounded in context was the answer)
# answer_relevancy:  0.89   (did the answer address the question)
```

### 2.2 RAGAS Metrics Explained

**Context Recall (0–1):**

"Did the retrieved chunks contain the information needed to answer the question?"

```
Ground truth: "Sale items are final sale and cannot be returned."
Retrieved:    ["Sale items are final sale.", "Returns are allowed within 30 days."]

RAGAS breaks ground truth into claims:
  Claim 1: "Sale items are final sale" → found in chunk 1 ✓
  Claim 2: "Cannot be returned" → found in chunk 1 ✓

Context recall = 2/2 = 1.0 (both claims are covered)

If relevant chunk was not retrieved: context recall would be lower
→ This tells you your RETRIEVAL is missing important information
```

**Context Precision (0–1):**

"What fraction of the retrieved chunks are actually relevant?"

```
Retrieved: [
  "Sale items are final sale.",            ← relevant ✓
  "Shipping takes 5-7 days.",              ← NOT relevant to return policy question ✗
  "Returns processed in 5-7 business days" ← somewhat relevant ✓
]

Context precision ≈ 2/3 = 0.67
→ If low: you're retrieving too much noise; better filtering needed
```

**Answer Faithfulness (0–1):**

"Does the generated answer only make claims that are in the retrieved context?"

```
Context:  "Sale items are final sale."
Answer:   "Sale items cannot be returned, and you must pay a 15% restocking fee."
                                                     ↑ NOT in context — hallucinated!

Faithfulness: RAGAS extracts claims from the answer, checks each against context
  Claim 1: "Sale items cannot be returned" → supported ✓
  Claim 2: "15% restocking fee" → NOT in context ✗

Faithfulness = 1/2 = 0.5
→ If low: your LLM is hallucinating or ignoring context
```

**Answer Relevancy (0–1):**

"Does the answer actually address the question asked?"

```
Question: "What is the return policy for sale items?"
Answer:   "Our shipping times are 5-7 days for standard."
→ Not relevant at all → Answer relevancy ≈ 0.1

Answer: "Sale items are final sale."
→ Directly answers the question → Answer relevancy ≈ 0.95
```

### 2.3 Building Your Eval Dataset

The eval dataset is everything. Garbage eval data = garbage metrics = false confidence.

```python
from dataclasses import dataclass
from typing import Optional
import json

@dataclass
class EvalCase:
    question:           str
    ground_truth:       str         # reference answer from domain expert
    relevant_doc_ids:   list[str]   # which documents should be retrieved
    category:           str         # type of question (factual, multi-hop, etc.)
    difficulty:         str         # easy/medium/hard
    notes:              Optional[str] = None

def build_eval_dataset_from_documents(
    documents: list[dict],
    n_cases: int = 50,
) -> list[EvalCase]:
    """
    Use an LLM to generate eval cases from your actual documents.
    Then have domain experts review and correct them.
    """
    cases = []
    import anthropic
    
    client = anthropic.Anthropic()
    
    for doc in documents[:n_cases]:
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=512,
            messages=[{"role": "user", "content": f"""Given this document excerpt, generate ONE evaluation question and answer:

Document:
{doc['text']}

Output as JSON:
{{
  "question": "A specific question that can be answered from this document",
  "ground_truth": "The precise answer from the document",
  "difficulty": "easy|medium|hard",
  "category": "factual|procedural|comparative|inferential"
}}"""}]
        )
        
        try:
            data = json.loads(response.content[0].text)
            cases.append(EvalCase(
                question         = data["question"],
                ground_truth     = data["ground_truth"],
                relevant_doc_ids = [doc.get("id", "")],
                category         = data.get("category", "factual"),
                difficulty       = data.get("difficulty", "medium"),
            ))
        except (json.JSONDecodeError, KeyError):
            pass  # Skip malformed responses
    
    return cases

# After generating: ALWAYS have domain experts review
# Remove cases with wrong ground truth
# Add edge cases and failure scenarios manually
# Ensure coverage of all document types and question categories
```

### 2.4 Retrieval-Specific Metrics

Before measuring end-to-end quality, measure retrieval alone:

```python
import numpy as np
from sentence_transformers import SentenceTransformer

embed = SentenceTransformer("BAAI/bge-large-en-v1.5")

def evaluate_retrieval(
    eval_cases: list[EvalCase],
    retrieve_fn,   # callable: (query: str) -> list[dict]
    k: int = 5,
) -> dict:
    """
    Measures retrieval quality independently from generation.
    If retrieval is poor, no prompt engineering will save your RAG.
    """
    recall_at_k_scores = []
    mrr_scores         = []
    
    for case in eval_cases:
        retrieved = retrieve_fn(case.question)
        retrieved_ids = [r.get("id") or r.get("source") for r in retrieved[:k]]
        
        # Recall@K: are the relevant docs in the top-K?
        hits = sum(1 for doc_id in case.relevant_doc_ids if doc_id in retrieved_ids)
        recall = hits / len(case.relevant_doc_ids) if case.relevant_doc_ids else 0
        recall_at_k_scores.append(recall)
        
        # MRR: rank of the first relevant doc
        for rank, doc_id in enumerate(retrieved_ids, start=1):
            if doc_id in case.relevant_doc_ids:
                mrr_scores.append(1.0 / rank)
                break
        else:
            mrr_scores.append(0.0)
    
    return {
        f"Recall@{k}":   round(np.mean(recall_at_k_scores), 3),
        "MRR":           round(np.mean(mrr_scores), 3),
        "n_eval_cases":  len(eval_cases),
    }

# Run retrieval eval
from functools import partial
retrieve = partial(search_knowledge_base, k=5)

metrics = evaluate_retrieval(eval_cases, retrieve_fn=lambda q: search_knowledge_base(q, k=5))
print(metrics)
# {"Recall@5": 0.82, "MRR": 0.71, "n_eval_cases": 50}
# If Recall@5 < 0.75: fix retrieval before touching generation
```

### 2.5 LLM-as-Judge for Faithfulness

A lightweight alternative to RAGAS for faithfulness checking:

```python
import anthropic
import re

client = anthropic.Anthropic()

def check_faithfulness(
    answer: str,
    context: str,
    threshold: float = 0.8,
) -> dict:
    """
    Use an LLM judge to assess if every claim in the answer is supported by context.
    Cheaper and faster than RAGAS for rapid iteration.
    """
    response = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=512,
        messages=[{"role": "user", "content": f"""Evaluate if the answer is fully grounded in the provided context.

Context:
{context}

Answer:
{answer}

For each factual claim in the answer, check if it's explicitly stated in the context.
Output as JSON:
{{
  "claims": [
    {{"claim": "...", "in_context": true/false, "evidence": "quote from context or 'not found'"}}
  ],
  "faithfulness_score": 0.0-1.0,
  "unsupported_claims": ["list of claims not in context"]
}}"""}]
    )
    
    try:
        result = json.loads(response.content[0].text)
        return {
            "faithfulness_score": result.get("faithfulness_score", 0),
            "is_faithful":        result.get("faithfulness_score", 0) >= threshold,
            "unsupported_claims": result.get("unsupported_claims", []),
            "claims":             result.get("claims", []),
        }
    except json.JSONDecodeError:
        return {"faithfulness_score": 0, "is_faithful": False, "error": "parse failure"}
```

## 3. Real-World Context

### The Metric That Matters Most for Each Failure Mode

```
If users say: "The answer is wrong"
→ Check: Faithfulness score
→ If low: LLM is hallucinating or not following context
→ Fix:    Stronger grounding instructions, citation forcing

If users say: "It doesn't answer my question"  
→ Check: Answer relevancy + Context recall
→ If context recall low: retrieval is missing the right chunk
→ Fix:   Chunking, hybrid search, query expansion

If users say: "Too much irrelevant information"
→ Check: Context precision
→ If low: Too many irrelevant chunks being retrieved
→ Fix:   Reranking, filtering, reduce top-K

If users say: "The answer is incomplete"
→ Check: Context recall
→ If low: Not all relevant chunks are being retrieved
→ Fix:   Increase K, hybrid search, better embedding model
```

### Target Metrics for Production RAG

Based on 2025–2026 production benchmarks:

| Metric | Minimum | Good | Excellent |
|---|---|---|---|
| Context Recall | > 0.70 | > 0.80 | > 0.90 |
| Context Precision | > 0.60 | > 0.75 | > 0.85 |
| Answer Faithfulness | > 0.80 | > 0.90 | > 0.95 |
| Answer Relevancy | > 0.75 | > 0.85 | > 0.92 |
| Retrieval Recall@5 | > 0.70 | > 0.82 | > 0.90 |

Below "Minimum" on any metric: production deployment is premature.

## 4. Complete Evaluation Script

```python
import anthropic
import numpy as np
from dataclasses import dataclass, asdict
import json

client = anthropic.Anthropic()

@dataclass
class RAGEvalResult:
    question:            str
    ground_truth:        str
    retrieved_contexts:  list[str]
    generated_answer:    str
    retrieval_recall:    float
    faithfulness_score:  float
    answer_relevant:     bool
    passed:              bool

def evaluate_rag_pipeline(
    eval_cases: list[EvalCase],
    rag_fn,   # callable: (question: str) -> dict with "answer" and "chunks"
    k: int = 5,
) -> dict:
    """Run full RAG eval and return aggregate metrics."""
    
    results: list[RAGEvalResult] = []
    
    for case in eval_cases:
        # Run the RAG pipeline
        rag_output = rag_fn(case.question)
        answer   = rag_output["answer"]
        chunks   = rag_output["chunks"]
        context  = "\n".join(c["text"] for c in chunks)
        
        # Retrieval recall (do retrieved chunks contain relevant doc?)
        retrieved_sources = [c.get("source", "") for c in chunks]
        hits = sum(1 for doc_id in case.relevant_doc_ids if doc_id in retrieved_sources)
        retrieval_recall = hits / len(case.relevant_doc_ids) if case.relevant_doc_ids else 0
        
        # Faithfulness check
        faith_result = check_faithfulness(answer, context)
        faith_score  = faith_result["faithfulness_score"]
        
        # Relevancy check (simple: does answer mention key terms from ground truth?)
        gt_terms = set(case.ground_truth.lower().split())
        ans_terms = set(answer.lower().split())
        term_overlap = len(gt_terms & ans_terms) / max(len(gt_terms), 1)
        answer_relevant = term_overlap > 0.3  # rough heuristic; RAGAS does this better
        
        passed = (retrieval_recall >= 0.7 and faith_score >= 0.8 and answer_relevant)
        
        results.append(RAGEvalResult(
            question=case.question,
            ground_truth=case.ground_truth,
            retrieved_contexts=[c["text"][:100] for c in chunks],
            generated_answer=answer,
            retrieval_recall=retrieval_recall,
            faithfulness_score=faith_score,
            answer_relevant=answer_relevant,
            passed=passed,
        ))
    
    # Aggregate metrics
    pass_rate   = sum(r.passed for r in results) / len(results)
    avg_recall  = np.mean([r.retrieval_recall  for r in results])
    avg_faith   = np.mean([r.faithfulness_score for r in results])
    avg_rel     = np.mean([r.answer_relevant    for r in results])
    
    # Find failing cases for analysis
    failures = [r for r in results if not r.passed]
    
    return {
        "summary": {
            "total_cases":       len(results),
            "pass_rate":         round(pass_rate, 3),
            "avg_retrieval_recall": round(float(avg_recall), 3),
            "avg_faithfulness":  round(float(avg_faith), 3),
            "avg_relevancy":     round(float(avg_rel), 3),
        },
        "failures":  [asdict(f) for f in failures[:5]],   # top 5 failures for debugging
        "all_results": [asdict(r) for r in results],
    }

# Run and print
eval_report = evaluate_rag_pipeline(eval_cases[:20], rag_fn=rag_query)
print(json.dumps(eval_report["summary"], indent=2))
print(f"\nTop failures:")
for f in eval_report["failures"]:
    print(f"  Q: {f['question']}")
    print(f"  Retrieval recall: {f['retrieval_recall']:.2f} | Faithfulness: {f['faithfulness_score']:.2f}")
```

## 6. Trade-offs

### LLM-as-Judge vs Human Evaluation

| Aspect | LLM-as-Judge (RAGAS) | Human Evaluation |
|---|---|---|
| Cost | Low ($0.01–0.05 per case) | High ($5–50 per case with expert) |
| Speed | Fast (automated) | Slow (days for 50 cases) |
| Reliability | Good for defined metrics | Better for nuanced quality |
| Bias | Positional bias, verbosity bias | Evaluator disagreement |
| Scale | Unlimited | Limited |

**Best practice:** Use LLM-as-judge for automated CI/CD eval. Use human evaluation for
calibrating the LLM judge and for final sign-off before major releases.

## 7. When To Use vs. When To Avoid

### RAGAS
✅ Always run before shipping any RAG system
✅ Run on every significant change (embedding model, chunk size, retrieval strategy)
✅ Use as a regression gate in CI/CD

### When to Build a Custom Eval
✅ When RAGAS doesn't capture your domain-specific quality requirements
✅ When you need to measure task-specific correctness (e.g., legal compliance)

⚠️ Common mistakes:
- Generating an eval dataset with the same LLM you're evaluating — leads to self-reinforcing quality theater
- Not including hard cases (ambiguous queries, questions the RAG should refuse)
- Running evals only once rather than continuously
- Optimizing for RAGAS scores without checking that improvements correlate with user satisfaction

## 8. Senior AI Engineer Perspective

### The Eval-First Discipline

The most impactful practice in RAG engineering: **write your evaluation before writing your pipeline.**

Before building anything:
1. Define what "good" looks like (the four RAGAS metrics + your domain-specific requirements)
2. Build 20 evaluation cases from actual user queries or representative questions
3. Define pass/fail thresholds per metric
4. Then build the pipeline and measure

This discipline prevents building a pipeline optimized for intuition rather than performance.

### Retrieval Recall Is the Foundation

If context recall < 0.75, nothing else matters. You can't fix a retrieval miss with better prompting.
The LLM can only answer from what it receives. Invest retrieval engineering effort proportionally:

```
If context recall < 0.75:    Fix retrieval (chunking, embedding, hybrid search)
If context precision < 0.70: Fix filtering (reranking, better top-K selection)
If faithfulness < 0.85:      Fix generation (stronger grounding prompt, citation forcing)
If answer relevancy < 0.80:  Fix both (often a symptom of both retrieval and generation issues)
```
