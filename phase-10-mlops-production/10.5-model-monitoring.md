## 1. What Is It? (Theory)

### The Monitoring Gap

You've deployed your AI system. CI/CD passed. Staging looked good. Production is live.
Now what?

Traditional software monitoring says: watch CPU, memory, error rates, latency. If they're
green, everything is fine.

For AI systems, this is insufficient. A system can have:
- 100% uptime ✓
- Sub-100ms latency ✓  
- 0% error rate ✓
- **Silently degraded quality** ✗

The API returns 200 OK. The model is answering questions. The answers are just wrong, or
increasingly low-quality, or no longer grounded in context. Standard metrics miss all of this.

AI monitoring must track **quality**, not just availability.

### The Five Drift Types

In 2026, production teams treat drift as a system-level metric:

**1. Data drift (input distribution shift):**
User queries change over time. Seasonal topics, new product launches, industry events —
all shift what users ask about. If the model wasn't trained for these patterns, quality degrades.

```
Example: A customer support bot trained on 2024 queries
         starts seeing queries about a new product launched in 2025.
         The new product isn't in the knowledge base or training data.
```

**2. Model drift (provider silent update):**
Your foundation model provider updates their model without version bumping the name.
`claude-sonnet-4-6` in January may not be the same as `claude-sonnet-4-6` in June.
Behavior subtly changes.

**3. Prompt drift (gradual degradation of prompt quality):**
As the context around the prompt changes (new products, new policies, system changes),
the prompt that worked 6 months ago may no longer be optimal.

**4. Retrieval drift (knowledge base staleness):**
For RAG systems: the documents get stale relative to current questions. Or the
vocabulary used in documents diverges from the vocabulary users use in queries.

**5. Tool drift (tool behavior changes):**
For agent systems: an external API or tool changes its response format, rate limits,
or behavior. The model's tool use patterns no longer match reality.

## 2. How It Actually Works — Deep Mechanics

### 2.1 The Monitoring Stack for LLM Systems

```python
from dataclasses import dataclass
from datetime import datetime, timedelta
from collections import deque
import statistics
import json

@dataclass
class MetricWindow:
    """Rolling window of metric values."""
    name:       str
    values:     deque
    window_size: int = 1000   # last N observations
    
    def __post_init__(self):
        self.values = deque(maxlen=self.window_size)
    
    def add(self, value: float):
        self.values.append(value)
    
    @property
    def mean(self) -> float:
        return statistics.mean(self.values) if self.values else 0
    
    @property
    def p10(self) -> float:
        return sorted(self.values)[int(len(self.values) * 0.1)] if len(self.values) >= 10 else 0

class ProductionAIMonitor:
    """
    Real-time quality monitoring for a production AI system.
    Fires alerts when quality degrades.
    """
    
    def __init__(
        self,
        quality_floor: float = 3.0,       # min acceptable mean quality score
        error_rate_ceiling: float = 0.05,  # max 5% error rate
        latency_ceiling_ms: float = 3000,  # max 3s p95 latency
    ):
        self.quality_floor    = quality_floor
        self.error_rate_ceil  = error_rate_ceiling
        self.latency_ceil     = latency_ceiling_ms
        
        self.quality_window   = MetricWindow("quality_score")
        self.latency_window   = MetricWindow("latency_ms")
        self.error_window     = MetricWindow("is_error")   # 0 or 1
        self.cost_window      = MetricWindow("cost_usd")
        
        self.alerts: list[dict] = []
    
    def record(
        self,
        quality_score:  float | None,
        latency_ms:     float,
        is_error:       bool,
        cost_usd:       float,
        metadata:       dict = {},
    ):
        """Record a single request's metrics."""
        if quality_score is not None:
            self.quality_window.add(quality_score)
        
        self.latency_window.add(latency_ms)
        self.error_window.add(1.0 if is_error else 0.0)
        self.cost_window.add(cost_usd)
        
        # Check thresholds and fire alerts
        self._check_quality()
        self._check_errors()
    
    def _check_quality(self):
        if len(self.quality_window.values) < 50:
            return  # not enough data for reliable signal
        
        mean_quality = self.quality_window.mean
        if mean_quality < self.quality_floor:
            self._fire_alert(
                "QUALITY_DEGRADATION",
                f"Mean quality score {mean_quality:.2f} below floor {self.quality_floor}",
                severity="HIGH"
            )
    
    def _check_errors(self):
        if len(self.error_window.values) < 100:
            return
        
        error_rate = self.error_window.mean
        if error_rate > self.error_rate_ceil:
            self._fire_alert(
                "HIGH_ERROR_RATE",
                f"Error rate {error_rate:.1%} above ceiling {self.error_rate_ceil:.1%}",
                severity="CRITICAL"
            )
    
    def _fire_alert(self, alert_type: str, message: str, severity: str = "MEDIUM"):
        alert = {
            "type":      alert_type,
            "message":   message,
            "severity":  severity,
            "timestamp": datetime.utcnow().isoformat(),
            "metrics":   {
                "quality_mean": round(self.quality_window.mean, 2),
                "error_rate":   round(self.error_window.mean, 4),
                "latency_p50":  round(self.latency_window.mean, 0),
            }
        }
        self.alerts.append(alert)
        
        # In production: send to PagerDuty, Slack, etc.
        print(f"🚨 ALERT [{severity}] {alert_type}: {message}")
    
    def status_report(self) -> dict:
        """Current health snapshot."""
        return {
            "quality_mean":   round(self.quality_window.mean, 2),
            "quality_p10":    round(self.quality_window.p10, 2),
            "error_rate":     round(self.error_window.mean, 4),
            "latency_mean_ms": round(self.latency_window.mean, 0),
            "cost_per_req":   round(self.cost_window.mean, 5),
            "total_requests": len(self.latency_window.values),
            "active_alerts":  len([a for a in self.alerts if a["severity"] != "RESOLVED"]),
        }
```

### 2.2 Statistical Drift Detection

For detecting subtle quality changes over time:

```python
import numpy as np
from scipy import stats

class DriftDetector:
    """
    Detect when quality distribution has shifted significantly.
    Uses Page-Hinkley change detection (online algorithm — no reference window needed).
    """
    
    def __init__(self, threshold: float = 50, delta: float = 0.005):
        self.threshold = threshold   # sensitivity; higher = less sensitive
        self.delta     = delta       # minimum detectable shift
        self.mean      = None
        self.ph_sum    = 0.0
        self.min_ph    = 0.0
        self.n         = 0
    
    def update(self, value: float) -> bool:
        """
        Update with a new observation.
        Returns True if a change/drift is detected.
        """
        self.n += 1
        
        if self.mean is None:
            self.mean = value
            return False
        
        # Update running mean
        self.mean = (self.mean * (self.n - 1) + value) / self.n
        
        # Page-Hinkley test
        self.ph_sum += value - self.mean - self.delta
        self.min_ph  = min(self.min_ph, self.ph_sum)
        
        ph_statistic = self.ph_sum - self.min_ph
        
        if ph_statistic > self.threshold:
            # Drift detected; reset
            self.ph_sum = 0.0
            self.min_ph = 0.0
            return True
        
        return False

class LLMDriftMonitor:
    """Monitor for the five LLM drift types."""
    
    def __init__(self):
        self.quality_detector  = DriftDetector(threshold=50)
        self.latency_detector  = DriftDetector(threshold=100)
        self.drift_events: list[dict] = []
    
    def record_request(
        self,
        quality_score: float,
        latency_ms:    float,
        input_text:    str,
        output_text:   str,
        model_version: str,
    ):
        """Process one request and check for drift signals."""
        
        # Quality drift
        if self.quality_detector.update(quality_score):
            self.drift_events.append({
                "type":      "QUALITY_DRIFT",
                "timestamp": datetime.utcnow().isoformat(),
                "model":     model_version,
                "message":   "Statistical quality drift detected — investigate"
            })
        
        # Latency drift
        if self.latency_detector.update(latency_ms):
            self.drift_events.append({
                "type":    "LATENCY_DRIFT",
                "timestamp": datetime.utcnow().isoformat(),
                "model":   model_version,
                "message": "Statistical latency drift detected"
            })
    
    def check_input_distribution(
        self,
        recent_queries: list[str],
        reference_queries: list[str],
        threshold: float = 0.1,   # Jensen-Shannon divergence threshold
    ) -> dict:
        """
        Check if query distribution has shifted from training distribution.
        High divergence = users are asking different things than training data covered.
        """
        from sentence_transformers import SentenceTransformer
        
        model = SentenceTransformer("all-MiniLM-L6-v2")
        
        recent_vecs = model.encode(recent_queries[:100])
        ref_vecs    = model.encode(reference_queries[:100])
        
        # Simplified: compare mean vectors
        recent_centroid = np.mean(recent_vecs, axis=0)
        ref_centroid    = np.mean(ref_vecs, axis=0)
        
        cosine_shift = 1 - np.dot(recent_centroid, ref_centroid) / (
            np.linalg.norm(recent_centroid) * np.linalg.norm(ref_centroid)
        )
        
        return {
            "cosine_shift":    round(float(cosine_shift), 4),
            "drift_detected":  cosine_shift > threshold,
            "message":         f"Query distribution {'HAS' if cosine_shift > threshold else 'has NOT'} drifted significantly",
        }
```

### 2.3 Practical Monitoring Dashboard

```python
import anthropic
import time
import random
from datetime import datetime

# Simulate a production monitoring loop
monitor = ProductionAIMonitor()
drift   = LLMDriftMonitor()

client = anthropic.Anthropic()

def monitored_rag_call(question: str) -> dict:
    """Production call with automatic monitoring."""
    start = time.time()
    
    try:
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=256,
            messages=[{"role": "user", "content": question}]
        )
        answer  = response.content[0].text
        latency = (time.time() - start) * 1000
        cost    = (response.usage.input_tokens * 3.00 + response.usage.output_tokens * 15.00) / 1_000_000
        
        # Score with lightweight judge (in production: sample 10% and score)
        quality_score = None
        if random.random() < 0.10:   # score 10% of requests
            quality_score = score_with_judge(question, answer)
        
        # Record to monitor
        monitor.record(quality_score, latency, is_error=False, cost_usd=cost)
        drift.record_request(
            quality_score=quality_score or 4.0,
            latency_ms=latency,
            input_text=question,
            output_text=answer,
            model_version="claude-sonnet-4-6"
        )
        
        return {"answer": answer, "latency_ms": latency, "cost_usd": cost}
    
    except Exception as e:
        latency = (time.time() - start) * 1000
        monitor.record(None, latency, is_error=True, cost_usd=0)
        return {"error": str(e)}

def score_with_judge(question: str, answer: str) -> float:
    # Lightweight judge — call quickly, don't block the main path
    return 4.0  # stub; in practice: call judge model asynchronously

# Periodic status check
def print_status():
    status = monitor.status_report()
    print(f"\n=== Monitoring Status ({datetime.utcnow().strftime('%H:%M:%S')}) ===")
    for k, v in status.items():
        print(f"  {k:25s}: {v}")
    
    if drift.drift_events:
        print(f"\n  Drift events: {len(drift.drift_events)}")
        for event in drift.drift_events[-3:]:
            print(f"    {event['type']}: {event['message']}")
```

## 3. Real-World Context

### The "Silent Degradation" Production Pattern

The most common production AI monitoring failure:

```
Month 1: Launch. Quality: 4.2/5. Good.
Month 2: Provider updates model. Quality: 4.0/5. Within noise.
Month 3: More provider updates. Quality: 3.7/5. Still "acceptable."
Month 4: Knowledge base gets stale. Quality: 3.3/5. Users complain.
Month 5: Engineering team investigates. Can't find the cause.
          "When did it start degrading?" Unknown — no monitoring.
```

With monitoring:
- Month 2 or 3: Alert fires. Team investigates immediately.
- Root cause identified (provider update) and remediated.
- No user complaints.

### Cost Monitoring Is Quality Monitoring

Unexpected cost spikes often signal quality problems:
- Runaway agent loop → 100× normal cost
- Users sending much longer queries (new feature launched?) → higher input tokens
- Model generating unusually long responses → quality issue (verbose output)

Monitor cost alongside quality. They're correlated.

## 5. Trade-offs

### Scoring Every Request vs Sampling

| | Score everything | Score 10% |
|---|---|---|
| Quality signal | Complete | Statistical (good for trends) |
| Cost | High (judge call per request) | Low |
| Latency impact | High (adds judge call time) | Minimal (async) |
| Use for | Dev/staging validation | Production monitoring |

**Rule:** Score 100% in staging. Sample 5–10% in production for trend monitoring.
Use deterministic checks on 100% for format/safety.

## 6. Senior AI Engineer Perspective

### The Three Numbers That Matter

Track these three daily:
1. **Mean quality score (rolling 24h)** — are responses still good?
2. **P10 quality score (rolling 24h)** — are the worst 10% of responses getting worse?
3. **Cost per request (rolling 24h)** — is anything running away?

Everything else is noise until these three are stable.

### Monitor for What You Didn't Expect

The drift types you can anticipate (model updates, knowledge staleness) are easy to check for.
The drift types that surprise you are more dangerous. Build your monitoring to surface
unexpected patterns:
- Unusual query lengths (users started sending much shorter or longer queries)
- New topics appearing in queries (users started asking about things your system wasn't designed for)
- Response length changes (model started being much more verbose or terse)

These unexpected shifts are the ones that cause the most damage because nobody was looking for them.
