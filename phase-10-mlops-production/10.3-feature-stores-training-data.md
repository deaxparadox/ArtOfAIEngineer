## 1. What Is It? (Theory)

### The Data Problem in Production AI

For traditional ML: features are computed from structured data (tables, databases).
The challenge: the same feature computed at training time must be computed identically at
inference time. If they differ — a common problem called **train-serve skew** — the model
gets inputs it wasn't trained on and performance degrades silently.

For LLM applications: the "features" are often:
- Documents in a knowledge base (for RAG)
- Fine-tuning datasets (conversation examples, preference pairs)
- Eval datasets (golden sets from Section 9.3)

The challenge is similar: ensuring the training/fine-tuning data is versioned, clean, and
reproducible, and that production data matches training data distribution.

### Feature Stores: What They Are

A **feature store** is a centralized system for storing, serving, and versioning ML features.
It solves the train-serve skew problem by ensuring the same computation is used in both:

```
Training:   Feature Store → batch features → Model training
Inference:  Feature Store → online features → Model serving

Same store, same computation = no skew
```

For LLM-specific applications, the analogue is:
- A versioned document store for RAG knowledge bases
- A versioned dataset for fine-tuning
- A clean, deduplicated evaluation dataset

## 2. How It Actually Works — Deep Mechanics

### 2.1 Training Data Management for LLMs

Managing fine-tuning data is often more important than managing model weights.
The fine-tuned model is only as good as the data it was trained on.

```python
from dataclasses import dataclass, field
from datetime import datetime
from pathlib import Path
import json
import hashlib
import shutil

@dataclass
class DatasetVersion:
    """Versioned snapshot of a training dataset."""
    version_id:   str
    name:         str
    created_at:   str
    n_examples:   int
    split:        dict   # {"train": 0.8, "val": 0.1, "test": 0.1}
    source:       str    # "production_logs" | "expert_labeled" | "synthetic"
    quality_score: float
    notes:        str = ""
    path:         str = ""

class TrainingDataManager:
    """
    Version-controlled training data management.
    Tracks every version, supports time-travel (reproduce any past dataset).
    """
    
    def __init__(self, base_path: str = "training_data"):
        self.base_path = Path(base_path)
        self.base_path.mkdir(exist_ok=True)
        self.registry_path = self.base_path / "registry.json"
        self.registry: dict[str, DatasetVersion] = self._load_registry()
    
    def _load_registry(self) -> dict:
        if self.registry_path.exists():
            data = json.loads(self.registry_path.read_text())
            return {k: DatasetVersion(**v) for k, v in data.items()}
        return {}
    
    def _save_registry(self):
        data = {k: vars(v) for k, v in self.registry.items()}
        self.registry_path.write_text(json.dumps(data, indent=2))
    
    def save_version(
        self,
        examples: list[dict],
        name: str,
        source: str,
        quality_score: float = 0.0,
        notes: str = "",
        split: dict | None = None,
    ) -> DatasetVersion:
        """
        Save a versioned snapshot of training data.
        Version ID is content-based (hash), so identical data = same version.
        """
        content_hash = hashlib.md5(json.dumps(examples, sort_keys=True).encode()).hexdigest()[:12]
        version_id   = f"{name}_{content_hash}"
        
        # Split examples
        split = split or {"train": 0.8, "val": 0.1, "test": 0.1}
        n = len(examples)
        train_end = int(n * split["train"])
        val_end   = train_end + int(n * split["val"])
        
        splits = {
            "train": examples[:train_end],
            "val":   examples[train_end:val_end],
            "test":  examples[val_end:]
        }
        
        # Save data files
        version_dir = self.base_path / version_id
        version_dir.mkdir(exist_ok=True)
        
        for split_name, data in splits.items():
            path = version_dir / f"{split_name}.jsonl"
            with open(path, "w") as f:
                for ex in data:
                    f.write(json.dumps(ex) + "\n")
        
        # Register version
        version = DatasetVersion(
            version_id    = version_id,
            name          = name,
            created_at    = datetime.utcnow().isoformat(),
            n_examples    = n,
            split         = {"train": len(splits["train"]), "val": len(splits["val"]), "test": len(splits["test"])},
            source        = source,
            quality_score = quality_score,
            notes         = notes,
            path          = str(version_dir),
        )
        self.registry[version_id] = version
        self._save_registry()
        
        print(f"Saved dataset version: {version_id} ({n} examples)")
        return version
    
    def load_version(self, version_id: str, split: str = "train") -> list[dict]:
        """Load a specific version of the training data."""
        if version_id not in self.registry:
            raise ValueError(f"Unknown dataset version: {version_id}")
        
        version = self.registry[version_id]
        path = Path(version.path) / f"{split}.jsonl"
        
        examples = []
        with open(path) as f:
            for line in f:
                examples.append(json.loads(line.strip()))
        return examples
    
    def compare_versions(self, v1: str, v2: str) -> dict:
        """Show what changed between two dataset versions."""
        data1 = set(json.dumps(e, sort_keys=True) for e in self.load_version(v1))
        data2 = set(json.dumps(e, sort_keys=True) for e in self.load_version(v2))
        
        added   = data2 - data1
        removed = data1 - data2
        
        return {
            "v1_size":  len(data1),
            "v2_size":  len(data2),
            "added":    len(added),
            "removed":  len(removed),
            "unchanged": len(data1 & data2),
        }

# Usage
manager = TrainingDataManager()

# Save a training dataset version
sft_data = [
    {"messages": [
        {"role": "user", "content": "What is our return policy?"},
        {"role": "assistant", "content": "Items can be returned within 30 days with receipt."}
    ]},
    {"messages": [
        {"role": "user", "content": "How long does shipping take?"},
        {"role": "assistant", "content": "Standard shipping is 5-7 business days."}
    ]},
    # ... more examples
]

v1 = manager.save_version(
    examples=sft_data,
    name="support_sft",
    source="expert_labeled",
    quality_score=4.5,
    notes="Initial expert-labeled dataset for customer support fine-tuning"
)
```

### 2.2 Data Quality Validation

Before using data for training, validate it:

```python
from collections import Counter

def validate_training_data(examples: list[dict]) -> dict:
    """
    Validate training data quality before fine-tuning.
    Returns a report with issues found.
    """
    issues = []
    warnings = []
    
    # 1. Check format
    for i, ex in enumerate(examples):
        if "messages" not in ex:
            issues.append(f"Example {i}: missing 'messages' key")
            continue
        
        msgs = ex["messages"]
        if not isinstance(msgs, list) or len(msgs) < 2:
            issues.append(f"Example {i}: messages must be a list with at least 2 items")
            continue
        
        # Check turn structure
        if msgs[0]["role"] != "user":
            warnings.append(f"Example {i}: first message should be 'user', got '{msgs[0]['role']}'")
        
        if msgs[-1]["role"] != "assistant":
            issues.append(f"Example {i}: last message must be 'assistant'")
    
    # 2. Check for duplicates
    content_hashes = [hashlib.md5(json.dumps(ex, sort_keys=True).encode()).hexdigest() for ex in examples]
    dup_count = len(content_hashes) - len(set(content_hashes))
    if dup_count > 0:
        warnings.append(f"Found {dup_count} duplicate examples ({dup_count/len(examples):.1%})")
    
    # 3. Check length distribution
    response_lengths = [
        len(ex["messages"][-1]["content"].split())
        for ex in examples
        if "messages" in ex and ex["messages"]
    ]
    
    if response_lengths:
        avg_len = sum(response_lengths) / len(response_lengths)
        if avg_len < 5:
            warnings.append(f"Very short average response length: {avg_len:.1f} words")
        if avg_len > 500:
            warnings.append(f"Very long average response length: {avg_len:.1f} words")
    
    # 4. Check label balance (for classification datasets)
    # (skip for generation datasets)
    
    return {
        "total_examples": len(examples),
        "issues": issues,
        "warnings": warnings,
        "valid": len(issues) == 0,
        "duplicate_count": dup_count,
        "avg_response_words": avg_len if response_lengths else 0,
    }
```

### 2.3 Feature Stores for Traditional ML Components

For any ML component using structured features (e.g., user profile features for routing,
historical context features for personalization):

```python
# Feast is the most popular open-source feature store
# pip install feast

# feast/feature_repo/features.py
from feast import Entity, Feature, FeatureView, FileSource, ValueType
from datetime import timedelta

# Define what a "user" entity is
user = Entity(name="user_id", value_type=ValueType.STRING)

# Define a source for user features
user_data_source = FileSource(
    path="data/user_features.parquet",
    timestamp_field="event_timestamp",
)

# Define the feature view (what features to store)
user_feature_view = FeatureView(
    name="user_features",
    entities=[user],
    ttl=timedelta(days=1),   # features expire after 1 day
    features=[
        Feature(name="total_purchases",       dtype=ValueType.INT64),
        Feature(name="days_since_last_visit",  dtype=ValueType.INT64),
        Feature(name="preferred_category",    dtype=ValueType.STRING),
        Feature(name="avg_order_value",        dtype=ValueType.FLOAT),
    ],
    source=user_data_source,
)

# Training: get historical features at the time each training example was created
# (prevents look-ahead bias)
from feast import FeatureStore

store = FeatureStore(repo_path=".")

training_df = store.get_historical_features(
    entity_df=labeled_df,   # has user_id + event_timestamp columns
    features=["user_features:total_purchases", "user_features:preferred_category"]
).to_df()

# Serving: get latest features in real-time
online_features = store.get_online_features(
    features=["user_features:total_purchases", "user_features:preferred_category"],
    entity_rows=[{"user_id": "user_123"}]
).to_dict()
```

## 3. Real-World Context

### Train-Serve Skew in LLM Applications

The classical train-serve skew problem manifests differently in LLM systems:

| Skew Type | Example | Impact |
|---|---|---|
| **Prompt skew** | Training used system prompt v1; serving uses v2 | Model learned patterns for v1 format |
| **Embedding skew** | Training chunks with model A; retrieval with model B | Embedding space incompatible |
| **Data distribution skew** | Training on formal support tickets; users send casual text | Model expects formal style |
| **Temporal skew** | Knowledge base updated but model not re-evaluated | Model "knows" old content better |

Preventing this requires tracking the version of every component used in training and ensuring
serving uses the same versions.

## 5. Trade-offs

### When You Need a Feature Store vs Not

**Need a feature store when:**
- ML pipeline has structured features computed from multiple data sources
- Same feature used in training and serving (skew prevention is critical)
- Features updated frequently and need point-in-time access
- Multiple models share the same features

**Can skip a feature store when:**
- Using LLM APIs only (no structured feature engineering)
- RAG pipeline (knowledge base is the "feature store")
- Simple applications where prompt + user message = complete context

Most LLM applications in 2026 don't need a traditional feature store — the prompts,
retrieved context, and model versions serve similar purposes. But teams training custom
models or using hybrid ML+LLM pipelines still need them.

## 6. Senior AI Engineer Perspective

### Data Quality Is the Ceiling for Fine-Tuned Models

For fine-tuning: the model cannot be better than the training data. A 10,000-example
dataset with 20% noisy or wrong examples produces a model with a quality ceiling at 80%.
No architecture, hyperparameter tuning, or training trick compensates for bad data.

The most common mistake: spending a week on fine-tuning and two hours on data quality.
The correct ratio is closer to the opposite.

### Data Versioning Is Non-Negotiable for Fine-Tuning

If you fine-tune and don't version the dataset:
- A model fails in production. What was in the training data?
- A dataset is accidentally corrupted. What did the last good version look like?
- You want to understand a model's behavior. What examples shaped it?

DVC + git is the minimum. 30 minutes to set up, indefinitely valuable.
