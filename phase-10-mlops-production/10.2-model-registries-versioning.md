## 1. What Is It? (Theory)

### The Rollback Nightmare

A team updates their customer support AI with a new fine-tuned model. Quality initially seems
fine. Three days later, a specific product category starts generating wrong answers. Tickets
spike. The engineering team scrambles to roll back.

Problem: they didn't version the model. They overwrote the previous one. Rolling back means
re-running a fine-tuning job (12 hours, $500), hoping they can reproduce the previous version.
They can't — the training dataset was also updated.

**A model registry** prevents this. It's the source of truth for what model version is running
where, with complete lineage back to data and code.

### What "Model" Means in 2026

For traditional ML: a model is weights + architecture + preprocessing code.

For LLM-based systems, "the model" is a much broader concept:
- The foundation model (and its version/snapshot)
- The fine-tuned adapter (LoRA weights + config)
- The prompt (system prompt + few-shot examples)
- The retrieval config (embedding model, chunk size, vector DB collection)
- The tool definitions and their implementations
- The evaluation baseline

**All of these must be versioned together** for a meaningful model registry.

### 2.1 MLflow Model Registry

```python
import mlflow
from mlflow.models import infer_signature
import anthropic

mlflow.set_tracking_uri("sqlite:///mlflow.db")

# Registering a custom LLM wrapper as an MLflow model
class RAGPipeline(mlflow.pyfunc.PythonModel):
    """Wrap the complete RAG pipeline as a versioned MLflow model."""
    
    def load_context(self, context):
        """Load artifacts saved with the model."""
        import json
        with open(context.artifacts["config_path"]) as f:
            self.config = json.load(f)
        self.client = anthropic.Anthropic()
    
    def predict(self, context, model_input: dict) -> dict:
        question = model_input["question"]
        
        # Use config from artifacts (versioned with the model)
        chunks = retrieve_with_config(
            question,
            embedding_model=self.config["embedding_model"],
            top_k=self.config["top_k"]
        )
        
        response = self.client.messages.create(
            model=self.config["llm_model"],
            max_tokens=512,
            system=self.config["system_prompt"],
            messages=[{"role": "user", "content": f"Context:\n{chunks}\n\nQ: {question}"}]
        )
        
        return {"answer": response.content[0].text}

# Register the pipeline with its configuration
import json, os

config = {
    "embedding_model": "voyage-3-large",
    "top_k": 5,
    "chunk_size": 512,
    "llm_model": "claude-sonnet-4-6",
    "system_prompt": "You are a helpful support assistant. Answer from provided sources only.",
    "version_note": "V2: Added citation requirement",
}

os.makedirs("artifacts", exist_ok=True)
config_path = "artifacts/config.json"
with open(config_path, "w") as f:
    json.dump(config, f)

with mlflow.start_run():
    # Log the config file as an artifact
    mlflow.log_artifact(config_path)
    mlflow.log_params(config)
    
    # Log quality metrics from eval run
    mlflow.log_metrics({
        "faithfulness": 0.92,
        "context_recall": 0.85,
        "answer_relevancy": 0.88,
        "mean_quality_score": 4.1,
    })
    
    # Save and register the model
    mlflow.pyfunc.log_model(
        artifact_path="rag_pipeline",
        python_model=RAGPipeline(),
        artifacts={"config_path": config_path},
        registered_model_name="customer-support-rag",
    )

# Transition to staging, then production
from mlflow import MlflowClient

client = MlflowClient()

# Get latest version
latest_version = client.get_latest_versions("customer-support-rag")[0]
print(f"Latest version: {latest_version.version}")

# Promote to staging
client.set_registered_model_alias(
    name="customer-support-rag",
    alias="staging",
    version=latest_version.version
)

# After staging tests pass:
client.set_registered_model_alias(
    name="customer-support-rag",
    alias="champion",
    version=latest_version.version
)

# Load the champion model in production
champion = mlflow.pyfunc.load_model("models:/customer-support-rag@champion")
result = champion.predict({"question": "What is the return policy?"})
```

### 2.2 Semantic Versioning for AI Systems

Apply semantic versioning to make breaking vs non-breaking changes explicit:

```python
from dataclasses import dataclass
from enum import Enum

class ChangeType(Enum):
    PATCH = "patch"    # 1.2.x → Bug fix, minor config tweak, no behavior change
    MINOR = "minor"    # 1.x.0 → Behavior improvement, new capability, backward-compatible
    MAJOR = "major"    # x.0.0 → Breaking change: different base model, API change, incompatible behavior

@dataclass
class ModelVersion:
    major: int
    minor: int
    patch: int
    
    # The complete configuration this version represents
    foundation_model:  str   # e.g., "claude-sonnet-4-6-20251001"
    prompt_hash:       str   # hash of system_prompt content
    embedding_model:   str   # e.g., "voyage-3-large"
    chunk_size:        int
    adapter_path:      str | None   # path to LoRA adapter if fine-tuned
    eval_score:        float
    deployed:          bool = False
    
    @property
    def string(self) -> str:
        return f"{self.major}.{self.minor}.{self.patch}"
    
    def bump(self, change_type: ChangeType) -> "ModelVersion":
        import copy
        new = copy.copy(self)
        if change_type == ChangeType.MAJOR:
            new.major += 1; new.minor = 0; new.patch = 0
        elif change_type == ChangeType.MINOR:
            new.minor += 1; new.patch = 0
        else:
            new.patch += 1
        return new

# Versioning decisions:
# Changed base model (claude-sonnet-4-6 → claude-opus-4-8)? MAJOR
# Added citation requirement to prompt? MINOR (behavior change, improvement)
# Fixed typo in system prompt? PATCH
# Changed top_k from 5 to 7? MINOR
# New fine-tuning run on same base model? MINOR
```

### 2.3 Blue-Green Deployment for Models

```python
import anthropic
import random
from dataclasses import dataclass

@dataclass
class ModelDeployment:
    version: str
    config: dict
    traffic_pct: float   # 0.0 to 1.0

class ModelRouter:
    """
    Blue-green router: gradually shift traffic from old to new model version.
    """
    
    def __init__(self):
        self.blue  = None   # stable production version
        self.green = None   # new candidate version
    
    def deploy_green(self, version: str, config: dict):
        """Deploy a new candidate version to receive 0% of traffic initially."""
        self.green = ModelDeployment(version, config, traffic_pct=0.0)
        print(f"Green deployment ready: v{version} (receiving 0% traffic)")
    
    def shift_traffic(self, green_pct: float):
        """Gradually shift traffic to green deployment."""
        if not self.green:
            raise ValueError("No green deployment configured")
        
        self.green.traffic_pct = green_pct
        if self.blue:
            self.blue.traffic_pct = 1.0 - green_pct
        
        print(f"Traffic: blue={self.blue.traffic_pct:.0%}, green={self.green.traffic_pct:.0%}")
    
    def promote_green(self):
        """Promote green to blue (new stable). Old blue is discarded."""
        self.blue  = self.green
        self.blue.traffic_pct = 1.0
        self.green = None
        print(f"Green promoted to blue: v{self.blue.version}")
    
    def rollback(self):
        """Emergency rollback: kill green, return all traffic to blue."""
        self.green = None
        if self.blue:
            self.blue.traffic_pct = 1.0
        print("Rolled back to blue (stable) deployment")
    
    def route(self, request: dict) -> dict:
        """Route a request to blue or green based on traffic split."""
        use_green = (
            self.green is not None and
            random.random() < self.green.traffic_pct
        )
        
        deployment = self.green if use_green else self.blue
        if not deployment:
            raise RuntimeError("No active deployment")
        
        return self._call_model(deployment.config, request)
    
    def _call_model(self, config: dict, request: dict) -> dict:
        client = anthropic.Anthropic()
        response = client.messages.create(
            model=config["model"],
            max_tokens=512,
            system=config["system_prompt"],
            messages=[{"role": "user", "content": request["question"]}]
        )
        return {"answer": response.content[0].text, "model_version": config.get("version", "unknown")}

# Deployment workflow
router = ModelRouter()

# Current stable (blue)
router.blue = ModelDeployment(
    version="1.2.0",
    config={"model": "claude-sonnet-4-6", "system_prompt": "Be helpful.", "version": "1.2.0"},
    traffic_pct=1.0
)

# Deploy new version (green) with 0% traffic
router.deploy_green(
    version="1.3.0",
    config={"model": "claude-sonnet-4-6", "system_prompt": "Be helpful. Always cite sources.", "version": "1.3.0"}
)

# Gradual rollout
router.shift_traffic(green_pct=0.05)   # 5% canary
# Monitor quality for 24 hours...
router.shift_traffic(green_pct=0.25)   # 25% if quality is good
# Monitor another 24 hours...
router.shift_traffic(green_pct=0.50)   # 50%
router.promote_green()                  # Full promotion

# Emergency: quality degraded
# router.rollback()
```

### 2.4 DVC for Data and Model Versioning

For teams managing model weights and large datasets, DVC (Data Version Control) is
the standard:

```bash
# Install: pip install dvc dvc-s3  (or dvc-gcs, dvc-azure)

# Initialize DVC alongside git
dvc init
git add .dvc .dvcignore
git commit -m "Initialize DVC"

# Track a model file
dvc add models/fine_tuned_adapter.bin
git add models/fine_tuned_adapter.bin.dvc .gitignore
git commit -m "Add fine-tuned adapter v1"

# Push model artifacts to remote storage
dvc remote add -d s3remote s3://my-bucket/models
dvc push

# Tag the git commit for this model version
git tag -a "model-v1.3.0" -m "Fine-tuned on customer support v3 dataset"
git push origin --tags

# Later: reproduce this exact model
git checkout model-v1.3.0
dvc pull    # downloads the exact model artifacts from S3
```

```python
# Python: DVC integration in MLOps pipeline
import subprocess

def get_model_artifact(version_tag: str) -> str:
    """Check out a specific model version by git tag."""
    subprocess.run(["git", "checkout", version_tag], check=True)
    subprocess.run(["dvc", "pull"], check=True)
    return f"models/fine_tuned_adapter.bin"  # now contains the right version

def save_model_artifact(model_path: str, version_tag: str, message: str):
    """Save and version a model artifact."""
    subprocess.run(["dvc", "add", model_path], check=True)
    subprocess.run(["git", "add", f"{model_path}.dvc"], check=True)
    subprocess.run(["git", "commit", "-m", message], check=True)
    subprocess.run(["dvc", "push"], check=True)
    subprocess.run(["git", "tag", "-a", version_tag, "-m", message], check=True)
    subprocess.run(["git", "push", "origin", "--tags"], check=True)
```

## 3. Real-World Context

### The Complete Versioning Strategy for LLM Systems

What needs versioning in a production LLM application:

| Component | Tool | Frequency |
|---|---|---|
| Foundation model | Pin to dated ID in config | On every model upgrade |
| System prompt | Langfuse / PromptLayer / git | On every prompt change |
| Fine-tuned adapter | DVC + MLflow | On every fine-tune run |
| Embedding model | Config + DVC (if local) | On every embedding model change |
| Vector DB collection | Snapshot ID + DVC | On every re-index |
| RAG config (chunk size, top_k) | git/MLflow | On every parameter change |
| Eval baseline | MLflow / file in git | On every intentional improvement |

A deployable artifact bundles ALL of these together with a single version tag.
Any single change creates a new version.

## 5. Trade-offs

### Single Monolithic Version vs Component Versions

**Monolithic (one version for everything):**
- Simple: "deploy v1.3" means everything is v1.3
- Harder to do partial updates (want to update only the prompt)
- Rollback is all-or-nothing

**Component-level versioning:**
- Prompt can be v3.2 with model v1.3 and embeddings v2.1
- More flexibility; can update components independently
- More complex: which combinations are compatible?

**Rule:** Start monolithic. Move to component-level when you have enough distinct components
that frequently update independently.

## 6. When To Use vs. When To Avoid

### Model Registry
✅ Always: Any production AI system where you need rollback capability
✅ Always: Fine-tuned models (expensive to reproduce)
✅ Always: When the system has multiple deployed environments (staging, prod)
❌ Can skip: Prototype using a public API with a pinned model version and a tracked system prompt in git

⚠️ Common mistakes:
- Not versioning the prompt alongside the model — the prompt IS part of the model for LLM systems
- Not recording what data was used for each training run — you need this to diagnose failures
- Not testing the rollback procedure until you actually need it — test rollback in staging first

## 7. Senior AI Engineer Perspective

### Every Deployment Must Be Reversible

The #1 rule: any deployed version must be rollable-back within 15 minutes. If you can't
guarantee this, your deployment infrastructure is incomplete.

What makes rollback fast:
- Model artifacts are versioned and accessible (DVC, S3, HF Hub)
- Deployment config is version-controlled (git)
- The rollback procedure is automated (one command or one button)
- You've tested the rollback in staging (otherwise it will fail when you need it most)
