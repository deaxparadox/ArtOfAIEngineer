## 1. What Is It? (Theory)

### The Golden Dataset

A **golden dataset** (also called a golden set or eval dataset) is a curated collection of
test cases that represents the behavior you want to measure. It's the foundation of your
entire eval pipeline. Everything builds on it.

"Garbage in, garbage out" applies with full force to eval datasets:
- Cherry-picked easy examples → metrics look good; real-world quality is poor
- No edge cases → you don't know what breaks at the boundaries
- Stale reference answers → correct score actually means wrong behavior
- Biased distribution → metrics measure the distribution you tested, not the one users send

Building a golden dataset is more engineering discipline than it is technical work. It requires
careful thought about what you're measuring and deliberate curation of diverse, representative cases.

### Why Most Eval Datasets Are Garbage

The most common ways eval datasets fail:

1. **Survivorship bias:** Cases are collected from examples that worked, not examples that failed
2. **Happy path only:** No adversarial inputs, edge cases, or out-of-scope queries
3. **Generated by the same model:** LLM writes the eval cases → they perfectly match the model's style
4. **Too small:** 10–20 cases don't represent the real distribution
5. **Static forever:** Never updated to reflect how usage evolves
6. **Wrong reference answers:** Domain experts weren't involved in creating the ground truth

## 2. How to Build a Good Eval Dataset

### 2.1 The Five Sources of Eval Cases

**Source 1: Real production logs (best)**
```python
from datetime import datetime, timedelta
import random

def sample_from_production_logs(
    logs_db,         # your production logging system
    n_cases: int = 100,
    lookback_days: int = 30,
) -> list[dict]:
    """
    Sample real user queries from production.
    This is the highest-fidelity source — real inputs, real distribution.
    """
    since = datetime.now() - timedelta(days=lookback_days)
    
    # Get all production queries from last N days
    all_queries = logs_db.query(
        "SELECT input, output, feedback FROM llm_logs WHERE created_at > ?",
        (since,)
    )
    
    # Stratified sampling: ensure we cover all query types
    # Not just random — intentionally sample from different categories
    sampled = []
    
    # 60%: typical queries (representative of common usage)
    typical = [q for q in all_queries if q.get("feedback") != "negative"]
    sampled.extend(random.sample(typical, min(n_cases * 6 // 10, len(typical))))
    
    # 30%: queries where user gave negative feedback (known failures)
    failures = [q for q in all_queries if q.get("feedback") == "negative"]
    sampled.extend(random.sample(failures, min(n_cases * 3 // 10, len(failures))))
    
    # 10%: long/complex queries (stress tests)
    long_queries = [q for q in all_queries if len(q.get("input", "")) > 500]
    sampled.extend(random.sample(long_queries, min(n_cases // 10, len(long_queries))))
    
    return sampled[:n_cases]
```

**Source 2: LLM-generated cases (with human review)**
```python
import anthropic
import json

client = anthropic.Anthropic()

def generate_eval_cases(
    domain_description: str,
    example_docs: list[str],
    n_cases: int = 50,
) -> list[dict]:
    """
    Use LLM to draft eval cases, then have humans review and correct them.
    NEVER trust generated cases without human review.
    """
    cases = []
    
    for doc in example_docs[:n_cases]:
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=512,
            messages=[{"role": "user", "content": f"""Generate ONE evaluation test case for this domain: {domain_description}

Based on this document:
{doc[:500]}

Output as JSON:
{{
  "input": "A realistic user question about this content",
  "ground_truth": "The precise, correct answer from the document",
  "difficulty": "easy|medium|hard",
  "category": "factual|procedural|comparative|edge_case",
  "should_refuse": false,
  "notes": "Any caveats or tricky aspects"
}}"""}]
        )
        
        try:
            case = json.loads(response.content[0].text)
            case["source"] = "generated"
            case["human_reviewed"] = False    # flag for review
            cases.append(case)
        except json.JSONDecodeError:
            pass
    
    print(f"Generated {len(cases)} cases. REQUIRES HUMAN REVIEW before use.")
    return cases
```

**Source 3: Edge cases and adversarial inputs**
```python
EDGE_CASE_TEMPLATES = [

    # Long inputs

    " ".join(["repeat"] * 200),        # repetitive input
]
```

**Source 4: Known failure cases (most valuable)**
```python

# 1. Save the input that caused the failure

FAILURE_DERIVED_CASES = [
    {
        "input": "Can I return a sale item if it's damaged?",

        "category": "edge_case",
        "source": "production_failure",
        "failure_description": "Model previously said 'yes, damaged items can be returned'"
    },
    # ... more cases from production failures
]
```

**Source 5: Expert-written cases**
For high-stakes domains (medical, legal, financial), have domain experts write both the
question and the ground truth answer. These are expensive but most reliable.

### 2.2 Structuring Cases

```python
from dataclasses import dataclass, field
from typing import Optional
from datetime import datetime

@dataclass
class EvalCase:
    # Required
    case_id:      str
    input:        str              # the prompt/question
    ground_truth: str              # the reference answer
    
    # Categorization
    category:     str              # "factual", "procedural", "comparison", "edge_case", "refusal"
    difficulty:   str              # "easy", "medium", "hard"
    source:       str              # "production_log", "generated+reviewed", "expert_written"
    
    # Metadata
    created_at:   str = field(default_factory=lambda: datetime.utcnow().isoformat())
    tags:         list[str] = field(default_factory=list)
    notes:        str = ""
    
    # For refusal cases
    should_refuse:        bool = False
    refusal_reason:       str = ""
    
    # For RAG cases
    relevant_doc_ids:     list[str] = field(default_factory=list)
    
    # Quality tracking
    human_reviewed:       bool = False
    reviewer_confidence:  float = 1.0   # 0-1: how confident is the reviewer in the ground truth?

    def is_valid(self) -> bool:
        """Basic validation."""
        return (
            len(self.input.strip()) > 0 and
            len(self.ground_truth.strip()) > 0 and
            self.category in ("factual", "procedural", "comparison", "edge_case", "refusal") and
            self.difficulty in ("easy", "medium", "hard") and
            0 < self.reviewer_confidence <= 1.0
        )
```

### 2.3 Dataset Composition Guidelines

A well-balanced golden dataset should have:

| Category | Percentage | Why |
|---|---|---|
| Typical/happy path | 50–60% | Most representative of real usage |
| Edge cases | 15–20% | Tests robustness at boundaries |
| Known failures | 10–15% | Regression protection for specific issues |
| Out-of-scope | 5–10% | Tests graceful refusal behavior |
| Adversarial | 5–10% | Tests security and robustness |

**Size guidelines:**
- Minimum viable: 30–50 cases (can catch major regressions)
- Good: 100–200 cases (reliable statistics; catches subtle issues)
- Excellent: 300–500 cases (good statistical power; covers distribution well)
- Overkill unless needed: >1000 cases (expensive to run and maintain)

### 3.2 Ground Truth Versioning

```python
import json
from pathlib import Path
from datetime import datetime

class EvalDatasetManager:
    """Version-controlled eval dataset management."""
    
    def __init__(self, dataset_path: str):
        self.path   = Path(dataset_path)
        self.cases  = self._load()
    
    def _load(self) -> list[EvalCase]:
        if not self.path.exists():
            return []
        with open(self.path) as f:
            return [EvalCase(**c) for c in json.load(f)]
    
    def save(self, note: str = ""):
        """Save with timestamp and change note."""
        data = {
            "metadata": {
                "last_updated": datetime.utcnow().isoformat(),
                "n_cases": len(self.cases),
                "change_note": note
            },
            "cases": [vars(c) for c in self.cases]
        }
        with open(self.path, "w") as f:
            json.dump(data, f, indent=2)
        print(f"Saved {len(self.cases)} cases. Note: {note}")
    
    def add_case(self, case: EvalCase):
        if not case.is_valid():
            raise ValueError(f"Invalid case: {case}")
        # Check for duplicates
        existing = [c for c in self.cases if c.input == case.input]
        if existing:
            print(f"Warning: similar case already exists: {existing[0].case_id}")
        self.cases.append(case)
    
    def statistics(self) -> dict:
        """Dataset health statistics."""
        return {
            "total":          len(self.cases),
            "by_category":    {
                cat: sum(1 for c in self.cases if c.category == cat)
                for cat in ("factual", "procedural", "comparison", "edge_case", "refusal")
            },
            "by_difficulty":  {
                diff: sum(1 for c in self.cases if c.difficulty == diff)
                for diff in ("easy", "medium", "hard")
            },
            "human_reviewed": sum(1 for c in self.cases if c.human_reviewed),
            "avg_confidence": sum(c.reviewer_confidence for c in self.cases) / len(self.cases),
            "stale_check":    sum(1 for c in self.cases
                                  if c.source == "production_log" and
                                  c.created_at < "2025-01-01"),  # cases >18 months old
        }
```

## 4. Real-World Context

### The Production Log Mining Pattern

The highest-value eval cases come from production. The workflow:

```
1. Log all LLM requests (input, output, any user feedback)
2. Weekly: sample 20-30 logs from the past week
3. Filter: pick diverse query types, especially failures
4. Annotate: have domain expert add ground truth
5. Review: second person validates the ground truth
6. Add to dataset: save with metadata
```

Teams that do this build eval datasets that actually predict production quality because
they're directly sampled from production distribution.

### The Canary Case Strategy

Include 3–5 "canary cases" in every dataset — cases with very well-known, stable correct answers
that any reasonable system should get right. If these fail, something fundamental is broken:

```python
CANARY_CASES = [
    EvalCase(
        case_id="canary_001",
        input="What is 2+2?",
        ground_truth="4",
        category="factual",
        difficulty="easy",
        source="expert_written",
        human_reviewed=True,
        notes="Canary: if this fails, something is fundamentally broken"
    ),
    # Add domain-specific canaries: the most basic, stable facts about your product/domain
]
```

If canary cases fail: stop the release. Don't investigate other failures first.

## 5. Code: Complete Dataset Pipeline

```python
import anthropic
import json
import hashlib
from pathlib import Path

client = anthropic.Anthropic()

def build_eval_dataset_from_docs(
    documents: list[dict],
    output_path: str,
    n_generated: int = 50,
    include_adversarial: bool = True,
) -> EvalDatasetManager:
    """
    Full pipeline: generate from docs → add edge cases → save versioned dataset.
    REMINDER: generated cases require human review before use in CI/CD.
    """
    manager = EvalDatasetManager(output_path)
    
    # 1. Generate cases from documents
    print("Generating cases from documents...")
    generated = generate_eval_cases(
        domain_description="Customer support for an e-commerce platform",
        example_docs=[d["text"] for d in documents],
        n_cases=n_generated
    )
    
    for i, case_data in enumerate(generated):
        manager.add_case(EvalCase(
            case_id=f"gen_{hashlib.md5(case_data['input'].encode()).hexdigest()[:8]}",
            input=case_data["input"],
            ground_truth=case_data["ground_truth"],
            category=case_data.get("category", "factual"),
            difficulty=case_data.get("difficulty", "medium"),
            source="generated",
            human_reviewed=False,    # MUST review before production use
        ))
    
    # 2. Add adversarial edge cases
    if include_adversarial:
        adversarial = [
            EvalCase(
                case_id=f"adv_{i:03d}",
                input=template,
                ground_truth="",    # fill in manually after review
                category="edge_case",
                difficulty="hard",
                source="adversarial_template",
                human_reviewed=False,
            )
            for i, template in enumerate(EDGE_CASE_TEMPLATES[:10])
        ]
        for case in adversarial:
            manager.add_case(case)
    
    # 3. Add canary cases
    for canary in CANARY_CASES:
        manager.add_case(canary)
    
    # 4. Save
    manager.save(note=f"Initial dataset: {n_generated} generated + {10 if include_adversarial else 0} adversarial + {len(CANARY_CASES)} canaries")
    
    print(f"\nDataset statistics:")
    stats = manager.statistics()
    for k, v in stats.items():
        print(f"  {k}: {v}")
    
    print("\n⚠️  ACTION REQUIRED: Review generated cases (human_reviewed=False) before using in CI/CD")
    return manager
```

## 6. Trade-offs

### Dataset Size vs Maintenance Cost

Larger datasets:
- More statistical power; can detect smaller regressions
- Higher cost to run (more LLM calls per eval run)
- More maintenance work (keeping ground truths current)

**Sweet spot:** 100–200 well-curated cases beats 1000 poorly curated cases every time.
Quality > quantity for eval datasets.

### Generated vs Human-Written Cases

| Aspect | Generated (with review) | Human-Written |
|---|---|---|
| Speed to create | Fast (~10 min for 50 cases) | Slow (~2 hours for 50 cases) |
| Quality | Medium (needs review) | High |
| Domain expertise | No (model-generated) | Yes |
| Diversity | Can be biased by model's style | Controlled by expert |
| Cost | Low | High |

Use generated cases for scale, human-written for critical paths.

## 7. When To Use vs. When To Avoid

### A Golden Dataset
✅ Every AI feature that has been deployed or will be deployed
✅ Any system where you care about catching regressions
❌ One-off throwaway prototypes you'll never iterate on

### LLM-Generated Eval Cases (without human review)
✅ For initial exploration of failure modes
❌ As a CI/CD gate — never without human review first

⚠️ Common mistakes:
- Using the eval model to generate the eval data for itself — circular validation
- Not adding cases from production failures — missing the most valuable signal
- Ground truths written by one person, never reviewed — stale/wrong baselines
- Running evals only before major releases — should run continuously

## 8. Senior AI Engineer Perspective

### The Dataset Is the Contract

The golden dataset is the formal specification of what your system must do. When you can't
agree on what "good" means, formalize it in the dataset. When you have a disagreement about
whether a change is an improvement, the dataset is the tie-breaker.

Teams that invest in high-quality golden datasets report that it clarifies product requirements
as much as it enables engineering quality. "What should the system do here?" becomes a question
with a precise, version-controlled answer.
