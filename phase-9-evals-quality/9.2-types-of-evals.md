## 1. What Is It? (Theory)

### The Four Evaluator Types

Every AI system needs a combination of evaluator types, because no single type covers all quality dimensions:

```
DETERMINISTIC (Code-based):
  - Exact match, regex, schema validation, unit tests
  - Fast, cheap, 100% reproducible
  - Only covers well-defined, unambiguous properties

STATISTICAL (Metrics):
  - BLEU, ROUGE, semantic similarity, edit distance
  - Quantitative signals for structure/overlap
  - Correlate with quality but don't measure it directly

HUMAN EVALUATION:
  - Expert or crowd-sourced human judgment
  - Highest validity, captures nuance and subjective quality
  - Slow (days) and expensive ($5–$50/example)
  - The ground truth; calibrates everything else

LLM-AS-JUDGE:
  - Use a frontier model to grade outputs
  - Scales to thousands of examples cheaply
  - Biased; not as reliable as humans; but often good enough
  - The workhorse of modern AI eval pipelines
```

The right eval strategy uses all four, applied to different quality dimensions.

## 2. Deterministic Evals — Fast, Exact, Narrow

### 2.1 What They Cover

Deterministic evals work best for properties with clear, unambiguous specifications:

```python
import json, re

def run_deterministic_evals(output: str, context: str, query: str) -> dict:
    """Suite of deterministic checks on LLM output."""
    
    results = {}
    
    # FORMAT CHECKS
    results["is_non_empty"] = len(output.strip()) > 0
    results["under_500_words"] = len(output.split()) <= 500
    results["no_pii_leak"] = not bool(re.search(r'\b\d{3}-\d{2}-\d{4}\b', output))  # SSN pattern
    
    # STRUCTURE CHECKS (for JSON output)
    try:
        data = json.loads(output)
        results["valid_json"] = True
        results["has_required_fields"] = all(
            k in data for k in ["sentiment", "confidence", "summary"]
        )
        results["confidence_in_range"] = 0 <= data.get("confidence", -1) <= 1
        results["sentiment_valid"] = data.get("sentiment") in ["positive", "negative", "neutral"]
    except json.JSONDecodeError:
        results["valid_json"] = False
    
    # CITATION CHECKS (for RAG outputs)
    citation_pattern = r'\[Source \d+\]'
    results["has_citations"] = bool(re.search(citation_pattern, output))
    
    # SAFETY CHECKS
    banned_phrases = ["I cannot help", "I'm not able to", "As an AI language model"]
    results["no_bad_refusals"] = not any(p.lower() in output.lower() for p in banned_phrases)
    
    # GROUNDING CHECK (does answer reference specific context terms?)
    context_terms = set(context.lower().split())
    answer_terms  = set(output.lower().split())
    overlap = len(context_terms & answer_terms)
    results["references_context"] = overlap >= 3
    
    passed = sum(1 for v in results.values() if v is True)
    total  = len(results)
    
    return {
        "checks": results,
        "pass_rate": passed / total,
        "all_passed": all(v is True for v in results.values() if isinstance(v, bool))
    }
```

### 2.2 Practical Usage

```python
# Check every RAG response for basic quality properties before returning to user
output = rag_query("What is the refund policy?")

checks = run_deterministic_evals(
    output=output["answer"],
    context=output["context"],
    query="What is the refund policy?"
)

if not checks["all_passed"]:
    failed = [k for k, v in checks["checks"].items() if v is False]
    logger.warning(f"Quality checks failed: {failed}")
    # Optionally: retry, escalate, or add disclaimer
```

## 3. Statistical Evals — Metrics-Based

### 3.1 Common Metrics

```python
from sentence_transformers import SentenceTransformer
import numpy as np

embed_model = SentenceTransformer("all-MiniLM-L6-v2")

def semantic_similarity(text1: str, text2: str) -> float:
    """Cosine similarity between two texts — higher is more similar."""
    vecs = embed_model.encode([text1, text2], normalize_embeddings=True)
    return float(np.dot(vecs[0], vecs[1]))

def answer_matches_reference(
    generated: str,
    reference: str,
    threshold: float = 0.75
) -> dict:
    """Check if a generated answer is semantically similar to the reference."""
    score = semantic_similarity(generated, reference)
    return {
        "similarity": score,
        "passed": score >= threshold,
        "threshold": threshold
    }

# ROUGE-L (recall-oriented: what fraction of reference is in the output?)
def rouge_l(generated: str, reference: str) -> float:
    """Simplified ROUGE-L: longest common subsequence ratio."""
    gen_tokens = generated.lower().split()
    ref_tokens = reference.lower().split()
    
    if not ref_tokens:
        return 0.0
    
    common = set(gen_tokens) & set(ref_tokens)
    precision = len(common) / len(gen_tokens) if gen_tokens else 0
    recall    = len(common) / len(ref_tokens)
    
    if precision + recall == 0:
        return 0.0
    return 2 * precision * recall / (precision + recall)
```

### 3.2 When Statistical Metrics Help and Don't

| Metric | Good for | Bad for |
|---|---|---|
| Semantic similarity | Paraphrase detection, summary quality | Factual accuracy |
| ROUGE | Summary recall (did we cover the reference?) | Precision, fluency |
| BLEU | Translation quality | Open-ended generation |
| Exact match | Structured output (JSON fields) | Natural language |
| BERTScore | Better than ROUGE for semantic overlap | Still not factuality |

**The practical rule:** Statistical metrics are useful as sanity checks and for tracking trends.
They are **not** reliable judges of quality on their own. A high ROUGE score doesn't mean
the answer is correct. A low semantic similarity doesn't mean the answer is wrong.

## 4. Human Evaluation — The Ground Truth

### 4.1 When Human Eval Is Required

Human evaluation is expensive but irreplaceable for:
- **Calibrating LLM judges:** Is your LLM judge actually aligned with human preferences?
- **Launching new features:** First-time quality validation before automation takes over
- **Sensitive/high-stakes domains:** Medical, legal, financial — where LLM judge errors are unacceptable
- **Disagreement resolution:** When your automated metrics conflict
- **Rubric development:** Building the scoring criteria that LLM judges will use

### 4.2 Designing a Human Eval Protocol

```python
from dataclasses import dataclass
from typing import Optional
from enum import Enum

class Rating(Enum):
    EXCELLENT     = 5
    GOOD          = 4
    ACCEPTABLE    = 3
    POOR          = 2
    UNACCEPTABLE  = 1

@dataclass
class HumanEvalInstructions:
    """Instructions shown to human evaluators."""
    
    task_description = """
You are evaluating AI-generated customer support responses.

For each response, rate these dimensions independently (1-5):

HELPFULNESS (1-5):
  5 = Completely answers the question; actionable
  3 = Partially answers; leaves something unclear
  1 = Doesn't answer the question at all

ACCURACY (1-5):
  5 = Every factual claim is correct
  3 = Mostly correct; minor inaccuracies
  1 = Contains significant factual errors

TONE (1-5):
  5 = Professional, warm, on-brand
  3 = Adequate but could be better
  1 = Inappropriate, off-brand, or offensive

CONCISENESS (1-5):
  5 = Right length; no unnecessary content
  3 = Somewhat padded or too brief
  1 = Way too long or dangerously too short

GUIDELINES:
- Rate based ONLY on what is asked; don't assume context not provided
- Do not give bonus points for long answers
- Flag any response that seems to make up information not in the provided context
"""
    
    dispute_criteria = """
Flag as DISPUTED if:
- You are less than 80% confident in your rating
- The response is ambiguous in a way that affects the rating
- The question is unclear or underspecified
"""

@dataclass
class HumanEvalResult:
    case_id:      str
    helpfulness:  Rating
    accuracy:     Rating
    tone:         Rating
    conciseness:  Rating
    notes:        str = ""
    disputed:     bool = False
    
    @property
    def overall(self) -> float:
        return (self.helpfulness.value + self.accuracy.value + 
                self.tone.value + self.conciseness.value) / 4
```

### 4.3 Inter-Rater Reliability

With two or more human raters, always compute agreement:

```python
import numpy as np

def cohens_kappa(rater1_scores: list[int], rater2_scores: list[int]) -> float:
    """
    Cohen's Kappa: inter-rater agreement beyond chance.
    > 0.6 = good agreement; > 0.8 = excellent
    """
    n = len(rater1_scores)
    if n != len(rater2_scores):
        raise ValueError("Must have same number of ratings")
    
    # Observed agreement
    po = sum(r1 == r2 for r1, r2 in zip(rater1_scores, rater2_scores)) / n
    
    # Expected agreement (by chance)
    categories = set(rater1_scores) | set(rater2_scores)
    pe = sum(
        (rater1_scores.count(c) / n) * (rater2_scores.count(c) / n)
        for c in categories
    )
    
    if pe == 1:
        return 1.0
    return (po - pe) / (1 - pe)

# If kappa < 0.4: raters disagree too much — clarify rubric, retrain raters
```

## 5. LLM-as-Judge — Scalable but Biased

### 5.1 The Basic Pattern

```python
import anthropic
import json

judge_client = anthropic.Anthropic()

def llm_judge(
    question:   str,
    answer:     str,
    rubric:     str,
    judge_model: str = "claude-sonnet-4-6",   # don't use same model being judged
) -> dict:
    """
    LLM-as-judge: score a response according to a rubric.
    Returns structured assessment.
    """
    response = judge_client.messages.create(
        model=judge_model,
        max_tokens=512,
        system="""You are a strict, objective evaluator. Follow the rubric exactly.
Do not be lenient. Do not give bonus points. Evaluate only what is asked.""",
        messages=[{
            "role": "user",
            "content": f"""Evaluate this response.

Question: {question}

Response to evaluate:
{answer}

Rubric:
{rubric}

Output as JSON:
{{
  "scores": {{
    "criterion_1_name": <integer 1-5>,
    "criterion_2_name": <integer 1-5>
  }},
  "overall_score": <float 1-5>,
  "reasoning": "<2-3 sentences explaining the scores>",
  "issues": ["<specific issue 1>", "<specific issue 2>"]
}}"""
        }]
    )
    
    raw = response.content[0].text
    if raw.startswith("```"):
        raw = re.sub(r"```[a-z]*\n?", "", raw).strip("` \n")
    
    try:
        return json.loads(raw)
    except json.JSONDecodeError:
        return {"overall_score": 0, "error": "parse failure", "raw": raw[:200]}

# Usage
HELPFULNESS_RUBRIC = """
5 (Excellent): Directly answers the question; provides all needed information; actionable
4 (Good): Mostly answers the question; minor gaps
3 (Acceptable): Partially answers; user needs to ask follow-up
2 (Poor): Mostly doesn't answer; misleading or incomplete
1 (Unacceptable): Doesn't answer; wrong; potentially harmful
"""

result = llm_judge(
    question="How long does shipping take?",
    answer="Standard shipping takes 5-7 business days. Express options are available.",
    rubric=HELPFULNESS_RUBRIC
)
print(f"Score: {result['overall_score']}")
print(f"Reasoning: {result['reasoning']}")
```

### 5.2 Mitigating LLM Judge Biases

The known biases and their fixes:

```python
def debiased_llm_judge(
    question: str,
    answer_a: str,
    answer_b: str | None = None,   # for pairwise comparison
    rubric: str = HELPFULNESS_RUBRIC,
    n_runs: int = 3,  # run multiple times and average
) -> dict:
    """
    Judges with position debiasing and consistency averaging.
    
    Position bias fix: if comparing A vs B, also compare B vs A.
    Consistency fix: run multiple times and average.
    """
    
    if answer_b is None:
        # Single answer: just run multiple times
        scores = []
        for _ in range(n_runs):
            result = llm_judge(question, answer_a, rubric)
            scores.append(result.get("overall_score", 0))
        
        return {
            "mean_score": np.mean(scores),
            "std_score":  np.std(scores),
            "stable":     np.std(scores) < 0.5,  # consistent if std < 0.5
            "raw_scores": scores
        }
    
    else:
        # Pairwise: evaluate both orderings to cancel position bias
        ab_result = llm_judge(question, f"Response A:\n{answer_a}\n\nResponse B:\n{answer_b}", rubric)
        ba_result = llm_judge(question, f"Response A:\n{answer_b}\n\nResponse B:\n{answer_a}", rubric)
        
        # If both agree, reliable. If they disagree, uncertain.
        return {
            "ab_prefers_a": ab_result.get("overall_score", 0) > 3,
            "ba_prefers_b": ba_result.get("overall_score", 0) > 3,  # B is A in this run
            "agreement": ab_result.get("overall_score", 0) > 3 == ba_result.get("overall_score", 0) > 3,
        }
```

## 6. Choosing the Right Eval for Each Property

| Quality Property | Best Evaluator | Why |
|---|---|---|
| JSON is valid | Deterministic | Objective, unambiguous |
| Required fields present | Deterministic | Objective |
| No PII leakage | Deterministic (regex) | Objective, critical safety |
| Answer length appropriate | Deterministic (word count) | Measurable |
| Answer mentions topic | Statistical (keyword) | Cheap proxy |
| Semantic similarity to reference | Statistical (embedding) | Scalable |
| Is the answer correct? | Human or LLM-judge | Subjective, needs judgment |
| Is the tone appropriate? | Human or LLM-judge | Subjective |
| Is the response helpful? | Human (ground truth) or LLM-judge | Subjective |
| Is the answer faithful to context? | LLM-judge (RAGAS faithfulness) | Validated metric |
| Is the answer safe? | Deterministic + human | Too important for LLM-judge alone |

## 7. The Eval Pyramid

Just like the testing pyramid in software engineering, AI evals should be shaped like a pyramid:

```
            ▲
           / \
          /   \  Human eval (few, expensive, high validity)
         /     \
        /       \
       /         \  LLM-as-judge (hundreds, moderate cost)
      /           \
     /             \
    /               \  Statistical metrics (thousands, cheap)
   /                 \
  /___________________\  Deterministic checks (thousands, free, always run)
```

Most checks are at the bottom (cheap, fast, always). Fewer at the top (expensive, selective).

## 9. Senior AI Engineer Perspective

### The Most Important Insight About Evaluator Choice

**Use the cheapest evaluator that answers the question you actually care about.**

For "is the JSON valid?" — deterministic. Zero reason to use LLM-judge.
For "is this response helpful?" — you need LLM-judge or human.
For "is this medical advice correct?" — human expert only. LLM-judge is too risky.

The mistake teams make: using LLM-judge for everything because it's easier to set up.
This misses cheap wins (deterministic checks are free and perfect for format) and
also misapplies LLM-judge to properties it can't reliably measure.

### Always Have a Deterministic Floor

Whatever else you do, always run the deterministic checks first. They're free, instant,
and catch the most obvious failures. If the output isn't valid JSON when you expect JSON,
no other eval matters.
