## 1. What Is It? (Theory)

### Why Software Testing Skills Don't Transfer Cleanly

In traditional software engineering, testing is well-understood:
- A function either returns the right value or it doesn't
- Tests are deterministic: same input → same output → same result
- "Correctness" is binary and unambiguous

AI systems break every one of these assumptions:

```
Same input → Different output each run (non-determinism)
"Correct" answer is often subjective or context-dependent
Quality exists on a spectrum, not a binary
The thing you're testing can change silently (model updates)
Your test data can become stale (world changes)
```

This is why experienced software engineers who are great at traditional testing often
underestimate how hard AI evals are. The *concept* is the same (measure whether the system
does what you want) but the *implementation* is fundamentally different.

### The Six Reasons Evals Are Hard

**1. Non-determinism**

```python
# Same prompt, two runs, different results
response1 = call_llm("Summarize quantum computing in one sentence.")
# → "Quantum computing uses quantum mechanical phenomena to solve complex problems."

response2 = call_llm("Summarize quantum computing in one sentence.")
# → "Quantum computers harness quantum superposition and entanglement to process information."
```

Both are correct. Neither matches a simple string comparison. How do you write a test?

**2. Subjective quality**

"Is this a good customer support response?" has no objective answer. Different humans
would rate the same response differently. There's no ground truth — only probability distributions
over human preferences.

**3. The reference problem**

To evaluate whether an output is "good," you need a reference. But:
- Writing reference answers takes expert time
- References go stale (correct answer in 2024 may be outdated in 2026)
- For generative tasks, there are multiple valid answers, not one
- The reference itself may have errors

**4. Distribution mismatch**

The inputs you test on aren't the inputs users actually send. Your carefully constructed
eval set may cover 80% of the quality space your users care about — while silently missing
the other 20% that actually causes complaints.

**5. The goodhart problem**

When you optimize for a metric, that metric stops being a good measure of what you care about.

```
Example: Optimize for "user gives 5-star rating"
Result:  Model becomes sycophantic — agrees with everything the user says
         Users rate it 5 stars... but it's giving wrong answers
         The metric is high; quality is low
```

**6. Evaluation costs money and time**

Human evaluation is the gold standard but costs $5–$50 per example with domain experts.
LLM-as-judge is cheaper but biased. Automated metrics are cheap but incomplete.
The tradeoff between cost, speed, and accuracy has no perfect solution.

## 2. The Consequence of Skipping Evals

### The Silent Quality Degradation Pattern

The most dangerous AI quality problem isn't sudden breakage — it's gradual, invisible decline.

```
Month 1:  You ship the feature. Manually test 10 examples. "Looks good!"
Month 2:  Model provider silently updates the model. Behavior shifts subtly.
Month 3:  Users start complaining. You don't know when it changed.
Month 4:  You can't revert — you don't know what to revert to.
Month 5:  You're debugging in the dark, manually testing variations.
```

This is the most common failure mode in production AI systems in 2025–2026.

With evals:
```
Month 2:  CI/CD runs eval suite after model update. Score drops 8%.
           Alert fires. You investigate. You find the regression.
           You either fix or stay on the old model version.
```

### The "But Our Users Love It" Trap

Teams without evals often rely on user satisfaction as their quality signal. This has several
problems:
- **Survivorship bias:** Only engaged users give feedback. Silent dissatisfied users churn.
- **Sycophancy hides quality issues:** Users may rate confident wrong answers higher than
  uncertain correct answers.
- **Lag:** User feedback comes days or weeks after the quality regression.
- **Resolution:** "Users are unhappy" doesn't tell you *what* broke.

Evals tell you what changed before users notice.

## 3. The Evaluation Maturity Ladder

Most teams go through these stages:

```
Level 0: No evaluation
  "I tested it manually, it works."
  Problem: No way to detect regressions or measure improvements.

Level 1: Manual spot checks
  "I test 5-10 examples before shipping."
  Problem: Too few examples; not representative; not repeatable.

Level 2: Automated string matching
  "I check if the output contains expected keywords."
  Problem: Brittle; misses semantic quality; high false-positive rate.

Level 3: LLM-as-judge
  "I use a model to grade model outputs."
  Problem: Judge bias, cost, and the chicken-and-egg problem.
  But: Scalable and better than manual for many tasks.

Level 4: Golden dataset + automated eval pipeline
  "I have 200 curated test cases with known-good answers, run automatically."
  Problem: Dataset maintenance; still misses edge cases.
  But: Reproducible, historical, catches regressions reliably.

Level 5: Continuous eval + observability
  "Every deployment runs evals. Production samples are scored continuously.
   Model drift alerts fire automatically."
  This is the target for production AI systems.
```

Most teams in 2026 are at Level 1–2. Level 4 is achievable for any team with ~2 weeks of
focused investment. Level 5 requires ongoing infrastructure investment.

## 4. What "Good Evals" Look Like

Good evals have five properties:

**Relevant:** The eval measures something that actually predicts user satisfaction,
not a proxy metric that games easily.

**Representative:** The test cases reflect the real distribution of user inputs, including
edge cases, adversarial inputs, and failure modes — not just the easy happy path.

**Sensitive:** The eval detects quality changes. If you make a meaningful improvement and
the eval score doesn't move, the eval isn't measuring what matters.

**Stable:** The eval produces consistent scores. Running the same suite twice should give
similar results (within statistical noise).

**Affordable:** The eval can be run frequently. An eval that costs $500 and takes 2 days
won't be run before every deployment.

## 5. The Right Mindset for Eval Work

### Evals as Specifications

The best mental model: **evals are the specification for your AI system's behavior**.

Before writing any code for a new AI feature:
1. Define what "good" looks like (the properties of correct outputs)
2. Write test cases that check those properties
3. Build the system until it passes

This is TDD (test-driven development) applied to AI. Teams that do this build better systems
faster because they know what they're targeting from day one.

### The Measurement-First Principle

You cannot improve what you cannot measure. Before optimizing any AI component:
- Establish a baseline metric
- Understand what it's measuring and its limitations
- Make changes and measure the delta
- Only keep changes that move the metric in the right direction

Intuition-based optimization ("I think this prompt sounds better") is unreliable for AI systems.
Measurement-based optimization is repeatable and compounding.

## 6. Code Examples

### The Simplest Possible Eval

```python
import anthropic
from dataclasses import dataclass

client = anthropic.Anthropic()

@dataclass
class EvalCase:
    input:          str
    expected:       str       # Reference answer or keywords
    check_fn:       callable  # How to grade the output

def run_single_eval(case: EvalCase, system: str, model: str = "claude-sonnet-4-6") -> dict:
    response = client.messages.create(
        model=model,
        max_tokens=256,
        system=system,
        messages=[{"role": "user", "content": case.input}],
        temperature=0   # deterministic for evals
    )
    output = response.content[0].text
    passed = case.check_fn(output, case.expected)
    
    return {
        "input":    case.input,
        "output":   output,
        "expected": case.expected,
        "passed":   passed,
        "tokens":   response.usage.input_tokens + response.usage.output_tokens
    }

def run_eval_suite(cases: list[EvalCase], system: str, model: str = "claude-sonnet-4-6") -> dict:
    results = [run_single_eval(c, system, model) for c in cases]
    
    n_passed  = sum(r["passed"] for r in results)
    pass_rate = n_passed / len(results)
    total_cost = sum(r["tokens"] for r in results) * 3 / 1_000_000  # rough Sonnet cost
    
    print(f"\n{'='*50}")
    print(f"Results: {n_passed}/{len(results)} passed ({pass_rate:.0%})")
    print(f"Estimated cost: ${total_cost:.4f}")
    
    failures = [r for r in results if not r["passed"]]
    if failures:
        print(f"\nFailures ({len(failures)}):")
        for f in failures[:3]:
            print(f"  Input: {f['input'][:60]}")
            print(f"  Got:   {f['output'][:80]}")
    
    return {"pass_rate": pass_rate, "results": results, "cost_usd": total_cost}

# Example
cases = [
    EvalCase(
        input="What is 2+2?",
        expected="4",
        check_fn=lambda output, exp: exp in output
    ),
    EvalCase(
        input="What is the capital of France?",
        expected="Paris",
        check_fn=lambda output, exp: exp.lower() in output.lower()
    ),
]

run_eval_suite(cases, system="Answer concisely.")
```

## 7. Senior AI Engineer Perspective

### The Eval Investment Is Never Wasted

The most common objection: "We don't have time for evals right now, we need to ship."

The math:
- Building evals for a feature: ~1–3 days
- Debugging a production regression without evals: ~3–7 days
- The cost of user trust lost during a quality regression: uncalculated

Teams that skip evals to ship faster end up slower over time. Every quality regression that
reaches users takes longer to diagnose and fix without the measurement infrastructure.
The compounding cost of undetected regressions dwarfs the upfront eval investment.

### Start Small, Stay Consistent

A 50-case golden dataset that you run consistently and maintain is worth more than a
500-case dataset you run once and forget. Start with 30–50 representative cases. Add 5–10
cases every time you encounter a production failure. Grow the dataset incrementally.

After 6 months of this discipline, you'll have a 150–200 case dataset that accurately reflects
your real-world distribution and has caught dozens of regressions.
