## 1. What Is It? (Theory)

### Why LLM-as-Judge Is Now the Default

By 2026, LLM judges agree with human reviewers ~85% of the time — higher than two humans
agree with each other. This makes LLM-as-judge the practical default for evaluating LLM
applications at scale.

The math:
- Human evaluation: $10–$50 per case, days to complete
- LLM-as-judge: $0.01–$0.05 per case, minutes to complete
- Scaling from 100 to 10,000 cases: human eval goes from $1K–$5K to $100K–$500K;
  LLM-judge goes from $100–$500 to $1K–$5K

For iterative development, you need to run evals frequently. At human eval cost, you run them
quarterly. At LLM-judge cost, you run them on every commit.

### The Critical Limitations

LLM-as-judge is not perfect. Knowing its failure modes is essential to using it correctly:

1. **Self-preference bias:** Models prefer outputs that look like their own training data
2. **Verbosity bias:** Longer responses score higher regardless of actual quality
3. **Position bias:** In pairwise comparisons, the first response is often preferred
4. **Sycophancy:** Judges tend to agree with authoritative-sounding statements
5. **Language bias:** Non-English responses score lower even at the same quality
6. **Rubric sensitivity:** Small changes in the judging prompt cause large score changes

These are not bugs — they're systematic biases baked into the model's training. Knowing them
lets you compensate.

## 2. How It Actually Works — Deep Mechanics

### 2.1 Single-Answer Scoring

The simplest pattern: judge one response on a numeric scale.

```python
import anthropic
import json
import re

judge_client = anthropic.Anthropic()

def score_response(
    question: str,
    response: str,
    criteria: dict[str, str],   # {criterion_name: description}
    judge_model: str = "claude-sonnet-4-6",
) -> dict:
    """
    Score a response on multiple criteria.
    Returns scores + reasoning for auditability.
    """
    criteria_text = "\n".join(
        f"{i+1}. {name}: {desc}"
        for i, (name, desc) in enumerate(criteria.items())
    )
    
    score_schema = {k: "integer 1-5" for k in criteria}
    
    response_obj = judge_client.messages.create(
        model=judge_model,
        max_tokens=512,
        system="""You are a strict, calibrated evaluator.
Score objectively. Do not reward length or confidence.
If uncertain, score lower. Explain your reasoning concisely.""",
        messages=[{
            "role": "user",
            "content": f"""Evaluate this AI response:

Question: {question}

AI Response:
{response}

Criteria (score each 1-5):
{criteria_text}

Score descriptions:
5 = Excellent: exceeds expectations
4 = Good: meets expectations
3 = Acceptable: partially meets
2 = Poor: mostly fails
1 = Unacceptable: completely fails

Output JSON: {{"scores": {score_schema}, "reasoning": "<2-3 sentences>", "issues": ["issue1"]}}"""
        }]
    )
    
    raw = response_obj.content[0].text
    # Strip markdown code fences
    if "```" in raw:
        raw = re.sub(r"```[a-z]*\n?", "", raw).strip("` \n")
    
    try:
        data = json.loads(raw)
        scores = data.get("scores", {})
        overall = sum(scores.values()) / len(scores) if scores else 0
        return {
            "scores":       scores,
            "overall":      round(overall, 2),
            "reasoning":    data.get("reasoning", ""),
            "issues":       data.get("issues", []),
            "judge_model":  judge_model,
        }
    except json.JSONDecodeError:
        return {"overall": 0, "error": "parse failure", "raw": raw[:200]}

# Usage
result = score_response(
    question="What is our return policy for electronics?",
    response="Electronics can be returned within 30 days if unopened.",
    criteria={
        "accuracy":     "Is every factual claim correct based on general knowledge?",
        "completeness": "Does the response cover all aspects of the question?",
        "clarity":      "Is the response easy to understand?",
    }
)
print(f"Overall: {result['overall']:.1f}/5")
print(f"Scores:  {result['scores']}")
print(f"Issues:  {result['issues']}")
```

### 2.2 Pairwise Comparison

Often more reliable than absolute scoring — compare two responses and pick the better one:

```python
def pairwise_compare(
    question: str,
    response_a: str,
    response_b: str,
    rubric: str,
    judge_model: str = "claude-sonnet-4-6",
    n_trials: int = 2,   # run in both orderings to cancel position bias
) -> dict:
    """
    Compare two responses. Runs A-vs-B and B-vs-A to cancel position bias.
    Returns: {"winner": "a"|"b"|"tie", "confidence": 0-1}
    """
    
    def judge_pair(first: str, second: str, first_label: str, second_label: str) -> str:
        r = judge_client.messages.create(
            model=judge_model, max_tokens=256,
            system="You are a strict evaluator. Pick the better response. Do not favor length.",
            messages=[{
                "role": "user",
                "content": f"""Question: {question}

Response {first_label}:
{first}

Response {second_label}:
{second}

Rubric: {rubric}

Which is better? Output JSON: {{"winner": "{first_label}"|"{second_label}"|"tie", "reason": "..."}}"""
            }]
        )
        raw = r.content[0].text
        if "```" in raw:
            raw = re.sub(r"```[a-z]*\n?", "", raw).strip("` \n")
        try:
            return json.loads(raw).get("winner", "tie")
        except json.JSONDecodeError:
            return "tie"
    
    # Trial 1: A first
    trial1 = judge_pair(response_a, response_b, "A", "B")
    
    # Trial 2: B first (swap order to cancel position bias)
    trial2_raw = judge_pair(response_b, response_a, "A", "B")
    # In trial 2, "A" is actually B and vice versa
    trial2 = "b" if trial2_raw == "A" else ("a" if trial2_raw == "B" else "tie")
    
    # Determine consensus
    if trial1.lower() == "a" and trial2 == "a":
        return {"winner": "a", "confidence": 1.0}
    elif trial1.lower() == "b" and trial2 == "b":
        return {"winner": "b", "confidence": 1.0}
    elif trial1.lower() == "a" and trial2 == "b":
        return {"winner": "tie", "confidence": 0.5, "note": "disagreement across orderings"}
    elif trial1.lower() == "b" and trial2 == "a":
        return {"winner": "tie", "confidence": 0.5, "note": "disagreement across orderings"}
    else:
        return {"winner": "tie", "confidence": 0.7}
```

### 2.3 Chain-of-Thought Judging

For complex rubrics, asking the judge to reason step-by-step before scoring improves reliability:

```python
def cot_judge(question: str, response: str, rubric: str) -> dict:
    """
    Chain-of-thought judging: reason before scoring.
    More reliable than direct scoring for complex criteria.
    """
    # Step 1: Ask the judge to analyze the response
    analysis = judge_client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=512,
        messages=[{
            "role": "user",
            "content": f"""Analyze this AI response step by step before scoring.

Question: {question}
Response: {response}
Rubric: {rubric}

Step 1: What does the response claim?
Step 2: Is each claim supported/correct?
Step 3: What's missing?
Step 4: What issues exist?
Step 5: Summary of strengths and weaknesses."""
        }]
    )
    reasoning = analysis.content[0].text
    
    # Step 2: Score based on the analysis
    scoring = judge_client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=128,
        messages=[{
            "role": "user",
            "content": f"""Based on this analysis:
{reasoning}

Score the response 1-5 on the rubric: {rubric}
Output JSON: {{"score": <1-5>, "justification": "<one sentence>"}}"""
        }]
    )
    
    try:
        raw = scoring.content[0].text
        if "```" in raw:
            raw = re.sub(r"```[a-z]*\n?", "", raw).strip("` \n")
        data = json.loads(raw)
        return {"score": data.get("score", 0), "reasoning": reasoning, "justification": data.get("justification", "")}
    except json.JSONDecodeError:
        return {"score": 0, "reasoning": reasoning, "error": "parse failure"}
```

### 2.4 Reference-Based Judging (G-Eval Pattern)

When you have a reference answer, include it as context for the judge:

```python
def reference_based_judge(
    question: str,
    generated: str,
    reference: str,   # the ground truth answer
    judge_model: str = "claude-sonnet-4-6",
) -> dict:
    """
    Judge a response relative to a reference answer.
    More reliable than rubric-only judging for factual tasks.
    """
    result = judge_client.messages.create(
        model=judge_model,
        max_tokens=256,
        system="You are an objective evaluator. Compare the response to the reference. Be strict about factual accuracy.",
        messages=[{
            "role": "user",
            "content": f"""Question: {question}

Reference answer (correct): {reference}

Generated response (to evaluate):
{generated}

Compare the generated response to the reference. Output JSON:
{{
  "factual_accuracy":  <1-5>,  // how well facts match reference
  "completeness":      <1-5>,  // how much of the reference is covered
  "no_hallucinations": <1-5>,  // 5=nothing in generated that contradicts reference
  "overall":           <1-5>,
  "missing_from_generated": ["key points in reference not in generated"],
  "extra_in_generated":     ["claims in generated not in reference"]
}}"""
        }]
    )
    
    raw = result.content[0].text
    if "```" in raw:
        raw = re.sub(r"```[a-z]*\n?", "", raw).strip("` \n")
    
    try:
        return json.loads(raw)
    except json.JSONDecodeError:
        return {"overall": 0, "error": "parse failure"}
```

## 3. Calibrating Your Judge

A judge you haven't calibrated against human labels is a judge you can't trust.

```python
import numpy as np
from scipy.stats import pearsonr, spearmanr

def calibrate_judge(
    judge_fn,          # your LLM judge function
    human_scores: list[dict],   # [{"question": ..., "response": ..., "score": 1-5}]
) -> dict:
    """
    Compare your judge's scores to human scores.
    Returns correlation and agreement statistics.
    """
    judge_scores = []
    
    for case in human_scores:
        result = judge_fn(case["question"], case["response"])
        judge_scores.append(result.get("overall", 0))
    
    human_vals = [c["score"] for c in human_scores]
    
    pearson_r,  p_pearson  = pearsonr(human_vals, judge_scores)
    spearman_r, p_spearman = spearmanr(human_vals, judge_scores)
    
    # Agreement within 1 point
    within_1 = sum(abs(h - j) <= 1 for h, j in zip(human_vals, judge_scores)) / len(human_vals)
    exact_match = sum(round(h) == round(j) for h, j in zip(human_vals, judge_scores)) / len(human_vals)
    
    print(f"Judge Calibration Results:")
    print(f"  Pearson r:     {pearson_r:.3f} (target: > 0.7)")
    print(f"  Spearman r:    {spearman_r:.3f} (target: > 0.7)")
    print(f"  Within-1:      {within_1:.0%} (target: > 80%)")
    print(f"  Exact match:   {exact_match:.0%}")
    
    # Diagnose biases
    mean_human = np.mean(human_vals)
    mean_judge = np.mean(judge_scores)
    
    if mean_judge > mean_human + 0.5:
        print(f"  ⚠️  Judge is LENIENT by {mean_judge - mean_human:.2f} points on average")
    elif mean_judge < mean_human - 0.5:
        print(f"  ⚠️  Judge is HARSH by {mean_human - mean_judge:.2f} points on average")
    
    return {
        "pearson_r":    pearson_r,
        "spearman_r":   spearman_r,
        "within_1_pct": within_1,
        "exact_match":  exact_match,
        "calibrated":   pearson_r > 0.7 and within_1 > 0.8
    }
```

## 4. Real-World Context

### When to Trust LLM-as-Judge

| Use Case | Reliability | Notes |
|---|---|---|
| Response helpfulness | High | Good correlation with human preference |
| Answer completeness | High | Judge can identify missing elements |
| Factual accuracy | Medium | Biased by model's own knowledge |
| Tone/safety | Medium | Good but misses cultural nuance |
| Medical/legal accuracy | Low | Use domain expert humans only |
| Code correctness | Low | Run the code instead; don't judge it |

**The 85% agreement figure** (LLM judge vs human) applies to general-purpose tasks.
For specialized domains, calibrate before trusting.

### The Meta-Evaluation Problem

If you use Claude to judge Claude outputs, there's a conflict of interest: the judge knows
how Claude writes and may prefer Claude-style outputs. This is **self-preference bias**.

Best practices:
- Use a different model family as judge when possible (e.g., evaluate Claude outputs with GPT-4o)
- Use a stronger/more capable model as judge than the one being judged
- If you must use the same family, use a different variant (e.g., Opus as judge, Sonnet as generator)

## 6. Trade-offs

### Direct Score vs Pairwise Comparison

**Direct scoring** (1–5 scale):
- Simpler to run (one API call per case)
- Useful for tracking trends over time
- Absolute scores are hard to calibrate
- More sensitive to verbosity bias

**Pairwise comparison** (A vs B):
- More natural for humans and LLMs
- Better for comparing two versions
- Cancels some biases (when done in both orderings)
- Can't easily build longitudinal trends

**Rule:** Use pairwise for comparing two system versions; use direct scoring for longitudinal tracking.

## 7. When To Use vs. When To Avoid

### LLM-as-Judge
✅ Use when: You need to evaluate thousands of outputs regularly; cost of human eval is prohibitive
✅ Use when: Task is general-purpose (helpfulness, clarity, appropriateness)
✅ Use after: Calibrating against at least 50 human-labeled examples
❌ Use for: Code correctness (run the code), mathematical accuracy (check the math), medical/legal
❌ Use without: Calibration against human labels — you don't know if the judge is reliable

⚠️ Common mistakes:
- Using the same model as judge and generation (self-preference bias)
- Not randomizing position in pairwise comparisons
- Using verbose rubrics with many criteria — causes score compression
- Not auditing judge outputs — judges can be confidently wrong

## 8. Senior AI Engineer Perspective

### The Rubric Is Everything

The single biggest driver of LLM judge quality is the rubric. A good rubric:
- Defines each point on the scale precisely (not just "1=bad, 5=good")
- Gives examples of what each score looks like
- Separates criteria that might conflict (helpfulness vs conciseness)
- Tells the judge what NOT to reward (length, confident tone, formatting)

Invest 2–4 hours in the rubric before running hundreds of cases. The rubric determines
what behavior you're selecting for. A poorly written rubric produces a system optimized
for the wrong thing.

### The Judge Audit Practice

Every week or sprint:
1. Sample 10 cases where the judge gave extreme scores (very high or very low)
2. Manually read the (question, response, score, reasoning)
3. Do you agree with the judge? Are the reasons coherent?
4. If not: update your rubric; investigate judge calibration

Without this audit loop, judge drift and bias go undetected. The judge may silently diverge
from human preferences over model updates.
