## 1. What Is It? (Theory)

### Why Regular Databases Fail at Vector Search

A relational database is optimized for exact lookups, range queries, and joins on structured data.
It stores rows with typed columns and indexes them with B-trees.

"Find the 10 vectors most similar to this query vector" is a fundamentally different operation:
- No exact match — you want approximate proximity
- High-dimensional space — B-trees don't work in 1,536 dimensions
- Large payloads — 1,536 floats × 4 bytes = 6KB per row, millions of rows

Vector databases are built specifically for this: they store embeddings alongside metadata,
build ANN indexes (HNSW, IVF) over the embedding space, and allow combined vector + metadata
filtering in a single query.

### The Production Feature Set

A production vector database needs:

```
1. Vector storage          — store embeddings efficiently
2. ANN indexing            — fast approximate nearest neighbor search
3. Metadata filtering      — find vectors WHERE category='tech' AND date>'2024-01-01'
4. CRUD operations         — update/delete vectors after indexing
5. Namespacing/tenancy     — separate collections per user/project
6. Persistence             — survive restarts
7. Scalability             — horizontal scaling for large collections
8. Batch ingestion         — ingest millions of vectors efficiently
```

Different databases prioritize different features — which is why choosing the right one matters.

## 2. Deep Dive: The Major Options

### 2.1 Qdrant

**The recommendation for most new projects in 2026.** Rust implementation → best latency
among open-source vector DBs. Strong filtering. Active development. Self-hosted or cloud.

```
Strengths:
  - Best query latency among open-source options (Rust, memory-safe)
  - Rich payload (metadata) filtering with HNSW + filter indexing
  - Flexible deployment: local, Docker, Kubernetes, Qdrant Cloud
  - gRPC and REST APIs
  - Native support for sparse vectors (for hybrid search)
  - Collection snapshots for backup

Weaknesses:
  - Newer than Pinecone/Weaviate; smaller ecosystem
  - Cloud tier less established than Pinecone
```

```python
from qdrant_client import QdrantClient
from qdrant_client.models import (
    VectorParams, Distance, PointStruct, Filter,
    FieldCondition, MatchValue, Range
)
import numpy as np

# Connect (local Docker or Qdrant Cloud)
client = QdrantClient(url="http://localhost:6333")
# or: client = QdrantClient(url="https://xyz.qdrant.tech", api_key="...")

COLLECTION = "documents"
DIM = 1536

# 1. Create collection with HNSW index
client.recreate_collection(
    collection_name=COLLECTION,
    vectors_config=VectorParams(size=DIM, distance=Distance.COSINE),
    # Optional HNSW tuning:
    # hnsw_config=HnswConfigDiff(m=16, ef_construct=100)
)

# 2. Ingest documents with metadata
points = [
    PointStruct(
        id=i,
        vector=np.random.rand(DIM).tolist(),
        payload={
            "text": f"Document {i}",
            "category": "tech" if i % 2 == 0 else "science",
            "year": 2020 + (i % 5),
            "score": float(i) / 100
        }
    )
    for i in range(1000)
]

client.upsert(collection_name=COLLECTION, points=points)
print(f"Indexed {len(points)} vectors")

# 3. Pure vector search
query_vector = np.random.rand(DIM).tolist()
results = client.search(
    collection_name=COLLECTION,
    query_vector=query_vector,
    limit=5
)

for r in results:
    print(f"Score: {r.score:.3f} | id={r.id} | {r.payload['text']}")

# 4. Filtered vector search (vector similarity + metadata filter)
filtered_results = client.search(
    collection_name=COLLECTION,
    query_vector=query_vector,
    query_filter=Filter(
        must=[
            FieldCondition(key="category", match=MatchValue(value="tech")),
            FieldCondition(key="year",     range=Range(gte=2022, lte=2024))
        ]
    ),
    limit=5
)

print(f"\nFiltered results (tech, 2022-2024): {len(filtered_results)}")

# 5. Delete a vector
client.delete(collection_name=COLLECTION, points_selector=[42])

# 6. Update payload (metadata) without changing vector
client.set_payload(
    collection_name=COLLECTION,
    payload={"category": "updated"},
    points=[1, 2, 3]
)
```

### 2.2 pgvector (PostgreSQL Extension)

**The right choice if you already run PostgreSQL.** Adds vector search as a first-class feature.
Huge advantage: your app already knows how to query Postgres. No new infrastructure, no new
SDK, no new concepts for your team.

```
Strengths:
  - Zero new infrastructure if you have Postgres
  - Full SQL power for metadata filtering and joins
  - Transactions, ACID compliance
  - Every Postgres hosting provider supports it (Supabase, Neon, AWS RDS, etc.)
  - Rich operator support: <=>, <#>, <->

Weaknesses:
  - Single-node (vertical scaling only, not horizontal)
  - IVFFlat and HNSW both good; HNSW added in pgvector 0.5.0+
  - Practical ceiling: ~10–50M vectors on a well-provisioned instance
```

```python
import psycopg2
import numpy as np
from pgvector.psycopg2 import register_vector

# Install: pip install pgvector psycopg2-binary
# SQL: CREATE EXTENSION IF NOT EXISTS vector;

conn = psycopg2.connect("postgresql://user:pass@localhost/db")
register_vector(conn)

cur = conn.cursor()

# 1. Create table with vector column
cur.execute("""
    CREATE TABLE IF NOT EXISTS documents (
        id         SERIAL PRIMARY KEY,
        content    TEXT,
        embedding  VECTOR(1536),
        category   VARCHAR(50),
        created_at TIMESTAMP DEFAULT NOW()
    )
""")

# 2. Create HNSW index for fast ANN search
cur.execute("""
    CREATE INDEX IF NOT EXISTS doc_embedding_hnsw
    ON documents
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64)
""")

conn.commit()

# 3. Insert documents with embeddings
docs = [
    ("Machine learning overview", np.random.rand(1536), "tech"),
    ("History of ancient Rome",   np.random.rand(1536), "history"),
    ("Python programming guide",  np.random.rand(1536), "tech"),
]

for content, embedding, category in docs:
    cur.execute(
        "INSERT INTO documents (content, embedding, category) VALUES (%s, %s, %s)",
        (content, embedding.tolist(), category)
    )
conn.commit()

# 4. Semantic search (cosine similarity = <=>)
query_vec = np.random.rand(1536).tolist()

cur.execute("""
    SELECT id, content, category,
           1 - (embedding <=> %s::vector) AS similarity
    FROM documents
    ORDER BY embedding <=> %s::vector
    LIMIT 5
""", (query_vec, query_vec))

for row in cur.fetchall():
    print(f"id={row[0]} | sim={row[3]:.3f} | {row[1]} ({row[2]})")

# 5. Filtered search (SQL WHERE + vector similarity — best of both worlds)
cur.execute("""
    SELECT id, content,
           1 - (embedding <=> %s::vector) AS similarity
    FROM documents
    WHERE category = 'tech'
    ORDER BY embedding <=> %s::vector
    LIMIT 5
""", (query_vec, query_vec))

print(f"\nTech documents only:")
for row in cur.fetchall():
    print(f"  {row[1]} | sim={row[2]:.3f}")

conn.close()
```

### 2.3 Pinecone

**Managed, serverless, zero ops.** Pay-per-use. The easiest to get started with for teams
that don't want to manage infrastructure.

```
Strengths:
  - Fully managed — zero infrastructure to maintain
  - Serverless (2024+) — no idle cost; scale to zero
  - Consistent < 5ms p99 query latency
  - Enterprise SLAs, SOC2, HIPAA compliance
  - Namespace support for multi-tenancy

Weaknesses:
  - Most expensive at high scale
  - Less flexible filtering than Qdrant
  - Data lives in Pinecone cloud
  - Vendor lock-in
```

```python
from pinecone import Pinecone, ServerlessSpec
import numpy as np

pc = Pinecone(api_key="your-api-key")

# 1. Create serverless index
pc.create_index(
    name="my-index",
    dimension=1536,
    metric="cosine",
    spec=ServerlessSpec(cloud="aws", region="us-east-1")
)

index = pc.Index("my-index")

# 2. Upsert vectors with metadata
vectors = [
    (f"doc-{i}", np.random.rand(1536).tolist(), {"text": f"Doc {i}", "category": "tech"})
    for i in range(100)
]

index.upsert(vectors=vectors)

# 3. Search
query = np.random.rand(1536).tolist()
results = index.query(vector=query, top_k=5, include_metadata=True)

for match in results.matches:
    print(f"Score: {match.score:.3f} | id={match.id} | {match.metadata}")

# 4. Filtered search
results = index.query(
    vector=query,
    top_k=5,
    filter={"category": {"$eq": "tech"}},   # Pinecone's filter syntax
    include_metadata=True
)
```

### 2.4 Weaviate

**Best native hybrid search** (vector + BM25 in one query). Good for applications that need
both semantic and keyword matching natively.

```
Strengths:
  - Native hybrid search (dense + BM25 in one query)
  - GraphQL and REST APIs
  - Multi-modal support (images, text in same collection)
  - Open source + Weaviate Cloud
  - Strong text2vec module integration (auto-embed on insert)

Weaknesses:
  - JVM-based (higher memory usage than Qdrant)
  - More complex setup than Qdrant
  - GraphQL API has a learning curve
```

```python
import weaviate
import weaviate.classes as wvc
import numpy as np

client = weaviate.connect_to_local()  # or connect_to_weaviate_cloud(...)

# 1. Create collection
collection = client.collections.create(
    name="Documents",
    vectorizer_config=wvc.config.Configure.Vectorizer.none(),   # bring your own vectors
    properties=[
        wvc.config.Property(name="text",     data_type=wvc.config.DataType.TEXT),
        wvc.config.Property(name="category", data_type=wvc.config.DataType.TEXT),
    ]
)

# 2. Insert
docs = client.collections.get("Documents")
with docs.batch.dynamic() as batch:
    for i in range(100):
        batch.add_object(
            properties={"text": f"Document {i}", "category": "tech"},
            vector=np.random.rand(1536).tolist()
        )

# 3. Vector search
results = docs.query.near_vector(
    near_vector=np.random.rand(1536).tolist(),
    limit=5,
    return_properties=["text", "category"]
)
for r in results.objects:
    print(f"{r.properties}")

# 4. Hybrid search (vector + BM25 keyword)
results = docs.query.hybrid(
    query="machine learning tutorial",
    vector=np.random.rand(1536).tolist(),
    alpha=0.5,   # 0 = pure keyword, 1 = pure vector, 0.5 = equal blend
    limit=5,
    return_properties=["text"]
)
for r in results.objects:
    print(r.properties)

client.close()
```

## 3. Head-to-Head Comparison (2026)

| Feature | Qdrant | pgvector | Pinecone | Weaviate |
|---|---|---|---|---|
| **Architecture** | Rust, purpose-built | PostgreSQL extension | Managed cloud | JVM, purpose-built |
| **Hosting** | Self-host or cloud | Self-host / any PG host | Cloud only | Self-host or cloud |
| **Cost (self-hosted)** | Free | Free (pay for Postgres) | N/A | Free |
| **Query latency** | 🥇 Fastest open-source | Good | 🥇 Fastest managed | Good |
| **Filtering** | Excellent | Full SQL | Good | Good |
| **Hybrid search** | Yes (sparse + dense) | Partial (pgvector 0.7+) | Partial | 🥇 Best native |
| **Multi-tenancy** | Namespaces/collections | Schema/databases | Namespaces | Tenants |
| **Max scale** | Billions (distributed) | ~50M (single node) | Billions (managed) | Billions |
| **Learning curve** | Low | Very low (SQL) | Very low | Medium (GraphQL) |
| **When to choose** | New projects, performance | Already on Postgres | Zero-ops needed | Hybrid-first apps |

## 4. Decision Framework

```
1. Do you already run PostgreSQL in production?
   YES → Use pgvector. Same stack, SQL filters, no new infra.
         Ceiling: ~50M vectors. Upgrade if you outgrow it.
   NO  → Continue...

2. Do you need zero infrastructure management?
   YES → Use Pinecone (serverless). More expensive at scale, easiest ops.
   NO  → Continue...

3. Is native hybrid search (BM25 + vector in one query) critical?
   YES → Use Weaviate.
   NO  → Continue...

4. Default choice:
   → Qdrant. Best price-performance, best filtering, active development.
```

## 5. Code Examples

### Qdrant: Production Ingestion Pipeline

```python
from qdrant_client import QdrantClient
from qdrant_client.models import PointStruct, VectorParams, Distance
from qdrant_client.http import models as rest
import numpy as np
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("BAAI/bge-large-en-v1.5")
client = QdrantClient(url="http://localhost:6333")

COLLECTION = "knowledge_base"
DIM = 1024

# Create collection with optimized settings
client.recreate_collection(
    collection_name=COLLECTION,
    vectors_config=VectorParams(size=DIM, distance=Distance.COSINE),
    optimizers_config=rest.OptimizersConfigDiff(
        indexing_threshold=20_000   # start indexing after 20K vectors
    ),
    hnsw_config=rest.HnswConfigDiff(
        m=16,
        ef_construct=100,
        full_scan_threshold=10_000  # exact search below 10K vectors
    )
)

def ingest_documents(documents: list[dict], batch_size: int = 100):
    """Ingest documents with embeddings in batches."""
    for i in range(0, len(documents), batch_size):
        batch = documents[i:i+batch_size]
        texts  = [d["text"] for d in batch]
        
        # Embed batch
        embeddings = model.encode(texts, normalize_embeddings=True, batch_size=64)
        
        # Create points
        points = [
            PointStruct(
                id=doc.get("id", i * batch_size + j),
                vector=emb.tolist(),
                payload={k: v for k, v in doc.items() if k != "id"}
            )
            for j, (doc, emb) in enumerate(zip(batch, embeddings))
        ]
        
        client.upsert(collection_name=COLLECTION, points=points)
        print(f"Ingested {min(i + batch_size, len(documents))}/{len(documents)}")

# Sample documents
docs = [
    {"id": 1, "text": "Machine learning uses statistical methods to make predictions.", "topic": "ai", "year": 2023},
    {"id": 2, "text": "Deep learning is a subset of machine learning.", "topic": "ai", "year": 2023},
    {"id": 3, "text": "The French Revolution began in 1789.", "topic": "history", "year": 2022},
]
ingest_documents(docs)

# Semantic search with filtering
query = "How does deep learning work?"
q_vec = model.encode([query], normalize_embeddings=True)[0].tolist()

results = client.search(
    collection_name=COLLECTION,
    query_vector=q_vec,
    query_filter=rest.Filter(
        must=[rest.FieldCondition(key="topic", match=rest.MatchValue(value="ai"))]
    ),
    limit=5,
    with_payload=True
)

for r in results:
    print(f"[{r.score:.3f}] {r.payload['text']}")
```

### pgvector: Supabase Integration (Cloud)

```python
import os
from supabase import create_client
import numpy as np

# Works with Supabase (managed Postgres + pgvector)
url = os.environ["SUPABASE_URL"]
key = os.environ["SUPABASE_KEY"]
supabase = create_client(url, key)

# Assuming table created:
# CREATE TABLE docs (id bigserial primary key, content text, embedding vector(1536));
# CREATE INDEX ON docs USING hnsw (embedding vector_cosine_ops);

def search_supabase(query_embedding: list[float], match_count: int = 5) -> list[dict]:
    """Semantic search via Supabase RPC (SQL function)."""
    result = supabase.rpc(
        "match_docs",   # RPC function wrapping the similarity query
        {
            "query_embedding": query_embedding,
            "match_count": match_count
        }
    ).execute()
    return result.data

# SQL function to create:
# CREATE OR REPLACE FUNCTION match_docs(query_embedding vector(1536), match_count int)
# RETURNS TABLE (id bigint, content text, similarity float)
# LANGUAGE SQL STABLE AS $$
#   SELECT id, content, 1 - (embedding <=> query_embedding) AS similarity
#   FROM docs
#   ORDER BY embedding <=> query_embedding
#   LIMIT match_count;
# $$;
```

## 6. Trade-offs

### Self-Hosted vs Managed

**Self-hosted (Qdrant, Weaviate, pgvector):**
- Free to run (just your compute)
- Full control, no vendor lock-in
- Data stays in your infrastructure (GDPR, HIPAA)
- You manage availability, backups, scaling

**Managed (Pinecone, Weaviate Cloud, Qdrant Cloud):**
- Pay per use or per tier
- Zero ops overhead
- Provider SLAs and uptime guarantees
- Faster to get started

**At scale economics (10M vectors, 1,000 queries/day):**
- Pinecone serverless: ~$30–$100/month
- Qdrant Cloud: ~$25/month
- Self-hosted Qdrant on $20/month VPS: free (just your server cost)

## 7. When To Use vs. When To Avoid

### pgvector
✅ Use when: You already run Postgres; < 50M vectors; want SQL filters; simple ops
❌ Avoid when: You need horizontal scaling; > 50M vectors; need real-time ingestion at high rates

### Qdrant
✅ Use when: New project; need best performance; rich filtering; self-hosting preferred
❌ Avoid when: You need native hybrid search without additional setup (Weaviate edges it here)

### Pinecone
✅ Use when: Zero infrastructure tolerance; need managed SLAs; team lacks infra expertise
❌ Avoid when: Cost at scale is a concern; data privacy requires self-hosting

⚠️ Common mistakes:
- Building with Chroma in production — it's great for prototyping, trails other options for production workloads
- Not creating a payload index for frequently-filtered fields (Qdrant) — filtering without index = full scan
- Forgetting to create the HNSW index in pgvector — queries without the index are O(N) exact search
- Using Pinecone with very large payloads — it charges for storage by vector count AND metadata size
- Not batching upserts — inserting one vector at a time is ~10–100× slower than batch upsert

## 8. Senior AI Engineer Perspective

### The Migration Problem

Changing vector databases after you've indexed 10M documents is expensive and risky:
- You need to re-read all original content
- Re-embed with the same or different model
- Re-ingest into the new database
- Cutover with minimal downtime

**Design for migration from day one:**
- Keep original text/content in your primary database (Postgres, S3)
- Use the vector database as a derived index (can be rebuilt)
- Store the embedding model name + version alongside each vector
- Never treat the vector database as the source of truth for content

### The "Pre-Filter vs Post-Filter" Problem

When you combine vector search with metadata filters, you have two approaches:

```
Pre-filter:   Apply metadata filter FIRST, then search only matching vectors
              → Returns top-K from matching subset
              → Problem: if filter is very restrictive, only 100 vectors remain;
                ANN on 100 vectors has poor recall

Post-filter:  Search vector space FIRST (get top-1000), then filter
              → Problem: after filtering, you may have fewer than K results

Qdrant solution: Indexed payload fields + smart filter/vector strategy
pgvector solution: WHERE clause filters the final result
```

For high-selectivity filters (< 1% of corpus matches), post-filter is better.
For low-selectivity filters (> 50% of corpus matches), pre-filter is better.
Qdrant automatically selects the optimal strategy based on filter selectivity.
