## 1. What Is It? (Theory)

### The Model Choice Defines Retrieval Quality

Given identical data and infrastructure, switching embedding models can change retrieval quality
(Recall@10) by 20–40%. The embedding model is the most impactful single decision in a retrieval
system — but it's also the most expensive to change after deployment (requires re-embedding the
entire corpus).

Make this decision carefully. Test on your actual data before committing.

### The Asymmetric Retrieval Problem

Most retrieval tasks have a fundamental asymmetry:

```
Query:    "What causes inflation?"            (short, interrogative, specific intent)
Document: "Inflation results from an increase in the money supply..."  (long, declarative, general)
```

If you embed both with the same model in the same way, the query and document may not be close
in embedding space — because they're stylistically different even if semantically related.

**Asymmetric embedding models** (Voyage AI, Cohere) solve this by using different encoding
strategies for queries vs documents. You specify `input_type="query"` or `input_type="document"`,
and the model adjusts accordingly. For retrieval tasks, asymmetric models consistently outperform
symmetric ones.

**OpenAI's models** are symmetric (no distinction) but still perform well because they were
trained on diverse query-document pairs.

## 2. How It Actually Works — Deep Mechanics

### 2.1 Model Architectures (2026)

**Bi-encoders (the standard):**

```
Query ──► Encoder ──► q_vec
                                  cos_sim(q_vec, d_vec)
Doc   ──► Encoder ──► d_vec
```

Both query and document go through the same encoder (or similar encoders for asymmetric models).
Pre-compute document vectors offline. At query time, only encode the query and compute
similarities. Fast and scalable.

**Cross-encoders (for reranking only):**

```
[Query + Document] ──► Single Encoder ──► Relevance Score
```

The query and document are concatenated and processed together. Much more accurate than bi-encoders
because every token attends to both query and document. But cannot pre-compute — must run at
query time for every (query, candidate) pair. Used only for reranking a small shortlist (top-20–50)
retrieved by a bi-encoder first.

**The standard production pipeline:**
```
Bi-encoder retrieval (fast, recall-focused) →
    Top-50 candidates →
        Cross-encoder reranking (slow, precision-focused) →
            Top-5 results
```

### 2.2 MTEB: The Benchmark That Matters

**MTEB (Massive Text Embedding Benchmark)** by Hugging Face is the standard evaluation benchmark
for embedding models. It covers 56 datasets across 8 task categories:

| Task Category | Measures |
|---|---|
| Retrieval | Query → relevant document (core RAG task) |
| Clustering | Group similar texts |
| Reranking | Score relevance of candidates |
| Classification | Classify text using embeddings |
| STS (Semantic Textual Similarity) | Score text pair similarity |
| Summarization | Match summary to source |
| Bitext Mining | Find parallel translations |
| Pair Classification | Classify text pairs |

**For RAG/retrieval systems:** focus on the Retrieval category scores, not the overall average.
A model with a high average MTEB score may be mediocre at retrieval if it was optimized for
classification or STS tasks.

Live leaderboard: [huggingface.co/spaces/mteb/leaderboard](https://huggingface.co/spaces/mteb/leaderboard)

### 2.3 Matryoshka Embeddings

OpenAI's `text-embedding-3` models support **Matryoshka Representation Learning (MRL)**: a
training technique where the model is trained such that the first K dimensions of the embedding
are themselves a good K-dimensional embedding.

```
Full embedding (3,072 dims): [d₁, d₂, ..., d₃₀₇₂]

Truncated embeddings (with ~graceful quality degradation):
  [d₁, ..., d₂₅₆]    → good quality (256 dims)
  [d₁, ..., d₅₁₂]    → better quality (512 dims)
  [d₁, ..., d₁₀₂₄]   → even better (1,024 dims)
  [d₁, ..., d₃₀₇₂]   → full quality (3,072 dims)
```

This lets you trade quality for speed and storage at query time without re-embedding.

**Practical use case:** Index everything at 1,536 dims for storage efficiency. If users demand
higher precision, re-retrieve with 3,072 dims for the shortlist only.

## 3. Provider Deep-Dive (2026)

### OpenAI text-embedding-3

**text-embedding-3-small:**
- 1,536 dimensions
- $0.020/M tokens
- Best value for general English text
- Matryoshka support (can truncate to 512, 256, etc.)
- The default choice for most English-only applications

**text-embedding-3-large:**
- 3,072 dimensions
- $0.130/M tokens (6.5× more expensive)
- ~5–8% better on MTEB retrieval vs small
- Worth it only when retrieval quality is critical and cost is secondary

```python
from openai import OpenAI
import numpy as np

client = OpenAI()

# Full dimensions
resp_full = client.embeddings.create(
    model="text-embedding-3-large",
    input="What is machine learning?",
)
full_vec = resp_full.data[0].embedding
print(f"Full dimensions: {len(full_vec)}")  # 3072

# Truncated via API (Matryoshka)
resp_truncated = client.embeddings.create(
    model="text-embedding-3-large",
    input="What is machine learning?",
    dimensions=512   # request truncated embedding
)
small_vec = resp_truncated.data[0].embedding
print(f"Truncated dimensions: {len(small_vec)}")  # 512
```

**Critical:** Always specify `input_type`. Not doing so is the most common Voyage API mistake:

```python
import voyageai

vo = voyageai.Client()

# WRONG: no input_type specified
bad = vo.embed(["What is ML?"], model="voyage-3-large")

# CORRECT: specify query vs document
query_emb = vo.embed(
    ["What is machine learning?"],
    model="voyage-3-large",
    input_type="query"      # for search queries
)

doc_embs = vo.embed(
    ["Machine learning is a subset of AI...", "Another document..."],
    model="voyage-3-large",
    input_type="document"   # for content being indexed
)
```

## 4. Evaluation: How to Choose for Your Task

### Running an Offline Retrieval Evaluation

Don't pick a model from benchmarks alone. Run a quick evaluation on your actual data:

```python
import numpy as np
from typing import Callable

def evaluate_retrieval(
    queries: list[str],
    documents: list[str],
    relevant_doc_indices: list[list[int]],   # for each query, which doc indices are relevant
    embed_fn: Callable[[list[str]], np.ndarray],
    k: int = 5,
) -> dict:
    """
    Measures:
    - Recall@K:  fraction of relevant docs in top-K results
    - MRR:       mean reciprocal rank of first relevant doc
    """
    # Embed queries with query-mode, documents with document-mode
    q_vecs = embed_fn(queries, "query")
    d_vecs = embed_fn(documents, "document")
    
    # Compute similarity matrix (queries × documents)
    sim_matrix = q_vecs @ d_vecs.T   # assumes normalized vectors
    
    recall_at_k_scores = []
    mrr_scores = []
    
    for i, (row, relevant) in enumerate(zip(sim_matrix, relevant_doc_indices)):
        top_k_indices = np.argsort(-row)[:k]
        
        # Recall@K: what fraction of relevant docs appear in top-K?
        hits = sum(1 for idx in top_k_indices if idx in relevant)
        recall_at_k_scores.append(hits / len(relevant) if relevant else 0)
        
        # MRR: 1/rank of first relevant doc
        for rank, idx in enumerate(np.argsort(-row), start=1):
            if idx in relevant:
                mrr_scores.append(1.0 / rank)
                break
        else:
            mrr_scores.append(0.0)
    
    return {
        f"Recall@{k}": np.mean(recall_at_k_scores),
        "MRR":         np.mean(mrr_scores),
    }


# Compare two embedding approaches on the same data
import voyageai
from openai import OpenAI
from sentence_transformers import SentenceTransformer

voyage_client = voyageai.Client()
openai_client  = OpenAI()
local_model    = SentenceTransformer("BAAI/bge-large-en-v1.5")

def embed_voyage(texts, input_type):
    result = voyage_client.embed(texts, model="voyage-3-large", input_type=input_type)
    vecs = np.array(result.embeddings)
    return vecs / np.linalg.norm(vecs, axis=1, keepdims=True)

def embed_openai(texts, input_type=None):  # OpenAI has no input_type
    resp = openai_client.embeddings.create(model="text-embedding-3-small", input=texts)
    vecs = np.array([d.embedding for d in sorted(resp.data, key=lambda x: x.index)])
    return vecs / np.linalg.norm(vecs, axis=1, keepdims=True)

def embed_local(texts, input_type=None):
    instruction = "Represent this sentence for retrieval: " if input_type == "query" else ""
    prefixed = [instruction + t for t in texts]
    return local_model.encode(prefixed, normalize_embeddings=True)

# Build a small test set (in production: use 50+ queries, 500+ docs)
test_queries = [
    "How does photosynthesis work?",
    "What are the causes of inflation?",
]
test_docs = [
    "Photosynthesis converts sunlight to chemical energy in plants.",
    "Inflation is caused by excess money supply and demand-pull factors.",
    "Machine learning uses statistical methods to make predictions.",
    "The atmosphere contains 78% nitrogen and 21% oxygen.",
]
# Query 0 is relevant to doc 0; query 1 to doc 1
relevant = [[0], [1]]

for name, fn in [("Voyage 3 Large", embed_voyage), ("OpenAI 3-small", embed_openai), ("BGE-large", embed_local)]:
    scores = evaluate_retrieval(test_queries, test_docs, relevant, fn, k=2)
    print(f"{name:20s}: Recall@2={scores['Recall@2']:.2f}, MRR={scores['MRR']:.2f}")
```

## 6. Trade-offs

### Commercial vs Open-Source

**Commercial APIs:**
- Latest models, best quality
- No infra to manage
- Cost scales with volume
- Data leaves your infrastructure

**Open-source / self-hosted:**
- Free per inference (just compute)
- Full data privacy
- Slightly behind commercial models
- Requires model hosting infrastructure

**Recommendation:** Start with commercial (OpenAI or Voyage). Move to open-source when:
(a) costs exceed ~$500/month, or (b) data privacy requires it, or (c) you need custom fine-tuning.

### Long Context vs Short Context

Most embedding models have context limits of 256–8,192 tokens. If your documents exceed
the context limit, tokens beyond the limit are truncated — the embedding only represents
the beginning of the document.

**Voyage AI** (32K context) is exceptional here — it can embed documents that are 4× longer
than any other commercial embedding model without truncation. Critical for legal, medical, and
technical documents.

## 7. When To Use vs. When To Avoid

### Domain-Specific Models
✅ Use Voyage `code-3` for: Code search, code completion, code documentation retrieval
✅ Use Voyage `law-2` for: Legal document retrieval, contract analysis
✅ Use Voyage `finance-2` for: Financial report analysis, earnings call retrieval
❌ Don't default to generic: A 4–6% MTEB improvement on domain-specific text is real and matters

### Asymmetric Models (Voyage, Cohere)
✅ Always use for: Query-document retrieval tasks (RAG, search)
❌ Not needed for: Clustering, deduplication, or semantic similarity between same-type texts

⚠️ Common mistakes:
- Using `input_type="document"` for both queries and documents with Voyage/Cohere — kills retrieval quality
- Choosing an embedding model based on MTEB average without checking the Retrieval subtask specifically
- Not testing on your actual domain data — benchmark performance and production performance can diverge
- Switching embedding models after indexing without re-embedding the corpus — vectors from different models are incompatible

## 8. Senior AI Engineer Perspective

### The Domain Gap Problem

A model trained primarily on general web text will embed "myocardial infarction" and "heart
attack" as different concepts — because they rarely co-occur in general web text. A
medical-domain-specific model trained on medical literature will embed them as near-synonyms.

If your application operates in a specialized domain (legal, medical, financial, scientific,
code), always test domain-specific models. The performance gap can be 10–25% on retrieval metrics.

### Embedding Model Versioning

OpenAI, Voyage, and Cohere periodically release new embedding model versions. When they do:

1. New model version vectors are incompatible with old version vectors
2. You must re-embed your entire corpus with the new model
3. At 10 million documents, this is a multi-day job and real money

**Strategy:** Version your embedding model like you version your schema. Tag every vector in
your database with its model version. When you upgrade, run both old and new in parallel until
the migration is complete. Never hard-code model names — reference them through a configuration.
