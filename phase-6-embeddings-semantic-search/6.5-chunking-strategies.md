## 1. What Is It? (Theory)

### The Problem Chunking Solves

LLM context windows accept a fixed number of tokens. A 500-page legal document has ~200,000 tokens
— far more than you can send in one query. Even with Gemini's 2M token window, you don't want to
send the entire corpus to the model — it would be prohibitively expensive and the "lost in the
middle" problem would degrade quality.

**Chunking** is the process of splitting documents into smaller pieces that:
1. Fit within the model's context window
2. Can be embedded into meaningful, focused vectors
3. Can be retrieved selectively — only the relevant chunks go to the LLM

The quality of your chunking strategy is often the #1 determinant of RAG retrieval quality.
A great embedding model on bad chunks produces bad retrieval. A mediocre model on well-designed
chunks often outperforms.

### The Core Tension

```
Larger chunks:
  + More context per chunk (avoids split-sentence problems)
  + Better embedding coherence for multi-sentence concepts
  - Dense embedding = "average" of many topics → poor retrieval for specific queries
  - May include irrelevant context when retrieved

Smaller chunks:
  + More focused embedding → better match to specific queries
  + Less irrelevant context sent to the LLM
  - May be incomplete (single sentence with no context)
  - Embedding of isolated sentences is noisy
  
Sweet spot: 256–512 tokens with appropriate overlap
```

## 2. Chunking Strategies — From Simple to Advanced

### 2.1 Fixed-Size Character Splitting

Split at N characters, regardless of content structure.

```python
def fixed_char_split(text: str, chunk_size: int = 1000, overlap: int = 200) -> list[str]:
    chunks = []
    start = 0
    while start < len(text):
        end   = start + chunk_size
        chunk = text[start:end]
        chunks.append(chunk)
        start = end - overlap   # overlap: repeat last N chars in next chunk
    return chunks
```

**Pros:** Simple. No dependencies.
**Cons:** Splits mid-sentence, mid-word. Embeddings are noisy. Works poorly on structured content.
**When to use:** Almost never. Use recursive character splitting instead.

### 2.2 Recursive Character Text Splitting (The Default)

The LangChain standard and the recommended starting point for 80% of applications. Tries to
split on paragraph boundaries → sentence boundaries → word boundaries → character boundaries,
in that order, to keep semantically coherent units together.

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter
import tiktoken

# Accurate token counting
enc = tiktoken.get_encoding("cl100k_base")

def token_len(text: str) -> int:
    return len(enc.encode(text))

splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,     # tokens per chunk (target)
    chunk_overlap=64,   # overlap between consecutive chunks
    length_function=token_len,   # accurate token-based splitting
    separators=[
        "\n\n",   # paragraphs (try first)
        "\n",     # line breaks
        ". ",     # sentences
        "? ",
        "! ",
        " ",      # words
        "",       # characters (last resort)
    ]
)

text = """
# Introduction to Machine Learning

Machine learning is a subset of artificial intelligence. It enables computers to learn
from data without being explicitly programmed.

## Types of Machine Learning

There are three main types: supervised, unsupervised, and reinforcement learning.
Supervised learning uses labeled examples.

Unsupervised learning finds patterns in unlabeled data.
"""

chunks = splitter.split_text(text)
for i, chunk in enumerate(chunks):
    print(f"Chunk {i}: {token_len(chunk)} tokens | '{chunk[:80].strip()}...'")
```

**Optimal parameters (from extensive production testing):**

| Document Type | chunk_size | chunk_overlap | Notes |
|---|---|---|---|
| General text | 512 tokens | 50–100 tokens | Good starting point |
| Technical documentation | 256–384 tokens | 64 tokens | More focused |
| Legal/financial | 1024–2048 tokens | 200 tokens | Context-dense; need larger chunks |
| Code | 384–512 tokens | 100 tokens | Function/class-level chunks preferred |
| Short articles/news | 256 tokens | 32 tokens | Already short |

### 2.3 Document Structure-Aware Chunking

Use document structure (markdown headers, HTML tags, PDF section breaks) to split at natural
semantic boundaries rather than arbitrary character counts.

```python
from langchain.text_splitter import MarkdownHeaderTextSplitter, HTMLHeaderTextSplitter

# For markdown documents: split at header boundaries
md_text = """
# Chapter 1: Introduction
This chapter covers the basics.

## 1.1 Overview
Machine learning is a field of study.

## 1.2 History
The field began in the 1950s.

# Chapter 2: Supervised Learning
This chapter explains supervised learning.
"""

md_splitter = MarkdownHeaderTextSplitter(
    headers_to_split_on=[
        ("#",  "h1"),
        ("##", "h2"),
        ("###","h3"),
    ],
    strip_headers=False
)

md_chunks = md_splitter.split_text(md_text)
for chunk in md_chunks:
    print(f"Metadata: {chunk.metadata}")
    print(f"Content:  {chunk.page_content[:80]}\n")
```

**Why this is powerful:** Each chunk carries structural metadata (which chapter, which section).
This metadata can be stored in the vector database payload and used for:
- Filtering results by section
- Adding section context to the retrieved chunk
- Improving the LLM's answer by providing structural context

### 2.4 Sentence-Level Chunking

Split on sentence boundaries, then merge small sentences into chunks of target size.

```python
import spacy   # pip install spacy && python -m spacy download en_core_web_sm
from langchain.text_splitter import SentenceTransformersTokenTextSplitter

nlp = spacy.load("en_core_web_sm")

def sentence_aware_split(text: str, max_tokens: int = 512, overlap_sentences: int = 1) -> list[str]:
    """Split by sentences, then merge into chunks of max_tokens."""
    doc = nlp(text)
    sentences = [sent.text.strip() for sent in doc.sents if sent.text.strip()]
    
    chunks = []
    current_chunk_sents = []
    current_tokens = 0
    
    for i, sent in enumerate(sentences):
        sent_tokens = len(enc.encode(sent))
        
        if current_tokens + sent_tokens > max_tokens and current_chunk_sents:
            chunks.append(" ".join(current_chunk_sents))
            # Overlap: keep last N sentences in next chunk
            current_chunk_sents = current_chunk_sents[-overlap_sentences:]
            current_tokens = sum(len(enc.encode(s)) for s in current_chunk_sents)
        
        current_chunk_sents.append(sent)
        current_tokens += sent_tokens
    
    if current_chunk_sents:
        chunks.append(" ".join(current_chunk_sents))
    
    return chunks
```

### 2.5 Semantic Chunking

Uses embedding similarity to detect topic boundaries. Split where the semantic distance between
consecutive sentences is highest — natural "chapter changes" in the text.

```python
import numpy as np
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("all-MiniLM-L6-v2")

def semantic_chunk(
    text: str,
    similarity_threshold: float = 0.4,  # split where similarity drops below this
    min_chunk_tokens: int = 100,
) -> list[str]:
    """
    Split at semantic breakpoints — where consecutive sentence similarity drops.
    More expensive but produces semantically coherent chunks.
    """
    # Split into sentences first
    import re
    sentences = re.split(r'(?<=[.!?])\s+', text.strip())
    sentences = [s for s in sentences if s.strip()]
    
    if len(sentences) <= 1:
        return [text]
    
    # Embed all sentences
    embeddings = model.encode(sentences, normalize_embeddings=True)
    
    # Compute consecutive similarity
    similarities = [
        float(np.dot(embeddings[i], embeddings[i+1]))
        for i in range(len(embeddings) - 1)
    ]
    
    # Find split points (where similarity drops)
    chunks = []
    current_chunk = [sentences[0]]
    current_tokens = len(enc.encode(sentences[0]))
    
    for i, (sent, sim) in enumerate(zip(sentences[1:], similarities), start=1):
        sent_tokens = len(enc.encode(sent))
        
        should_split = (
            sim < similarity_threshold           # semantic break detected
            and current_tokens >= min_chunk_tokens  # chunk is large enough
        )
        
        if should_split:
            chunks.append(" ".join(current_chunk))
            current_chunk  = [sent]
            current_tokens = sent_tokens
        else:
            current_chunk.append(sent)
            current_tokens += sent_tokens
    
    if current_chunk:
        chunks.append(" ".join(current_chunk))
    
    return chunks

# Note on performance: semantic chunking requires an embedding call per sentence during indexing.
# It's 5-10x more expensive than recursive splitting but produces better retrieval results
# for documents with multiple distinct topics.
```

**The 2025 research note:** A NAACL 2025 paper found that for many RAG benchmarks, fixed-size
chunking (256-512 tokens) matched or beat semantic chunking in end-to-end evaluation. Semantic
chunking is not a guaranteed win — test on your specific documents.

### 2.6 Document-Specific Strategies

**Code:**
```python
from langchain.text_splitter import Language, RecursiveCharacterTextSplitter

# Language-aware splitter: splits at function/class boundaries
code_splitter = RecursiveCharacterTextSplitter.from_language(
    language=Language.PYTHON,
    chunk_size=1000,
    chunk_overlap=100
)

python_code = """
def process_data(df):
    \"\"\"Process the dataframe.\"\"\"
    df = df.dropna()
    return df.apply(transform)

class DataPipeline:
    def __init__(self, config):
        self.config = config
    
    def run(self):
        data = self.load()
        return self.process(data)
"""

code_chunks = code_splitter.split_text(python_code)
# Splits at function/class boundaries, not mid-function
```

**Tables and structured data:**
- Don't split tables across chunks — a half-table chunk embeds as gibberish
- Use HTML/markdown parsers to identify table boundaries
- Consider converting tables to text descriptions before chunking
- Or store tables as separate metadata rather than in the chunk text

**PDFs:**
- Use `pdfplumber` or `pymupdf` for better text extraction than basic PDF parsers
- Handle multi-column layouts explicitly
- Page-level chunking is often better than paragraph-level for PDFs

## 3. The Chunk Metadata Strategy

Every chunk should carry metadata that enables downstream filtering and context enrichment:

```python
@dataclass
class DocumentChunk:
    text:        str
    doc_id:      str
    chunk_index: int
    start_char:  int
    end_char:    int
    token_count: int
    metadata: dict   # document-level metadata (source, date, category, etc.)
    section:   str   # structural context (chapter, section name)

# When retrieving, you can augment the retrieved chunk with:
# - The preceding chunk (for context)
# - The document title and section name
# - The original document URL
# This dramatically improves LLM answer quality vs sending just the raw chunk text
```

## 4. Parent-Child Chunking (Advanced)

A technique that solves the tension between "small chunks for precise retrieval" and
"large chunks for sufficient context":

```
Index: Small child chunks (128-256 tokens) → for precise retrieval
       Large parent chunks (1024-2048 tokens) → for context

Query time:
  1. Search against small child chunks (precise match)
  2. Retrieve the parent chunk that contains each matched child
  3. Send parent chunks to the LLM (full context)
```

This combines the retrieval precision of small chunks with the context richness of large chunks.

```python
from langchain.retrievers import ParentDocumentRetriever
from langchain.storage import InMemoryStore
from langchain.vectorstores import Qdrant as LangchainQdrant
from langchain.text_splitter import RecursiveCharacterTextSplitter

# Parent splitter: large chunks for context
parent_splitter = RecursiveCharacterTextSplitter(chunk_size=2000, chunk_overlap=200)

# Child splitter: small chunks for indexing
child_splitter = RecursiveCharacterTextSplitter(chunk_size=400, chunk_overlap=50)

# Parent chunks stored in a simple store (parent documents)
store = InMemoryStore()

retriever = ParentDocumentRetriever(
    vectorstore=...,        # your vector store (Qdrant, pgvector, etc.)
    docstore=store,         # stores the full parent documents
    child_splitter=child_splitter,
    parent_splitter=parent_splitter,
)
```

## 5. Real-World Context

### What Actually Improves Retrieval Quality

Based on production experience and 2025 research:

| Intervention | Expected Recall@5 Delta | Notes |
|---|---|---|
| 512 → 256 token chunks (if docs have focused topics) | +3–8% | More focused embeddings |
| Adding structural metadata (section, chapter) | +5–10% | Better filtering |
| Parent-child chunking | +8–15% | Retrieval precision + answer quality |
| Better embedding model | +10–20% | Model choice matters most |
| Hybrid search vs pure vector | +10–20% | Covers BM25 blind spots |

**Hierarchy of impact:** Embedding model > hybrid search > parent-child chunking > chunk size > overlap

### Production Chunking Statistics

From RAG systems in production (2025 survey):
- Most common chunk size: 512 tokens
- Most common overlap: 10–15% of chunk size
- Most common strategy: RecursiveCharacterTextSplitter (75% of production systems)
- Semantic chunking adoption: ~20% (growing)

## 6. Code Examples

### Production Chunking Pipeline

```python
import hashlib
from dataclasses import dataclass, field
from langchain.text_splitter import RecursiveCharacterTextSplitter
import tiktoken

enc = tiktoken.get_encoding("cl100k_base")

@dataclass
class Chunk:
    chunk_id:    str
    doc_id:      str
    text:        str
    token_count: int
    chunk_index: int
    metadata:    dict = field(default_factory=dict)

def chunk_document(
    text: str,
    doc_id: str,
    metadata: dict,
    chunk_size: int = 512,
    chunk_overlap: int = 64,
) -> list[Chunk]:
    """Production chunking with stable chunk IDs and metadata."""
    
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=chunk_size,
        chunk_overlap=chunk_overlap,
        length_function=lambda t: len(enc.encode(t)),
        separators=["\n\n", "\n", ". ", "? ", "! ", " ", ""],
    )
    
    raw_chunks = splitter.split_text(text)
    chunks = []
    
    for i, chunk_text in enumerate(raw_chunks):
        # Deterministic chunk ID based on content (enables deduplication)
        chunk_id = hashlib.md5(f"{doc_id}:{chunk_text}".encode()).hexdigest()
        
        chunks.append(Chunk(
            chunk_id    = chunk_id,
            doc_id      = doc_id,
            text        = chunk_text,
            token_count = len(enc.encode(chunk_text)),
            chunk_index = i,
            metadata    = {**metadata, "chunk_index": i, "total_chunks": len(raw_chunks)}
        ))
    
    return chunks


# Test
sample_doc = """
Introduction to Python

Python is a high-level programming language known for its readability.
It was created by Guido van Rossum in 1991.

Key Features

Python supports multiple programming paradigms including procedural, object-oriented,
and functional programming. It has a comprehensive standard library.

Getting Started

To start with Python, install it from python.org. Then create your first script:
  print("Hello, World!")
"""

chunks = chunk_document(
    text=sample_doc,
    doc_id="doc-001",
    metadata={"source": "python_guide.txt", "category": "programming"},
    chunk_size=100,
    chunk_overlap=20
)

for c in chunks:
    print(f"[{c.chunk_index}] {c.token_count} tokens | {c.text[:60]}...")
```

### Evaluating Chunk Quality

```python
import numpy as np
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("all-MiniLM-L6-v2")

def evaluate_chunk_quality(chunks: list[str]) -> dict:
    """
    Measures:
    - Average intra-chunk coherence (tokens within a chunk should be related)
    - Inter-chunk diversity (chunks should cover different aspects)
    """
    if len(chunks) < 2:
        return {"error": "Need at least 2 chunks"}
    
    embeddings = model.encode(chunks, normalize_embeddings=True)
    
    # Inter-chunk similarity (lower = more diverse = better coverage)
    n = len(embeddings)
    inter_sims = []
    for i in range(n):
        for j in range(i+1, n):
            inter_sims.append(float(np.dot(embeddings[i], embeddings[j])))
    
    return {
        "num_chunks":          len(chunks),
        "avg_tokens":          np.mean([len(c.split()) for c in chunks]),
        "inter_chunk_sim_mean": np.mean(inter_sims),
        "inter_chunk_sim_std":  np.std(inter_sims),
        "verdict": "Good diversity" if np.mean(inter_sims) < 0.7 else "Low diversity (chunks too similar)"
    }

# Compare strategies
from langchain.text_splitter import RecursiveCharacterTextSplitter

long_text = "Machine learning..." * 200   # simulate a long document

for size in [128, 256, 512, 1024]:
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=size, chunk_overlap=size//8,
        length_function=lambda t: len(t.split())
    )
    chunks = splitter.split_text(long_text)
    metrics = evaluate_chunk_quality(chunks)
    print(f"Chunk size {size:4d}: {len(chunks):3d} chunks, inter-sim={metrics['inter_chunk_sim_mean']:.3f} ({metrics['verdict']})")
```

## 7. Trade-offs

### Chunk Size
| Size | Embedding quality | Retrieval precision | Context for LLM |
|---|---|---|---|
| Very small (64–128 tokens) | Noisy (too little context) | High (focused) | Poor (incomplete thoughts) |
| Small (256–512 tokens) | Good | High | Adequate |
| Medium (512–1024 tokens) | Good | Moderate | Good |
| Large (1024–2048 tokens) | Diffuse | Low | Excellent |

### Overlap
- No overlap: risk of query answers being split across chunks
- 10–20% overlap: standard; catches most split-answer cases
- > 30% overlap: wasteful; high redundancy; more storage

## 8. When To Use vs. When To Avoid

### RecursiveCharacterTextSplitter
✅ Use when: Starting a new RAG system; general text; need to get something working now
✅ The default for 80% of production RAG systems

### Semantic Chunking
✅ Use when: Documents contain multiple distinct topics; you've already optimized other factors
❌ Don't use when: You haven't yet measured that recursive splitting is insufficient

### Parent-Child Chunking
✅ Use when: You've verified that small chunks give better retrieval but poor answer quality
✅ Use when: Documents have clear hierarchical structure

⚠️ Common mistakes:
- Ignoring chunk boundaries at sentence/paragraph boundaries — split-sentence chunks produce poor embeddings
- Setting chunk_overlap=0 — queries that span chunk boundaries will miss results
- Not storing the original document alongside chunks — you need the source to rebuild chunks if strategy changes
- Chunking PDFs without handling multi-column layouts — text extraction order may be wrong
- Using the same chunk size for all document types — code, legal text, and short articles have very different optimal sizes

## 9. Senior AI Engineer Perspective

### Chunking Is Retrieval Schema Design

The choice of chunk size and strategy defines what "a unit of information" means in your system.
Get this wrong and no amount of embedding model tuning or prompt engineering will fix it.

The questions to ask before chunking:
1. What is a typical user query? (This informs chunk size — the chunk should fully answer the query)
2. What is a typical document structure? (Respect natural boundaries)
3. What context does the LLM need? (Larger chunks if the LLM needs full paragraphs to answer)

### The Context Window Enrichment Pattern

Retrieving raw chunks is the baseline. Production-quality RAG typically enriches chunks before
sending to the LLM:

```python
def enrich_chunk(chunk: Chunk, all_chunks: list[Chunk]) -> str:
    """Add context around the retrieved chunk."""
    parts = []
    
    # Add source metadata
    parts.append(f"[Source: {chunk.metadata.get('source', 'unknown')}]")
    parts.append(f"[Section: {chunk.metadata.get('section', '')}]")
    
    # Add preceding chunk for context (if available)
    if chunk.chunk_index > 0:
        preceding = next((c for c in all_chunks 
                         if c.doc_id == chunk.doc_id and c.chunk_index == chunk.chunk_index - 1), None)
        if preceding:
            parts.append(f"[Preceding context]: {preceding.text[-200:]}")
    
    parts.append(f"[Retrieved text]: {chunk.text}")
    
    return "\n".join(parts)
```

This simple enrichment often improves LLM answer quality more than any chunking strategy change.
