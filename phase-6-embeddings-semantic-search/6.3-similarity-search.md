## 1. What Is It? (Theory)

### The Core Problem

You have 1 million document vectors (embeddings). A user sends a query. You embed the query
and need to find the top-10 most similar document vectors.

**Brute force:** Compute similarity against all 1 million vectors. For 1,536-dimensional embeddings
that's 1.5 billion multiplications. At 100 queries/second: 150 billion multiplications/second.
On modern hardware this is feasible (GPUs can do this in ~50ms), but expensive and doesn't scale
to 100M+ vectors.

**Approximate Nearest Neighbor (ANN):** Build an index structure that lets you skip most of the
1M vectors and find the approximate top-K in milliseconds, with <5% recall loss. This is what
vector databases do under the hood.

Understanding similarity metrics tells you how to configure your system correctly. Understanding
ANN tells you *why* vector databases behave the way they do.

## 2. How It Actually Works — Deep Mechanics

### 2.1 Similarity Metrics

**Cosine Similarity:**

```
cos_sim(a, b) = (a · b) / (||a|| × ||b||)
```

- Range: [-1, 1] where 1 = identical direction, 0 = orthogonal, -1 = opposite
- Measures the **angle** between vectors, not their magnitude
- **Best for:** Comparing texts of different lengths where magnitude varies

```python
import numpy as np

def cosine_similarity(a: np.ndarray, b: np.ndarray) -> float:
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

# Example
a = np.array([0.9, 0.1, 0.4])
b = np.array([0.8, 0.2, 0.5])
c = np.array([-0.7, 0.3, -0.2])

print(f"cos_sim(a, b) = {cosine_similarity(a, b):.3f}")  # ~0.99 (very similar direction)
print(f"cos_sim(a, c) = {cosine_similarity(a, c):.3f}")  # ~-0.74 (opposite direction)
```

**Dot Product (Inner Product):**

```
dot(a, b) = Σ aᵢ × bᵢ
```

- Equivalent to cosine similarity when both vectors are **L2-normalized** (unit length)
- Faster to compute (no normalization step)
- **Best for:** When vectors are pre-normalized (most production embedding pipelines normalize)

```python
# When vectors are L2-normalized:
def l2_normalize(v: np.ndarray) -> np.ndarray:
    return v / np.linalg.norm(v)

a_norm = l2_normalize(a)
b_norm = l2_normalize(b)

dot_result   = np.dot(a_norm, b_norm)
cosine_result = cosine_similarity(a, b)
print(f"dot(norm_a, norm_b) ≈ cosine(a, b): {dot_result:.4f} ≈ {cosine_result:.4f}")  # same!
```

**Production practice:** Always L2-normalize your embeddings before storing them. Then use
dot product for similarity computation — it's equivalent to cosine similarity but faster (one
matrix multiply instead of multiply + divide).

**Euclidean Distance (L2):**

```
euclidean(a, b) = sqrt(Σ (aᵢ - bᵢ)²)
```

- Range: [0, ∞) where 0 = identical
- Measures actual geometric distance
- **Rarely used for text embeddings** — text vectors can have different magnitudes,
  making Euclidean distance misleading (a short document's embedding may be closer to
  a query than a long, relevant document purely due to magnitude differences)
- **Used for:** Image embeddings, structured numeric features, when magnitude is meaningful

### 2.2 Choosing the Right Metric

```
Text embeddings → Cosine similarity (or dot product after L2 normalization)
Image embeddings → Cosine similarity (generally)
Numeric features (age, price) → Euclidean distance
Learned embeddings with Matryoshka → Cosine or dot product
Binary vectors (sparse) → Hamming distance or Jaccard

Rule of thumb: if the embedding model normalized its outputs, use dot product.
               If not, use cosine similarity.
               Check the model card.
```

### 2.3 Approximate Nearest Neighbor (ANN) Algorithms

Exact nearest neighbor search scales as O(N × D) where N = vectors, D = dimensions. ANN
achieves sub-linear search time by building index structures during ingestion.

**HNSW (Hierarchical Navigable Small World):**

The dominant ANN algorithm in production. Builds a multi-layer graph where:
- Bottom layer: dense graph connecting all vectors to neighbors
- Upper layers: progressively sparser "highway" connections

Search: enter at the top layer, greedily navigate to the closest node, descend layers.

```
Layer 2 (sparse): ①──────────────⑤──────────③
                  │                           │
Layer 1:      ①──②────④────⑥────⑤────⑦────③
              │   │    │    │    │    │    │
Layer 0:  ①──②──③──④──⑤──⑥──⑦──⑧──③──⑨──⑩
              ↑ query entry       ↑ target
```

Key parameters you must tune:
- `M` (connectivity): edges per node at construction (typical: 16–64). Higher = better quality, more memory.
- `ef_construction`: search quality during index building (typical: 100–400). Higher = better quality, slower build.
- `ef_search`: search beam width at query time (typical: 50–200). Higher = better recall, slower query.

**IVF (Inverted File Index):**

Partitions the vector space into clusters (Voronoi cells). At search time, only searches
within the nearest clusters.

```
Training: K-means → centroids
Query:    find nearest centroids → search only vectors in those cells
```

Key parameter: `nprobe` — how many cells to search. Higher = better recall, slower query.

**IVF-PQ (Product Quantization):**

Compresses vectors to save memory. Splits each vector into sub-vectors and quantizes each
sub-vector to a codebook entry. Allows storing billions of vectors in RAM.

**In 2026:** HNSW is the dominant choice for most applications (fast, high recall, in-memory).
IVF-PQ is used when dataset size exceeds available RAM (billions of vectors).

### 2.4 The Recall vs Latency Trade-off

ANN has a fundamental trade-off:

```
High ef_search → High recall (finds the true nearest neighbor) → Slow queries
Low ef_search  → Low recall (misses some true nearest neighbors) → Fast queries

Typical target: Recall@10 > 0.95 at < 10ms query latency
```

Tuning example for Qdrant/HNSW:
```python
# At indexing time (one-time cost):
hnsw_config = {"m": 16, "ef_construct": 100}   # balance quality vs build time

# At query time (per-query cost):
# ef >= top_k for meaningful recall
search_params = {"hnsw_ef": 128}   # higher = better recall, slower
```

## 3. Real-World Context

### When ANN Is Not Needed

Exact search is viable and sometimes preferable:

| Corpus Size | Approach | Reason |
|---|---|---|
| < 10,000 vectors | Brute force (numpy) | Fast enough; no index overhead |
| 10K–500K vectors | ANN (HNSW) | Exact gets slow |
| > 500K vectors | ANN + quantization | Memory becomes a concern |
| > 100M vectors | ANN + PQ + batching | Requires specialized infrastructure |

For internal tools, prototypes, and small collections: a simple numpy brute-force search is
perfectly fine and much simpler than setting up a vector database.

## 4. Code Examples

### Brute-Force Search (Small Collections)

```python
import numpy as np
from dataclasses import dataclass
from typing import Any

@dataclass
class SearchResult:
    index: int
    score: float
    metadata: Any

class SimpleVectorStore:
    """
    In-memory brute-force semantic search.
    Fast enough for up to ~50K vectors.
    No external dependencies beyond numpy.
    """
    
    def __init__(self):
        self.vectors:   np.ndarray | None = None
        self.metadatas: list[Any]         = []
    
    def add(self, vectors: np.ndarray, metadatas: list[Any]):
        """Add normalized vectors and associated metadata."""
        # L2-normalize (so dot product = cosine similarity)
        norms   = np.linalg.norm(vectors, axis=1, keepdims=True)
        normed  = vectors / np.maximum(norms, 1e-8)
        
        self.vectors = np.vstack([self.vectors, normed]) if self.vectors is not None else normed
        self.metadatas.extend(metadatas)
    
    def search(self, query_vector: np.ndarray, k: int = 5) -> list[SearchResult]:
        """Return top-K most similar results."""
        if self.vectors is None:
            return []
        
        # Normalize query
        q = query_vector / max(np.linalg.norm(query_vector), 1e-8)
        
        # Dot product with all vectors = cosine similarity (since all normalized)
        scores = self.vectors @ q   # shape: (N,)
        
        # Get top-K indices
        top_k_indices = np.argpartition(-scores, min(k, len(scores) - 1))[:k]
        top_k_indices = top_k_indices[np.argsort(-scores[top_k_indices])]
        
        return [
            SearchResult(
                index    = int(idx),
                score    = float(scores[idx]),
                metadata = self.metadatas[idx]
            )
            for idx in top_k_indices
        ]


# Usage
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("all-MiniLM-L6-v2")
store = SimpleVectorStore()

documents = [
    "Photosynthesis converts sunlight to chemical energy.",
    "The stock market closed higher on Tuesday.",
    "Neural networks are inspired by the human brain.",
    "Dogs are domesticated mammals related to wolves.",
    "Python is a high-level programming language.",
]

# Index documents
doc_embeddings = model.encode(documents, normalize_embeddings=True)
store.add(doc_embeddings, [{"text": d, "id": i} for i, d in enumerate(documents)])

# Search
query = "How do plants make energy from sunlight?"
q_vec = model.encode([query], normalize_embeddings=True)[0]
results = store.search(q_vec, k=3)

print(f"Query: {query}\n")
for r in results:
    print(f"Score: {r.score:.3f} | {r.metadata['text']}")
```

### ANN with faiss (Medium-Scale)

```python
import faiss
import numpy as np
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("all-MiniLM-L6-v2")

# Simulate 100K documents
documents = [f"Document {i}: This is a sample document about topic {i % 20}." for i in range(100_000)]
print("Encoding documents...")
embeddings = model.encode(documents, batch_size=256, normalize_embeddings=True, show_progress_bar=True)
# shape: (100_000, 384)

dim = embeddings.shape[1]

# === Option 1: Exact Search (IndexFlatIP = inner product = cosine when normalized) ===
index_exact = faiss.IndexFlatIP(dim)
index_exact.add(embeddings.astype(np.float32))

# === Option 2: HNSW (Approximate, faster at scale) ===
M = 16  # connections per node — higher = better quality, more memory
index_hnsw = faiss.IndexHNSWFlat(dim, M, faiss.METRIC_INNER_PRODUCT)
index_hnsw.hnsw.efConstruction = 200  # quality during building
index_hnsw.add(embeddings.astype(np.float32))
index_hnsw.hnsw.efSearch = 64         # quality during searching

# Search comparison
query = "Sample document about topic 5"
q_vec = model.encode([query], normalize_embeddings=True)[0].astype(np.float32)

import time

for name, index in [("Exact (FlatIP)", index_exact), ("HNSW", index_hnsw)]:
    start = time.perf_counter()
    scores, indices = index.search(q_vec.reshape(1, -1), k=5)
    elapsed = (time.perf_counter() - start) * 1000
    
    print(f"\n{name} ({elapsed:.1f}ms):")
    for score, idx in zip(scores[0], indices[0]):
        print(f"  Score: {score:.3f} | doc {idx}: {documents[idx][:60]}")
```

### Batched Search for Production Throughput

```python
import faiss
import numpy as np

def batch_search(
    index: faiss.Index,
    query_vectors: np.ndarray,
    k: int = 10,
    batch_size: int = 128
) -> tuple[np.ndarray, np.ndarray]:
    """
    Search multiple queries efficiently in batches.
    faiss.search can process multiple queries simultaneously — much faster than one at a time.
    """
    all_scores  = []
    all_indices = []
    
    for i in range(0, len(query_vectors), batch_size):
        batch = query_vectors[i:i+batch_size].astype(np.float32)
        scores, indices = index.search(batch, k)
        all_scores.append(scores)
        all_indices.append(indices)
    
    return np.vstack(all_scores), np.vstack(all_indices)
```

## 6. Alternatives & Comparisons

### Exact vs Approximate Search

| Aspect | Exact | ANN (HNSW) |
|---|---|---|
| Recall | 100% | 95–99% |
| Query time (1M vecs) | ~100ms | ~5ms |
| Index build time | None | Minutes (HNSW) |
| Memory | Dense (N×D floats) | Dense + graph overhead |
| Dynamic updates | Easy | HNSW supports it; some ANN methods don't |
| Use when | <100K vectors | >100K vectors |

### HNSW vs IVF

| Aspect | HNSW | IVF |
|---|---|---|
| Query latency | Lower (5–20ms) | Higher (10–50ms) |
| Memory | Higher (graph overhead) | Lower (no graph) |
| Build time | Slow | Fast (just K-means + flat) |
| Dynamic inserts | Easy | Harder (need to re-cluster) |
| Billions of vectors | Impractical | Yes (with PQ compression) |
| Use when | <100M vectors, memory available | Billions of vectors, memory-constrained |

## 7. Trade-offs

### Recall vs Speed vs Memory

The eternal triangle in ANN search:
- High recall + low memory = slow query
- High recall + fast query = high memory (large HNSW graph)
- Low memory + fast query = lower recall

**Typical production tuning:**

For user-facing search (latency < 10ms): Use HNSW with `M=16`, `ef_search=64`. Accept ~97% recall.
For batch processing (throughput > latency): Use IVF-PQ with larger `nprobe`. Save memory.
For exact compliance requirements: Use brute force on a GPU. ~50ms but 100% recall.

### Distance Metric Choice

The distance metric must be chosen before building the index — you can't change it without
rebuilding. Wrong metric = poor retrieval quality:

```
Problem: Using Euclidean distance on unnormalized text embeddings
  Query:    [0.1, 0.2, 0.3]  (short document embedding)
  Doc A:    [0.1, 0.2, 0.3]  (identical direction — same topic) — Euclidean distance: 0 ✓
  Doc B:    [0.5, 1.0, 1.5]  (same direction, longer doc) — Euclidean distance: 1.4 ✗

Cosine would correctly identify A and B as similar (same direction).
Euclidean penalizes B for having a longer document.

Solution: L2-normalize all vectors before indexing. Then all metrics produce equivalent results.
```

## 8. When To Use vs. When To Avoid

### Cosine vs Dot Product
✅ Dot product when: All vectors are pre-normalized (faster, same result)
✅ Cosine when: Vectors may not be normalized, or you're not sure
❌ Euclidean for text: Almost always wrong for text embeddings

### HNSW vs Exact Search
✅ HNSW when: > 50K vectors, latency matters
✅ Exact when: < 50K vectors, 100% recall required, simplicity preferred
❌ Building ANN for <10K vectors: over-engineering; brute force numpy is fine

⚠️ Common mistakes:
- Not normalizing vectors before using dot product — yields wrong rankings
- Using Euclidean distance for text embeddings — sensitive to document length, not meaning
- Setting `ef_search` too low — fast but poor recall
- Not measuring actual recall on your data — tune HNSW params for your distribution, not defaults

## 9. Senior AI Engineer Perspective

### The 1% Recall Loss That Doesn't Matter

Teams agonize over getting HNSW recall from 98% to 100%. The question to ask first:
"What's the downstream impact of missing 2% of the top-10 results?"

For most RAG applications: the LLM can generate a good answer from 9 relevant chunks; 10 isn't
noticeably better. For search: users care about the top-3 results, not the top-10.

Spend your tuning budget on chunk quality and embedding model selection — those have 10× more
impact than tuning from 98% to 100% ANN recall.

### When to Use faiss Directly vs Vector Database

**Use faiss directly when:**
- You're doing batch offline processing
- You need maximum control over index configuration
- You're building a custom pipeline
- Your data fits in memory on one machine

**Use a vector database when:**
- You need multi-tenancy (different users, different collections)
- You need metadata filtering (find vectors AND filter by date/category)
- You need dynamic updates (add/delete vectors after indexing)
- You need persistence across restarts
- You need horizontal scaling

Most production systems need at least some of the vector database features — building faiss
from scratch with persistence and filtering is essentially reimplementing a vector database.
