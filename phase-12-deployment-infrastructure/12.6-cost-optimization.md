## 1. What Is It? (Theory)

### The Three Cost Levers for Self-Hosted Inference

When you self-host LLMs, your costs are GPU time (compute) and memory. Unlike API pricing
(pay per token), self-hosting means paying for the GPU whether it's busy or idle.

```
Cost per token = (GPU hourly cost) / (tokens per hour)

A100 @ $3/hr:
  Naive serving (200 tok/s):  $3 / (200 × 3600) = $0.0000042/token = $4.17/M tokens
  Optimized (4000 tok/s):     $3 / (4000 × 3600) = $0.00000021/token = $0.21/M tokens

20× throughput improvement = 20× cost reduction per token
```

The three cost levers:
1. **Maximize GPU utilization** (Section 12.2, 12.5) — squeeze more tokens per GPU-hour
2. **Use the smallest model that works** (model routing) — small = faster and cheaper
3. **Avoid regenerating what you already computed** (caching) — free tokens

## 2. How It Actually Works — Deep Mechanics

### 2.1 Prompt Caching (Prefix Caching)

KV-cache from previous requests can be reused if they share a common prefix (same system
prompt, same document, etc.). This is the biggest single cost optimization available.

**vLLM automatic prefix caching:**

```bash
# Enable prefix caching in vLLM
python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Llama-3-8B-Instruct \
    --enable-prefix-caching \
    --prefix-caching-hash-algo sha256 \
    --port 8000
```

```python
from openai import OpenAI

client = OpenAI(api_key="not-needed", base_url="http://localhost:8000/v1")

# Scenario: 1000-token system prompt, shared by many requests
LONG_SYSTEM = "You are a senior Python engineer...\n" + "..." * 500   # ~1000 tokens

# Request 1: full prefill (1000 tokens processed)
r1 = client.chat.completions.create(
    model="meta-llama/Llama-3-8B-Instruct",
    messages=[
        {"role": "system", "content": LONG_SYSTEM},
        {"role": "user",   "content": "How do decorators work?"}
    ]
)

# Request 2: same system prompt → prefix cache HIT (0 tokens prefilled for system)
r2 = client.chat.completions.create(
    model="meta-llama/Llama-3-8B-Instruct",
    messages=[
        {"role": "system", "content": LONG_SYSTEM},  # same prefix → from cache
        {"role": "user",   "content": "Explain list comprehensions."}
    ]
)

# Cost: Request 1 = 1010 tokens prefilled
#       Request 2 = 10 tokens prefilled (just the user message)
# 99% cheaper for the cached portion!
```

**SGLang RadixAttention (more aggressive caching):**

SGLang's RadixAttention shares KV-cache across requests more aggressively using a radix tree
structure, enabling cache hits even for partial prefix matches. Especially effective for:
- RAG systems where the same documents appear in many requests
- Agent systems where the same tool definitions appear repeatedly
- Multi-turn conversations where early turns are shared

### 2.2 Response Caching (Semantic Caching)

Cache entire responses for semantically similar queries:

```python
import hashlib
import json
import numpy as np
from sentence_transformers import SentenceTransformer
from redis import Redis

embed_model = SentenceTransformer("all-MiniLM-L6-v2")
redis = Redis(host="localhost", port=6379, decode_responses=True)

CACHE_TTL = 3600     # 1 hour
SIM_THRESHOLD = 0.95  # Cosine similarity threshold for cache hit

def get_cached_response(
    query: str,
    system_prompt: str,
) -> str | None:
    """
    Semantic cache: return a cached response if a very similar query was answered before.
    Effective for FAQ-style workloads where users ask the same questions repeatedly.
    """
    cache_key = f"cache:{hashlib.md5((system_prompt + query).encode()).hexdigest()}"
    
    # Exact match first (fast)
    exact = redis.get(cache_key)
    if exact:
        return json.loads(exact)["response"]
    
    # Semantic match (check recent cache entries for similar queries)
    q_vec = embed_model.encode([query], normalize_embeddings=True)[0]
    
    # Scan recent cache keys (in production: use a vector store for this)
    recent_keys = redis.keys("cache_vec:*")[:200]  # check last 200 entries
    
    for key in recent_keys:
        entry = json.loads(redis.get(key) or "{}")
        if not entry.get("embedding"):
            continue
        
        cached_vec = np.array(entry["embedding"])
        similarity  = float(np.dot(q_vec, cached_vec))
        
        if similarity >= SIM_THRESHOLD:
            return entry["response"]
    
    return None   # cache miss

def cache_response(query: str, system_prompt: str, response: str):
    """Cache a response with its embedding for semantic matching."""
    q_vec  = embed_model.encode([query], normalize_embeddings=True)[0]
    
    entry = {
        "query":     query,
        "response":  response,
        "embedding": q_vec.tolist()
    }
    
    cache_key = f"cache_vec:{hashlib.md5((system_prompt + query).encode()).hexdigest()}"
    redis.setex(cache_key, CACHE_TTL, json.dumps(entry))

def cached_llm_call(query: str, system_prompt: str, llm_fn) -> dict:
    """LLM call with semantic caching."""
    # Try cache first
    cached = get_cached_response(query, system_prompt)
    if cached:
        return {"response": cached, "cache_hit": True, "cost": 0}
    
    # Cache miss → call LLM
    response = llm_fn(query, system_prompt)
    cache_response(query, system_prompt, response)
    
    return {"response": response, "cache_hit": False, "cost": "normal"}
```

**When semantic caching is worth it:**
- FAQ bots where the same 100 questions account for 60%+ of traffic
- Search interfaces where popular queries repeat
- Evaluation pipelines where the same questions run repeatedly

**When it's not:**
- User-specific queries (personalized; never repeat exactly)
- Queries about current events (cached answers go stale)

### 2.3 Model Routing — The Most Impactful Cost Lever

Not every query needs the most powerful (and expensive) model. A routing layer sends:
- Simple queries → small, fast, cheap model
- Complex queries → large, capable, expensive model

```python
import anthropic
from openai import OpenAI

anthropic_client = anthropic.Anthropic()
openai_client    = OpenAI(api_key="not-needed", base_url="http://localhost:8000/v1")

# Local cheap model (vLLM)
CHEAP_MODEL   = "meta-llama/Llama-3-8B-Instruct"
# Cloud expensive model (Anthropic)
EXPENSIVE_MODEL = "claude-sonnet-4-6"

def classify_query_complexity(query: str) -> str:
    """
    Classify whether a query needs the expensive model.
    Returns "simple" or "complex".
    
    Simple: factual questions, short transformations, classification
    Complex: multi-step reasoning, code generation, analysis, synthesis
    """
    # Heuristics (fast, no LLM call needed)
    words = query.lower().split()
    n_words = len(words)
    
    # Signals of complexity
    complex_keywords = {
        "analyze", "synthesize", "compare", "debug", "architect",
        "refactor", "implement", "why", "how would you", "evaluate"
    }
    
    has_complex_keyword = any(kw in query.lower() for kw in complex_keywords)
    is_long_query       = n_words > 50
    has_code_block      = "```" in query or "def " in query
    is_multi_part       = query.count("?") > 1 or query.count("and") > 2
    
    complexity_score = sum([
        has_complex_keyword,
        is_long_query,
        has_code_block,
        is_multi_part,
    ])
    
    return "complex" if complexity_score >= 2 else "simple"

def routed_call(query: str, system: str) -> dict:
    """Route to appropriate model based on query complexity."""
    
    complexity = classify_query_complexity(query)
    
    if complexity == "simple":
        # Local 8B model: ~$0.00002/query at self-hosted cost
        response = openai_client.chat.completions.create(
            model=CHEAP_MODEL,
            messages=[
                {"role": "system", "content": system},
                {"role": "user",   "content": query}
            ],
            max_tokens=512
        )
        return {
            "response": response.choices[0].message.content,
            "model":    CHEAP_MODEL,
            "cost_tier": "cheap"
        }
    
    else:
        # Claude Sonnet: ~$0.005/query average
        response = anthropic_client.messages.create(
            model=EXPENSIVE_MODEL,
            max_tokens=1024,
            system=system,
            messages=[{"role": "user", "content": query}]
        )
        return {
            "response": response.content[0].text,
            "model":    EXPENSIVE_MODEL,
            "cost_tier": "expensive"
        }

# Expected routing distribution for typical support chatbot:
# Simple (60–70% of queries): classification, yes/no, simple factual lookup
# Complex (30–40% of queries): analysis, multi-step, nuanced judgment
# Cost savings: 0.65 × cheap + 0.35 × expensive vs 1.0 × expensive
# ≈ 0.65 × $0.00002 + 0.35 × $0.005 = $0.00177 vs $0.005 = 65% cost reduction
```

### 2.4 Batch API for Non-Realtime Workloads

For any workload that doesn't need real-time response, use batch APIs:

```python
import anthropic
import json

client = anthropic.Anthropic()

def batch_classify_documents(texts: list[str]) -> list[str]:
    """
    Classify 1000 documents using the Batch API.
    50% cheaper and can process overnight.
    """
    requests = [
        {
            "custom_id": f"doc_{i}",
            "params": {
                "model": "claude-haiku-4-5-20251001",
                "max_tokens": 50,
                "messages": [{
                    "role": "user",
                    "content": f"Classify as 'positive', 'negative', or 'neutral': {text}"
                }]
            }
        }
        for i, text in enumerate(texts)
    ]
    
    batch = client.beta.messages.batches.create(requests=requests)
    print(f"Batch submitted: {batch.id}")
    print(f"Cost: 50% off vs real-time — paid at completion")
    
    # Returns batch.id; collect results later
    return batch.id

# For self-hosted vLLM: batch via offline inference
from vllm import LLM, SamplingParams

llm = LLM(model="meta-llama/Llama-3-8B-Instruct", gpu_memory_utilization=0.92)
params = SamplingParams(temperature=0, max_tokens=50)

def batch_process_offline(texts: list[str]) -> list[str]:
    """Maximum throughput offline batch processing with vLLM."""
    prompts  = [f"Classify as positive/negative/neutral: {t}" for t in texts]
    outputs  = llm.generate(prompts, params)
    return [o.outputs[0].text.strip() for o in outputs]
```

## 3. Complete Cost Optimization Stack

```
Layer 1: Response cache (Redis)
         Cache complete responses for identical/near-identical queries
         Hit rate: 20–60% for FAQ/repetitive workloads
         Cost reduction: up to 60% of LLM calls eliminated

Layer 2: KV-cache / prefix caching (vLLM)
         Reuse computed KV-cache for shared system prompt prefixes
         Automatic in vLLM with --enable-prefix-caching
         Cost reduction: 50–90% of prefill cost for requests with long shared context

Layer 3: Model routing
         Route simple queries to cheap small model, complex to large
         Cost reduction: 40–70% depending on traffic distribution

Layer 4: Batch API
         Use batch endpoints (50% off) for non-realtime workloads
         Applicable to: eval runs, data processing, bulk classification

Layer 5: Quantization (Section 12.3)
         INT4 AWQ: 2-4× more throughput per GPU-hour at <3% quality loss
         Cost reduction: 50-75% on self-hosted inference cost

Layer 6: Infrastructure efficiency (Section 12.2)
         Continuous batching, PagedAttention, optimized configuration
         Cost reduction: 10-20× vs naive serving
```

## 4. Real-World Cost Example

```python
# Example: 1M requests/month, 500-token average response

BASELINE_COST_SONNET = 1_000_000 * (500 * 15 + 200 * 3) / 1_000_000  # input + output
print(f"Baseline (all Sonnet): ${BASELINE_COST_SONNET:,.0f}/month")
# = $9,600/month

# With optimization stack:
RESPONSE_CACHE_HIT  = 0.35   # 35% requests served from cache (free)
PREFIX_CACHE_SAVING = 0.60   # 60% savings on prefill for remaining 65%
ROUTING_SMALL_PCT   = 0.55   # 55% go to cheap local model (near-zero marginal cost)
ROUTING_LARGE_PCT   = 0.10   # remaining 10% go to Sonnet

optimized_sonnet_cost = (
    1_000_000 *
    (1 - RESPONSE_CACHE_HIT) *   # not cached
    ROUTING_LARGE_PCT *          # routed to Sonnet
    (500 * 15 + 200 * 3 * (1 - PREFIX_CACHE_SAVING)) /  # with prefix caching discount
    1_000_000
)

local_serving_cost = 3.00   # A100 @ $3/hr handles 4000 tok/s = ~$720/month for 1M requests

print(f"Optimized: ${optimized_sonnet_cost + local_serving_cost:,.0f}/month")
print(f"Savings: {(1 - (optimized_sonnet_cost + local_serving_cost)/BASELINE_COST_SONNET):.0%}")
# Typical: ~85% cost reduction
```

## 6. Senior AI Engineer Perspective

### The 80% Rule for Cost Optimization

Before optimizing serving infrastructure, verify that the 80% gains are captured first:
1. **Are you using prompt caching?** If not, add it. 30 minutes of work, 50–90% savings on prefill.
2. **Are you using the right model size?** If a 7B model works as well as 70B for your task, use it.
3. **Are you batching?** If you're doing serial API calls for a batch job, switch to the batch API.

These three capture 80%+ of available cost savings. Complex infrastructure optimization
(quantization, distributed inference, speculative decoding) adds the remaining 20%.

### The Engineering Completion Mindset

You've now built knowledge across the entire AI engineering stack — from neural network fundamentals
to production GPU infrastructure. This is not common. Most engineers specialize in one or two layers.

The value of end-to-end understanding: you can reason about tradeoffs across the whole stack.
When a product manager asks "why is this feature slow?", you can diagnose whether it's a:
- Prompt design issue (Phase 4)
- Context window issue (Phase 3)
- Retrieval quality issue (Phase 6–7)
- Serving infrastructure issue (Phase 12)
- Model capability issue (Phase 3, 11)

That diagnosis ability is what distinguishes senior AI engineers from developers who can only
connect APIs.
