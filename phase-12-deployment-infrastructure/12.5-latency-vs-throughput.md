## 1. What Is It? (Theory)

### The Fundamental Tension

In LLM serving, latency and throughput pull in opposite directions:

**Latency:** How fast does a single request get a response?
**Throughput:** How many requests per second can the system handle?

```
Optimizing for latency:
  → Small batch sizes (process requests immediately)
  → Less waiting for other requests to join the batch
  → GPU is often underutilized (~20% utilization for single requests)

Optimizing for throughput:
  → Large batch sizes (wait to accumulate requests)
  → More waiting time per request
  → GPU approaches ~90% utilization

The GPU can't be both fast for one request AND fully utilized by many.
```

You cannot maximize both simultaneously. You must decide which matters more
for your specific product and configure accordingly.

### The Two Time Metrics

```
TTFT (Time to First Token):
  Time from when user sends request to when first token appears.
  Affects perceived responsiveness.
  Dominated by: prompt processing (prefill), request queuing time.
  Target for interactive apps: < 500ms

TBT (Time Between Tokens) / Generation Speed:
  Time between consecutive tokens during generation.
  Affects how fast the response "types out".
  Target for interactive apps: < 50ms/token (20+ tokens/sec)

E2E Latency (End-to-End):
  Total time from request to complete response.
  = TTFT + (n_output_tokens × TBT)
  For 100-token response at 20ms TTFT, 20ms/token: 20 + 100×20 = 2020ms
```

## 2. How It Actually Works — Deep Mechanics

### 2.1 The Latency-Throughput Curve

```python
import asyncio, httpx, time, statistics

async def measure_at_concurrency(url: str, model: str, n: int, prompt: str) -> dict:
    """Run n concurrent requests and measure latency + throughput."""
    
    async def one_request(client):
        start = time.perf_counter()
        r = await client.post(
            f"{url}/v1/chat/completions",
            json={
                "model": model,
                "messages": [{"role": "user", "content": prompt}],
                "max_tokens": 100,
                "stream": False,
            },
            timeout=120
        )
        elapsed = time.perf_counter() - start
        tokens  = r.json()["usage"]["completion_tokens"]
        return elapsed, tokens
    
    async with httpx.AsyncClient() as client:
        start_all = time.perf_counter()
        results   = await asyncio.gather(*[one_request(client) for _ in range(n)])
        wall_time = time.perf_counter() - start_all
    
    latencies   = [r[0] for r in results]
    total_toks  = sum(r[1] for r in results)
    throughput  = total_toks / wall_time
    
    return {
        "concurrency":   n,
        "p50_latency_s": round(statistics.median(latencies), 2),
        "p95_latency_s": round(sorted(latencies)[int(n*0.95)], 2),
        "throughput_tps": round(throughput, 0),
        "gpu_util_pct":   "~" + str(min(100, round(n * 20))),  # rough estimate
    }

async def characterize_server():
    print("Concurrency | P50 Latency | P95 Latency | Throughput | GPU~")
    print("-" * 70)
    
    for c in [1, 2, 4, 8, 16, 32, 64]:
        result = await measure_at_concurrency(
            "http://localhost:8000",
            "meta-llama/Llama-3-8B-Instruct",
            n=c,
            prompt="Explain machine learning in 50 words."
        )
        print(f"  {result['concurrency']:4d}      | "
              f"  {result['p50_latency_s']:.2f}s       | "
              f"  {result['p95_latency_s']:.2f}s       | "
              f"  {result['throughput_tps']:5.0f} tok/s | "
              f"  {result['gpu_util_pct']}")

asyncio.run(characterize_server())

# Typical output for 8B model on A100:
# Concurrency |  P50 Latency  |  P95 Latency  | Throughput   | GPU~
# ────────────────────────────────────────────────────────────────────
#    1         |  0.52s        |  0.54s        |    185 tok/s |  ~20%
#    4         |  0.55s        |  0.60s        |    720 tok/s |  ~40%
#    8         |  0.65s        |  0.90s        |   1320 tok/s |  ~70%
#   16         |  1.10s        |  1.50s        |   2100 tok/s |  ~85%
#   32         |  2.20s        |  3.00s        |   2800 tok/s |  ~92%
#   64         |  4.80s        |  7.00s        |   3100 tok/s |  ~95%
```

### 2.3 Priority Queues for Mixed Workloads

Real production systems have both latency-sensitive (user-facing) and throughput-oriented
(batch processing) requests. A priority queue separates them:

```python
# Conceptual: vLLM supports request priority
# High priority requests (interactive) get served first
# Low priority requests (batch) fill in remaining capacity

from vllm import AsyncLLMEngine, SamplingParams

engine = AsyncLLMEngine.from_engine_args(engine_args)

async def serve_request(prompt: str, priority: int = 0, request_id: str = ""):
    """
    priority: 0 = highest (interactive), 10 = lowest (batch)
    vLLM processes higher priority requests first.
    """
    sampling_params = SamplingParams(max_tokens=512)
    
    async for output in engine.generate(
        inputs={"prompt": prompt},
        sampling_params=sampling_params,
        request_id=request_id,
        priority=priority,   # supported in vLLM 0.6.0+
    ):
        yield output.outputs[0].text
```

### 2.4 SLA Definition and Monitoring

```python
from dataclasses import dataclass

@dataclass
class ServingSLA:
    """Service Level Agreement for an LLM serving endpoint."""
    
    # Latency SLAs
    ttft_p50_ms:  float = 300    # median first token: <300ms
    ttft_p95_ms:  float = 800    # 95th percentile: <800ms
    e2e_p50_ms:   float = 3000   # end-to-end p50: <3s (for 100-token response)
    e2e_p95_ms:   float = 8000
    
    # Throughput SLAs
    min_throughput_tps: float = 500  # minimum tokens/second
    
    # Error SLAs
    max_error_rate: float = 0.01  # <1% errors
    max_timeout_rate: float = 0.005  # <0.5% timeouts

class SLAMonitor:
    def __init__(self, sla: ServingSLA):
        self.sla = sla
        self.ttft_samples:   list[float] = []
        self.e2e_samples:    list[float] = []
        self.error_count:    int = 0
        self.total_count:    int = 0
    
    def record(self, ttft_ms: float, e2e_ms: float, error: bool = False):
        self.total_count += 1
        self.ttft_samples.append(ttft_ms)
        self.e2e_samples.append(e2e_ms)
        if error:
            self.error_count += 1
    
    def check_sla(self) -> dict:
        import numpy as np
        
        violations = []
        
        if len(self.ttft_samples) >= 20:
            ttft_p95 = np.percentile(self.ttft_samples, 95)
            if ttft_p95 > self.sla.ttft_p95_ms:
                violations.append(f"TTFT P95: {ttft_p95:.0f}ms > SLA {self.sla.ttft_p95_ms}ms")
        
        if self.total_count > 0:
            error_rate = self.error_count / self.total_count
            if error_rate > self.sla.max_error_rate:
                violations.append(f"Error rate: {error_rate:.1%} > SLA {self.sla.max_error_rate:.1%}")
        
        return {
            "sla_met": len(violations) == 0,
            "violations": violations,
            "ttft_p95_ms": round(np.percentile(self.ttft_samples, 95), 0) if self.ttft_samples else None,
        }
```

## 3. Real-World Context

### The Interactive vs Batch Decision

| Use Case | Latency Tolerance | Throughput Need | Strategy |
|---|---|---|---|
| User-facing chat | Low (< 1s TTFT) | Medium | Low concurrency, fast model |
| Background summarization | High (minutes OK) | High | Large batches, slow model OK |
| Real-time code completion | Very low (< 100ms) | Low | Smallest viable model, dedicated GPU |
| RAG pipeline (user-facing) | Low (< 2s total) | Medium | Cache embeddings, fast generation |
| Overnight batch analysis | None | Maximize | Full batch API, cheapest GPU |

### The Prefill vs Decode Split

Modern systems increasingly separate the prefill phase (prompt processing — compute-bound)
from the decode phase (token generation — memory-bound), assigning each to optimized hardware.

This is called "prefill-decode disaggregation" and is an emerging production pattern in 2026:

```
Prefill servers:  high compute GPUs (H100) → fast prompt processing
Decode servers:   high memory bandwidth GPUs (H200) → fast token generation
Load balancer:    routes prefill → decode on completion
```

This is complex but achieves ~2× improvement in both TTFT and throughput for mixed workloads.
vLLM 0.7+ supports disaggregated prefill as an experimental feature.

## 4. Senior AI Engineer Perspective

### Set SLAs Before Choosing Hardware

The right hardware choice follows directly from your SLA:

1. Define: "What's our TTFT target? E2E latency target? Tokens/second minimum?"
2. Benchmark: Run the characterization script on different GPU options
3. Choose: The cheapest GPU that meets your SLA at your expected peak concurrency

Teams that buy hardware first and define SLAs later often end up with expensive hardware
that's still not meeting user experience targets — because they optimized for the wrong metric.

### The Long-Tail Latency Problem

Users experiencing P99 or P99.9 latency are your most likely churners.
P50 (median) latency can look great while P99 is completely unacceptable:

```
vLLM batch processing at concurrency=32:
  P50 latency: 2.2s   ← great for median user
  P95 latency: 3.0s   ← acceptable
  P99 latency: 12.0s  ← terrible; this user won't come back
  
Root cause: tail latency spikes when the request queues behind a very long generation.
Fix: set max_tokens limits, use preemption, or separate long/short requests into different queues.
```
