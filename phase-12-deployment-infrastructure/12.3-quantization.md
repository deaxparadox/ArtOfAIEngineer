## 1. What Is It? (Theory)

### The Precision-Size Trade-off

A 7B parameter model in float16 (the standard training format) takes 14 GB of VRAM.
The same model in int4 (4-bit integer) takes 3.5 GB — a 4× reduction.

Quantization converts model weights from high-precision floats (float16/bfloat16) to
lower-precision integers or compressed floats. The trade-off: smaller size and faster
inference, at a small accuracy cost.

```
Precision  → Size (7B model) → Memory bandwidth savings → Typical accuracy loss
─────────────────────────────────────────────────────────────────────────────────
FP32       → 28 GB           → baseline               → None (reference)
BF16/FP16  → 14 GB           → 2×                     → Negligible
FP8        →  7 GB           → 4×                     → <0.5% (production-ready)
INT8       →  7 GB           → 4×                     → <1%   (production-ready)
INT4/NF4   →  3.5 GB         → 8×                     → 1–3%  (acceptable for most)
INT2       →  1.75 GB        → 16×                    → 5–15% (usually unacceptable)
```

**The 2026 standard:** INT8 for quality-sensitive production. AWQ INT4 for memory-constrained
deployment. FP8 increasingly viable with NVIDIA Hopper (H100/H200) native support.

## 2. How It Actually Works — Deep Mechanics

### 2.1 INT8 Quantization (LLM.int8())

The simplest production-ready quantization. Tim Dettmers' `bitsandbytes` library implements
LLM.int8() which handles the "emergent outliers" problem:

```
The emergent outliers problem:
  During LLM inference, ~0.1% of weight values are extreme outliers (100× larger than typical)
  Naive int8 quantization: scale to fit all values → outliers cause large precision loss
  LLM.int8(): detect outliers → keep them in fp16, quantize the rest in int8
              → Mixed-precision matrix multiply → nearly identical output quality
```

```python
from transformers import AutoModelForCausalLM, AutoTokenizer
import torch

model_id = "meta-llama/Llama-3-8B-Instruct"

# INT8 quantization via bitsandbytes (automatic)
model_int8 = AutoModelForCausalLM.from_pretrained(
    model_id,
    load_in_8bit=True,        # INT8 (LLM.int8())
    device_map="auto",
)

tokenizer = AutoTokenizer.from_pretrained(model_id)

# Usage: identical to normal inference
inputs = tokenizer("What is machine learning?", return_tensors="pt").to("cuda")
outputs = model_int8.generate(**inputs, max_new_tokens=100)
print(tokenizer.decode(outputs[0], skip_special_tokens=True))

# Memory check
print(f"Model size (INT8): {model_int8.get_memory_footprint() / 1e9:.1f} GB")
# vs ~14 GB for bfloat16 → ~7 GB for INT8
```

### 2.2 GPTQ — Post-Training Weight Quantization

GPTQ (Frantar et al., 2022) is a post-training quantization method that optimizes the
quantization to minimize output difference from the original model.

```
GPTQ process:
  1. Load the original fp16 model
  2. Run calibration data (small dataset) through the model
  3. For each layer: find the quantization values that minimize
     ||W_original - W_quantized||² weighted by activation patterns
  4. Save quantized weights (typically 4-bit)
  
Result: quantized model that closely matches original output
```

```python
# Using pre-quantized GPTQ models from Hugging Face
# (Most popular models have community GPTQ versions)

from transformers import AutoModelForCausalLM, AutoTokenizer
from auto_gptq import AutoGPTQForCausalLM

# Load a pre-quantized GPTQ model
model_id = "TheBloke/Llama-3-8B-Instruct-GPTQ"   # GPTQ versions on HuggingFace

model = AutoGPTQForCausalLM.from_quantized(
    model_id,
    use_safetensors=True,
    device_map="auto",
    use_triton=True,   # Triton kernels for faster inference
)
tokenizer = AutoTokenizer.from_pretrained(model_id)

# Or via AutoModelForCausalLM with AutoGPTQ backend
model_auto = AutoModelForCausalLM.from_pretrained(
    model_id,
    device_map="auto",
    torch_dtype="auto",
)

# Quantize from scratch (if no pre-quantized version exists)
from auto_gptq import AutoGPTQForCausalLM, BaseQuantizeConfig

quantize_config = BaseQuantizeConfig(
    bits=4,                  # 4-bit quantization
    group_size=128,          # weight groups of 128 (larger = better quality, less compression)
    damp_percent=0.01,
    desc_act=False,          # activation ordering
)

model_to_quantize = AutoGPTQForCausalLM.from_pretrained(
    "meta-llama/Llama-3-8B-Instruct",
    quantize_config=quantize_config
)

# Calibration data (representative inputs)
calibration_data = [
    "What is machine learning?",
    "Explain quantum computing.",
    # ... 128+ diverse examples
]

examples = [
    tokenizer(text, return_tensors="pt")
    for text in calibration_data[:128]
]

# Quantize (takes 10-30 minutes for 8B model)
model_to_quantize.quantize(examples)

# Save
model_to_quantize.save_quantized("./llama3-8b-gptq-4bit")
```

### 2.3 AWQ — Activation-Aware Weight Quantization

AWQ (Lin et al., 2023) makes a key observation: not all weights are equally important.
About 1% of weight channels corresponding to high-activation features have outsized impact.
Protecting these channels during quantization preserves quality far better than naive int4.

```
GPTQ approach:      find best quantization for each weight independently
AWQ approach:       identify the ~1% "salient" weights (large activations)
                    scale those channels up before quantization
                    scale them down at inference time
                    → uniform INT4 but better effective precision where it matters
```

**AWQ vs GPTQ in practice (2025–2026):**

| Aspect | GPTQ | AWQ |
|---|---|---|
| Quality (academic benchmarks) | AWQ +0.23–0.35 better | Slightly better |
| Quality (real-world tasks) | GPTQ better by 2.9 points | Surprising reversal |
| Inference speed | Similar | Similar |
| Quantization time | 10–30 min | 5–15 min |
| vLLM native support | Yes | Yes (default recommendation) |

```python
# AWQ quantization
# pip install autoawq

from awq import AutoAWQForCausalLM
from transformers import AutoTokenizer

model_id = "meta-llama/Llama-3-8B-Instruct"
quant_path = "./llama3-8b-awq-4bit"

# Quantize
model = AutoAWQForCausalLM.from_pretrained(model_id)
tokenizer = AutoTokenizer.from_pretrained(model_id)

quant_config = {
    "zero_point": True,
    "q_group_size": 128,
    "w_bit": 4,
    "version": "GEMM"
}

model.quantize(tokenizer, quant_config=quant_config)
model.save_quantized(quant_path)
tokenizer.save_pretrained(quant_path)

# Load and use
model_awq = AutoAWQForCausalLM.from_quantized(quant_path, fuse_layers=True)

# Via vLLM (recommended for production)
# python -m vllm.entrypoints.openai.api_server \
#     --model ./llama3-8b-awq-4bit \
#     --quantization awq
```

### 2.4 FP8 — The 2025–2026 Standard

FP8 (8-bit floating point) is increasingly the preferred quantization for NVIDIA Hopper and
newer architectures (H100, H200, B100):

```python
# vLLM native FP8 support (H100+ required)
# python -m vllm.entrypoints.openai.api_server \
#     --model meta-llama/Llama-3-8B-Instruct \
#     --dtype fp8 \
#     --quantization fp8

# Or with vLLM Python API
from vllm import LLM

llm = LLM(
    model="meta-llama/Llama-3-8B-Instruct",
    quantization="fp8",         # native H100 FP8
    dtype="bfloat16",           # compute in bf16, store in fp8
    kv_cache_dtype="fp8",       # also quantize the KV-cache
)
# FP8 KV-cache: halves KV-cache memory → larger effective batch sizes
```

**Why FP8 matters:** FP8 has hardware support on H100/H200 via Transformer Engine. Unlike
software-emulated INT4 (where CUDA cores do fp16 math but store INT4), FP8 uses dedicated
hardware → ~2× faster matrix multiplications.

### 2.5 GGUF for CPU / Local Inference

For running models on CPU or consumer hardware (no dedicated GPU):

```python
# llama.cpp / GGUF format
# Download a GGUF quantized model from Hugging Face
# (e.g., TheBloke/Llama-3-8B-GGUF)

# Python via llama-cpp-python
from llama_cpp import Llama

llm = Llama(
    model_path="./llama-3-8b.Q4_K_M.gguf",
    n_ctx=4096,
    n_threads=8,     # CPU threads
    n_gpu_layers=0,  # 0 = CPU only; >0 = partial GPU offload
)

response = llm("What is machine learning?", max_tokens=100)
print(response["choices"][0]["text"])
```

## 3. Quality vs Compression Trade-offs

### Measured Quality Degradation by Format

Based on 2025 benchmarks across LLaMA-3, Mistral, and Qwen models:

| Format | MMLU | Code (HumanEval) | Reasoning | Memory |
|---|---|---|---|---|
| BF16 (reference) | 100% | 100% | 100% | 100% |
| FP8 | ~99.8% | ~99.5% | ~99.7% | 50% |
| INT8 | ~99.5% | ~99.2% | ~99.4% | 50% |
| AWQ INT4 | ~98.2% | ~97.5% | ~97.9% | 25% |
| GPTQ INT4 | ~97.9% | ~98.3% | ~98.7% | 25% |
| GGUF Q4_K_M | ~97.5% | ~96.8% | ~97.2% | 25% |

**Practical interpretation:**
- FP8 / INT8: barely measurable difference for most production tasks
- AWQ / GPTQ INT4: 1–3% degradation — acceptable for most applications
- INT2: 5–15% degradation — only for extreme memory constraints

## 4. Quick Decision Guide

```
Do you have a NVIDIA H100 or H200?
  YES → Use FP8. Native hardware support, minimal quality loss.

Do you have 80+ GB VRAM for a 70B model in bfloat16?
  YES → Don't quantize. Use bfloat16 for maximum quality.

Do you need a 70B model but only have one A100 (80GB)?
  → AWQ INT4 (70B fits in ~35GB)
  
Do you need a 7B model but only have 16GB VRAM?
  → GGUF Q4_K_M via llama.cpp (fits on consumer GPU or even CPU)

Default production recommendation for 7B–70B?
  → AWQ INT4 served via vLLM
  → Or FP8 if on H100+
  
Need absolute maximum quality with quantization?
  → INT8 via bitsandbytes (7B: ~7GB, 70B: ~70GB)
```

## 5. Senior AI Engineer Perspective

### Always Evaluate Quantized Models on Your Specific Task

Published benchmarks (MMLU, HumanEval) don't predict quality on your specific use case.
A model quantized to INT4 might perform near-perfectly on customer support classification
and 5% worse on complex code generation. Test on your eval dataset before deploying.

### The Group Size Parameter

Both GPTQ and AWQ have a `group_size` parameter (typically 32–256). It controls the granularity
of quantization:
- Smaller group size (32): better quality, more memory overhead
- Larger group size (256): worse quality, less memory overhead
- Standard: 128 is the most common production choice
