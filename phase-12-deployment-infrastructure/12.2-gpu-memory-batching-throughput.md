## 1. What Is It? (Theory)

### Why LLM Inference Is Memory-Bound, Not Compute-Bound

An A100 GPU has 312 TFLOPS of compute but only 2 TB/s of memory bandwidth. For LLM inference:
- Each token requires loading all model weights from GPU memory
- At 7B × 2 bytes = 14GB of weights, loading everything for one token = 14GB of memory reads
- At 2 TB/s bandwidth: 14 GB / 2 TB/s = **7ms just to load weights for one token**

This means: adding more compute doesn't help unless you also increase how much you do with each memory load. **Batching** is the solution — load weights once, compute for many tokens simultaneously.

```
Without batching:     Load 14GB → generate 1 token = 7ms/token
With batch size 8:    Load 14GB → generate 8 tokens = 7ms/8 = 0.875ms/token
With batch size 64:   Load 14GB → generate 64 tokens = 7ms/64 = 0.11ms/token

8× batch = 8× throughput improvement from the same hardware
```

But there's a constraint: each request in the batch needs its own KV-cache. Larger batch = more KV-cache memory required.

## 2. How It Actually Works — Deep Mechanics

### 2.1 GPU Memory Layout During Inference

```
A100 80GB VRAM allocation:
┌────────────────────────────────────────────────────────────┐
│  MODEL WEIGHTS (fixed)                                      │
│  7B model  = ~14 GB (bfloat16)                             │
│  13B model = ~26 GB                                        │
│  70B model = ~140 GB (doesn't fit on single A100)          │
├────────────────────────────────────────────────────────────┤
│  KV-CACHE (grows with context)                             │
│  Per token, per layer:                                     │
│    LLaMA-3 8B: 0.5 MB/token                               │
│  At 8K context, batch=16:                                  │
│    16 × 8000 × 0.5MB = 64 GB                              │
├────────────────────────────────────────────────────────────┤
│  ACTIVATIONS (temporary, during computation)               │
│  ~5–10 GB depending on batch size                          │
└────────────────────────────────────────────────────────────┘
```

The key insight: **KV-cache dominates memory usage at high context lengths and large batch sizes**.
The "model fits on my GPU" check is insufficient — you also need to account for KV-cache.

### 2.2 PagedAttention: The Game-Changer

Traditional KV-cache allocation: reserve the maximum possible space for each sequence upfront.

```
Traditional: 
  Request arrives (will generate up to 2048 tokens)
  → Allocate 2048 token slots in KV-cache immediately
  → Most slots empty until generation reaches them
  → 80% of reserved memory is wasted on average (allocation waste)

PagedAttention (vLLM):
  Request arrives
  → Allocate KV-cache pages on demand as tokens are generated
  → Pages = fixed-size blocks (e.g., 16 tokens each)
  → No reservation; no waste
  → Pages from different requests can be interleaved in memory
  → Average memory utilization: ~95% (vs ~30% traditional)
```

This is directly analogous to OS virtual memory management. vLLM treats GPU memory like
virtual pages, enabling much larger effective batch sizes.

### 2.3 Continuous Batching (Iteration-Level Batching)

Traditional batching: wait until you have N requests, process all N together, then wait again.

```
Static batching:
  t=0: Requests [A, B, C, D] arrive → start batch
  t=5: Request A finishes (short response)
       Request B, C, D still running
       Request E, F arrive → WAIT (can't join mid-batch)
  t=10: All of B, C, D finish → batch complete
  t=10: Start new batch with E, F (late!)

Continuous batching:
  t=0: Requests [A, B, C, D] arrive → start batch
  t=5: Request A finishes
       → Immediately slot in Request E
  t=7: Request B finishes
       → Immediately slot in Request F
  Batch is always full → GPU never idle between requests
```

Continuous batching improves GPU utilization dramatically for mixed-length requests.
vLLM uses continuous batching by default — you don't need to configure it.

### 2.4 Measuring and Configuring Throughput

```python
import time
import asyncio
import httpx

async def benchmark_vllm(
    base_url: str,
    model:    str,
    prompts:  list[str],
    max_tokens: int = 100,
    concurrency: int = 10,
) -> dict:
    """
    Benchmark vLLM throughput at a given concurrency level.
    """
    
    async def single_request(client: httpx.AsyncClient, prompt: str) -> dict:
        start = time.perf_counter()
        resp = await client.post(
            f"{base_url}/v1/chat/completions",
            json={
                "model":      model,
                "messages":   [{"role": "user", "content": prompt}],
                "max_tokens": max_tokens,
            },
            timeout=120.0
        )
        elapsed = time.perf_counter() - start
        data = resp.json()
        tokens = data["usage"]["completion_tokens"]
        return {"latency_s": elapsed, "tokens": tokens}
    
    async with httpx.AsyncClient() as client:
        start_all = time.perf_counter()
        tasks = [single_request(client, p) for p in prompts[:concurrency]]
        results = await asyncio.gather(*tasks)
        total_elapsed = time.perf_counter() - start_all
    
    total_tokens = sum(r["tokens"] for r in results)
    avg_latency  = sum(r["latency_s"] for r in results) / len(results)
    throughput   = total_tokens / total_elapsed
    
    return {
        "concurrency":      concurrency,
        "requests":         len(results),
        "total_tokens":     total_tokens,
        "total_time_s":     round(total_elapsed, 2),
        "throughput_tok_s": round(throughput, 1),
        "avg_latency_s":    round(avg_latency, 2),
        "p95_latency_s":    round(sorted(r["latency_s"] for r in results)[int(len(results)*0.95)], 2),
    }

# Run benchmark at different concurrency levels
async def sweep_concurrency():
    prompts = ["Explain machine learning in 3 sentences."] * 100
    
    for concurrency in [1, 4, 8, 16, 32]:
        result = await benchmark_vllm(
            base_url="http://localhost:8000",
            model="meta-llama/Llama-3-8B-Instruct",
            prompts=prompts,
            concurrency=concurrency,
        )
        print(f"Concurrency={concurrency:3d}: "
              f"{result['throughput_tok_s']:6.0f} tok/s | "
              f"latency={result['avg_latency_s']:.2f}s")

asyncio.run(sweep_concurrency())
```

### 2.5 Configuring vLLM for Maximum Throughput

```bash
# Throughput-optimized configuration
python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Llama-3-8B-Instruct \
    --dtype bfloat16 \
    
    # Memory allocation
    --gpu-memory-utilization 0.90 \    # use 90% of GPU VRAM for KV-cache
    
    # Batching
    --max-num-batched-tokens 8192 \    # max tokens across all requests in one batch
    --max-num-seqs 256 \               # max concurrent sequences
    
    # KV-cache tuning
    --block-size 32 \                  # page size in tokens (default 16; larger = less overhead)
    
    # For long contexts
    --max-model-len 16384 \            # max context length
    
    # Enable CUDA graphs (faster kernels for fixed-size batches)
    --enforce-eager \                  # remove this to enable CUDA graphs
    
    --port 8000
```

**Latency-optimized configuration:**
```bash
python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Llama-3-8B-Instruct \
    --dtype bfloat16 \
    --gpu-memory-utilization 0.80 \
    --max-num-seqs 4 \                 # keep batch small for low latency
    --max-num-batched-tokens 1024 \
    --port 8000
```

## 3. Real-World Context

### Throughput Benchmarks (2026, A100 80GB, LLaMA-3 8B)

| Configuration | Throughput | Latency (p50) |
|---|---|---|
| Naive (batch=1) | ~200 tok/s | ~50ms/tok |
| vLLM default | ~2,000 tok/s | ~20ms/tok |
| vLLM optimized | ~4,000+ tok/s | ~15ms/tok |
| vLLM + INT4 AWQ | ~6,000+ tok/s | ~10ms/tok |

**The 20× gap** between naive and optimized serving is the infrastructure opportunity.

### vLLM Benchmarks: PagedAttention Impact

Anyscale benchmarks (2024) showed vLLM achieves 23× higher throughput vs HuggingFace Transformers
naive serving, primarily through:
- Continuous batching: ~5× improvement
- PagedAttention: ~4× improvement (larger effective batch size)
- CUDA graph optimization: ~1.5× improvement

## 5. Trade-offs

### Batch Size vs Latency

```
Batch size 1:   Latency ~50ms first token, ~5ms/subsequent   | Low throughput
Batch size 8:   Latency ~60ms first token, ~15ms/subsequent  | Good balance
Batch size 64:  Latency ~200ms first token, ~50ms/subsequent | High throughput, high latency
```

There is no configuration that simultaneously maximizes throughput AND minimizes latency.
You must decide which matters more for your use case (Section 12.5 covers this in depth).

## 6. Senior AI Engineer Perspective

### The gpu-memory-utilization Parameter Is Crucial

The most impactful single parameter in vLLM is `--gpu-memory-utilization`. Setting it too
low wastes KV-cache capacity (smaller batches). Setting it too high causes OOM crashes.

Practical approach:
1. Start at 0.85 (default)
2. Monitor actual VRAM usage via `nvidia-smi`
3. Increase to 0.90–0.92 if you have headroom
4. Set no higher than 0.95 — leave room for spikes

The difference between 0.80 and 0.90 utilization can mean 2× more concurrent requests.

### Always Profile Before Scaling Horizontally

The instinct when performance is insufficient: "add more GPUs." The correct approach: first
verify you're extracting maximum throughput from the current GPU. Adding GPUs is 5–10× more
expensive than fixing a configuration issue that leaves 80% of capacity idle.

Profile first with the benchmark script above. If you're getting <70% of theoretical throughput,
there's a configuration problem to solve.
