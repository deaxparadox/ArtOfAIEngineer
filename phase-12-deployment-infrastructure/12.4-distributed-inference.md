## 1. What Is It? (Theory)

### When a Single GPU Isn't Enough

A 70B parameter model in bfloat16 requires ~140 GB of VRAM. A single A100 has 80 GB. You
need at least 2 GPUs just to hold the model weights, before accounting for KV-cache.

**Distributed inference** splits the computation across multiple GPUs. There are three
fundamental strategies — and choosing the right one depends on whether you're constrained
by memory, latency, or throughput.

```
TENSOR PARALLELISM (TP):
  Split each layer across multiple GPUs.
  Each GPU computes a fraction of every matrix multiplication.
  Communication needed: all-reduce after each layer.
  Best for: latency (all GPUs work on every request together)
  Constraint: requires high-bandwidth GPU interconnects (NVLink)

PIPELINE PARALLELISM (PP):
  Assign consecutive layers to different GPUs.
  GPU 1 processes layers 1–8, GPU 2 processes layers 9–16, etc.
  Best for: throughput (multiple batches in the pipeline simultaneously)
  Constraint: pipeline bubbles reduce efficiency; higher latency

DATA PARALLELISM (DP):
  Replicate the full model on multiple GPUs.
  Each GPU handles different requests.
  Best for: throughput scaling when model fits on one GPU
  Constraint: requires model to fit in single GPU VRAM
```

## 2. How It Actually Works — Deep Mechanics

### 2.1 Tensor Parallelism with vLLM

```bash
# 2-GPU tensor parallelism for LLaMA-3 70B
python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Llama-3-70B-Instruct \
    --tensor-parallel-size 2 \   # split across 2 GPUs
    --dtype bfloat16 \
    --gpu-memory-utilization 0.90 \
    --port 8000

# 4-GPU tensor parallelism (for very large models or more throughput)
python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Llama-3-70B-Instruct \
    --tensor-parallel-size 4 \
    --dtype bfloat16 \
    --port 8000

# Multi-node tensor parallelism (example: 2 nodes, 4 GPUs each = 8 GPUs total)
# Node 1 (head):
VLLM_HOST_IP=<head-node-ip> \
python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Llama-3-405B-Instruct \
    --tensor-parallel-size 8 \
    --pipeline-parallel-size 1 \
    --dist-init-addr <head-node-ip>:12355 \
    --port 8000
```

### 2.2 What Tensor Parallelism Does to Attention

For a transformer with hidden size d_model:

```
Standard attention (single GPU):
  Q = X @ W_Q        [d_model × d_model] computation on one GPU

Tensor parallel attention (4 GPUs):
  GPU 0 computes Q_0 = X @ W_Q[0:256]   # head 0 of 32
  GPU 1 computes Q_1 = X @ W_Q[256:512] # head 8 of 32
  GPU 2 computes Q_2 = X @ W_Q[512:768] # head 16 of 32
  GPU 3 computes Q_3 = X @ W_Q[768:d]   # head 24 of 32
  → All-reduce to combine
```

Each GPU holds 1/N of each weight matrix, does 1/N of the computation, then combines results.

### 2.3 Pipeline Parallelism

```python
# In vLLM: combine tensor + pipeline parallelism for very large models
# Example: 4-node cluster, 4 GPUs per node = 16 GPUs total for 405B model

import subprocess

command = [
    "python", "-m", "vllm.entrypoints.openai.api_server",
    "--model", "meta-llama/Llama-3-405B-Instruct",
    "--tensor-parallel-size", "4",     # 4 GPUs per node do tensor parallelism
    "--pipeline-parallel-size", "4",   # 4 nodes do pipeline parallelism
    "--dtype", "bfloat16",
]
# Total: 4 × 4 = 16 GPUs
# 405B model: 405B × 2 bytes = 810 GB → needs 10+ A100 80GB → 16 GPUs works
```

### 2.4 Data Parallelism: The Right Choice When Model Fits

If your model fits in a single GPU but you need more throughput:

```bash
# Instead of splitting one model across GPUs,
# run multiple independent model instances behind a load balancer

# docker-compose with multiple vLLM instances
version: "3.8"
services:
  vllm_0:
    image: vllm/vllm-openai:latest
    command: "--model meta-llama/Llama-3-8B-Instruct --port 8001"
    deploy:
      resources:
        reservations:
          devices: [{driver: nvidia, device_ids: ['0'], capabilities: [gpu]}]
    ports: ["8001:8001"]

  vllm_1:
    image: vllm/vllm-openai:latest
    command: "--model meta-llama/Llama-3-8B-Instruct --port 8002"
    deploy:
      resources:
        reservations:
          devices: [{driver: nvidia, device_ids: ['1'], capabilities: [gpu]}]
    ports: ["8002:8002"]

  nginx:
    image: nginx:alpine
    volumes: ["./nginx.conf:/etc/nginx/nginx.conf"]
    ports: ["8000:80"]
    depends_on: [vllm_0, vllm_1]
```

```nginx
# nginx.conf — round-robin load balancing
upstream vllm_backends {
    server vllm_0:8001;
    server vllm_1:8002;
}

server {
    listen 80;
    location /v1/ {
        proxy_pass http://vllm_backends;
    }
}
```

### 2.5 Speculative Decoding (Cross-Model Collaboration)

A newer distributed inference technique: use a small, fast "draft" model to propose
multiple tokens, then verify them in parallel with the large target model:

```
Without speculative decoding:
  Large model generates 1 token → 10ms
  For 100 tokens: 1000ms

With speculative decoding:
  Small model (7B) generates 5 draft tokens → 2ms
  Large model (70B) verifies all 5 in parallel → 12ms
  Accept on average 3 tokens → 14ms for 3 tokens = 4.7ms/token
  
  Speedup: 10ms → 4.7ms per accepted token = ~2.1× faster
```

```bash
# vLLM speculative decoding
python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Llama-3-70B-Instruct \
    --speculative-model meta-llama/Llama-3-8B-Instruct \  # draft model
    --num-speculative-tokens 5 \  # tokens draft model proposes
    --tensor-parallel-size 2 \
    --port 8000
```

## 3. Real-World Context

### When to Use Each Strategy

| Model Size | GPU VRAM | Strategy | Notes |
|---|---|---|---|
| ≤ 8B | 24 GB single GPU | Single GPU | No parallelism needed |
| 13B | 24 GB single (INT4) or 40 GB (bf16) | Single GPU | Quantize if needed |
| 70B | 2× A100 80GB | Tensor parallelism (TP=2) | Standard deployment |
| 70B | 4× A100 40GB | Tensor parallelism (TP=4) | Works, slightly less efficient |
| 405B | 8× A100 80GB | TP=4 + PP=2 or TP=8 | Need NVLink for TP |
| Any | Multiple nodes, high throughput | Data parallelism | Each node has full model |

### NVLink vs PCIe for Tensor Parallelism

Tensor parallelism requires an all-reduce after every attention layer — potentially GB of
data per forward pass. The interconnect bandwidth determines TP efficiency:

| Interconnect | Bandwidth | TP Efficiency |
|---|---|---|
| NVLink 4.0 (H100 SXM) | 900 GB/s | Excellent — TP scales well |
| NVLink 3.0 (A100 SXM) | 600 GB/s | Good — TP scales well |
| PCIe 4.0 | 64 GB/s | Poor — communication bottleneck |

**Rule:** Only use tensor parallelism between GPUs connected by NVLink. For PCIe-connected
GPUs (e.g., cloud instances without NVLink), data parallelism is almost always better.

## 4. Trade-offs

### Tensor Parallelism Scaling Efficiency

TP doesn't scale linearly — communication overhead grows with more GPUs:

| TP Size | Throughput (relative) | Latency improvement |
|---|---|---|
| 1× (baseline) | 1× | — |
| 2× (NVLink) | ~1.9× | ~1.8× better |
| 4× (NVLink) | ~3.4× | ~3.0× better |
| 8× (NVLink) | ~5.5× | ~4.5× better |

Communication overhead prevents perfect linear scaling. Beyond TP=4 per node, the returns
diminish significantly.

## 5. Senior AI Engineer Perspective

### The NVLink Tax

Cloud instances with NVLink (A100 SXM, H100 SXM) are significantly more expensive than
PCIe variants. Before paying for NVLink for tensor parallelism, verify that your use case
actually benefits:

- If you're latency-constrained and your model needs multi-GPU: NVLink TP is worth it
- If you're throughput-constrained: data parallelism on PCIe GPUs is often cheaper

The A100 PCIe at 40 GB VRAM running data parallelism can match A100 SXM 80 GB with TP=2
on throughput tasks, at lower cost.

### Speculative Decoding Is Worth Testing

Speculative decoding achieves 2–3× speedup with no quality degradation when:
- The draft model is from the same family as the target model (LLaMA-3 8B + 70B)
- Generation is relatively predictable (lower temperature, short outputs)
- You have GPU memory to spare for the draft model

For user-facing chat with 200–500 token outputs, speculative decoding often doubles the
perceived response speed at modest memory cost.
