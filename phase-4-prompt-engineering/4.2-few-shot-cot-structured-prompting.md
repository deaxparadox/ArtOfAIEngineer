## 1. What Is It? (Theory)

### The Three Techniques That Changed Everything

Between 2020 and 2022, three prompting discoveries dramatically changed what was possible with LLMs:

1. **Few-shot prompting** (Brown et al., 2020, GPT-3 paper): showing examples inside the prompt
   dramatically improves output quality and format adherence — without any fine-tuning.

2. **Chain-of-thought** (Wei et al., 2022): asking the model to reason step-by-step before answering
   dramatically improves accuracy on reasoning tasks.

3. **Self-consistency** (Wang et al., 2022): sampling multiple reasoning chains and majority-voting
   the answers reduces hallucinations on reasoning tasks.

These aren't tricks. They're evidence about how transformer-based language models work:
- Examples in context teach format better than instructions
- More output tokens = more "working memory" for complex reasoning
- Diverse reasoning paths catch errors that a single path misses

## 2. How It Actually Works — Deep Mechanics

### 2.1 Zero-Shot Prompting

No examples. Just task description + question.

```
Prompt:
"Classify the sentiment of the following tweet as positive, negative, or neutral.

Tweet: 'Just got my order. It's okay I guess.'"

Response: "Neutral"
```

Zero-shot works for:
- Well-defined tasks with clear semantics (sentiment, classification, summarization)
- Models that have seen many examples of the task in pre-training
- When format flexibility is acceptable

Zero-shot fails for:
- Niche output formats the model doesn't default to
- Tasks requiring specific domain conventions
- Complex reasoning requiring multiple steps

**2026 update:** Modern frontier models (GPT-4o, Claude Sonnet) handle many tasks zero-shot that
required few-shot in 2022. Try zero-shot first. Add examples only when you observe failures.

### 2.2 Few-Shot Prompting

Include 2–10 (prompt, completion) examples before the actual question. The model learns:
- The output format
- The expected response style and length
- Domain-specific patterns and conventions

```
System: "You are a customer intent classifier."

User:
Example 1:
Input: "Can I get a refund for my order from last week?"
Output: {"intent": "refund_request", "urgency": "medium"}

Example 2:
Input: "My package hasn't arrived and it's been 3 weeks!!!"
Output: {"intent": "shipping_inquiry", "urgency": "high"}

Example 3:
Input: "Do you have this in blue?"
Output: {"intent": "product_inquiry", "urgency": "low"}

Now classify:
Input: "I need to cancel my subscription immediately"
Output:
```

**Why it works:** The examples demonstrate the mapping from input to output in a way that pure
instructions can't. "Output a JSON object with intent and urgency" is less precise than seeing
three examples of correct output.

**How many examples?**

| Task complexity | Recommended examples |
|---|---|
| Simple classification | 2–3 |
| Format adherence | 3–5 |
| Complex domain task | 5–10 |
| Subtle nuance (tone, style) | 10+ |

**Example quality beats quantity.** Three diverse, high-quality examples outperform ten similar,
mediocre ones. Cover edge cases and difficult inputs in your few-shot set.

**Dynamic few-shot:** Select examples based on similarity to the current input (retrieval over your
example library). Closest examples produce better results than static examples. Used in production
RAG systems and advanced prompt pipelines.

### 2.3 Chain-of-Thought (CoT) Prompting

CoT instructs the model to reason step-by-step before giving the final answer.

**Zero-shot CoT:**
```
"Solve this step by step: If a train travels at 60 mph and needs to cover 150 miles,
how many minutes will the journey take?"
```

The magic phrase "step by step" (or "think through this carefully") unlocks better reasoning on
most frontier models — because the model generates intermediate reasoning tokens that constrain
the final answer.

**Few-shot CoT (original Wei et al. 2022 style):**
```
Q: Roger has 5 tennis balls. He buys 2 more cans of tennis balls. Each can has 3 balls.
   How many tennis balls does he have now?
A: Roger started with 5 balls. 2 cans × 3 balls = 6 balls. 5 + 6 = 11. The answer is 11.

Q: The cafeteria had 23 apples. If they used 20 to make lunch and bought 6 more, how many do they have?
A: [model continues the step-by-step pattern]
```

**Why CoT works (mechanistically):**
- Intermediate reasoning tokens act as working memory
- Each step is constrained by previous steps — the reasoning path is coherent
- The model can't jump to a wrong final answer without contradicting an intermediate step
- More output tokens = more capacity to work through a complex problem

**When CoT doesn't help:**
- Simple factual questions: "What is the capital of France?" — CoT adds no value
- Tasks where the answer requires domain knowledge, not reasoning steps
- Very short-context reasoning where the model can handle it in one step

**2025 research finding:** For strong modern models (GPT-4o, Claude Sonnet), zero-shot CoT
("think step by step") often matches few-shot CoT quality. The few-shot examples in CoT
contribute primarily to format standardization, not reasoning improvement. Start with zero-shot CoT.

### 2.4 Self-Consistency

Sample multiple reasoning chains (temperature > 0), aggregate the final answers, return the majority.

```
Question: "If all roses are flowers, and some flowers fade quickly, can we conclude all roses fade quickly?"

Sample 1 chain: "All roses are flowers. Some flowers fade. But 'some' doesn't mean 'all flowers'.
                 So we can't conclude all roses fade quickly." → Answer: No
                 
Sample 2 chain: "The premise says some flowers fade, not all. Roses are a subset of flowers.
                 We can't generalize from 'some flowers' to 'all roses'." → Answer: No
                 
Sample 3 chain: "Some flowers fade → not necessarily all roses fade → No conclusion." → Answer: No

Majority vote: No (3/3). High confidence.
```

**When to use self-consistency:**
- High-stakes reasoning questions where accuracy matters more than cost
- Math problems where wrong answers are common
- Logic puzzles, multi-step reasoning chains

**Cost:** N× more tokens (N samples). Use sparingly. For production systems, self-consistency
is usually reserved for the hardest/most important queries, not every request.

### 2.5 Structured Prompting Patterns

Beyond the basics, these patterns address specific production needs:

**XML / Delimited blocks for long contexts:**

```xml
<documents>
  <doc id="1">First document content...</doc>
  <doc id="2">Second document content...</doc>
</documents>

<task>
Summarize the key differences between the two documents above.
</task>

<format>
Return a bullet list. Maximum 5 bullets.
</format>
```

XML tags prevent content/instruction bleed in long prompts. They also allow the model to
reference specific sections. Anthropic's models respond particularly well to XML structure.

**Role-play / persona prompting:**

```
"You are a skeptical senior engineer reviewing this code for security vulnerabilities.
Your job is to find flaws, not to compliment good code. Be specific about attack vectors."
```

Establishing a skeptical, critical persona increases the quality of code reviews compared to
a neutral "review this code" prompt.

**Constrained generation / COSTAR framework:**

```
Context:    [background information]
Objective:  [what the task should accomplish]
Style:      [writing style, tone]
Tone:       [formal, casual, etc.]
Audience:   [who will read this]
Response:   [format requirements]
```

**Step-back prompting (Zheng et al., 2023):**

Ask the model to first identify the general principle, then apply it:

```
"Before answering, first identify what category of problem this is and what
general principles apply. Then apply those principles to the specific question."
```

Step-back prompting improves performance on questions requiring abstract reasoning or
domain knowledge transfer.

## 3. Real-World Context

### Where Each Technique Is Used in Production

| Technique | Production Use Case | Typical Impact |
|---|---|---|
| Zero-shot | Simple classification, extraction, summarization | Baseline quality |
| Few-shot | Format-critical outputs, domain-specific tasks | +10–40% format adherence |
| Zero-shot CoT | Math, logic, multi-step reasoning | +15–40% on reasoning benchmarks |
| Few-shot CoT | Complex domain reasoning with strict format | Best quality for hard tasks |
| Self-consistency | High-stakes answers, medical/legal use cases | +5–20% accuracy at 3–5× cost |
| XML structure | Long-context multi-document tasks | Better attention to instructions |

### The Dynamic Few-Shot Production Pattern

```
User query arrives
    │
    ▼
Embed the query
    │
    ▼
Retrieve top-K most similar examples from example library
    │
    ▼
Build prompt: system + retrieved_examples + user_query
    │
    ▼
Send to model
```

Used at companies like Stripe (fraud classification), Cohere (document processing), and many
RAG systems for increasing classification accuracy without fine-tuning.

## 4. Code Examples

### Zero-Shot vs Few-Shot vs CoT Comparison

```python
import anthropic

client = anthropic.Anthropic()

def prompt_model(system: str, user: str) -> str:
    r = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=500,
        system=system,
        messages=[{"role": "user", "content": user}]
    )
    return r.content[0].text


question = """A store sells apples for $0.50 each and oranges for $0.75 each.
Alice buys 4 apples and some oranges. She pays exactly $5.00.
How many oranges did she buy?"""

# === Zero-shot ===
zero_shot = prompt_model(
    system="You are a math assistant. Answer concisely.",
    user=question
)
print("ZERO-SHOT:", zero_shot[:200])

# === Zero-shot CoT ===
cot = prompt_model(
    system="You are a math assistant. Think step by step before giving your final answer.",
    user=question
)
print("\nZERO-SHOT CoT:", cot[:300])

# === Few-shot CoT ===
few_shot_examples = """
Example 1:
Q: A jar has red and blue marbles. There are 3 more red than blue.
   Total = 15. How many blue?
A: Let blue = b, then red = b + 3. So b + (b + 3) = 15. 2b = 12. b = 6. Blue marbles = 6.

Example 2:
Q: Movie tickets cost $8 adult and $5 child. A family pays $41 for 2 adults and some children.
A: 2 adults = $16. Remaining = $41 - $16 = $25. Children = $25 / $5 = 5. Answer: 5 children.
"""

few_shot_cot = prompt_model(
    system="You are a math assistant. Solve problems step by step as shown in the examples.",
    user=few_shot_examples + "\n\nNow solve:\n" + question
)
print("\nFEW-SHOT CoT:", few_shot_cot[:300])
```

### Self-Consistency Implementation

```python
import anthropic
from collections import Counter
import re

client = anthropic.Anthropic()

def self_consistent_answer(question: str, n_samples: int = 5) -> dict:
    """
    Sample N reasoning chains, extract final answers, majority-vote.
    """
    chains = []
    final_answers = []
    
    for i in range(n_samples):
        response = client.messages.create(
            model="claude-haiku-4-5-20251001",  # cheaper for sampling
            max_tokens=400,
            system="Solve step by step. End with 'FINAL ANSWER: [answer]'",
            messages=[{"role": "user", "content": question}],
            temperature=0.7   # variation enables diverse reasoning paths
        )
        
        chain = response.content[0].text
        chains.append(chain)
        
        # Extract final answer
        match = re.search(r"FINAL ANSWER:\s*(.+?)(?:\n|$)", chain, re.IGNORECASE)
        if match:
            final_answers.append(match.group(1).strip().lower())
    
    # Majority vote
    if final_answers:
        vote_counts = Counter(final_answers)
        best_answer, best_count = vote_counts.most_common(1)[0]
        confidence = best_count / n_samples
    else:
        best_answer = "could not extract"
        confidence = 0
    
    return {
        "question": question,
        "answer": best_answer,
        "confidence": f"{confidence:.0%} ({best_count}/{n_samples} agree)",
        "all_answers": final_answers,
        "sample_chain": chains[0][:300]
    }

result = self_consistent_answer(
    "A factory produces 240 widgets per day. If they need to fill an order of 3,360 widgets "
    "and they work 5 days per week, how many complete weeks does it take?",
    n_samples=5
)
print(f"Answer:     {result['answer']}")
print(f"Confidence: {result['confidence']}")
```

### Dynamic Few-Shot with Semantic Similarity

```python
import anthropic
import numpy as np
from dataclasses import dataclass

client = anthropic.Anthropic()

@dataclass
class Example:
    input: str
    output: str
    embedding: list[float] = None

def get_embedding(text: str) -> list[float]:
    """Get embedding using Voyage AI (Anthropic's recommended embedding model)."""
    # In practice: use voyage-3 via Anthropic's integration or OpenAI embeddings
    # Pseudocode for demonstration
    import hashlib
    # Placeholder: returns a deterministic pseudo-embedding
    h = hashlib.md5(text.encode()).hexdigest()
    return [int(h[i:i+2], 16) / 255.0 for i in range(0, 32, 2)]

def cosine_similarity(a: list[float], b: list[float]) -> float:
    a, b = np.array(a), np.array(b)
    return float(np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b) + 1e-8))

class DynamicFewShotClassifier:
    """
    Builds prompts using the most relevant examples for each query.
    """
    
    def __init__(self, examples: list[dict], n_examples: int = 3):
        self.n_examples = n_examples
        self.examples = []
        for ex in examples:
            emb = get_embedding(ex["input"])
            self.examples.append(Example(ex["input"], ex["output"], emb))
    
    def get_relevant_examples(self, query: str) -> list[Example]:
        query_emb = get_embedding(query)
        scored = [(cosine_similarity(query_emb, ex.embedding), ex) for ex in self.examples]
        scored.sort(key=lambda x: x[0], reverse=True)
        return [ex for _, ex in scored[:self.n_examples]]
    
    def classify(self, query: str) -> str:
        relevant = self.get_relevant_examples(query)
        
        # Build dynamic few-shot prompt
        examples_text = "\n\n".join([
            f"Input: {ex.input}\nOutput: {ex.output}"
            for ex in relevant
        ])
        
        prompt = f"""Examples:
{examples_text}

Now classify:
Input: {query}
Output:"""
        
        response = client.messages.create(
            model="claude-haiku-4-5-20251001",
            max_tokens=100,
            system="You are an intent classifier. Output only the classification, nothing else.",
            messages=[{"role": "user", "content": prompt}]
        )
        return response.content[0].text.strip()


# Example: support ticket classification
example_library = [
    {"input": "I need a refund for my order",        "output": "billing"},
    {"input": "My package hasn't arrived",           "output": "shipping"},
    {"input": "How do I reset my password?",         "output": "account"},
    {"input": "The app keeps crashing on my phone",  "output": "technical"},
    {"input": "I was charged twice for one item",    "output": "billing"},
    {"input": "Can I change my delivery address?",   "output": "shipping"},
    {"input": "I can't log into my account",         "output": "account"},
]

classifier = DynamicFewShotClassifier(example_library, n_examples=3)
test_queries = [
    "My credit card was charged incorrectly",
    "The tracking number shows delivered but I didn't get it",
    "I forgot my login details",
]

for q in test_queries:
    print(f"'{q}' → {classifier.classify(q)}")
```

### XML-Structured Long-Context Prompt

```python
import anthropic

client = anthropic.Anthropic()

def analyze_documents(docs: list[dict], question: str) -> str:
    """
    Use XML structure to clearly separate documents from instructions
    in a long-context multi-document prompt.
    """
    docs_xml = "\n".join([
        f'<document id="{d["id"]}" title="{d["title"]}">\n{d["content"]}\n</document>'
        for d in docs
    ])
    
    user_message = f"""<documents>
{docs_xml}
</documents>

<question>
{question}
</question>

<instructions>
1. Answer using only information from the provided documents.
2. Cite each claim with the document ID: [doc-N].
3. If documents contradict each other, note the discrepancy.
4. If the answer is not in the documents, say so explicitly.
</instructions>"""

    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system="You are a research analyst. Follow the instructions in the user message precisely.",
        messages=[{"role": "user", "content": user_message}]
    )
    return response.content[0].text


docs = [
    {"id": "1", "title": "Q3 Report", "content": "Revenue was $1.82B, up 8% YoY."},
    {"id": "2", "title": "Press Release", "content": "The company achieved record revenue of $1.82 billion in Q3 2024."},
    {"id": "3", "title": "Analyst Note", "content": "Growth decelerated from 14% in Q2 to 8% in Q3."},
]

answer = analyze_documents(docs, "What was the revenue trend from Q2 to Q3?")
print(answer)
```

### Extended Thinking (Anthropic)

Claude's extended thinking feature allows the model to "think" internally before responding:

```python
import anthropic

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=16000,
    thinking={
        "type": "enabled",
        "budget_tokens": 10000   # tokens the model can use for internal reasoning
    },
    messages=[{
        "role": "user",
        "content": "Prove that sqrt(2) is irrational."
    }]
)

# Response includes thinking blocks + final answer
for block in response.content:
    if block.type == "thinking":
        print("INTERNAL REASONING:", block.thinking[:200], "...")
    elif block.type == "text":
        print("FINAL ANSWER:", block.text)
```

## 6. Alternatives & Comparisons

### Prompting vs Fine-tuning for Format Adherence

| Approach | Data Needed | Speed to Implement | Format Consistency |
|---|---|---|---|
| Zero-shot instruction | None | Minutes | ~70–80% |
| Few-shot (3–5 examples) | 3–5 examples | 30 min | ~85–95% |
| Structured output API (JSON mode) | None | 1 hour | ~99%+ |
| Fine-tuning on format examples | 100–1K examples | Days | ~99%+ |

For pure format consistency, **use structured output APIs** (Section 4.3). Few-shot is better for
complex format nuances that can't be expressed in a JSON schema.

### Self-Consistency vs Extended Thinking

| Method | Mechanism | Cost | Best For |
|---|---|---|---|
| Self-consistency | Multiple separate API calls | N× model cost | When reasoning model not available |
| Extended thinking | Single call, internal reasoning | 2–5× base cost | Best accuracy on complex problems |

For Anthropic models: use extended thinking. For other providers without reasoning mode: self-consistency.

## 7. Trade-offs

### Few-Shot Examples
**Pros:** Dramatically improves format adherence; teaches domain conventions without fine-tuning
**Cons:** Increases token count; wrong examples can mislead; static examples may not cover all cases

### Chain-of-Thought
**Pros:** Significant accuracy gains on reasoning tasks; makes errors visible and debuggable
**Cons:** More output tokens = higher cost + higher latency; not beneficial for simple tasks

### Self-Consistency
**Pros:** Higher accuracy than single chain; especially useful for math/logic
**Cons:** N× more expensive; slow; only meaningful for tasks with deterministic answers

## 8. When To Use vs. When To Avoid

### Zero-Shot CoT
✅ Use when: Math, logic, multi-step planning — any task requiring reasoning
✅ Use when: The model produces inconsistent answers to the same question
❌ Avoid when: Simple factual questions, summarization, basic extraction

### Few-Shot
✅ Use when: Specific output format required (especially non-JSON)
✅ Use when: Domain-specific classification/labeling tasks
✅ Use when: Zero-shot produces wrong tone, length, or structure
❌ Avoid when: Structured output API (Section 4.3) handles the format need

### Self-Consistency
✅ Use when: High-stakes answers where accuracy justifies cost
✅ Use when: Production evals show high variance in answers to the same question
❌ Avoid when: Open-ended generation tasks (no "correct answer" to vote on)

⚠️ Common mistakes:
- Writing more examples when the examples are the wrong ones — fix quality before adding quantity
- Using CoT for simple tasks (adds latency/cost, no benefit)
- Selecting few-shot examples that are too similar (diverse coverage beats repeated themes)
- Prompting CoT on tasks where reasoning tokens can hurt (creative tasks where overthinking stifles output)

## 9. Senior AI Engineer Perspective

### The "Zero-Shot First" Rule

Every prompting decision should start with zero-shot and only add complexity when you've measured
a gap. Teams that jump to few-shot or CoT without first establishing a zero-shot baseline don't
know what they're actually getting from the complexity.

Workflow:
1. Zero-shot → measure baseline accuracy on your eval set
2. Zero-shot CoT → measure delta (costs more tokens; worth it?)
3. Few-shot → measure delta (costs more context; worth it?)
4. Fine-tuning → only if prompting can't close the gap

### The Most Overlooked Prompting Technique

**Negative examples in few-shot.** Most teams show only positive (correct) examples. Showing
a pair — "here is a WRONG output, here is the CORRECT output" — dramatically improves the model's
ability to avoid specific error types.

```
BAD output (avoid this):
Input: "I'm happy with my order."
Output: {"sentiment": "positive", "score": 0.9, "reason": "The word 'happy' indicates positive sentiment."}

CORRECT output:
Input: "I'm happy with my order."
Output: {"sentiment": "positive", "score": 0.9}
[Reason: no explanation needed, just the classification]
```

This is especially useful when you know the model's default behavior for a task is wrong in a
specific way.

### What Changed in 2025

Frontier models are now significantly better at following instructions without elaborate prompting.
The techniques from 2022 (elaborate CoT templates, large few-shot sets) are less necessary on
GPT-4o and Claude Sonnet. The value of prompting expertise shifted from "unlock capabilities" to
"reliably constraint behavior" — preventing hallucinations, enforcing format, maintaining persona.
