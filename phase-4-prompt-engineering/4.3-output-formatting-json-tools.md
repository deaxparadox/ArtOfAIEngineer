## 1. What Is It? (Theory)

### The Problem: Prose When You Need Data

LLMs default to generating prose. Your application often needs machine-readable data. The gap
between these two is where a lot of production AI engineering lives.

```
What you asked:  "Extract the company name and founding year from this text."
What you got:    "Based on the text, the company mentioned is Acme Corp,
                  which was founded in the year 1987."
What you needed: {"company": "Acme Corp", "founded": 1987}
```

Parsing unstructured prose in application code is brittle, error-prone, and a debugging nightmare.
Structured output is the solution: force the model to emit machine-parseable formats that your code
can consume directly.

### Three Approaches, One Goal

```
Approach 1: Prompt engineering (unreliable)
  "Output ONLY valid JSON. No prose."
  → Works ~80–90% of the time. Breaks on edge cases.

Approach 2: JSON mode / response format (reliable)
  response_format={"type": "json_object"}
  → Guarantees valid JSON. No schema enforcement.

Approach 3: Structured output / JSON schema (most reliable)
  response_format={"type": "json_schema", "json_schema": {...}}
  → Guarantees the output matches your exact schema.
  → Use this in production.
```

## 2. How It Actually Works — Deep Mechanics

### 2.1 JSON Mode

JSON mode instructs the model to produce valid JSON output. The model is constrained to generate
tokens that form valid JSON — no markdown, no prose.

**What it guarantees:** Valid JSON syntax
**What it doesn't guarantee:** Your specific schema, field names, or data types

```python
# OpenAI
response = client.chat.completions.create(
    model="gpt-4o",
    response_format={"type": "json_object"},
    messages=[{"role": "user", "content": "Extract: name, age, city from: 'Alice is 29, from NYC'"}]
)

# Anthropic does not have a dedicated JSON mode — use prefill or structured output instead
```

**JSON mode pitfall:** You must mention JSON in your prompt, or OpenAI will return an error.
You also get no schema guarantee — the model decides field names. For production, use JSON schema.

### 2.2 Structured Output (JSON Schema Enforcement)

JSON schema structured output enforces your exact schema at the token level. The model's next
token choices are constrained to only produce valid tokens for the current position in the schema.

This is done via **constrained decoding** — the provider's sampling logic masks out invalid tokens
at each generation step.

```
Schema: {"name": str, "age": int, "city": str}

At each token position, only valid tokens are allowed:
  After '{"': only '"name"' is valid (schema says first key is name)
  After '"name":': only '"' is valid (value must be string)
  After '"name": "Alice",': only '"age"' is valid (next key is age)
  After '"age":': only digits are valid (value must be integer)
```

**OpenAI Structured Outputs (2024+):**

```python
from openai import OpenAI
from pydantic import BaseModel

client = OpenAI()

class PersonExtraction(BaseModel):
    name: str
    age: int
    city: str
    confidence: float   # 0.0 to 1.0

response = client.beta.chat.completions.parse(
    model="gpt-4o-2024-08-06",  # requires this version or later
    messages=[
        {"role": "user", "content": "Alice is 29 years old and lives in New York City."}
    ],
    response_format=PersonExtraction,
)

person = response.choices[0].message.parsed
print(person.name)    # "Alice"
print(person.age)     # 29
print(person.city)    # "New York City"
```

**Anthropic Structured Output:**

Anthropic uses tool use for structured output. Define a tool, instruct the model to use it.
The tool's input schema enforces structure:

```python
import anthropic
import json
from typing import Any

client = anthropic.Anthropic()

def structured_extract(text: str, schema: dict, description: str) -> Any:
    """Extract structured data using Anthropic tool use."""
    
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        tools=[{
            "name": "extract_data",
            "description": description,
            "input_schema": schema
        }],
        tool_choice={"type": "tool", "name": "extract_data"},  # force tool use
        messages=[{"role": "user", "content": text}]
    )
    
    # Extract the tool input (our structured data)
    for block in response.content:
        if block.type == "tool_use":
            return block.input
    
    return None

# Define schema
person_schema = {
    "type": "object",
    "properties": {
        "name":       {"type": "string"},
        "age":        {"type": "integer"},
        "city":       {"type": "string"},
        "confidence": {"type": "number", "minimum": 0, "maximum": 1}
    },
    "required": ["name", "age", "city", "confidence"]
}

result = structured_extract(
    text="Alice is 29 years old and lives in New York City.",
    schema=person_schema,
    description="Extract person information from text"
)

print(result)
# {'name': 'Alice', 'age': 29, 'city': 'New York City', 'confidence': 0.98}
```

### 2.3 Tool Calls (Function Calling)

Tool calling is both a structured output mechanism AND an agentic action mechanism. The model
decides to call a tool, provides structured arguments, and your code executes the tool.

```
User: "What's the current stock price of AAPL?"

Model decides: I need to call get_stock_price with {"symbol": "AAPL"}

Your code: executes get_stock_price("AAPL") → returns "$182.50"

Model continues: generates final response using the tool result: "Apple's current price is $182.50"
```

The key insight: tool calling lets the model declare *intent* (I want to do X with these parameters),
and your code executes it. The model never directly runs code — it outputs structured data that
describes what it wants to do.

```python
import anthropic
import json

client = anthropic.Anthropic()

# Define available tools
tools = [
    {
        "name": "get_weather",
        "description": "Get current weather for a location. Use when asked about weather.",
        "input_schema": {
            "type": "object",
            "properties": {
                "location": {
                    "type": "string",
                    "description": "City and country code, e.g. 'London, UK'"
                },
                "unit": {
                    "type": "string",
                    "enum": ["celsius", "fahrenheit"],
                    "description": "Temperature unit"
                }
            },
            "required": ["location"]
        }
    },
    {
        "name": "search_web",
        "description": "Search the web for current information. Use for recent events or facts.",
        "input_schema": {
            "type": "object",
            "properties": {
                "query": {"type": "string", "description": "Search query"}
            },
            "required": ["query"]
        }
    }
]

# Simulated tool implementations
def execute_tool(name: str, inputs: dict) -> str:
    if name == "get_weather":
        return f"Current weather in {inputs['location']}: 22°C, partly cloudy"
    elif name == "search_web":
        return f"Search results for '{inputs['query']}': [simulated results]"
    return "Tool not found"

def agent_turn(user_message: str) -> str:
    messages = [{"role": "user", "content": user_message}]
    
    while True:
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            tools=tools,
            messages=messages
        )
        
        # End of turn — model gave a final text response
        if response.stop_reason == "end_turn":
            return next(b.text for b in response.content if b.type == "text")
        
        # Model wants to call a tool
        if response.stop_reason == "tool_use":
            # Add assistant's tool use request to messages
            messages.append({"role": "assistant", "content": response.content})
            
            # Execute all requested tools
            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    result = execute_tool(block.name, block.input)
                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": result
                    })
            
            # Return tool results to model
            messages.append({"role": "user", "content": tool_results})

# Test
print(agent_turn("What's the weather like in Tokyo?"))
```

### 2.4 Complex Schema Design

Real production schemas need careful design. Key principles:

**Make fields required when you need them guaranteed:**
```json
{
  "type": "object",
  "properties": {
    "intent":     {"type": "string", "enum": ["question", "complaint", "praise", "request"]},
    "urgency":    {"type": "string", "enum": ["low", "medium", "high", "critical"]},
    "summary":    {"type": "string", "maxLength": 200},
    "action_needed": {"type": "boolean"},
    "entities": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "type":  {"type": "string", "enum": ["product", "order_id", "date", "amount"]},
          "value": {"type": "string"}
        },
        "required": ["type", "value"]
      }
    }
  },
  "required": ["intent", "urgency", "summary", "action_needed", "entities"]
}
```

**Use enums to constrain categorical fields:**
Instead of `"string"` for sentiment, use `"enum": ["positive", "negative", "neutral"]`.
The model can't invent a new category, and your downstream code never hits an unexpected value.

**Use descriptions to guide extraction:**
```json
{
  "name": "order_amount",
  "type": "number",
  "description": "The dollar amount of the order. Extract from the text if present, otherwise null."
}
```

Descriptions matter. The model uses them to understand what to extract/generate for each field.

### 2.5 Output Formats Beyond JSON

JSON isn't always the right choice:

| Format | When to Use | Pros | Cons |
|---|---|---|---|
| **JSON** | API consumption, structured data | Universal, typed | Verbose, no comments |
| **YAML** | Config files, human-readable structured data | More readable | Less universal |
| **XML** | Legacy systems, document-centric | Hierarchical, namespaces | Verbose |
| **Markdown** | User-facing content | Human readable, rich formatting | Hard to parse programmatically |
| **CSV** | Tabular data export | Simple, spreadsheet-friendly | No nested structures |
| **TOML** | Config files | Very readable | Limited ecosystem |
| **Custom** | Domain-specific parsers | Maximum control | Requires custom parsing |

For machine-readable pipeline data: **JSON with schema enforcement**.
For user-facing content: **Markdown**.
For tabular exports: **CSV**.

## 3. Real-World Context

### Production Usage Patterns

**Pattern 1: Extraction pipeline**
```
Document → LLM with schema → Structured JSON → Database insert
                                              → Downstream processing
```
Used by: legal document processing, financial data extraction, medical records structuring.

**Pattern 2: Classification service**
```
User message → LLM with enum schema → {"intent": "refund", "urgency": "high"}
                                    → Route to correct support queue
```

**Pattern 3: Agent action dispatch**
```
User request → LLM with tools → tool_call: search_inventory({"sku": "ABC123"})
                               → Your code executes → returns stock level
                               → LLM generates response with real data
```

### 99%+ Schema Adherence

Production teams using native structured output APIs (OpenAI's json_schema, Anthropic's tool use
with forced tool choice) report 99%+ schema adherence. Compared to prompt-engineering-only approaches
(~80–90%), this is a meaningful reliability improvement for high-volume pipelines.

## 4. Code Examples

### Full Production Extraction Pipeline

```python
import anthropic
import json
from typing import Optional
from pydantic import BaseModel, Field

client = anthropic.Anthropic()

class InvoiceData(BaseModel):
    invoice_number:  str
    vendor_name:     str
    invoice_date:    str                # ISO 8601 format
    due_date:        Optional[str]
    total_amount:    float
    currency:        str = Field(default="USD")
    line_items:      list[dict]
    payment_terms:   Optional[str]
    extraction_confidence: float = Field(ge=0, le=1)

def extract_invoice(text: str) -> InvoiceData:
    """Extract structured invoice data from raw text."""
    
    schema = InvoiceData.model_json_schema()
    
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=2048,
        tools=[{
            "name": "extract_invoice",
            "description": "Extract all invoice data from the provided text into a structured format.",
            "input_schema": schema
        }],
        tool_choice={"type": "tool", "name": "extract_invoice"},
        messages=[{
            "role": "user",
            "content": f"Extract all invoice information from this document:\n\n{text}"
        }]
    )
    
    for block in response.content:
        if block.type == "tool_use":
            return InvoiceData(**block.input)
    
    raise ValueError("No tool use block in response")

# Test
sample_invoice = """
INVOICE #INV-2024-0847
Acme Corp → CloudServices Ltd
Date: March 15, 2024  |  Due: April 14, 2024

Services:
  Cloud hosting (Jan–Mar 2024) ........ $2,400.00
  Setup fee ........................... $150.00
  Support package ..................... $350.00

TOTAL: $2,900.00 USD
Terms: Net 30
"""

invoice = extract_invoice(sample_invoice)
print(f"Vendor:  {invoice.vendor_name}")
print(f"Total:   ${invoice.total_amount}")
print(f"Due:     {invoice.due_date}")
print(f"Items:   {len(invoice.line_items)}")
```

### Multi-Tool Agent with Error Handling

```python
import anthropic
import json

client = anthropic.Anthropic()

tools = [
    {
        "name": "calculator",
        "description": "Perform arithmetic calculations. Use for any math.",
        "input_schema": {
            "type": "object",
            "properties": {
                "expression": {
                    "type": "string",
                    "description": "Mathematical expression to evaluate, e.g. '(150 * 0.08) + 150'"
                }
            },
            "required": ["expression"]
        }
    },
    {
        "name": "lookup_exchange_rate",
        "description": "Look up current currency exchange rates.",
        "input_schema": {
            "type": "object",
            "properties": {
                "from_currency": {"type": "string", "description": "Source currency code (e.g. USD)"},
                "to_currency":   {"type": "string", "description": "Target currency code (e.g. EUR)"}
            },
            "required": ["from_currency", "to_currency"]
        }
    }
]

def execute_tool(name: str, inputs: dict) -> str:
    if name == "calculator":
        try:
            # Safe eval for math expressions (not arbitrary code)
            allowed = set("0123456789+-*/()., ")
            if all(c in allowed for c in inputs["expression"]):
                result = eval(inputs["expression"])
                return f"{result:.2f}"
            return "Error: unsafe expression"
        except Exception as e:
            return f"Calculation error: {e}"
    
    if name == "lookup_exchange_rate":
        # In production: call a real rates API
        mock_rates = {"USD→EUR": 0.92, "USD→GBP": 0.79, "EUR→USD": 1.09}
        key = f"{inputs['from_currency']}→{inputs['to_currency']}"
        rate = mock_rates.get(key, 0.0)
        return f"1 {inputs['from_currency']} = {rate} {inputs['to_currency']}"
    
    return "Unknown tool"

def run_with_tools(question: str) -> str:
    messages = [{"role": "user", "content": question}]
    max_iterations = 10   # safety: prevent infinite loops
    
    for _ in range(max_iterations):
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            tools=tools,
            messages=messages
        )
        
        if response.stop_reason == "end_turn":
            # Extract final text response
            for block in response.content:
                if hasattr(block, "text"):
                    return block.text
        
        if response.stop_reason == "tool_use":
            messages.append({"role": "assistant", "content": response.content})
            
            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    result = execute_tool(block.name, block.input)
                    print(f"  Tool: {block.name}({block.input}) → {result}")
                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": result
                    })
            
            messages.append({"role": "user", "content": tool_results})
    
    return "Max iterations reached"

print(run_with_tools(
    "I have $850 USD. If I convert to EUR and buy something for €650, "
    "how much EUR would I have left? Round to 2 decimal places."
))
```

### Streaming Structured Output

```python
import anthropic
import json

client = anthropic.Anthropic()

def stream_structured_extraction(text: str) -> dict:
    """
    Stream the response while accumulating the tool use block.
    Useful for long extractions where you want to show progress.
    """
    schema = {
        "type": "object",
        "properties": {
            "key_findings":   {"type": "array", "items": {"type": "string"}},
            "sentiment":      {"type": "string", "enum": ["positive", "negative", "neutral", "mixed"]},
            "action_items":   {"type": "array", "items": {"type": "string"}},
            "summary":        {"type": "string"}
        },
        "required": ["key_findings", "sentiment", "summary"]
    }
    
    accumulated_input = ""
    tool_use_id = None
    
    with client.messages.stream(
        model="claude-sonnet-4-6",
        max_tokens=2048,
        tools=[{"name": "analyze", "description": "Analyze the provided text.", "input_schema": schema}],
        tool_choice={"type": "tool", "name": "analyze"},
        messages=[{"role": "user", "content": f"Analyze:\n{text}"}]
    ) as stream:
        for event in stream:
            # Stream tool input as it arrives
            if hasattr(event, "delta") and hasattr(event.delta, "partial_json"):
                accumulated_input += event.delta.partial_json
                print(".", end="", flush=True)  # progress indicator
    
    print()  # newline after progress dots
    
    if accumulated_input:
        return json.loads(accumulated_input)
    return {}
```

## 5. Providers, Tools & Ecosystem

### Structured Output Support by Provider (2026)

| Provider | JSON Mode | Schema Enforcement | Tool Use | Notes |
|---|---|---|---|---|
| **OpenAI** | Yes | Yes (`json_schema`) | Yes | Best native structured output |
| **Anthropic** | Prefill hack | Via tool use | Yes | Tool use is the recommended approach |
| **Google Gemini** | Yes | Yes (`response_mime_type`) | Yes | Schema via response_schema |
| **Mistral** | Yes | Yes (`response_format`) | Yes | OpenAI-compatible API |
| **Ollama** | Yes (`format: json`) | Partial | Yes | Depends on model |
| **Groq** | Yes | Yes | Yes | OpenAI-compatible |

### Schema Generation Libraries

| Library | Language | Notes |
|---|---|---|
| **Pydantic** | Python | Industry standard; `.model_json_schema()` |
| **Instructor** | Python | Wraps OpenAI + Anthropic; Pydantic validation |
| **Zod** | TypeScript | TypeScript schema validation + generation |
| **jsonschema** | Python | Pure JSON Schema validation |

**Instructor** is worth knowing — it wraps the LLM client and handles retries automatically when
schema validation fails:

```python
import instructor
import anthropic
from pydantic import BaseModel

# Patch the Anthropic client with instructor
client = instructor.from_anthropic(anthropic.Anthropic())

class UserProfile(BaseModel):
    name: str
    age: int
    interests: list[str]

# instructor handles retries if validation fails
user = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Extract: John is 25, loves hiking and reading."}],
    response_model=UserProfile,
)

print(user.name)       # "John"
print(user.age)        # 25
print(user.interests)  # ["hiking", "reading"]
```

## 6. Alternatives & Comparisons

### Approach Reliability Comparison

| Method | Reliability | Schema Guarantee | Notes |
|---|---|---|---|
| Prompt only ("output JSON") | ~80–90% | No | Breaks on complex schemas |
| JSON mode | ~98%+ | Valid JSON only | No field guarantee |
| JSON schema (OpenAI) | ~99.9%+ | Yes | Best choice for OpenAI |
| Tool use forced (Anthropic) | ~99.9%+ | Yes | Best choice for Anthropic |
| Instructor with retries | ~99.9%+ | Yes | Provider-agnostic |
| Outlines / LM-format | ~100% | Yes | Self-hosted only |

### JSON vs YAML vs CSV for LLM Output

JSON is usually the right choice for structured output because:
- Universal parsing support in all languages
- Native to JavaScript/TypeScript frontends
- LLMs have seen vastly more JSON in training than YAML or TOML
- Most LLM APIs' structured output features are JSON-native

Exception: If your pipeline consumes YAML config files, having the LLM output YAML directly
avoids a conversion step. But enforce structure via prompt rather than schema, since schema
enforcement is JSON-only in most providers.

## 7. Trade-offs

### Strict Schema vs Flexible Extraction

**Strict schema:**
- Guaranteed structure; your code is simple
- Model can't add useful information outside the schema
- Harder to iterate (schema changes require code changes)

**Flexible extraction (prompt-only):**
- Model can add useful context beyond the schema
- Easier to iterate
- Harder to parse; more application-layer validation needed

**Rule:** Use strict schema for machine-to-machine pipelines. Use flexible extraction when a human
reviews the output before it's consumed.

### Schema Complexity vs Model Performance

Very complex schemas (deeply nested, many constraints) increase the chance of schema constraint
conflicts — where valid JSON can't be produced given all constraints simultaneously. Keep schemas
as simple as possible. Decompose complex extractions into multiple simpler calls if needed.

---

## 8. When To Use vs. When To Avoid

### Structured Output (JSON Schema / Tool Use)
✅ Use when: Output feeds into code (database, API, further processing)
✅ Use when: Downstream system has strict type requirements
✅ Use when: High volume production pipelines need reliability
❌ Avoid when: Output is user-facing prose — schema enforcement makes it feel mechanical

### Tool Calls
✅ Use when: The model needs to fetch data, run code, or take actions
✅ Use when: You want structured arguments for a function your code will execute
❌ Avoid when: You just need structured output — tool use adds round-trip complexity

⚠️ Common mistakes:
- Using JSON mode without specifying the schema in the prompt (OpenAI will error)
- Not validating schema output even with native structured output (schema enforcement is ~99.9%, not 100%)
- Building deeply nested schemas on first iteration (start flat, add nesting when needed)
- Forgetting to handle the tool result → model response round-trip in agentic patterns
- Not using `tool_choice` to force tool use (model may choose to respond in prose instead)

## 9. Senior AI Engineer Perspective

### The "Output Contract" Mental Model

Every structured output endpoint is a contract:
- **Input:** (prompt + schema)
- **Output:** data conforming to schema

Write the schema before writing the prompt. The schema IS the specification of what the model
must extract. If you can't write a clean schema, you don't have a clear enough idea of what
you need — solve that problem first.

### Validation Is Not Optional

Even with 99.9% schema adherence, at 1M requests/day you'll see ~1,000 malformed outputs daily.
Always validate:

```python
from pydantic import BaseModel, ValidationError

try:
    result = MySchema(**raw_dict)
except ValidationError as e:
    # Log the error, flag for review, return a safe default
    logger.error(f"Schema validation failed: {e}")
    return fallback_response
```

Never assume the model followed the schema. Trust but verify.

### Tool Design Is API Design

When designing tools for an LLM agent, apply the same principles as API design:
- Clear, unambiguous names (`search_documents` not `search`)
- Precise descriptions that distinguish from other tools
- Input schemas with descriptions for every field
- Narrow scope — one tool does one thing

A well-designed tool set is one where, given a task description, there is one obviously correct
tool to use. Ambiguous tool sets produce unpredictable routing.
