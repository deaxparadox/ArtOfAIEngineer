## 1. What Is It? (Theory)

### The Most Expensive Failure Mode: Silent Degradation

LLM failures aren't like server crashes. There's no stack trace. There's no 500 error.
The API returns 200 OK. The output is well-formed text. It just isn't correct, useful, or
consistent with what it was yesterday.

The most dangerous failure mode in production LLM engineering isn't outright breakage — it's
**silent degradation**: quality slowly declining, users noticing before you do, no monitoring
alert firing because the system technically "works."

In 2025–2026, practitioners report:
- Most production LLM regressions are caused by **prompt changes** or **model version updates**
- Failures are often **intermittent** — the same prompt works 85% of the time and fails 15%
- **Debugging is hard** because LLMs are opaque: you can observe inputs/outputs but not the
  reasoning path that produced a bad output

This section gives you a systematic taxonomy and diagnostic protocol.

## 2. The Failure Taxonomy

### Category 1: Instruction-Following Failures

The model ignores or misinterprets your instructions.

```
Instruction: "Respond in JSON only. No prose."
Output:      "Sure! Here's the JSON you requested: {"result": "value"}"
              ↑ preamble prose despite instruction
```

**Subtypes:**
- **Instruction omission:** Model ignores a specific rule
- **Instruction override:** Model's defaults override your instructions
- **Instruction conflict:** Two instructions contradict; model picks arbitrarily
- **Instruction drift:** Model follows instructions early in the conversation but "forgets" them later

**Diagnose with:**
```python
# Test: does the instruction appear in the right position?
# Is it specific enough? (vague instructions = vague compliance)
# Does any other instruction contradict it?
# Is it too buried in a long system prompt?
```

### Category 2: Format Failures

Output doesn't match the required format.

```
Required: {"intent": "refund", "urgency": "high"}
Got:      "Based on the message, the intent appears to be a refund request with high urgency."
```

Or:
```
Required: {"urgency": "high" | "medium" | "low"}
Got:      {"urgency": "HIGH"}   ← wrong case
Got:      {"urgency": "urgent"} ← wrong value entirely
```

**Subtypes:**
- **Schema mismatch:** Wrong field names, types, or values
- **Extra fields:** Model adds fields not in the schema
- **Missing required fields:** Model omits a field it should always produce
- **Inconsistent casing:** Field values use inconsistent capitalization

**Fix:** Use structured output APIs (Section 4.3) instead of relying on prompt-only format control.

### Category 3: Hallucination Failures

Model generates plausible but incorrect content. Covered thoroughly in Section 3.5.

```
Input: [document with no mention of revenue]
Output: "The company reported revenue of $2.3M" ← invented
```

### Category 4: Refusal Failures

Model refuses a legitimate request.

```
User: "What's the chemical formula for table salt?"
Model: "I can't provide information about chemicals as it could be misused."
       ← over-refusal on a completely safe question
```

**Causes:**
- Overly cautious alignment training
- Topic-based refusal matching keywords rather than intent
- System prompt that's too broad in its safety scope

**Subtypes:**
- **False positive refusal:** Refuses safe content due to keyword matching
- **Capability refusal:** Claims it can't do something it can (alignment-training artifact)
- **Context blindness:** Refuses without reading the full context

### Category 5: Consistency Failures

Same prompt, different outputs across runs — variance beyond acceptable tolerance.

```
Run 1: {"sentiment": "positive", "score": 0.85}
Run 2: {"sentiment": "positive", "score": 0.62}
Run 3: {"sentiment": "negative", "score": 0.71}  ← flipped!
```

**Causes:**
- Temperature too high (increase randomness)
- Prompt is ambiguous (model interprets differently each time)
- Edge cases in training distribution

**Diagnose with:**
```python
# Run the same prompt 10 times at temperature=0
# If variance is high → your prompt is ambiguous
# If variance is low → the problem is edge-case inputs
```

### Category 6: Reasoning Failures

Model makes logical errors, arithmetic mistakes, or multi-step reasoning failures.

```
Q: "If I buy 3 items at $12.50 each with a 15% discount, what do I pay?"
A: "$31.87"   ← wrong (correct: $31.875 → $31.88, but the computation path was wrong)
```

**Causes:**
- Tokenization makes arithmetic hard (see Section 3.3)
- Multi-step problems exceed "working memory" without CoT
- Model confuses the ordering of operations

**Fix:** Add CoT ("think step by step"), or delegate arithmetic to a tool/code interpreter.

### Category 7: Context Failures

Model fails to use (or misuses) information provided in the context.

```
Context: "User's subscription: Pro plan, $99/month, renews Feb 1 2025"
User: "What plan am I on?"
Model: "I don't have information about your subscription." ← ignores context
```

**Or:**
```
Context: 10,000 words of product documentation
User: "What are the API rate limits?"
Model: [gives rate limits from training data, not from the documentation] ← prefers prior
```

**Causes:**
- "Lost in the middle" effect for long contexts
- Model's training-time priors override provided context
- Information structure makes it hard to locate the relevant passage

## 3. How to Diagnose Systematically

### The LLM Debugging Protocol

```
Step 1: Reproduce reliably
  - Collect the exact input that caused the failure
  - Can you reproduce it deterministically at temperature=0?
  - If not: is it input-dependent (edge case) or random (prompt ambiguity)?

Step 2: Isolate the failure category
  - Is output format wrong? → Category 2
  - Is it factually wrong? → Category 3 (hallucination)
  - Is it refusing? → Category 4
  - Is it inconsistent? → Category 5
  - Is it wrong reasoning? → Category 6
  - Is it ignoring context? → Category 7

Step 3: Find the minimum reproducing case
  - Simplify the prompt to the smallest version that still fails
  - Remove few-shot examples one by one — does removing one fix it?
  - Remove sentences from the system prompt — does removing one fix it?

Step 4: Hypothesize the cause
  - Is the instruction absent?
  - Is it contradicted by another instruction?
  - Is it too ambiguous?
  - Is it buried in a long prompt?

Step 5: Fix and validate with evals
  - Make the smallest change that fixes the failure
  - Run your full eval suite to verify no regressions
  - Never fix by adding complexity — prefer clarifying existing instructions
```

### The Five Diagnostic Questions

When a prompt fails, ask these in order:

1. **Is the instruction present?** (Most common fix: just add it explicitly)
2. **Is the instruction specific enough?** ("Be concise" → "Max 3 sentences")
3. **Is there a conflicting instruction?** (Check system prompt top-to-bottom)
4. **Is the instruction in the right position?** (Critical rules near the beginning AND end)
5. **Does a worked example exist?** (Few-shot often fixes ambiguous instructions)

### The "Bisect" Method for Prompt Regression

When a prompt worked last week but doesn't work today:

```
Possible causes:
  A. Model was updated (provider pushed a new version silently)
  B. System prompt was changed
  C. User input distribution shifted
  D. Retrieval changed (for RAG-based systems)

Bisect:
  - Pin model version → does the failure go away? → cause is A
  - Revert system prompt → does failure go away? → cause is B
  - Test on old representative inputs → does failure go away? → cause is C
  - Check retrieval quality → is the right context being retrieved? → cause is D
```

## 4. Real-World Context

### The Most Common Production Failures in 2025–2026

From practitioner reports and engineering blogs:

| Failure | Frequency | Root Cause | Fix |
|---|---|---|---|
| Format inconsistency | Very common | Prompt-only format control | Use structured output API |
| Silent context ignore | Common | Long context + lost in middle | Shorten context; move info to start |
| Over-refusal | Common | Broad safety keywords | Narrow scope in system prompt; add clarifying context |
| Reasoning errors on numbers | Common | Tokenization + no CoT | Add "think step by step"; use calculator tool |
| Prompt regression after update | Common | Model version change | Pin model version; run evals on update |
| Sycophantic drift | Common | RLHF preference alignment | Test with adversarial pushback; add "maintain position when correct" instruction |

### The Silent Degradation Story

A real pattern: a company deploys a support chatbot. It performs well in testing. They don't build
an eval suite — just spot-check outputs. Six months later, a model provider silently updates their
model. The chatbot's responses become subtly less accurate and more verbose. Users start
complaining. The engineering team can't tell if it was always like this or got worse recently.
They have no baseline to compare against.

This is why evals are the non-negotiable discipline of production AI engineering.

## 5. Code Examples

### Failure Detection Eval Suite

```python
import anthropic
import json
from dataclasses import dataclass

client = anthropic.Anthropic()

@dataclass
class EvalCase:
    name: str
    system: str
    user: str
    checks: list  # list of callables(response: str) -> bool

def run_evals(cases: list[EvalCase], model: str = "claude-sonnet-4-6") -> dict:
    """Run all eval cases and report pass/fail for each check."""
    results = []
    
    for case in cases:
        response = client.messages.create(
            model=model,
            max_tokens=512,
            system=case.system,
            messages=[{"role": "user", "content": case.user}],
            temperature=0   # deterministic for evals
        ).content[0].text
        
        check_results = []
        for check_fn in case.checks:
            passed = check_fn(response)
            check_results.append({"check": check_fn.__name__, "passed": passed})
        
        all_passed = all(r["passed"] for r in check_results)
        results.append({
            "name": case.name,
            "passed": all_passed,
            "checks": check_results,
            "response_preview": response[:100]
        })
    
    total = len(results)
    passed = sum(1 for r in results if r["passed"])
    print(f"\nResults: {passed}/{total} passed ({100*passed/total:.0f}%)")
    for r in results:
        status = "✓" if r["passed"] else "✗"
        print(f"  {status} {r['name']}")
        if not r["passed"]:
            for c in r["checks"]:
                if not c["passed"]:
                    print(f"      FAILED: {c['check']}")
            print(f"      Response: {r['response_preview']}")
    
    return {"total": total, "passed": passed, "results": results}


# Define eval cases covering different failure categories

SYSTEM = "You are a customer support agent. Always respond in JSON with 'answer' and 'action_needed' (boolean) fields only."

def is_valid_json(r): 
    try: json.loads(r); return True
    except: return False
is_valid_json.__name__ = "output_is_valid_json"

def has_answer_field(r):
    try: return "answer" in json.loads(r)
    except: return False
has_answer_field.__name__ = "has_answer_field"

def has_action_field(r):
    try: return "action_needed" in json.loads(r)
    except: return False
has_action_field.__name__ = "has_action_needed_field"

def action_is_bool(r):
    try: d = json.loads(r); return isinstance(d.get("action_needed"), bool)
    except: return False
action_is_bool.__name__ = "action_needed_is_boolean"

def no_extra_fields(r):
    try: d = json.loads(r); return set(d.keys()) <= {"answer", "action_needed"}
    except: return False
no_extra_fields.__name__ = "no_extra_fields"

def refuses_off_topic(r):
    # For off-topic request, model should indicate it can't help, not answer the question
    return "action_needed" in r.lower() or "out of scope" in r.lower() or "can't" in r.lower()
refuses_off_topic.__name__ = "handles_off_topic_gracefully"

format_checks = [is_valid_json, has_answer_field, has_action_field, action_is_bool, no_extra_fields]

cases = [
    EvalCase("normal_query", SYSTEM, "My order hasn't arrived.", format_checks),
    EvalCase("format_check_billing", SYSTEM, "I need a refund please.", format_checks),
    EvalCase("edge_case_empty", SYSTEM, ".", format_checks),
    EvalCase("long_message", SYSTEM, "I placed an order three weeks ago " * 20 + " Where is it?", format_checks),
]

run_evals(cases)
```

### Consistency Testing for Prompt Ambiguity

```python
import anthropic
from collections import Counter
import json

client = anthropic.Anthropic()

def test_consistency(
    system: str,
    user: str,
    extract_key: str,   # JSON key to compare across runs
    n_runs: int = 10
) -> dict:
    """
    Run the same prompt N times and measure consistency.
    High variance = prompt is ambiguous or the task is inherently hard.
    """
    answers = []
    
    for i in range(n_runs):
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=256,
            system=system,
            messages=[{"role": "user", "content": user}],
            temperature=0.3  # low but not zero — tests realistic variance
        ).content[0].text
        
        try:
            data = json.loads(response)
            answers.append(str(data.get(extract_key, "parse_error")))
        except:
            answers.append("parse_error")
    
    counts = Counter(answers)
    most_common, count = counts.most_common(1)[0]
    consistency = count / n_runs
    
    return {
        "consistency_rate": f"{consistency:.0%}",
        "most_common_answer": most_common,
        "answer_distribution": dict(counts),
        "is_consistent": consistency >= 0.8,  # 80% threshold
    }

# Test a classification prompt for consistency
result = test_consistency(
    system='Classify message sentiment. Return JSON: {"sentiment": "positive"|"negative"|"neutral"}',
    user="The product works okay I guess. Not thrilling but not terrible.",
    extract_key="sentiment",
    n_runs=10
)
print("Consistency test:")
for k, v in result.items():
    print(f"  {k}: {v}")

# If consistency < 80%: the prompt is too ambiguous for this edge case
# Fix: add a definition of "neutral" to the system prompt, or add few-shot examples
```

### Regression Testing After Model/Prompt Changes

```python
import anthropic
import json
from datetime import datetime

client = anthropic.Anthropic()

def snapshot_eval(
    cases: list[dict],   # [{"input": str, "expected_output": dict}]
    system: str,
    model: str,
    snapshot_name: str
) -> dict:
    """
    Run an eval suite and save results as a snapshot.
    Compare against previous snapshots to detect regressions.
    """
    results = []
    
    for case in cases:
        response = client.messages.create(
            model=model,
            max_tokens=512,
            system=system,
            messages=[{"role": "user", "content": case["input"]}],
            temperature=0
        ).content[0].text
        
        try:
            actual = json.loads(response)
        except:
            actual = {"parse_error": response[:100]}
        
        # Check each expected field
        field_results = {}
        for field, expected_val in case["expected_output"].items():
            actual_val = actual.get(field)
            field_results[field] = {
                "expected": expected_val,
                "actual": actual_val,
                "passed": actual_val == expected_val
            }
        
        case_passed = all(r["passed"] for r in field_results.values())
        results.append({
            "input": case["input"][:80],
            "passed": case_passed,
            "fields": field_results
        })
    
    score = sum(r["passed"] for r in results) / len(results)
    
    snapshot = {
        "name": snapshot_name,
        "model": model,
        "timestamp": datetime.now().isoformat(),
        "score": score,
        "results": results
    }
    
    # In production: save to a database or file for comparison
    print(f"\n=== Eval Snapshot: {snapshot_name} ===")
    print(f"Score: {score:.0%} ({sum(r['passed'] for r in results)}/{len(results)})")
    print(f"Model: {model}")
    
    return snapshot

# Example: run before and after a prompt change
test_cases = [
    {"input": "My order hasn't arrived after 3 weeks.",   "expected_output": {"intent": "shipping", "urgency": "high"}},
    {"input": "Can I get a refund?",                      "expected_output": {"intent": "refund", "urgency": "medium"}},
    {"input": "Do you have this in size L?",              "expected_output": {"intent": "product_inquiry", "urgency": "low"}},
]

system_v1 = "Classify customer intent. Return JSON: {intent, urgency}"
system_v2 = 'Classify customer intent. Return JSON: {"intent": "shipping"|"refund"|"product_inquiry"|"other", "urgency": "low"|"medium"|"high"}'

snap1 = snapshot_eval(test_cases, system_v1, "claude-sonnet-4-6", "v1_vague_system")
snap2 = snapshot_eval(test_cases, system_v2, "claude-sonnet-4-6", "v2_specific_enums")
# v2 should score higher due to enum constraints
```

## 7. Trade-offs

### Deterministic Testing (temperature=0) vs Realistic Testing

**temperature=0 testing:**
- Reproducible; easy to compare results
- Misses failures that only appear at normal sampling temperatures
- Good for regression tests

**temperature=0.3–0.7 testing:**
- Closer to production behavior
- Non-deterministic; harder to compare
- Use multiple runs + consistency metrics (see code above)

**Best practice:** Run eval suite at temperature=0 for regression detection. Periodically run
consistency tests at production temperature to catch ambiguity failures.

### Coverage vs Depth

Building a comprehensive eval suite is an investment. Start with:
1. **10 representative examples** of the most common use case (core happy path)
2. **5 edge cases** you've already seen fail in testing or production
3. **5 adversarial cases** designed to trigger known failure modes

Expand as you discover new failure patterns. An eval suite of 30–50 targeted cases is more
valuable than 200 random examples.

## 8. When To Use vs. When To Avoid

### Building an Eval Suite
✅ Always build: Before shipping any LLM feature to users
✅ Always build: Before upgrading a model version
✅ Always build: Before making significant prompt changes
❌ Skip only for: Throwaway prototypes you won't maintain

### Temperature=0 for Testing
✅ Use for: Regression testing, snapshot comparison
✅ Use for: Establishing baseline quality
❌ Use for: Testing creative tasks where temperature is intentional (use 0.7+)

⚠️ Common mistakes:
- Debugging by adding more instructions to an already-complex prompt — often this creates new conflicts
- Not testing edge cases (empty inputs, very long inputs, non-English inputs, adversarial inputs)
- Treating "it works on my test cases" as "it works" — your test cases aren't the full distribution
- Not monitoring production model behavior over time (silently degrading quality is invisible without monitoring)
- Forgetting to pin the model version — provider updates cause unexplained regressions

## 9. Senior AI Engineer Perspective

### The Most Valuable Debugging Habit

When you encounter a prompt failure, resist the urge to immediately add more text to the prompt.

Before touching the prompt, ask: "What specifically did I ask for, and what specifically did the
model do instead?" Write both down as precise observations. Most of the time, the gap reveals either:
- You didn't actually specify the constraint you assumed was implicit
- You specified it, but another instruction contradicts it
- The model has a different default behavior than you expected for this input type

The fix is almost always clarification, not addition.

### Prompts Are Code — Treat Them as Such

In 2026, the industry consensus: prompts need the same engineering discipline as code.

| Software engineering | Prompt engineering |
|---|---|
| Version control (git) | Version prompts in a prompt management system |
| Unit tests | Eval cases per prompt |
| Code review | Review prompt changes with the team |
| Staging environment | Test on staging data before prod |
| Deployment monitoring | Track quality metrics post-deploy |
| Rollback plan | Keep previous prompt version |

Teams that treat prompts as "just text" and edit them directly in production are the ones that
show up in incident retrospectives saying "we don't know when this broke."

### The Compound Failure Problem

Many production failures aren't a single failure mode — they're two or three interacting:
- Format failure (missing field) causes the downstream code to crash
- The crash hides the hallucination that would have been caught by validation
- The lack of logging means you can't reproduce the exact input that caused it

Defense in depth: structured output + validation + logging + eval suite. Each layer catches what
the others miss.
