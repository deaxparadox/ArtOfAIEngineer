## 1. What Is It? (Theory)

### Why Multiple Agents?

A single agent handles a task from start to finish. Multiple agents cooperate when:

1. **Context overload:** A single agent handling a complex task accumulates too much context
   and loses track. Specialized subagents with narrow scope maintain focus.

2. **Specialization:** A coding agent, a research agent, and a writing agent each have
   optimized system prompts and tool sets. A generalist does all three worse.

3. **Parallelism:** Independent tasks run faster when separate agents handle them concurrently.

4. **Scale:** A single powerful model is expensive for every step. Route simple tasks to
   cheap/fast agents; hard tasks to capable ones.

5. **Trust boundaries:** Some agents should have access to sensitive systems; others shouldn't.
   Multi-agent lets you enforce least-privilege per agent.

### The Three Fundamental Patterns

```
1. ORCHESTRATOR → WORKERS
   One planner coordinates multiple specialist workers.
   Workers report back; orchestrator synthesizes.

2. PIPELINE (SEQUENTIAL)
   Agent A → Agent B → Agent C
   Each agent's output is the next agent's input.
   Like an assembly line.

3. HANDOFF (TRANSFER OF CONTROL)
   Agent A decides: "This needs Agent B"
   Control transfers to Agent B completely.
   Agent B handles the rest.
```

## 2. How It Actually Works — Deep Mechanics

### 2.1 Orchestrator-Worker Pattern

The orchestrator is a strategic planner that delegates tasks to specialized workers.
Workers are themselves agents (with their own tools and system prompts).

```python
import anthropic
import asyncio
from dataclasses import dataclass

client = anthropic.Anthropic()

# ── WORKER AGENTS ────────────────────────────────────────────────────────────

def research_agent(query: str) -> str:
    """Specialized agent for information gathering."""
    response = client.messages.create(
        model="claude-haiku-4-5-20251001",  # cheap: simple retrieval task
        max_tokens=1024,
        system="""You are a research specialist. Your ONLY job is to gather factual information.
Output concise, factual findings. No opinions. Cite sources where possible.""",
        messages=[{"role": "user", "content": f"Research: {query}"}]
    )
    return response.content[0].text

def analysis_agent(data: str, question: str) -> str:
    """Specialized agent for analysis and reasoning."""
    response = client.messages.create(
        model="claude-sonnet-4-6",   # capable: complex reasoning
        max_tokens=1024,
        system="""You are an analytical specialist. Your job is to analyze data and draw insights.
Be rigorous. State assumptions. Quantify uncertainty.""",
        messages=[{"role": "user", "content": f"Data:\n{data}\n\nAnalysis question: {question}"}]
    )
    return response.content[0].text

def writing_agent(content: str, format_spec: str) -> str:
    """Specialized agent for content creation."""
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=2048,
        system="""You are a professional writer. Transform raw content into polished,
well-structured documents. Match the specified format exactly.""",
        messages=[{
            "role": "user",
            "content": f"Content to format:\n{content}\n\nFormat specification: {format_spec}"
        }]
    )
    return response.content[0].text

# ── ORCHESTRATOR ─────────────────────────────────────────────────────────────

# Worker agents exposed as tools to the orchestrator
WORKER_TOOLS = [
    {
        "name": "call_research_agent",
        "description": "Delegate information gathering to the research specialist. "
                       "Use when you need to find facts, verify information, or gather data.",
        "input_schema": {
            "type": "object",
            "properties": {
                "query": {"type": "string", "description": "What to research. Be specific."}
            },
            "required": ["query"]
        }
    },
    {
        "name": "call_analysis_agent",
        "description": "Delegate data analysis to the analysis specialist. "
                       "Use when you have data and need insight, patterns, or conclusions drawn.",
        "input_schema": {
            "type": "object",
            "properties": {
                "data":     {"type": "string", "description": "The data or content to analyze"},
                "question": {"type": "string", "description": "The analytical question to answer"}
            },
            "required": ["data", "question"]
        }
    },
    {
        "name": "call_writing_agent",
        "description": "Delegate document creation to the writing specialist. "
                       "Use when you have content that needs to be written up in a specific format.",
        "input_schema": {
            "type": "object",
            "properties": {
                "content":     {"type": "string", "description": "Raw content to be written up"},
                "format_spec": {"type": "string", "description": "How to format the output (e.g., 'executive summary, bullet points, 300 words')"}
            },
            "required": ["content", "format_spec"]
        }
    }
]

WORKER_EXECUTORS = {
    "call_research_agent":  lambda query: {"result": research_agent(query)},
    "call_analysis_agent":  lambda data, question: {"result": analysis_agent(data, question)},
    "call_writing_agent":   lambda content, format_spec: {"result": writing_agent(content, format_spec)},
}

def orchestrator_agent(task: str) -> str:
    """Master orchestrator: delegates to specialists, synthesizes results."""
    from phase_8_examples import react_agent   # the run_agent from Section 8.3
    
    result = react_agent(
        task=task,
        tools=WORKER_TOOLS,
        tool_executors=WORKER_EXECUTORS,
        max_steps=8,
    )
    return result["answer"]
```

### 2.2 Parallel Worker Execution

When worker tasks are independent, run them concurrently:

```python
import asyncio

async def parallel_orchestrator(task: str, sub_queries: list[str]) -> dict:
    """
    Run multiple research agents in parallel on independent sub-queries.
    Total time ≈ max(individual times), not sum.
    """
    
    async def run_research(query: str) -> str:
        # Wrap synchronous agent in async
        loop = asyncio.get_event_loop()
        return await loop.run_in_executor(None, research_agent, query)
    
    # Run all sub-queries concurrently
    print(f"Running {len(sub_queries)} research agents in parallel...")
    results = await asyncio.gather(*[run_research(q) for q in sub_queries])
    
    # Synthesize
    combined = "\n\n".join(
        f"Sub-query: {q}\nResult: {r}"
        for q, r in zip(sub_queries, results)
    )
    
    return {
        "sub_results": dict(zip(sub_queries, results)),
        "synthesis_input": combined
    }
```

### 2.3 Handoff Pattern

Handoffs transfer complete control from one agent to another. The source agent decides
to hand off and specifies what the target agent should do:

```python
@dataclass
class HandoffResult:
    target_agent: str
    task:         str
    context:      str

def triage_agent(user_message: str) -> HandoffResult | str:
    """
    Triage agent: routes to the right specialist.
    Returns either a handoff or a direct answer.
    """
    response = client.messages.create(
        model="claude-haiku-4-5-20251001",   # cheap: simple routing
        max_tokens=256,
        system="""You are a customer service triage agent.
Classify the request and return ONE of these routing decisions as JSON:
- {"route": "billing", "reason": "..."}
- {"route": "technical", "reason": "..."}
- {"route": "general", "reason": "..."}
- {"route": "self", "answer": "..."} for simple questions you can answer directly""",
        messages=[{"role": "user", "content": user_message}]
    )
    
    import json
    try:
        routing = json.loads(response.content[0].text)
    except json.JSONDecodeError:
        routing = {"route": "general", "reason": "routing failed"}
    
    if routing["route"] == "self":
        return routing.get("answer", "")
    
    return HandoffResult(
        target_agent = routing["route"],
        task         = user_message,
        context      = f"Routed from triage: {routing.get('reason', '')}"
    )

SPECIALIST_AGENTS = {
    "billing":   lambda task, ctx: billing_specialist_agent(task, ctx),
    "technical": lambda task, ctx: technical_specialist_agent(task, ctx),
    "general":   lambda task, ctx: general_agent(task, ctx),
}

def handle_request(user_message: str) -> str:
    """Complete handoff-based routing flow."""
    result = triage_agent(user_message)
    
    if isinstance(result, str):
        # Triage answered directly
        return result
    
    # Handoff to specialist
    specialist = SPECIALIST_AGENTS.get(result.target_agent)
    if specialist:
        print(f"Routing to {result.target_agent} specialist...")
        return specialist(result.task, result.context)
    
    return "No specialist available for this request."
```

### 2.4 Supervisor Pattern (Hierarchical)

A supervisor agent monitors worker agents and intervenes when they go wrong:

```python
def supervised_execution(task: str, max_retries: int = 2) -> str:
    """
    Worker attempts the task. Supervisor reviews and either accepts or requests revision.
    """
    for attempt in range(max_retries + 1):
        # Worker generates output
        worker_output = react_agent(task, tools=TOOLS, tool_executors=EXECUTORS)["answer"]
        
        # Supervisor evaluates
        supervisor_response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=256,
            system="You are a quality supervisor. Evaluate if the worker's output correctly "
                   "and completely addresses the task. Be strict.",
            messages=[{
                "role": "user",
                "content": f"Task: {task}\n\nWorker output: {worker_output}\n\n"
                           f"Does this fully and correctly address the task? "
                           f"Output: {{\"approved\": true/false, \"feedback\": \"...\"}}"
            }]
        )
        
        import json
        try:
            evaluation = json.loads(supervisor_response.content[0].text)
        except json.JSONDecodeError:
            evaluation = {"approved": True}
        
        if evaluation.get("approved"):
            return worker_output
        
        if attempt < max_retries:
            # Revise task with supervisor feedback
            task = f"{task}\n\nPrevious attempt feedback: {evaluation.get('feedback', '')}"
            print(f"Supervisor rejected. Retrying with feedback...")
    
    return worker_output  # Return last attempt after max retries
```

## 3. Multi-Agent Frameworks (2026)

```python
# LangGraph: the most production-ready multi-agent framework
# pip install langgraph

from langgraph.graph import StateGraph, END
from typing import TypedDict, Annotated
import operator

class AgentState(TypedDict):
    messages:    list[dict]
    task:        str
    current_agent: str
    result:      str

def research_node(state: AgentState) -> AgentState:
    """LangGraph node: runs research agent"""
    result = research_agent(state["task"])
    state["messages"].append({"agent": "research", "result": result})
    state["current_agent"] = "analysis"
    return state

def analysis_node(state: AgentState) -> AgentState:
    """LangGraph node: runs analysis agent"""
    research_result = next(m["result"] for m in state["messages"] if m["agent"] == "research")
    result = analysis_agent(research_result, state["task"])
    state["messages"].append({"agent": "analysis", "result": result})
    state["result"] = result
    return state

# Build graph
workflow = StateGraph(AgentState)
workflow.add_node("research",  research_node)
workflow.add_node("analysis",  analysis_node)
workflow.set_entry_point("research")
workflow.add_edge("research", "analysis")
workflow.add_edge("analysis", END)

app = workflow.compile()

# Run
result = app.invoke({
    "task": "Analyze the impact of interest rates on tech stocks",
    "messages": [],
    "current_agent": "research",
    "result": ""
})
```

## 5. Trade-offs

### Single Agent vs Multi-Agent

| Factor | Single Agent | Multi-Agent |
|---|---|---|
| Complexity | Low | High |
| Reliability | Higher | Lower (more failure points) |
| Cost | Predictable | Variable (orchestration overhead) |
| Specialization | Limited | Excellent |
| Parallelism | None | Possible |
| Debugging | Simpler | Complex (which agent failed?) |

**Guideline:** Start with a single agent. Only add multi-agent architecture when you've hit
a wall that single-agent can't solve: context overload, specialization needs, or parallelism.

From a 2026 survey of 16 practitioners: 7 were still at Level 1 (AI Assistants), 4 at
Level 2 (AI Compensators), and only 1 had reached Level 3 (Multi-Agent Orchestration).
Most teams don't need multi-agent yet.

## 6. When To Use vs. When To Avoid

### Multi-Agent
✅ Use when: Context accumulates so much a single agent loses focus
✅ Use when: Different sub-tasks need very different expertise (specialized system prompts)
✅ Use when: Independent sub-tasks can run in parallel
✅ Use when: You need cost optimization (cheap workers for simple steps)
❌ Avoid when: Single-agent handles the task well — added complexity has no benefit
❌ Avoid when: Reliability is critical — multi-agent has more failure points

⚠️ Common mistakes:
- Building multi-agent systems before validating the single-agent baseline
- Not propagating context properly across agent boundaries (specialist doesn't know what orchestrator found)
- No circuit breaker — a misbehaving subagent can consume resources indefinitely
- Not instrumenting which agent did what — debugging is impossible without traces per agent

## 7. Senior AI Engineer Perspective

### The Coordination Tax

Multi-agent introduces a "coordination tax": each agent boundary requires passing context,
handling errors from the other agent's outputs, and managing additional API calls.

For a task that takes 1 API call single-agent, it may take 4–8 calls multi-agent (orchestrator
planning + worker calls + orchestrator synthesis). The coordination overhead is real.

Only the specialization benefit and parallelism benefit can justify this overhead. If neither
applies, keep it single-agent.

### State Management Is the Hard Part

Every multi-agent system needs to answer:
- What state does each agent see?
- How does state pass between agents?
- What happens when an agent modifies shared state?
- How do you checkpoint and resume after a failure?

LangGraph's checkpointing is the most mature answer to these questions in 2026. If you're
building production multi-agent systems, start with LangGraph.
