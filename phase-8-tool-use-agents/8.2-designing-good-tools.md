## 1. What Is It? (Theory)

### Tool Design Is API Design for LLMs

When you design a REST API, you design it for developers. When you design a tool for an LLM,
you design it for a reasoning engine that makes decisions based on:
1. The tool's **name** — is this relevant to what I'm trying to do?
2. The tool's **description** — what does it do, and when should I use it?
3. The tool's **input schema** — what arguments can I provide?
4. The tool's **output format** — what will I get back that I can reason about?

The LLM cannot ask clarifying questions about the tool. It uses the schema and description
to make decisions. **Ambiguous or incomplete tool definitions produce wrong tool calls.**

### The Consequences of Bad Tool Design

```
Bad tool name:      "process" → Model has no idea what this does or when to call it
Bad description:    "Does the thing" → Model guesses wrong about its purpose
Bad input schema:   No field descriptions → Model fills in wrong argument names/types
Bad output format:  Raw bytes or unstructured text → Model can't reason about the result
Overlapping tools:  Two tools that sound similar → Model picks the wrong one
```

A bad tool design causes subtle failures that are hard to debug because the model is
technically calling the tool — just with wrong arguments or at wrong times.

## 2. How It Actually Works — Deep Mechanics

### 2.1 The Tool Name

The name is the first signal the model uses for routing. It should be:
- **Specific and verb-first:** `search_documents`, `create_calendar_event`, `send_email`
- **Unambiguous:** `get_user` vs `lookup_user_by_id` vs `find_user_by_email` are different tools
- **Snake_case by convention** — all major providers use snake_case for tool names

```
Bad names:               Good names:
  "process"          →     "process_refund"
  "data"             →     "query_customer_database"
  "helper"           →     "calculate_shipping_cost"
  "search"           →     "search_product_catalog"
  "do_thing"         →     "update_ticket_status"
```

### 2.2 The Tool Description

The description is the most important field. It tells the model:
- What the tool does
- When to use it (and critically, when NOT to use it)
- What kinds of inputs are appropriate
- What the model will get back

```python
# BAD: Too vague
{
    "name": "search",
    "description": "Search for information",
    ...
}
# Model doesn't know: search where? for what? returns what format?

# GOOD: Specific and complete
{
    "name": "search_product_catalog",
    "description": (
        "Search our product catalog by name, category, or description. "
        "Returns a list of matching products with name, SKU, price, and inventory count. "
        "Use for: finding products, checking availability, looking up SKUs. "
        "Do NOT use for: order status (use get_order_status), "
        "customer info (use lookup_customer), or pricing history."
    ),
    ...
}
```

**The "when to use" and "when NOT to use" pattern** is especially valuable when you have
multiple similar tools. It prevents the model from calling the wrong one.

### 2.3 Input Schema Design

Each parameter needs:
- A clear, descriptive name
- The correct type
- A description that explains what to put there (not just what the field is named)
- Whether it's required or optional (with a sensible default for optional fields)

```python
# BAD: No descriptions, no examples
{
    "type": "object",
    "properties": {
        "id":    {"type": "string"},
        "flag":  {"type": "boolean"},
        "n":     {"type": "integer"}
    },
    "required": ["id"]
}
# Model: What kind of ID? What does flag do? What is n?

# GOOD: Fully described
{
    "type": "object",
    "properties": {
        "order_id": {
            "type": "string",
            "description": "The order ID from the order confirmation, e.g. 'ORD-2024-1234567'"
        },
        "include_shipping_history": {
            "type": "boolean",
            "description": "Whether to include the full shipping history timeline. Default: false. "
                           "Set to true only if the user explicitly asks about shipping updates.",
            "default": False
        },
        "max_line_items": {
            "type": "integer",
            "description": "Maximum number of line items to return. Use 5 for summaries, 50 for full details.",
            "minimum": 1,
            "maximum": 50,
            "default": 5
        }
    },
    "required": ["order_id"]
}
```

### 2.4 Output Format Design

The tool's return value must be **LLM-readable** — the model will reason about it.
This means:

- **Structured JSON** with clear field names (not raw bytes, not arbitrary objects)
- **Human-readable values** where possible (`"In Progress"` not `2`)
- **Error messages** as part of the schema, not Python exceptions
- **Explicit null handling** — tell the model what missing data means

```python
# BAD: Opaque output
def get_order(order_id: str) -> dict:
    order = db.query(order_id)
    return order.to_dict()   # Raw DB object with internal field names

# GOOD: LLM-readable output
def get_order(order_id: str) -> dict:
    try:
        order = db.query(order_id)
        if not order:
            return {
                "found": False,
                "order_id": order_id,
                "message": "Order not found. Verify the order ID or check if it's in a different system."
            }
        
        return {
            "found": True,
            "order_id":       order.id,
            "status":         order.status_label,   # "In Progress", not 2
            "placed_at":      order.created_at.isoformat(),
            "estimated_delivery": order.eta.isoformat() if order.eta else "No estimate available",
            "total_usd":      order.total / 100,   # cents → dollars
            "items":          [{"name": i.name, "qty": i.quantity, "price_usd": i.price/100}
                               for i in order.items[:5]],
            "can_cancel":     order.status in ("pending", "processing"),
            "can_return":     order.delivered_at is not None and order.days_since_delivery <= 30,
        }
    except Exception as e:
        return {
            "found": False,
            "error": str(e),
            "message": "Failed to retrieve order. Please try again or contact support."
        }
```

### 2.5 The Tool Set: Avoiding Overlap and Gaps

When you have multiple tools, their combined design matters:

**No overlapping tools (model picks the wrong one):**
```python
# BAD: Two tools that sound similar
tools = [
    {"name": "search_users",    "description": "Find users in the database"},
    {"name": "lookup_customer", "description": "Look up customer information"},
]
# Model: "I need user info. Do I call search_users or lookup_customer?"

# GOOD: Clear separation of concern
tools = [
    {"name": "search_users", "description": "Search users by name or email. Returns a list."},
    {"name": "get_user_by_id", "description": "Get full profile for a specific user ID."},
]
# Different use cases, different inputs, no ambiguity
```

**No gaps (model can't complete the task):**
```python
# BAD: Can read but not write
tools = [get_order, get_customer]  # No update_order, no create_case

# Model: "I need to update this order but I have no tool for that"
# → Model hallucinates an action or fails

# GOOD: Complete set of operations the agent needs
tools = [get_order, update_order, get_customer, create_support_case, send_email]
```

### 2.6 Tool Granularity

A common mistake: tools that are too coarse-grained (do too many things) or too fine-grained
(require too many calls for simple tasks).

```python
# TOO COARSE: "do_everything"
{
    "name": "manage_customer",
    "description": "Can look up, update, delete, or communicate with a customer",
    "input_schema": {
        "action": "string",   # "lookup" | "update" | "delete" | "email"
        "customer_id": "string",
        "data": "object"      # meaning depends on action — nightmare
    }
}
# Model must guess the right "action" string. Input schema is underdefined.

# TOO FINE: "one-field-at-a-time"
tools = [
    set_customer_first_name,
    set_customer_last_name,
    set_customer_email,
    set_customer_phone,
]
# Updating a customer profile takes 4 tool calls

# JUST RIGHT: atomic but complete operations
tools = [
    lookup_customer,          # one call to read
    update_customer_profile,  # one call to write multiple fields at once
    delete_customer,          # separate high-risk operation
    send_customer_email,      # different concern
]
```

## 3. Code Examples

### Complete Well-Designed Tool Set for a Support Agent

```python
# A well-designed tool set for a customer support agent

SUPPORT_TOOLS = [
    {
        "name": "get_order_details",
        "description": (
            "Retrieve full details about a specific order including status, "
            "items, shipping, and return eligibility. "
            "Use this first whenever a user mentions an order ID or order-related issue. "
            "Do NOT use this to search by customer name — use search_orders for that."
        ),
        "input_schema": {
            "type": "object",
            "properties": {
                "order_id": {
                    "type": "string",
                    "description": "The order ID (format: ORD-YYYY-NNNNNNN, e.g. 'ORD-2024-1234567')"
                }
            },
            "required": ["order_id"]
        }
    },
    {
        "name": "search_orders",
        "description": (
            "Search orders by customer email or name. Returns a list of matching orders "
            "with basic details. Use when the customer doesn't have their order ID handy."
        ),
        "input_schema": {
            "type": "object",
            "properties": {
                "customer_email": {"type": "string", "description": "Customer email address"},
                "customer_name":  {"type": "string", "description": "Full or partial customer name"},
                "limit": {
                    "type": "integer",
                    "description": "Max results to return (1-20). Default: 5.",
                    "default": 5
                }
            },
            "required": []
        }
    },
    {
        "name": "initiate_refund",
        "description": (
            "Initiate a refund for a specific order or line item. "
            "ONLY use this after: (1) confirming the order exists, "
            "(2) verifying the item is return-eligible, "
            "(3) explicitly confirming with the customer that they want a refund. "
            "This action cannot be undone without manager approval."
        ),
        "input_schema": {
            "type": "object",
            "properties": {
                "order_id": {"type": "string"},
                "item_skus": {
                    "type": "array",
                    "items": {"type": "string"},
                    "description": "SKUs of items to refund. Empty array = full order refund."
                },
                "reason": {
                    "type": "string",
                    "enum": ["defective", "wrong_item", "not_received", "changed_mind", "other"],
                    "description": "Refund reason for record-keeping"
                },
                "customer_confirmed": {
                    "type": "boolean",
                    "description": "Must be true — confirms customer explicitly requested the refund"
                }
            },
            "required": ["order_id", "reason", "customer_confirmed"]
        }
    },
    {
        "name": "add_support_note",
        "description": (
            "Add an internal note to a customer's support ticket. "
            "Use to document actions taken, customer preferences, or context for future agents. "
            "Notes are internal-only — the customer cannot see them."
        ),
        "input_schema": {
            "type": "object",
            "properties": {
                "ticket_id": {"type": "string"},
                "note":      {"type": "string", "description": "The internal note text"},
                "tags":      {
                    "type": "array",
                    "items": {"type": "string"},
                    "description": "Optional tags like ['refund', 'escalation', 'vip-customer']"
                }
            },
            "required": ["ticket_id", "note"]
        }
    }
]
```

### Tool Output Validator

```python
from pydantic import BaseModel, Field
from typing import Optional
import json

class OrderDetails(BaseModel):
    found:        bool
    order_id:     Optional[str] = None
    status:       Optional[str] = None
    total_usd:    Optional[float] = None
    can_cancel:   Optional[bool] = None
    can_return:   Optional[bool] = None
    error:        Optional[str] = None
    message:      Optional[str] = None

def get_order_details_validated(order_id: str) -> dict:
    """
    Validates tool output before returning it to the model.
    Prevents bad data from corrupting agent reasoning.
    """
    raw_result = get_order_from_db(order_id)
    
    try:
        validated = OrderDetails(**raw_result)
        return validated.model_dump(exclude_none=True)
    except Exception as e:
        return {
            "found": False,
            "error": "Tool output validation failed",
            "message": "Unable to retrieve order details. Please try again."
        }
```

## 4. Tool Design Checklist

Before shipping any tool, verify:

```
□ Name is specific, verb-first, snake_case
□ Description explains WHAT, WHEN to use, and WHEN NOT to use
□ Every parameter has a clear description with examples
□ Optional parameters have sensible defaults
□ Output is structured JSON the model can reason about
□ Errors return {"error": "..."} not Python exceptions
□ No overlap with other tools in the same tool set
□ Tool set has no gaps for the agent's tasks
□ High-risk operations have a confirmation field (e.g., customer_confirmed: bool)
□ Tested with 10+ varied inputs to verify model uses it correctly
```

## 5. Trade-offs

### Fewer, Broader Tools vs Many, Narrow Tools

**Fewer tools:**
- Model has fewer choices → less decision-making overhead
- Harder to make descriptions specific enough
- One tool doing too much → harder to validate and secure

**Many narrow tools:**
- Each tool is focused and testable
- Model has more choices → may pick wrong tool
- More schema to maintain

**Sweet spot:** One tool per distinct operation. Group related read operations (search + get)
separately from write operations (update, delete, create).

## 6. Senior AI Engineer Perspective

### The Confirmation Field Pattern

For any tool with irreversible side effects (delete, send, submit, pay), add a boolean
confirmation field:

```python
{
    "name": "delete_account",
    "input_schema": {
        "properties": {
            "account_id": {"type": "string"},
            "confirmed_by_user": {
                "type": "boolean",
                "description": "Must be true. Only set to true after the user explicitly "
                               "typed 'yes' or 'confirm' to confirm account deletion."
            }
        },
        "required": ["account_id", "confirmed_by_user"]
    }
}
```

The model must explicitly confirm the user said yes before the flag is set. Your tool executor
also checks this:

```python
def delete_account(account_id: str, confirmed_by_user: bool) -> dict:
    if not confirmed_by_user:
        return {"success": False, "message": "Deletion requires explicit user confirmation."}
    # proceed with deletion
```

This double-confirmation (model + executor) prevents accidental or injected deletions.
