## 1. What Is It? (Theory)

### The Core Idea

An LLM by itself produces text. Function calling gives it the ability to *request actions* in
the real world: query a database, call an API, run code, read a file, send a message.

The critical thing to understand: **the LLM never executes code**. It outputs a structured
declaration saying "I want to call function X with arguments Y." Your code executes the
function and returns the result. The LLM reads the result and continues reasoning.

```
LLM:  "I want to call get_weather({city: 'London'})"
      ↑ This is just text output with special structure

YOU:  Actually call get_weather("London")
      Get back: {"temp": 18, "condition": "cloudy"}

LLM:  Reads the result, generates: "The weather in London is 18°C and cloudy."
```

This separation is intentional. Your code is the execution sandbox. The LLM is the reasoning
engine. Neither does the other's job.

### Why This Pattern Unlocks Agents

Without tool use:
```
User:  "What's the current price of AAPL?"
Model: "[Training data from 2024] Apple's stock price was around $185..."
→ Outdated, hallucinated, unreliable
```

With tool use:
```
User:  "What's the current price of AAPL?"
Model: calls get_stock_price("AAPL")
Tool:  returns {"price": 212.50, "timestamp": "2026-06-11T14:23:00Z"}
Model: "Apple (AAPL) is currently trading at $212.50."
→ Accurate, verifiable, real-time
```

The same pattern works for: database queries, file operations, web search, code execution,
form submission, calendar booking, email sending — any action your code can perform.

## 2. How It Actually Works — Deep Mechanics

### 2.1 The Tool Schema

Every tool is defined by a JSON schema that tells the model:
- What the function does (description)
- What arguments it accepts (input_schema)
- Which arguments are required

```python
get_weather_tool = {
    "name": "get_weather",
    "description": "Get the current weather for a given city. Returns temperature in Celsius and weather condition.",
    "input_schema": {
        "type": "object",
        "properties": {
            "city": {
                "type": "string",
                "description": "The city name, e.g. 'London, UK' or 'New York, USA'"
            },
            "units": {
                "type": "string",
                "enum": ["celsius", "fahrenheit"],
                "description": "Temperature units. Defaults to celsius.",
                "default": "celsius"
            }
        },
        "required": ["city"]
    }
}
```

The model reads this schema at inference time and uses it to decide:
1. Whether this tool is relevant to the current request
2. What arguments to fill in
3. Which argument names and types to use

### 2.2 The Full Tool Use Cycle (Anthropic)

```python
import anthropic
import json

client = anthropic.Anthropic()

# Step 1: Define tools
tools = [get_weather_tool]

# Step 2: Send initial request
messages = [{"role": "user", "content": "What's the weather in London?"}]

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=tools,
    messages=messages
)

print(f"Stop reason: {response.stop_reason}")   # "tool_use"
print(f"Content blocks: {[b.type for b in response.content]}")

# Step 3: Handle tool call
if response.stop_reason == "tool_use":
    tool_use_block = next(b for b in response.content if b.type == "tool_use")
    
    print(f"Tool called: {tool_use_block.name}")        # "get_weather"
    print(f"Arguments:   {tool_use_block.input}")       # {"city": "London"}
    print(f"Tool use ID: {tool_use_block.id}")           # unique ID for this call
    
    # Step 4: Execute the tool (YOUR code)
    result = get_weather(tool_use_block.input["city"])  # {"temp": 18, "condition": "cloudy"}
    
    # Step 5: Return result to the model
    messages.append({"role": "assistant", "content": response.content})
    messages.append({
        "role": "user",
        "content": [{
            "type": "tool_result",
            "tool_use_id": tool_use_block.id,    # must match the ID from the request
            "content": json.dumps(result)
        }]
    })
    
    # Step 6: Get final answer
    final = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        tools=tools,
        messages=messages
    )
    
    print(final.content[0].text)
    # "The weather in London is currently 18°C and cloudy."
```

### 2.3 Parallel Tool Calls

Modern LLMs (Claude, GPT-4o) can request multiple tool calls in a single response when it's
efficient to do so:

```python
# User: "What's the weather in London and Paris?"
# Model may request both in one response:

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=[get_weather_tool],
    messages=[{"role": "user", "content": "What's the weather in London and Paris?"}]
)

# response.content may contain TWO tool_use blocks
tool_calls = [b for b in response.content if b.type == "tool_use"]
# [ToolUseBlock(name="get_weather", input={"city": "London"}, id="toolu_01"),
#  ToolUseBlock(name="get_weather", input={"city": "Paris"}, id="toolu_02")]

# Execute both
results = {}
for call in tool_calls:
    results[call.id] = get_weather(call.input["city"])

# Return all results to the model
messages.append({"role": "assistant", "content": response.content})
messages.append({
    "role": "user",
    "content": [
        {"type": "tool_result", "tool_use_id": call.id, "content": json.dumps(results[call.id])}
        for call in tool_calls
    ]
})
```

Parallel tool calls reduce round-trips and latency significantly for independent operations.

### 2.4 Tool Use Loop (The Complete Production Pattern)

```python
import anthropic
import json
from typing import Callable

client = anthropic.Anthropic()

def run_agent(
    system: str,
    user_message: str,
    tools: list[dict],
    tool_executors: dict[str, Callable],
    model: str = "claude-sonnet-4-6",
    max_iterations: int = 10,
) -> str:
    """
    Complete tool-use agent loop.
    Runs until model returns end_turn or max_iterations is reached.
    """
    messages = [{"role": "user", "content": user_message}]
    
    for iteration in range(max_iterations):
        response = client.messages.create(
            model=model,
            max_tokens=4096,
            system=system,
            tools=tools,
            messages=messages
        )
        
        # Model finished — return text answer
        if response.stop_reason == "end_turn":
            return next((b.text for b in response.content if hasattr(b, "text")), "")
        
        # Model wants to call tools
        if response.stop_reason == "tool_use":
            messages.append({"role": "assistant", "content": response.content})
            
            tool_results = []
            for block in response.content:
                if block.type != "tool_use":
                    continue
                
                executor = tool_executors.get(block.name)
                if executor is None:
                    result = {"error": f"Unknown tool: {block.name}"}
                else:
                    try:
                        result = executor(**block.input)
                    except Exception as e:
                        result = {"error": str(e)}
                
                tool_results.append({
                    "type":        "tool_result",
                    "tool_use_id": block.id,
                    "content":     json.dumps(result)
                })
            
            messages.append({"role": "user", "content": tool_results})
    
    return "Max iterations reached without completing."


# Usage
def get_weather(city: str) -> dict:
    # In production: call a real weather API
    return {"city": city, "temp_c": 18, "condition": "cloudy", "humidity": 75}

def get_time(timezone: str = "UTC") -> dict:
    from datetime import datetime, timezone as tz
    return {"timezone": timezone, "time": datetime.now(tz.utc).isoformat()}

answer = run_agent(
    system="You are a helpful assistant with access to weather and time tools.",
    user_message="What's the weather in Tokyo, and what time is it there?",
    tools=[
        {
            "name": "get_weather",
            "description": "Get current weather for a city",
            "input_schema": {
                "type": "object",
                "properties": {"city": {"type": "string"}},
                "required": ["city"]
            }
        },
        {
            "name": "get_time",
            "description": "Get the current time in a timezone",
            "input_schema": {
                "type": "object",
                "properties": {"timezone": {"type": "string", "description": "Timezone name like 'Asia/Tokyo'"}},
                "required": []
            }
        }
    ],
    tool_executors={"get_weather": get_weather, "get_time": get_time},
)
print(answer)
```

### 2.5 tool_choice: Forcing Tool Use

Sometimes you want to guarantee the model uses a specific tool:

```python
# Force the model to use a specific tool (no prose, structured output only)
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=[extract_data_tool],
    tool_choice={"type": "tool", "name": "extract_data"},   # force this specific tool
    messages=[{"role": "user", "content": "Extract entities from: 'Alice at Acme Corp met Bob...'"}]
)
# Model MUST call extract_data — cannot refuse or respond in prose
# This is the structured output pattern from Phase 4.3

# Other options:
# {"type": "auto"}       — model decides (default)
# {"type": "any"}        — must call some tool, model's choice
# {"type": "tool", "name": "..."}  — must call this specific tool
```

## 3. Real-World Context

### What "Tool Use" Looks Like in Production

| Agent Type | Tools | Actions |
|---|---|---|
| Customer support | query_orders, update_ticket, send_email | Look up order status, escalate issues |
| Code assistant | run_code, read_file, write_file, search_web | Execute code, edit files, look up docs |
| Research assistant | search_web, extract_pdf, summarize | Gather information from multiple sources |
| Data analyst | query_database, create_chart, export_csv | Pull data, visualize, report |
| Calendar assistant | read_calendar, create_event, find_availability | Schedule meetings, check conflicts |
| DevOps agent | run_command, check_logs, deploy_service | Deploy, monitor, incident response |

### The Model Context Protocol (MCP)

In 2025, Anthropic introduced MCP (Model Context Protocol) — an open standard for connecting
AI models to external tools and data sources. By 2026, MCP is under vendor-neutral governance.

MCP standardizes how tools are defined and exposed, allowing:
- Tool servers that any MCP-compatible model can use
- Pre-built integrations for common services (GitHub, Slack, databases, etc.)
- Standardized security and permission models

MCP is the "USB standard" for AI tools — you write a tool server once, any MCP-compatible
model can use it without custom integration code.

```python
# With MCP, instead of defining tools manually, you connect to an MCP server:
# The MCP server exposes tools; the client discovers them automatically

# MCP server (example - you or a third party runs this):
# - Exposes: search_github, create_pr, get_issue, list_repositories
# - Any model with MCP support can use these without custom integration

# For direct Anthropic SDK use without MCP, define tools manually as shown above
```

## 5. Trade-offs

### Parallel vs Sequential Tool Execution

**Parallel (model requests multiple tools at once):**
- Faster (one round-trip for multiple actions)
- Model must reason about them independently
- Can't use result of tool A to decide arguments for tool B

**Sequential (model calls one tool, reads result, calls next):**
- Slower (multiple round-trips)
- Model can chain reasoning: "Based on the weather, now call get_clothing_recommendation"
- More tokens consumed per tool call

**Rule:** Use parallel for independent operations. Sequential for dependent operations where each step depends on the previous result.

## 6. When To Use vs. When To Avoid

### Tool Use
✅ When: Model needs real-time data (stock prices, weather, news)
✅ When: Model needs to take persistent actions (write files, send emails, update DB)
✅ When: Model needs to perform exact computation (call a calculator, not do arithmetic itself)
✅ When: Model needs to search an external knowledge base (RAG via tool)
❌ Avoid when: The model can answer reliably from training knowledge (adds latency)
❌ Avoid when: Tools have irreversible side effects without human confirmation first

⚠️ Common mistakes:
- Not handling `stop_reason == "tool_use"` — code crashes silently when model calls a tool
- Not handling tool execution errors — return `{"error": "..."}` to the model, don't crash
- Not setting `max_iterations` — loops can run indefinitely and burn tokens
- Not logging which tools are called — impossible to debug failures without this

## 7. Senior AI Engineer Perspective

### The Model Is Your Caller, Not Your Executor

The most important mental model shift: the LLM is a very smart function **caller**, not a function **executor**. It decides *what* to call and *with what arguments*. Your code decides *whether* and *how* to execute it.

This means: you have full control over what actually happens. You can:
- Validate tool arguments before execution
- Require human confirmation for dangerous actions
- Enforce an allowlist of permitted operations
- Rate-limit tool calls
- Log everything for audit

The model can ask for anything. You control what gets approved.

### Tool Descriptions Are API Documentation

Write tool descriptions as if a smart developer (who doesn't know your system) will read them.
The model uses the description to decide:
1. Whether this tool is relevant to the current task
2. What arguments to provide

Vague description → wrong tool selection, wrong arguments → wrong actions.
